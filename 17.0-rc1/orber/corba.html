<h1>corba</h1> <h2 class="modsummary">The functions on CORBA module level</h2> <div class="description"> <p>This module contains functions that are specified on the CORBA module level. It also contains some functions for creating and disposing objects.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="create/2">create(Module, TypeID) -&gt; Object</h3> <h3 id="create/3">create(Module, TypeID, Env) -&gt; Object</h3> <h3 id="create/4">create(Module, TypeID, Env, Optons1) -&gt; Object</h3> <h3 id="create_link/2">create_link(Module, TypeID) -&gt; Object</h3> <h3 id="create_link/3">create_link(Module, TypeID, Env) -&gt; Object</h3> <h3 id="create_link/4">create_link(Module, TypeID, Env, Options2) -&gt; Reply</h3> <ul class="type"> <li><code>Module = atom()</code></li> <li><code>TypeID = string()</code></li> <li><code>Env = term()</code></li> <li><code>Options1 = [{persistent, Bool} | {regname, RegName} | {local_typecheck, Bool}]</code></li> <li><code>Options2 = [{sup_child, Bool} | {persistent, Bool} | {regname, RegName} | {pseudo, Bool} | {local_typecheck, Bool}]</code></li> <li><code>RegName = {local, atom()} | {global, term()}</code></li> <li><code>Reply = #objref | {ok, Pid, #objref}</code></li> <li><code>Bool = true | false</code></li> <li><code>Object = #objref</code></li> </ul> <div class="description"> <p>These functions start a new server object. If you start it without <em>RegName</em> it can only be accessed through the returned object key. Started with a <em>RegName</em> the name is registered locally or globally. </p> <p><em>TypeID</em> is the repository ID of the server object type and could for example look like "IDL:StackModule/Stack:1.0". </p> <p><em>Module</em> is the name of the interface API module. </p> <p><em>Env</em> is the arguments passed which will be passed to the implementations <em>init</em> call-back function.</p> <p>A server started with create/2, create/3 or create/4 does not care about the parent, which means that the parent is not handled explicitly in the generic process part. </p> <p>A server started with create_link2, create_link/3 or create_link/4 is initially linked to the caller, the parent, and it will terminate whenever the parent process terminates, and with the same reason as the parent. If the server traps exits, the terminate/2 call-back function is called in order to clean up before the termination. These functions should be used if the server is a worker in a supervision tree.</p> <p>If you use the option <code>{sup_child, true}</code> create_link/4 will return <code>{ok, Pid, #objref}</code>, otherwise <code>#objref</code>, and make it possible to start a server as a supervisor child (stdlib-1.7 or later).</p> <p>If you use the option <code>{persistent, true}</code> you also must use the option <code>{regname, {global, Name}}</code>. This combination makes it possible to tell the difference between a server permanently terminated or in the process of restarting.</p> <p>The option <code>{pseudo, true}</code>, allow us to create an object which is not a server. Using <code>{pseudo, true}</code> overrides all other start options. For more information see section <code>Module_Interface</code>.</p> <p>If a server is started using the option <code>{persistent, true}</code> the object key will not be removed unless it terminates with reason <em>normal</em> or <em>shutdown</em>. Hence, if persistent servers is used as supervisor children they should be <em>transient</em> and the <em>objectkeys_gc_time</em> should be modified (default equals <code>infinity</code>).</p> <p>The option <code>{local_typecheck, boolean()}</code>, which overrides the <a href="ch_install.html#flags" class="seealso">Local Typechecking</a> environment flag, turns on or off typechecking. If activated, parameters, replies and raised exceptions will be checked to ensure that the data is correct, when invoking operations on CORBA Objects within the same Orber domain. Due to the extra overhead, this option <em>MAY ONLY</em> be used during testing and development.</p> <pre class="sh_erlang"> Example: corba:create('StackModule_Stack', "IDL:StackModule/Stack:1.0", {10, test}) </pre> </div> </div> <div class="function"> <h3 id="dispose/1">dispose(Object) -&gt; ok</h3> <ul class="type"> <li><code>Object = #objref</code></li> </ul> <div class="description"> <p>This function is used for terminating the execution of a server object. Invoking this operation on a NIL object reference, e.g., the return value of <code>corba:create_nil_objref/0</code>, always return ok. For valid object references, invoking this operation more than once, will result in a system exception.</p> </div> </div> <div class="function"> <h3 id="create_nil_objref/0">create_nil_objref() -&gt; Object</h3> <ul class="type"> <li><code>Object = #objref representing NIL.</code></li> </ul> <div class="description"> <p>Creates an object reference that represents the NIL value. Attempts to invoke operations using the returned object reference will return a system exception.</p> </div> </div> <div class="function"> <h3 id="create_subobject_key/2">create_subobject_key(Object, Key) -&gt; Result</h3> <ul class="type"> <li><code>Object = #objref</code></li> <li><code>Key = term()</code></li> <li><code>Result = #objref</code></li> </ul> <div class="description"> <p>This function is used to create a subobject in a server object. It can for example be useful when one wants unique access to separate rows in a mnesia or an ETS table. The <em>Result</em> is an object reference that will be seen as a unique reference to the outside world but will access the same server object where one can use the <em>get_subobject_key/1</em> function to get the private key value.</p> <p><em>Key</em> is stored in the object reference <em>Object</em>. If it is a binary it will be stored as is and otherwise it is converted to a binary before storage.</p> </div> </div> <div class="function"> <h3 id="get_subobject_key/1">get_subobject_key(Object) -&gt; Result</h3> <ul class="type"> <li><code>Object = #objref</code></li> <li><code>Result = #binary</code></li> </ul> <div class="description"> <p>This function is used to fetch a subobject key from the object reference <em>Object</em>. The result is a always a binary, if it was an Erlang term that was stored with <em>create_subobject_key/2</em> one can to do <em>binary_to_term/1</em> to get the real value. </p> </div> </div> <div class="function"> <h3 id="get_pid/1">get_pid(Object) -&gt; Result</h3> <ul class="type"> <li><code>Object = #objref</code></li> <li><code>Result = #pid | {error, Reason} | {'EXCEPTION',E}</code></li> </ul> <div class="description"> <p>This function is to get the process id from an object, which is a must when CORBA objects is started/handled in a supervisor tree. The function will throw exceptions if the key is not found or some other error occurs.</p> </div> </div> <div class="function"> <h3 id="raise/1">raise(Exception)</h3> <ul class="type"> <li><code>Exception = record()</code></li> </ul> <div class="description"> <p>This function is used for raising corba exceptions as an Erlang user generated exit signal. It will throw the tuple <code>{'EXCEPTION', </code><em>Exception</em><code>}</code>.</p> </div> </div> <div class="function"> <h3 id="reply/2">reply(To, Reply) -&gt; true</h3> <ul class="type"> <li><code>To = client reference</code></li> <li><code>Reply = IDL type</code></li> </ul> <div class="description"> <p>This function can be used by a CORBA object to explicitly send a reply to a client that invoked a two-way operation. If this operation is used, it is <em>not</em> possible to return a reply in the call-back module. <br /> <em>To</em> must be the <em>From</em> argument provided to the callback function, which requires that the IC option <em>from</em> was used when compiling the IDL-file.</p> </div> </div> <div class="function"> <h3 id="resolve_initial_references/1">resolve_initial_references(ObjectId) -&gt; Object</h3> <h3 id="resolve_initial_references/2">resolve_initial_references(ObjectId, Contexts) -&gt; Object</h3> <ul class="type"> <li><code>ObjectId = string()</code></li> <li><code>Contexts = [Context]</code></li> <li><code>Context = #'IOP_ServiceContext'{context_id = CtxId, context_data = CtxData}</code></li> <li><code>CtxId = ?ORBER_GENERIC_CTX_ID</code></li> <li><code>CtxData = {interface, Interface} | {userspecific, term()} | {configuration, Options}</code></li> <li><code>Interface = string()</code></li> <li><code>Options = [{Key, Value}]</code></li> <li><code>Key = ssl_client_options</code></li> <li><code>Value = allowed value associated with the given key</code></li> <li><code>Object = #objref</code></li> </ul> <div class="description"> <p>This function returns the object reference associated with the given object id. Initially, only <code>"NameService"</code> is available. To add or remove services use <code>add_initial_service/2</code> or <code>remove_initial_service/1</code>.</p> <p>The <em>configuration</em> context is used to override the global SSL client side <a href="ch_install.html#config" class="seealso">configuration</a>.</p> </div> </div> <div class="function"> <h3 id="add_initial_service/2">add_initial_service(ObjectId, Object) -&gt; boolean()</h3> <ul class="type"> <li><code>ObjectId = string()</code></li> <li><code>Object = #objref</code></li> </ul> <div class="description"> <p>This operation allows us to add initial services, which can be accessed by using <code>resolve_initial_references/1</code> or the <code>corbaloc</code> schema. If using an Id defined by the OMG, the given object must be of the correct type; for more information see the <a href="ch_naming_service.html#interop_ns" class="seealso">Interoperable Naming Service</a>. Returns <code>false</code> if the given id already exists.</p> </div> </div> <div class="function"> <h3 id="remove_initial_service/1">remove_initial_service(ObjectId) -&gt; boolean()</h3> <ul class="type"> <li><code>ObjectId = string()</code></li> </ul> <div class="description"> <p>If we don not want a certain service to be accessible, invoking this function will remove the association. Returns <code>true</code> if able to terminate the binding. If no such binding existed <code>false</code> is returned.</p> </div> </div> <div class="function"> <h3 id="list_initial_services/0">list_initial_services() -&gt; [ObjectId]</h3> <ul class="type"> <li><code>ObjectId = string()</code></li> </ul> <div class="description"> <p>This function returns a list of allowed object id's.</p> </div> </div> <div class="function"> <h3 id="resolve_initial_references_remote/2">resolve_initial_references_remote(ObjectId, Address) -&gt; Object</h3> <h3 id="resolve_initial_references_remote/3">resolve_initial_references_remote(ObjectId, Address, Contexts) -&gt; Object</h3> <ul class="type"> <li><code>ObjectId = string()</code></li> <li><code>Address = [RemoteModifier]</code></li> <li><code>RemoteModifier = string()</code></li> <li><code>Contexts = [Context]</code></li> <li><code>Context = #'IOP_ServiceContext'{context_id = CtxId, context_data = CtxData}</code></li> <li><code>CtxId = ?ORBER_GENERIC_CTX_ID</code></li> <li><code>CtxData = {interface, Interface} | {userspecific, term()} | {configuration, Options}</code></li> <li><code>Interface = string()</code></li> <li><code>Options = [{Key, Value}]</code></li> <li><code>Key = ssl_client_options</code></li> <li><code>Value = allowed value associated with the given key</code></li> <li><code>Object = #objref</code></li> </ul> <div class="description"> <p>This function returns the object reference for the object id asked for. Hence, use <code>corba:string_to_object/1</code> instead.</p> </div> </div> </div> <div class="function"> <h3 id="list_initial_services_remote/1">list_initial_services_remote(Address) -&gt; [ObjectId]</h3> <h3 id="list_initial_services_remote/2">list_initial_services_remote(Address, Contexts) -&gt; [ObjectId]</h3> <ul class="type"> <li><code>Address = [RemoteModifier]</code></li> <li><code>RemoteModifier = string()</code></li> <li><code>Contexts = [Context]</code></li> <li><code>Context = #'IOP_ServiceContext'{context_id = CtxId, context_data = CtxData}</code></li> <li><code>CtxId = ?ORBER_GENERIC_CTX_ID</code></li> <li><code>CtxData = {interface, Interface} | {userspecific, term()} | {configuration, Options}</code></li> <li><code>Interface = string()</code></li> <li><code>Options = [{Key, Value}]</code></li> <li><code>Key = ssl_client_options</code></li> <li><code>Value = allowed value associated with the given key</code></li> <li><code>ObjectId = string()</code></li> </ul> <div class="description"> <p>This function returns a list of allowed object id's. Only the new instance contains the new component. When this object is passed to another ORB, which supports the ALTERNATE_IIOP_ADDRESS, requests will be routed to the alternate address if it is not possible to communicate with the main address.</p> <p>The ALTERNATE_IIOP_ADDRESS component requires that IIOP-1.2 is used. Hence, make sure both Orber and the other ORB is correctly configured.</p> <p /> <div class="note"><h2>Note!</h2> <p>Make sure that the given <code>Object</code> is accessible via the alternate Host/port.
