<h1>Module_Interface</h1> <h2 class="modsummary">Orber generated stubs/skeletons.</h2> <div class="description"> <p>This module contains the stub/skeleton functions generated by IC.</p> <p>Starting a Orber server can be done in three ways:</p> <list type="bulleted"> <item>Normal - when the server dies Orber forgets all knowledge of the server.</item> <item>Supervisor child - adding the configuration parameter <code>{sup_child, true}</code> the <code>oe_create_link/2</code> function returns <code>{ok, Pid, ObjRef}</code> which can be handled by the application <em>supervisor/stdlib-1.7</em> or later. </item> <item>Persistent object reference - adding the configuration parameters <code>{persistent, true}</code> and <code>{regname, {global, term()}}</code> Orber will remember the object reference until the server terminates with reason <em>normal</em> or <em>shutdown</em>. Hence, if the server is started as a <em>transient</em> supervisor child we do not receive a 'OBJECT_NOT_EXIST' exception when it has crashed and is being restarted.</item> </list> <p>The Orber stub can be used to start a <code>pseudo object</code>, which will create a non-server implementation. A pseudo object introduce some limitations:</p> <list type="bulleted"> <item>The functions <code>oe_create_link/2</code> is equal to <code>oe_create/2</code>, i.e., no link can or will be created.</item> <item>The <code>BIF:s self()</code> and <code>process_flag(trap_exit,true)</code> behaves incorrectly.</item> <item>The <code>IC</code> option <code>{{impl, "M::I"}, "other_impl"}</code> has no effect. The call-back functions must be implemented in a file called <code>M_I_impl.erl</code></item> <item>The <code>IC</code> option <code>from</code> has no effect. </item> <item>The call-back functions must be implemented as if the <code>IC</code> option <code>{this, "M::I"}</code> was used.</item> <item>Server <code>State</code> changes have no effect. The user can provide information via the <code>Env</code> start parameter and the State returned from <code>init/2</code> will be the State passed in following invocations.</item> <item>If a call-back function replies with the <code>Timeout</code> parameter set it have no effect.</item> <item>Operations defined as <code>oneway</code> are blocking until the operation replies.</item> <item>The option <code>{pseudo, true}</code> overrides all other start options.</item> <item>Only the functions, besides own definitions, <code>init/2</code> (called via oe_create*/2) and <code>terminate/2</code> (called via corba:dispose/1) must be implemented.</item> </list> <p>By adopting the rules for <code>pseudo</code> objects described above we can use <code>oe_create/2</code> to create <code>server</code> or <code>pseudo</code> objects, by excluding or including the option <code>{pseudo, true}</code>, without changing the call-back module. </p> <p>If you start a object without <code>{regname, RegName}</code> it can only be accessed through the returned object key. Started with a <code>{regname, RegName}</code> the name is registered locally or globally. </p> <div class="warning"><h2>Warning!</h2> <p>To avoid flooding Orber with old object references start erlang using the flag <em>-orber objectkeys_gc_time Time</em>, which will remove all object references related to servers being dead for Time seconds. To avoid extra overhead, i.e., performing garbage collect if no persistent objects are started, the objectkeys_gc_time default value is <em>infinity</em>. For more information, see the orber and corba documentation.</p> </div> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="typeID/0">Module_Interface:typeID() -&gt; ObjRef</h3> <ul class="type"> <li><code>Env = term()</code></li> <li><code>ObjRef = #object reference</code></li> <li><code>Options = [{sup_child, false} | {persistent, Bool} | {regname, RegName} | {pseudo, Bool} | {local_typecheck, Bool} | {survive_exit, Bool} | {create_options, [CreateOpts]}]</code></li> <li><code>Bool = true | false</code></li> <li><code>RegName = {global, term()} | {local, atom()}</code></li> <li><code>CreateOpts = {debug, [Dbg]} | {timeout, Time}</code></li> <li><code>Dbg = trace | log | statistics | {log_to_file, FileName}</code></li> </ul> <div class="description"> <p>Start a Orber server passing Env to <code>init/1</code>.</p> <p>If the option <code>{pseudo, true}</code> is used, all other options are overridden. As default, this option is set to false.</p> <p>This function cannot be used for starting a server as supervisor child. If started as <code>persistent</code>, the options <code>[{persistent, true}, {regname, {global, term()}}]</code> must be used and Orber will only forget the object reference if it terminates with reason <em>normal</em> or <em>shutdown</em>.</p> <p>The option <code>{local_typecheck, boolean()}</code>, which overrides the <a href="ch_install.html#flags" class="seealso">Local Typechecking</a> environment flag, turns on or off typechecking. If activated, parameters, replies and raised exceptions will be checked to ensure that the data is correct, when invoking operations on CORBA Objects within the same Orber domain. Due to the extra overhead, this option <em>MAY ONLY</em> be used during testing and development.</p> <p><code>{survive_exit, boolean()}</code> overrides the <a href="ch_install.html#flags" class="seealso">EXIT Tolerance</a> environment flag. If activated, the server will not terminate, even though the call-back module returns EXIT.</p> <p><code>Time</code> specifies how long time, in milliseconds, the server is allowed to spend initializing. For more information about the <code>Dbg</code> options, see the <code>sys</code> module.</p> </div> </div> <div class="function"> <h3 id="oe_create_link/2">Module_Interface:oe_create_link(Env, Options) -&gt; Return</h3> <ul class="type"> <li><code>Env = term()</code></li> <li><code>Return = ObjRef | {ok, Pid, ObjRef}</code></li> <li><code>ObjRef = #object reference</code></li> <li><code>Options = [{sup_child, Bool} | {persistent, Bool} | {regname, RegName} | {pseudo, Bool} | {local_typecheck, Bool} | {survive_exit, Bool} | {create_options, [CreateOpts]}]</code></li> <li><code>Bool = true | false</code></li> <li><code>RegName = {global, term()} | {local, atom()}</code></li> <li><code>CreateOpts = {debug, [Dbg]} | {timeout, Time}</code></li> <li><code>Dbg = trace | log | statistics | {log_to_file, FileName}</code></li> <li> </li> <li> </li> <li> </li> </ul> <div class="description"> <p>Start a linked Orber server passing Env to <code>init/1</code>.</p> <p>If the option <code>{pseudo, true}</code> is used, all other options are overridden and no link will be created. As default, this option is set to false.</p> <p>This function can be used for starting a server as persistent or supervisor child. At the moment <code>[{persistent, true}, {regname, {global, term()}}]</code> must be used to start a server as persistent, i.e., if a server died and is in the process of being restarted a call to the server will not raise <code>'OBJECT_NOT_EXIST'</code> exception. Orber will only forget the object reference if it terminates with reason <em>normal</em> or <em>shutdown</em>, hence, the server must be started as <em>transient</em> (for more information see the supervisor documentation).</p> <p>The options <code>{local_typecheck, boolean()}</code> and <code>{survive_exit, boolean()}</code> behaves in the same way as for <code>oe_create/2</code>.</p> <p><code>Time</code> specifies how long time, in milliseconds, the server is allowed to spend initializing.
