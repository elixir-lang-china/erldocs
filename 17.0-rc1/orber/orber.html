<h1>orber</h1> <h2 class="modsummary">The main module of the Orber application</h2> <div class="description"> <p>This module contains the functions for starting and stopping the application. It also has some utility functions to get some of the configuration information from running application.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start/0">start() -&gt; ok</h3> <h3 id="start/1">start(Type) -&gt; ok</h3> <ul class="type"> <li><code>Type = temporary | permanent</code></li> </ul> <div class="description"> <p>Starts the Orber application (it also starts mnesia if it is not running). Which <code>Type</code> parameter is supplied determines the behavior. If not supplied Orber is started as <code>temporary</code>. See the Reference Manual <em>application(3)</em> for further information. </p> </div> </div> <div class="function"> <h3 id="jump_start/1">jump_start(Attributes) -&gt; ok | {'EXIT', Reason}</h3> <ul class="type"> <li><code>Attributes = Port | Options</code></li> <li><code>Port = integer()</code></li> <li><code>Options = [{Key, Value}]</code></li> <li><code>Key = any key listed in the configuration chapter</code></li> <li><code>Value = allowed value associated with the given key</code></li> </ul> <div class="description"> <p>Installs and starts the Orber and the Mnesia applications with the configuration parameters <code>domain</code> and <code>iiop_port</code> set to <code>"IP-number:Port"</code> and the supplied Port respectively. Theses settings are in most cases sufficient to ensure that no clash with any other Orber instance occur. If this operation fails, check if the listen port (iiop_port) is already in use. This function <em>MAY ONLY</em> be used during development and tests; how Orber is configured when using this operation may change at any time without warning.</p> </div> </div> <div class="function"> <h3 id="stop/0">stop() -&gt; ok</h3> <div class="description"> <p>Stops the Orber application.</p> </div> </div> <div class="function"> <h3 id="info/0">info() -&gt; ok</h3> <h3 id="info/1">info(IoType) -&gt; ok | {'EXIT', Reason} | string()</h3> <ul class="type"> <li><code>IoType = info_msg | string | io | {io, IoDevice}</code></li> </ul> <div class="description"> <p>Generates an Info Report, which contain Orber's configuration settings. Otherwise it returns -1.</p> </div> </div> <div class="function"> <h3 id="iiop_timeout/0">iiop_timeout() -&gt; int() (milliseconds)</h3> <div class="description"> <p>This function returns the timeout value after which outgoing IIOP requests terminate. It can be configured by setting the application variable <em>iiop_timeout TimeVal (seconds)</em>, if it is not set it will have the default value <em>infinity</em>. If a request times out a system exception, e.g. <em>TIMEOUT</em>, is raised.</p> <p>Note: the iiop_timeout configuration parameter (TimeVal) may only range between 0 and 1000000 seconds. Otherwise, the default value is used.</p> <p>Note: Earlier IC versions required that the compile option <code>{timeout,"module::interface"}</code>, was used, which allow the user to add an extra timeout parameter, e.g., <code>module_interface:function(ObjRef, Timeout, ... Arguments ...)</code> or <code>module_interface:function(ObjRef, [{timeout, Timeout}], ... Arguments ...)</code>, instead of <code>module_interface:function(ObjRef, ... Arguments ...)</code>. This is no longer the case and if the extra Timeout is used, argument will override the configuration parameter <code>iiop_timeout</code>. It is, however, not possible to use <code>infinity</code> to override the Timeout parameter. The Timeout option is also valid for objects which resides within the same Orber domain.</p> </div> </div> <div class="function"> <h3 id="iiop_connection_timeout/0">iiop_connection_timeout() -&gt; int() (milliseconds)</h3> <div class="description"> <p>This function returns the timeout value after which outgoing IIOP connections terminate. The connection will not be terminated if there are pending requests.</p> <p>Note: the iiop_connection_timeout configuration parameter (TimeVal) may only range between 0 and 1000000 seconds. Otherwise, the default value is used.</p> </div> </div> <div class="function"> <h3 id="iiop_connections/0">iiop_connections() -&gt; Result</h3> <h3 id="iiop_connections/1">iiop_connections(Direction) -&gt; Result</h3> <ul class="type"> <li><code>Direction = in | out | inout</code></li> <li><code>Result = [{Host, Port}] | [{Host, Port, Interface}] | {'EXIT',Reason}</code></li> <li><code>Host = string()</code></li> <li><code>Port = integer()</code></li> <li><code>Interface = string()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>The list returned by this operation contain tuples of remote hosts/ports Orber is currently connected to. The <code>Filter</code> uses a extended format of Classless Inter Domain Routing (CIDR). For example, <code>"123.123.123.10"</code> limits the connection to that particular host, while <code>"123.123.123.10/17"</code> allows connections to or from any host equal to the 17 most significant bits. Orber also allow the user to specify a certain port or port range, for example, <code>"123.123.123.10/17#4001"</code> and <code>"123.123.123.10/17#4001/5001"</code> respectively. IPv4 or none compressed IPv6 strings are accepted. <br /> The list of <code>Interfaces</code>, IPv4 or IPv6 strings, are currently only used for outgoing connections and may only contain <em>one</em> address. If set and access is granted, Orber will use that local interface when connecting to the other ORB. The module <a href="orber_acl.html" class="seealso">orber_acl</a> provides operations for evaluating the access control for filters and addresses.</p> </div> </div> <div class="function"> <h3 id="activate_audit_trail/0">activate_audit_trail() -&gt; Result</h3> <h3 id="activate_audit_trail/1">activate_audit_trail(Verbosity) -&gt; Result</h3> <ul class="type"> <li><code>Verbosity = stealth | normal | verbose</code></li> <li><code>Result = ok | {error, Reason}</code></li> <li><code>Reason = string()</code></li> </ul> <div class="description"> <p>Activates audit/trail for all existing incoming and outgoing IIOP connections.
