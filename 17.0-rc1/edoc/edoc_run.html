<h1>edoc_run</h1> <h2 class="modsummary">Interface for calling EDoc from Erlang startup options.</h2> <div class="description"> <p>Interface for calling EDoc from Erlang startup options.</p> <p>The following is an example of typical usage in a Makefile: </p><pre class="sh_erlang"> docs: erl -noshell -run edoc_run application "'$(APP_NAME)'" \ '"."' '[{def,{vsn,"$(VSN)"}}]'</pre><p> (note the single-quotes to avoid shell expansion, and the double-quotes enclosing the strings).</p> <p><em>New feature in version 0.6.9</em>: It is no longer necessary to write <code>-s init stop</code> last on the command line in order to make the execution terminate. The termination (signalling success or failure to the operating system) is now built into these functions.</p></div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="application/1">application(Args::[string()]) -&gt; none()</h3> <div class="description"><span id="application-1"> </span> <p>Calls <a href="edoc.html#application-3" class="seealso">edoc:application/3</a> with the corresponding arguments. The strings in the list are parsed as Erlang constant terms. The list can be either <code>[App]</code>, <code>[App, Options]</code> or <code>[App, Dir, Options]</code>. In the first case <a href="edoc.html#application-1" class="seealso">edoc:application/1</a> is called instead; in the second case, <a href="edoc.html#application-2" class="seealso">edoc:application/2</a> is called.</p> <p>The function call never returns; instead, the emulator is automatically terminated when the call has completed, signalling success or failure to the operating system.</p> </div></div> <div class="function"> <h3 id="files/1">files(Args::[string()]) -&gt; none()</h3> <div class="description"><span id="files-1"> </span> <p>Calls <a href="edoc.html#files-2" class="seealso">edoc:files/2</a> with the corresponding arguments. The strings in the list are parsed as Erlang constant terms. The list can be either <code>[Files]</code> or <code>[Files, Options]</code>. In the first case, <a href="edoc.html#files-1" class="seealso">edoc:files/1</a> is called instead.</p> <p>The function call never returns; instead, the emulator is automatically terminated when the call has completed, signalling success or failure to the operating system.</p> </div></div> <div class="function"> <h3 id="packages/1">packages(Args::[string()]) -&gt; none()</h3> <div class="description"><span id="packages-1"> </span> <p>Calls <a href="edoc.html#application-2" class="seealso">edoc:application/2</a> with the corresponding arguments. The strings in the list are parsed as Erlang constant terms. The list can be either <code>[Packages]</code> or <code>[Packages, Options]</code>. In the first case <a href="edoc.html#application-1" class="seealso">edoc:application/1</a> is called instead.</p> <p>The function call never returns; instead, the emulator is automatically terminated when the call has completed, signalling success or failure to the operating system.</p> </div></div> <div class="function"> <h3 id="file/1">file(Args::[string()]) -&gt; none()</h3> <div class="description"><span id="file-1"> </span> <p><em>This function is deprecated: </em>This is part of the old interface to EDoc and is mainly kept for backwards compatibility. The preferred way of generating documentation is through one of the functions <a href="#application-1" class="seealso">application/1</a>, <a href="#packages-1" class="seealso">packages/1</a> and <a href="#files-1" class="seealso">files/1</a>. </p> <p>Calls <a href="edoc.html#file-2" class="seealso">edoc:file/2</a> with the corresponding arguments. The strings in the list are parsed as Erlang constant terms. The list can be either <code>[File]</code> or <code>[File, Options]</code>. In the first case, an empty list of options is passed to <a href="edoc.html#file-2" class="seealso">edoc:file/2</a>.</p> <p>The following is an example of typical usage in a Makefile: </p><pre class="sh_erlang"> $(DOCDIR)/%.html:%.erl erl -noshell -run edoc_run file '"$&lt;"' '[{dir,"$(DOCDIR)"}]' \ -s init stop</pre> <p>The function call never returns; instead, the emulator is automatically terminated when the call has completed, signalling success or failure to the operating system.</p> </div></div></div> <div class="section"><h4>See also</h4><p><a href="edoc.html" class="seealso">edoc</a></p></div> <authors> <aname>Richard Carlsson</aname> <email>carlsson.richard@gmail.com</email></authors>
