The preferred way of generating documentation is through one of the functions <a href="#application-2" class="seealso">application/2</a>, <a href="#packages-2" class="seealso">packages/2</a> and <a href="#files-2" class="seealso">files/2</a>. </p> <p>Reads a source code file and outputs formatted documentation to a corresponding file.</p> <p>Options: </p><taglist> <dt><code>{dir, filename()}</code> </dt> <item><p>Specifies the output directory for the created file. The default value is <code>".html"</code>. </p></item> </taglist> <p>See <a href="#get_doc-2" class="seealso">get_doc/2</a> and <a href="#layout-2" class="seealso">layout/2</a> for further options.</p> <p>For running EDoc from a Makefile or similar, see <a href="edoc_run.html#file-1" class="seealso">edoc_run:file/1</a>. </p> <p><em>See also:</em> <a href="#read-2" class="seealso">read/2</a>.</p> </div></div> <div class="function"> <h3 id="files/1">files(Files::[filename() | {package(), [filename()]}]) -&gt; ok</h3> <div class="description"><span id="files-1"> </span> <p>Equivalent to <a href="#packages-2" class="seealso">packages(Packages, [])</a>.</p> </div></div> <div class="function"> <h3 id="files/1-1">files(Files::[filename() | {package(), [filename()]}], Options::proplist()) -&gt; ok</h3> <div class="description"><span id="files-2"> </span> <p>Runs EDoc on a given set of source files. Tries to automatically set up good defaults. Unless the user specifies otherwise: </p><list> <item><p>The <code>doc</code> subdirectory will be used as the target directory, if it exists; otherwise the application directory is used. </p></item> <item><p>The source code is assumed to be located in the <code>src</code> subdirectory, if it exists, or otherwise in the application directory itself. </p></item> <item><p>The <a href="#run-3" class="seealso">subpackages</a> option is turned on. All found source files will be processed. </p></item> <item><p>The <code>include</code> subdirectory is automatically added to the include path. (Only important if <a href="#read_source-2" class="seealso"> preprocessing</a> is turned on.) </p></item> </list><p> See <a href="#run-3" class="seealso">run/3</a> for details, including options. </p> <p><em>See also:</em> <a href="#application-2" class="seealso">application/2</a>.</p> </div></div> <div class="function"> <h3 id="run/1">run(Packages::[package()], Files::[filename() | {package(), [filename()]}], Options::proplist()) -&gt; ok</h3> <div class="description"><span id="run-3"> </span> <p>Runs EDoc on a given set of source files and/or packages. Note that the doclet plugin module has its own particular options; see the <code>doclet</code> option below.</p> <p>Also see <a href="#layout-2" class="seealso">layout/2</a> for layout-related options, and <a href="#get_doc-2" class="seealso">get_doc/2</a> for options related to reading source files.</p> <p>Options: </p><taglist> <dt><code>{app_default, string()}</code> </dt> <item><p>Specifies the default base URI for unknown applications. </p></item> <dt><code>{application, App::atom()}</code> </dt> <item><p>Specifies that the generated documentation describes the application <code>App</code>. This mainly affects generated references. </p></item> <dt><code>{dir, filename()}</code> </dt> <item><p>Specifies the target directory for the generated documentation. </p></item> <dt><code>{doc_path, [string()]}</code> </dt> <item><p>Specifies a list of URI:s pointing to directories that contain EDoc-generated documentation. URI without a <code>scheme://</code> part are taken as relative to <code>file://</code>. (Note that such paths must use <code>/</code> as separator, regardless of the host operating system.) </p></item> <dt><code>{doclet, Module::atom()}</code> </dt> <item><p>Specifies a callback module to be used for creating the documentation. The module must export a function <code>run(Cmd, Ctxt)</code>. The default doclet module is <a href="edoc_doclet.html" class="seealso">edoc_doclet</a>; see <a href="edoc_doclet.html#run-2" class="seealso">edoc_doclet:run/2</a> for doclet-specific options. </p></item> <dt><code>{exclude_packages, [package()]}</code> </dt> <item><p>Lists packages to be excluded from the documentation. Typically used in conjunction with the <code>subpackages</code> option. </p></item> <dt><code>{file_suffix, string()}</code> </dt> <item><p>Specifies the suffix used for output files. The default value is <code>".html"</code>. Note that this also affects generated references. </p></item> <dt><code>{new, boolean()}</code> </dt> <item><p>If the value is <code>true</code>, any existing <code>edoc-info</code> file in the target directory will be ignored and overwritten. The default value is <code>false</code>. </p></item> <dt><code>{packages, boolean()}</code> </dt> <item><p>If the value is <code>true</code>, it it assumed that packages (module namespaces) are being used, and that the source code directory structure reflects this. The default value is <code>true</code>. (Usually, this does the right thing even if all the modules belong to the top-level "empty" package.) <code>no_packages</code> is an alias for <code>{packages, false}</code>. See the <code>subpackages</code> option below for further details.</p> <p>If the source code is organized in a hierarchy of subdirectories although it does not use packages, use <code>no_packages</code> together with the recursive-search <code>subpackages</code> option (on by default) to automatically generate documentation for all the modules. </p></item> <dt><code>{source_path, [filename()]}</code> </dt> <item><p>Specifies a list of file system paths used to locate the source code for packages. </p></item> <dt><code>{source_suffix, string()}</code> </dt> <item><p>Specifies the expected suffix of input files. The default value is <code>".erl"</code>. </p></item> <dt><code>{subpackages, boolean()}</code> </dt> <item><p>If the value is <code>true</code>, all subpackages of specified packages will also be included in the documentation. The default value is <code>false</code>. <code>no_subpackages</code> is an alias for <code>{subpackages, false}</code>. See also the <code>exclude_packages</code> option.</p> <p>Subpackage source files are found by recursively searching for source code files in subdirectories of the known source code root directories. (Also see the <code>source_path</code> option.) Directory names must begin with a lowercase letter and contain only alphanumeric characters and underscore, or they will be ignored. (For example, a subdirectory named <code>test-files</code> will not be searched.) </p></item> </taglist><p> </p> <p><em>See also:</em> <a href="#application-2" class="seealso">application/2</a>, <a href="#files-2" class="seealso">files/2</a>, <a href="#packages-2" class="seealso">packages/2</a>.</p> </div></div> <div class="function"> <h3 id="read/1">read(File::filename()) -&gt; string()</h3> <div class="description"><span id="read-1"> </span> <p>Equivalent to <a href="#read-2" class="seealso">read(File, [])</a>.</p> </div></div> <div class="function"> <h3 id="read/1-1">read(File::filename(), Options::proplist()) -&gt; string()</h3> <div class="description"><span id="read-2"> </span> <p>Reads and processes a source file and returns the resulting EDoc-text as a string. The default value is <code>false</code>. <code>no_preprocess</code> is an alias for <code>{preprocess, false}</code>.</p> <p>Normally, preprocessing is not necessary for EDoc to work, but if a file contains too exotic definitions or uses of macros, it will not be possible to read it without preprocessing. <em>Note: comments in included files will not be available to EDoc, even with this option enabled.</em> </p></item> <dt><code>{includes, Path::[string()]}</code> </dt> <item><p>Specifies a list of directory names to be searched for include files, if the <code>preprocess</code> option is turned on. The directory of the source file is always automatically appended to the search path. </p></item> <dt><code>{macros, [{atom(), term()}]}</code> </dt> <item><p>Specifies a list of pre-defined Erlang preprocessor (<code>epp</code>) macro definitions, used if the <code>preprocess</code> option is turned on. The default value is the empty list.</p></item> </taglist><p> <dt><code>{report_missing_types, boolean()}</code> </dt> <item><p>If the value is <code>true</code>, warnings are issued for missing types. The default value is <code>false</code>. <code>no_report_missing_types</code> is an alias for <code>{report_missing_types, false}</code>. </p></item> </p> <p><em>See also:</em> <a href="../http///www.erlang.org/edoc/doc/syntax_tools/doc/erl_syntax.xml.html" class="seealso">//syntax_tools/erl_syntax</a>, <a href="#get_doc-2" class="seealso">get_doc/2</a>.</p> </div></div> <div class="function"> <h3 id="get_doc/1">get_doc(File::filename()) -&gt; {ModuleName, edoc_module()}</h3> <div class="description"><span id="get_doc-1"> </span> <p>Equivalent to <a href="#get_doc-2" class="seealso">get_doc(File, [])</a>.</p> </div></div> <div class="function"> <h3 id="get_doc/1-1">get_doc(File::filename(), Options::proplist()) -&gt; {ModuleName, edoc_module()}</h3> <ul class="type"> <li><code>ModuleName = atom()</code></li></ul> <div class="description"><span id="get_doc-2"> </span> <p>Reads a source code file and extracts EDoc documentation data.
