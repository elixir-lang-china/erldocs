<h1>leex</h1> <h2 class="modsummary">Lexical analyzer generator for Erlang</h2> <div class="description"> <p>A regular expression based lexical analyzer generator for Erlang, similar to lex or flex.</p> <div class="note"><h2>Note!</h2><p>The Leex module should be considered experimental as it will be subject to changes in future releases.</p></div> </div> <div class="section"> <h4>DATA TYPES</h4> <pre class="sh_erlang"> ErrorInfo = {ErrorLine,module(),error_descriptor()} ErrorLine = integer() Token = tuple()</pre> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="file/3">file(FileName, [, Options]) -&gt; LeexRet</h3> <ul class="type"> <li><code>FileName = filename()</code></li> <li><code>Options = Option | [Option]</code></li> <li><code>Option =&nbsp;-&nbsp;see below&nbsp;-</code></li> <li><code>LeexRet = {ok, Scannerfile} | {ok, Scannerfile, Warnings} | error | {error, Warnings, Errors}</code></li> <li><code>Scannerfile = filename()</code></li> <li><code>Warnings = Errors = [{filename(), [ErrorInfo]}]</code></li> <li><code>ErrorInfo = {ErrorLine, module(), Reason}</code></li> <li><code>ErrorLine = integer()</code></li> <li><code>Reason =&nbsp;-&nbsp;formatable by format_error/1&nbsp;-</code></li> </ul> <div class="description"> <p>Generates a lexical analyzer from the definition in the input file. The input file has the extension <code>.xrl</code>. This is added to the filename if it is not given. The resulting module is the Xrl filename without the <code>.xrl</code> extension.</p> <p>The current options are:</p> <taglist> <dt><code>dfa_graph</code></dt> <item><p>Generates a <code>.dot</code> file which contains a description of the DFA in a format which can be viewed with Graphviz, <code>www.graphviz.com</code>.</p> </item> <dt><code>{includefile,Includefile}</code></dt> <item><p>Uses a specific or customised prologue file instead of default <code>lib/parsetools/include/leexinc.hrl</code> which is otherwise included.</p> </item> <dt><code>{report_errors, bool()}</code></dt> <item><p>Causes errors to be printed as they occur. Default is <code>true</code>.</p> </item> <dt><code>{report_warnings, bool()}</code></dt> <item><p>Causes warnings to be printed as they occur. Default is <code>true</code>.</p> </item> <dt><code>warnings_as_errors</code></dt> <item> <p>Causes warnings to be treated as errors.</p> </item> <dt><code>{report, bool()}</code></dt> <item><p>This is a short form for both <code>report_errors</code> and <code>report_warnings</code>.</p> </item> <dt><code>{return_errors, bool()}</code></dt> <item><p>If this flag is set, <code>{error, Errors, Warnings}</code> is returned when there are errors. Default is <code>false</code>.</p> </item> <dt><code>{return_warnings, bool()}</code></dt> <item><p>If this flag is set, an extra field containing <code>Warnings</code> is added to the tuple returned upon success. Default is <code>false</code>.</p> </item> <dt><code>{return, bool()}</code></dt> <item><p>This is a short form for both <code>return_errors</code> and <code>return_warnings</code>.</p> </item> <dt><code>{scannerfile, Scannerfile}</code></dt> <item><p><code>Scannerfile</code> is the name of the file that will contain the Erlang scanner code that is generated. The default (<code>""</code>) is to add the extension <code>.erl</code> to <code>FileName</code> stripped of the <code>.xrl</code> extension.</p> </item> <dt><code>{verbose, bool()}</code></dt> <item><p>Outputs information from parsing the input file and generating the internal tables.</p> </item> </taglist> <p>Any of the Boolean options can be set to <code>true</code> by stating the name of the option. For example, <code>verbose</code> is equivalent to <code>{verbose, true}</code>.</p> <p>Leex will add the extension <code>.hrl</code> to the <code>Includefile</code> name and the extension <code>.erl</code> to the <code>Scannerfile</code> name, unless the extension is already there.</p> </div> </div> <div class="function"> <h3 id="format_error/1">format_error(ErrorInfo) -&gt; Chars</h3> <ul class="type"> <li><code>Chars = [char() | Chars]</code></li> </ul> <div class="description"> <p>Returns a string which describes the error <code>ErrorInfo</code> returned when there is an error in a regular expression.</p> </div> </div> </div> <div class="section"> <h4>GENERATED SCANNER EXPORTS</h4> <p>The following functions are exported by the generated scanner.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="string/1">string(String) -&gt; StringRet</h3> <h3 id="string/2">string(String, StartLine) -&gt; StringRet</h3> <ul class="type"> <li><code>String = string()</code></li> <li><code>StringRet = {ok,Tokens,EndLine} | ErrorInfo</code></li> <li><code>Tokens = [Token]</code></li> <li><code>EndLine = StartLine = integer()</code></li> </ul> <div class="description"> <p>Scans <code>String</code> and returns all the tokens in it, or an error.</p> <div class="note"><h2>Note!</h2><p>It is an error if not all of the characters in <code>String</code> are consumed.</p></div> </div> </div> <div class="function"> <h3 id="token/2">token(Cont, Chars) -&gt; {more,Cont1} | {done,TokenRet,RestChars} </h3> <h3 id="token/3">token(Cont, Chars, StartLine) -&gt; {more,Cont1} | {done,TokenRet,RestChars} </h3> <ul class="type"> <li><code>Cont = [] | Cont1</code></li> <li><code>Cont1 = tuple()</code></li> <li><code>Chars = RestChars = string() | eof</code></li> <li><code>TokenRet = {ok, Token, EndLine} | {eof, EndLine} | ErrorInfo</code></li> <li><code>StartLine = EndLine = integer()</code></li> </ul> <div class="description"> <p>This is a re-entrant call to try and scan one token from <code>Chars</code>. If there are enough characters in <code>Chars</code> to either scan a token or detect an error then this will be returned with <code>{done,...}</code>. Otherwise <code>{cont,Cont}</code> will be returned where <code>Cont</code> is used in the next call to <code>token()</code> with more characters to try an scan the token. This is continued until a token has been scanned. <code>Cont</code> is initially <code>[]</code>.</p> <p>It is not designed to be called directly by an application but used through the i/o system where it can typically be called in an application by:</p> <code> io:request(InFile, {get_until,Prompt,Module,token,[Line]}) -&gt; TokenRet</code> </div> </div> <div class="function"> <h3 id="tokens/2">tokens(Cont, Chars) -&gt; {more,Cont1} | {done,TokensRet,RestChars} </h3> <h3 id="tokens/3">tokens(Cont, Chars, StartLine) -&gt; {more,Cont1} | {done,TokensRet,RestChars} </h3> <ul class="type"> <li><code>Cont = [] | Cont1</code></li> <li><code>Cont1 = tuple()</code></li> <li><code>Chars = RestChars = string() | eof</code></li> <li><code>TokensRet = {ok, Tokens, EndLine} | {eof, EndLine} | ErrorInfo</code></li> <li><code>Tokens = [Token]</code></li> <li><code>StartLine = EndLine = integer()</code></li> </ul> <div class="description"> <p>This is a re-entrant call to try and scan tokens from <code>Chars</code>. If there are enough characters in <code>Chars</code> to either scan tokens or detect an error then this will be returned with <code>{done,...}</code>. Otherwise <code>{cont,Cont}</code> will be returned where <code>Cont</code> is used in the next call to <code>tokens()</code> with more characters to try an scan the tokens. This is continued until all tokens have been scanned. <code>Cont</code> is initially <code>[]</code>.</p> <p>This functions differs from <code>token</code> in that it will continue to scan tokens upto and including an <code>{end_token,Token}</code> has been scanned (see next section). It will then return all the tokens. This is typically used for scanning grammars like Erlang where there is an explicit end token, <code>'.'</code>. If no end token is found then the whole file will be scanned and returned. If an error occurs then all tokens upto and including the next end token will be skipped.</p> <p>It is not designed to be called directly by an application but used through the i/o system where it can typically be called in an application by:</p> <code> io:request(InFile, {get_until,Prompt,Module,tokens,[Line]}) -&gt; TokensRet</code> </div> </div> </div> <div class="section"> <h4>Input File Format</h4> <p>Erlang style comments starting with a <code>%</code> are allowed in scanner files. A definition file has the following format:</p> <code> &lt;Header&gt; Definitions. &lt;Macro Definitions&gt; Rules. &lt;Token Rules&gt; Erlang code. &lt;Erlang code&gt;</code> <p>The "Definitions.", "Rules." and "Erlang code." headings are mandatory and must occur at the beginning of a source line. The &lt;Header&gt;, &lt;Macro Definitions&gt; and &lt;Erlang code&gt; sections may be empty but there must be at least one rule.</p> <p>Macro definitions have the following format:</p> <code> NAME = VALUE</code> <p>and there must be spaces around <code>=</code>. Macros can be used in the regular expressions of rules by writing <code>{NAME}</code>.</p> <div class="note"><h2>Note!</h2><p>When macros are expanded in expressions the macro calls are replaced by the macro value without any form of quoting or enclosing in parentheses.</p></div> <p>Rules have the following format:</p> <code> &lt;Regexp&gt; : &lt;Erlang code&gt;.</code> <p>The &lt;Regexp&gt; must occur at the start of a line and not include any blanks; use <code>\t</code> and <code>\s</code> to include TAB and SPACE characters in the regular expression. If &lt;Regexp&gt; matches then the corresponding &lt;Erlang code&gt; is evaluated to generate a token. With the Erlang code the following predefined variables are available:</p> <taglist> <dt><code>TokenChars</code></dt> <item><p>A list of the characters in the matched token.</p> </item> <dt><code>TokenLen</code></dt> <item><p>The number of characters in the matched token.</p> </item> <dt><code>TokenLine</code></dt> <item><p>The line number where the token occurred.</p> </item> </taglist> <p>The code must return:</p> <taglist> <dt><code>{token,Token}</code></dt> <item><p>Return <code>Token</code> to the caller.</p> </item> <dt><code>{end_token,Token}</code></dt> <item><p>Return <code>Token</code> and is last token in a tokens call.</p> </item> <dt><code>skip_token</code></dt> <item><p>Skip this token completely.</p> </item> <dt><code>{error,ErrString}</code></dt> <item><p>An error in the token, <code>ErrString</code> is a string describing the error.</p> </item> </taglist> <p>It is also possible to push back characters into the input characters with the following returns:</p> <list> <item><code>{token,Token,PushBackList}</code></item> <item><code>{end_token,Token,PushBackList}</code></item> <item><code>{skip_token,PushBackList}</code></item> </list> <p>These have the same meanings as the normal returns but the characters in <code>PushBackList</code> will be prepended to the input characters and scanned for the next token. Note that pushing back a newline will mean the line numbering will no longer be correct.</p> <div class="note"><h2>Note!</h2><p>Pushing back characters gives you unexpected possibilities to cause the scanner to loop!</p></div> <p>The following example would match a simple Erlang integer or float and return a token which could be sent to the Erlang parser:</p> <code> D = [0-9] {D}+ : {token,{integer,TokenLine,list_to_integer(TokenChars)}}. {D}+\.{D}+((E|e)(\+|\-)?{D}+)? : {token,{float,TokenLine,list_to_float(TokenChars)}}.</code> <p>The Erlang code in the "Erlang code." section is written into the output file directly after the module declaration and predefined exports declaration so it is possible to add extra exports, define imports and other attributes which are then visible in the whole file.</p> </div> <div class="section"> <h4>Regular Expressions</h4> <p>The regular expressions allowed here is a subset of the set found in <code>egrep</code> and in the AWK programming language, as defined in the book, The AWK Programming Language, by A. V. Aho, B. W. Kernighan, P. J. Weinberger. They are composed of the following characters:</p> <taglist> <dt><code>c</code></dt> <item><p>Matches the non-metacharacter c.</p> </item> <dt><code>\c</code></dt> <item><p>Matches the escape sequence or literal character c.</p> </item> <dt><code>.</code></dt> <item><p>Matches any character.</p> </item> <dt><code>^</code></dt> <item><p>Matches the beginning of a string.</p> </item> <dt><code>$</code></dt> <item><p>Matches the end of a string.</p></item> <dt><code>[abc...]</code></dt> <item><p>Character class, which matches any of the characters <code>abc...</code>. Character ranges are specified by a pair of characters separated by a <code>-</code>.</p> </item> <dt><code>[^abc...]</code></dt> <item><p>Negated character class, which matches any character except <code>abc...</code>.</p> </item> <dt><code>r1 | r2</code></dt> <item><p>Alternation. It matches either <code>r1</code> or <code>r2</code>.</p> </item> <dt><code>r1r2</code></dt> <item><p>Concatenation. It matches <code>r1</code> and then <code>r2</code>.</p> </item> <dt><code>r+</code></dt> <item><p>Matches one or more <code>rs</code>.</p> </item> <dt><code>r*</code></dt> <item><p>Matches zero or more <code>rs</code>.</p> </item> <dt><code>r?</code></dt> <item><p>Matches zero or one <code>rs</code>.</p> </item> <dt><code>(r)</code></dt> <item><p>Grouping. It matches <code>r</code>.</p> </item> </taglist> <p>The escape sequences allowed are the same as for Erlang strings:</p> <taglist> <dt><code>\b</code></dt> <item><p>Backspace.</p></item> <dt><code>\f</code></dt> <item><p>Form feed.</p></item> <dt><code>\n</code></dt> <item><p>Newline (line feed).</p></item> <dt><code>\r</code></dt> <item><p>Carriage return.</p></item> <dt><code>\t</code></dt> <item><p>Tab.</p></item> <dt><code>\e</code></dt> <item><p>Escape.</p></item> <dt><code>\v</code></dt> <item><p>Vertical tab.</p></item> <dt><code>\s</code></dt> <item><p>Space.</p></item> <dt><code>\d</code></dt> <item><p>Delete.</p></item> <dt><code>\ddd</code></dt> <item><p>The octal value <code>ddd</code>.</p></item> <dt><code>\xhh</code></dt> <item><p>The hexadecimal value <code>hh</code>.</p></item> <dt><code>\x{h...}</code></dt> <item><p>The hexadecimal value <code>h...</code>.</p></item> <dt><code>\c</code></dt> <item><p>Any other character literally, for example <code>\\</code> for backslash, <code>\"</code> for <code>"</code>.</p> </item> </taglist> <p>The following examples define Erlang data types:</p> <code> Atoms [a-z][0-9a-zA-Z_]* Variables [A-Z_][0-9a-zA-Z_]* Floats (\+|-)?[0-9]+\.[0-9]+((E|e)(\+|-)?[0-9]+)?</code> <div class="note"><h2>Note!</h2><p>Anchoring a regular expression with <code>^</code> and <code>$</code> is not implemented in the current version of Leex and just generates a parse error.</p></div> </div>
