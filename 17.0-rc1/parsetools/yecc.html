<h1>yecc</h1> <h2 class="modsummary">LALR-1 Parser Generator</h2> <div class="description"> <p>An LALR-1 parser generator for Erlang, similar to <code>yacc</code>. Takes a BNF grammar definition as input, and produces Erlang code for a parser. </p> <p>To understand this text, you also have to look at the <code>yacc</code> documentation in the UNIX(TM) manual. This is most probably necessary in order to understand the idea of a parser generator, and the principle and problems of LALR parsing with finite look-ahead.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="file/2">file(Grammarfile [, Options]) -&gt; YeccRet</h3> <ul class="type"> <li><code>Grammarfile = filename()</code></li> <li><code>Options = Option | [Option]</code></li> <li><code>Option =&nbsp;-&nbsp;see below&nbsp;-</code></li> <li><code>YeccRet = {ok, Parserfile} | {ok, Parserfile, Warnings} | error | {error, Warnings, Errors}</code></li> <li><code>Parserfile = filename()</code></li> <li><code>Warnings = Errors = [{filename(), [ErrorInfo]}]</code></li> <li><code>ErrorInfo = {ErrorLine, module(), Reason}</code></li> <li><code>ErrorLine = integer()</code></li> <li><code>Reason =&nbsp;-&nbsp;formatable by format_error/1&nbsp;-</code></li> </ul> <div class="description"> <p><code>Grammarfile</code> is the file of declarations and grammar rules. Returns <code>ok</code> upon success, or <code>error</code> if there are errors. An Erlang file containing the parser is created if there are no errors. The options are: </p> <taglist> <dt><code>{parserfile, Parserfile}</code>.</dt> <item><code>Parserfile</code> is the name of the file that will contain the Erlang parser code that is generated. The default (<code>""</code>) is to add the extension <code>.erl</code> to <code>Grammarfile</code> stripped of the <code>.yrl</code> extension. </item> <dt><code>{includefile, Includefile}</code>.</dt> <item>Indicates a customized prologue file which the user may want to use instead of the default file <code>lib/parsetools/include/yeccpre.hrl</code> which is otherwise included at the beginning of the resulting parser file. <em>N.B.</em> The <code>Includefile</code> is included 'as is' in the parser file, so it must not have a module declaration of its own, and it should not be compiled. It must, however, contain the necessary export declarations. The default is indicated by <code>""</code>. </item> <dt><code>{report_errors, bool()}</code>.</dt> <item>Causes errors to be printed as they occur. Default is <code>true</code>. </item> <dt><code>{report_warnings, bool()}</code>.</dt> <item>Causes warnings to be printed as they occur. Default is <code>true</code>. </item> <dt><code>{report, bool()}</code>.</dt> <item>This is a short form for both <code>report_errors</code> and <code>report_warnings</code>. </item> <dt><code>warnings_as_errors</code></dt> <item> <p>Causes warnings to be treated as errors.</p> </item> <dt><code>{return_errors, bool()}</code>.</dt> <item>If this flag is set, <code>{error, Errors, Warnings}</code> is returned when there are errors. Default is <code>false</code>. </item> <dt><code>{return_warnings, bool()}</code>.</dt> <item>If this flag is set, an extra field containing <code>Warnings</code> is added to the tuple returned upon success. Default is <code>false</code>. </item> <dt><code>{return, bool()}</code>.</dt> <item>This is a short form for both <code>return_errors</code> and <code>return_warnings</code>. </item> <dt><code>{verbose, bool()}</code>. </dt> <item>Determines whether the parser generator should give full information about resolved and unresolved parse action conflicts (<code>true</code>), or only about those conflicts that prevent a parser from being generated from the input grammar (<code>false</code>, the default). </item> </taglist> <p>Any of the Boolean options can be set to <code>true</code> by stating the name of the option. For example, <code>verbose</code> is equivalent to <code>{verbose, true}</code>. </p> <p>The value of the <code>Parserfile</code> option stripped of the <code>.erl</code> extension is used by Yecc as the module name of the generated parser file.</p> <p>Yecc will add the extension <code>.yrl</code> to the <code>Grammarfile</code> name, the extension <code>.hrl</code> to the <code>Includefile</code> name, and the extension <code>.erl</code> to the <code>Parserfile</code> name, unless the extension is already there.</p> </div> </div> <div class="function"> <h3 id="format_error/1">format_error(Reason) -&gt; Chars</h3> <ul class="type"> <li><code>Reason =&nbsp;-&nbsp;as returned by yecc:file/1,2&nbsp;-</code></li> <li><code>Chars = [char() | Chars]</code></li> </ul> <div class="description"> <p>Returns a descriptive string in English of an error tuple returned by <code>yecc:file/1,2</code>. This function is mainly used by the compiler invoking Yecc.</p> </div> </div> </div> <div class="section"> <h4>Pre-Processing</h4> <p>A <code>scanner</code> to pre-process the text (program, etc.) to be parsed is not provided in the <code>yecc</code> module. The scanner serves as a kind of lexicon look-up routine. It is possible to write a grammar that uses only character tokens as terminal symbols, thereby eliminating the need for a scanner, but this would make the parser larger and slower.</p> <p>The user should implement a scanner that segments the input text, and turns it into one or more lists of tokens. Each token should be a tuple containing information about syntactic category, position in the text (e.g. line number), and the actual terminal symbol found in the text: <code>{Category, LineNumber, Symbol}</code>.</p> <p>If a terminal symbol is the only member of a category, and the symbol name is identical to the category name, the token format may be <code>{Symbol, LineNumber}</code>.</p> <p>A list of tokens produced by the scanner should end with a special <code>end_of_input</code> tuple which the parser is looking for. The format of this tuple should be <code>{Endsymbol, LastLineNumber}</code>, where <code>Endsymbol</code> is an identifier that is distinguished from all the terminal and non-terminal categories of the syntax rules. The <code>Endsymbol</code> may be declared in the grammar file (see below).</p> <p>The simplest case is to segment the input string into a list of identifiers (atoms) and use those atoms both as categories and values of the tokens. For example, the input string <code>aaa bbb 777, X</code> may be scanned (tokenized) as:</p> <pre class="sh_erlang"> [{aaa, 1}, {bbb, 1}, {777, 1}, {',' , 1}, {'X', 1}, {'$end', 1}]. </pre> <p>This assumes that this is the first line of the input text, and that <code>'$end'</code> is the distinguished <code>end_of_input</code> symbol.</p> <p>The Erlang scanner in the <code>io</code> module can be used as a starting point when writing a new scanner. Study <code>yeccscan.erl</code> in order to see how a filter can be added on top of <code>io:scan_erl_form/3</code> to provide a scanner for Yecc that tokenizes grammar files before parsing them with the Yecc parser. A more general approach to scanner implementation is to use a scanner generator. A scanner generator in Erlang called <code>leex</code> is under development.</p> </div> <div class="section"> <h4>Grammar Definition Format</h4> <p>Erlang style <code>comments</code>, starting with a <code>'%'</code>, are allowed in grammar files.</p> <p>Each <code>declaration</code> or <code>rule</code> ends with a dot (the character <code>'.'</code>).</p> <p>The grammar starts with an optional <code>header</code> section. The header is put first in the generated file, before the module declaration. The purpose of the header is to provide a means to make the documentation generated by <code>EDoc</code> look nicer. Each header line should be enclosed in double quotes, and newlines will be inserted between the lines. For example:</p> <code> Header "%% Copyright (C)" "%% @private" "%% @Author John"</code> <p>Next comes a declaration of the <code>nonterminal categories</code> to be used in the rules. For example:</p> <pre class="sh_erlang"> Nonterminals sentence nounphrase verbphrase. </pre> <p>A non-terminal category can be used at the left hand side (= <code>lhs</code>, or <code>head</code>) of a grammar rule. It can also appear at the right hand side of rules.</p> <p>Next comes a declaration of the <code>terminal categories</code>, which are the categories of tokens produced by the scanner. For example:</p> <pre class="sh_erlang"> Terminals article adjective noun verb. </pre> <p>Terminal categories may only appear in the right hand sides (= <code>rhs</code>) of grammar rules.</p> <p>Next comes a declaration of the <code>rootsymbol</code>, or start category of the grammar. For example:</p> <pre class="sh_erlang"> Rootsymbol sentence. </pre> <p>This symbol should appear in the lhs of at least one grammar rule. This is the most general syntactic category which the parser ultimately will parse every input string into.</p> <p>After the rootsymbol declaration comes an optional declaration of the <code>end_of_input</code> symbol that your scanner is expected to use. For example:</p> <pre class="sh_erlang"> Endsymbol '$end'. </pre> <p>Next comes one or more declarations of <code>operator precedences</code>, if needed. These are used to resolve shift/reduce conflicts (see <code>yacc</code> documentation).</p> <p>Examples of operator declarations:</p> <pre class="sh_erlang"> Right 100 '='. Nonassoc 200 '==' '=/='. Left 300 '+'. Left 400 '*'. Unary 500 '-'. </pre> <p>These declarations mean that <code>'='</code> is defined as a <code>right associative binary</code> operator with precedence 100, <code>'=='</code> and <code>'=/='</code> are operators with <code>no associativity</code>, <code>'+'</code> and <code>'*'</code> are <code>left associative binary</code> operators, where <code>'*'</code> takes precedence over <code>'+'</code> (the normal case), and <code>'-'</code> is a <code>unary</code> operator of higher precedence than <code>'*'</code>. The fact that '==' has no associativity means that an expression like <code>a == b == c</code> is considered a syntax error.</p> <p>Certain rules are assigned precedence: each rule gets its precedence from the last terminal symbol mentioned in the right hand side of the rule. It is also possible to declare precedence for non-terminals, "one level up". This is practical when an operator is overloaded (see also example 3 below).</p> <p>Next come the <code>grammar rules</code>. Each rule has the general form</p> <pre class="sh_erlang"> Left_hand_side -&gt; Right_hand_side : Associated_code. </pre> <p>The left hand side is a non-terminal category. The right hand side is a sequence of one or more non-terminal or terminal symbols with spaces between. The associated code is a sequence of zero or more Erlang expressions (with commas <code>','</code> as separators). If the associated code is empty, the separating colon <code>':'</code> is also omitted. A final dot marks the end of the rule.</p> <p>Symbols such as <code>'{'</code>, <code>'.'</code>, etc., have to be enclosed in single quotes when used as terminal or non-terminal symbols in grammar rules. The use of the symbols <code>'$empty'</code>, <code>'$end'</code>, and <code>'$undefined'</code> should be avoided.</p> <p>The last part of the grammar file is an optional section with Erlang code (= function definitions) which is included 'as is' in the resulting parser file. This section must start with the pseudo declaration, or key words</p> <pre class="sh_erlang"> Erlang code. </pre> <p>No syntax rule definitions or other declarations may follow this section. To avoid conflicts with internal variables, do not use variable names beginning with two underscore characters ('__') in the Erlang code in this section, or in the code associated with the individual syntax rules.</p> <p>The optional <code>expect</code> declaration can be placed anywhere before the last optional section with Erlang code. It is used for suppressing the warning about conflicts that is ordinarily given if the grammar is ambiguous. An example:</p> <pre class="sh_erlang"> Expect 2. </pre> <p>The warning is given if the number of shift/reduce conflicts differs from 2, or if there are reduce/reduce conflicts. </p> </div> <div class="section"> <h4>Examples</h4> <p>A grammar to parse list expressions (with empty associated code):</p> <pre class="sh_erlang"> Nonterminals list elements element. Terminals atom '(' ')'. Rootsymbol list. list -&gt; '(' ')'. list -&gt; '(' elements ')'. elements -&gt; element. elements -&gt; element elements. element -&gt; atom. element -&gt; list. </pre> <p>This grammar can be used to generate a parser which parses list expressions, such as <code>(), (a), (peter charles), (a (b c) d (())), ...</code> provided that your scanner tokenizes, for example, the input <code>(peter charles)</code> as follows:</p> <pre class="sh_erlang"> [{'(', 1} , {atom, 1, peter}, {atom, 1, charles}, {')', 1}, {'$end', 1}] </pre> <p>When a grammar rule is used by the parser to parse (part of) the input string as a grammatical phrase, the associated code is evaluated, and the value of the last expression becomes the value of the parsed phrase. This value may be used by the parser later to build structures that are values of higher phrases of which the current phrase is a part. The values initially associated with terminal category phrases, i.e. input tokens, are the token tuples themselves.</p> <p>Below is an example of the grammar above with structure building code added:</p> <pre class="sh_erlang"> list -&gt; '(' ')' : nil. list -&gt; '(' elements ')' : '$2'. elements -&gt; element : {cons, '$1', nil}. elements -&gt; element elements : {cons, '$1', '$2'}. element -&gt; atom : '$1'. element -&gt; list : '$1'. </pre> <p>With this code added to the grammar rules, the parser produces the following value (structure) when parsing the input string <code>(a b c).</code>. This still assumes that this was the first input line that the scanner tokenized:</p> <pre class="sh_erlang"> {cons, {atom, 1, a,} {cons, {atom, 1, b}, {cons, {atom, 1, c}, nil}}} </pre> <p>The associated code contains <code>pseudo variables</code> <code>'$1'</code>, <code>'$2'</code>, <code>'$3'</code>, etc. which refer to (are bound to) the values associated previously by the parser with the symbols of the right hand side of the rule. When these symbols are terminal categories, the values are token tuples of the input string (see above).</p> <p>The associated code may not only be used to build structures associated with phrases, but may also be used for syntactic and semantic tests, printout actions (for example for tracing), etc. during the parsing process. Since tokens contain positional (line number) information, it is possible to produce error messages which contain line numbers. If there is no associated code after the right hand side of the rule, the value <code>'$undefined'</code> is associated with the phrase.</p> <p>The right hand side of a grammar rule may be empty. This is indicated by using the special symbol <code>'$empty'</code> as rhs. Then the list grammar above may be simplified to:</p> <pre class="sh_erlang"> list -&gt; '(' elements ')' : '$2'. elements -&gt; element elements : {cons, '$1', '$2'}. elements -&gt; '$empty' : nil. element -&gt; atom : '$1'. element -&gt; list : '$1'. </pre> </div> <div class="section"> <h4>Generating a Parser</h4> <p>To call the parser generator, use the following command:</p> <pre class="sh_erlang"> yecc:file(Grammarfile). </pre> <p>An error message from Yecc will be shown if the grammar is not of the LALR type (for example too ambiguous). Shift/reduce conflicts are resolved in favor of shifting if there are no operator precedence declarations. Refer to the <code>yacc</code> documentation on the use of operator precedence.</p> <p>The output file contains Erlang source code for a parser module with module name equal to the <code>Parserfile</code> parameter. After compilation, the parser can be called as follows (the module name is assumed to be <code>myparser</code>):</p> <pre class="sh_erlang"> myparser:parse(myscanner:scan(Inport)) </pre> <p>The call format may be different if a customized prologue file has been included when generating the parser instead of the default file <code>lib/parsetools/include/yeccpre.hrl</code>.</p> <p>With the standard prologue, this call will return either <code>{ok, Result}</code>, where <code>Result</code> is a structure that the Erlang code of the grammar file has built, or <code>{error, {Line_number, Module, Message}}</code> if there was a syntax error in the input.</p> <p><code>Message</code> is something which may be converted into a string by calling <code>Module:format_error(Message)</code> and printed with <code>io:format/3</code>.</p> <div class="note"><h2>Note!</h2> <p>By default, the parser that was generated will not print out error messages to the screen. The user will have to do this either by printing the returned error messages, or by inserting tests and print instructions in the Erlang code associated with the syntax rules of the grammar file.</p> </div> <p>It is also possible to make the parser ask for more input tokens when needed if the following call format is used:</p> <pre class="sh_erlang"> myparser:parse_and_scan({Function, Args}) myparser:parse_and_scan({Mod, Tokenizer, Args}) </pre> <p>The tokenizer <code>Function</code> is either a fun or a tuple <code>{Mod, Tokenizer}</code>. The call <code>apply(Function, Args)</code> or <code>apply({Mod, Tokenizer}, Args)</code> is executed whenever a new token is needed. This, for example, makes it possible to parse from a file, token by token.</p> <p>The tokenizer used above has to be implemented so as to return one of the following:</p> <pre class="sh_erlang"> {ok, Tokens, Endline} {eof, Endline} {error, Error_description, Endline} </pre> <p>This conforms to the format used by the scanner in the Erlang <code>io</code> library module.</p> <p>If <code>{eof, Endline}</code> is returned immediately, the call to <code>parse_and_scan/1</code> returns <code>{ok, eof}</code>. If <code>{eof, Endline}</code> is returned before the parser expects end of input, <code>parse_and_scan/1</code> will, of course, return an error message (see above). Otherwise <code>{ok, Result}</code> is returned.</p> </div> <div class="section"> <h4>More Examples</h4> <p>1. A grammar for parsing infix arithmetic expressions into prefix notation, without operator precedence:</p> <pre class="sh_erlang"> Nonterminals E T F. Terminals '+' '*' '(' ')' number. Rootsymbol E. E -&gt; E '+' T: ['$2', '$1', '$3']. E -&gt; T : '$1'. T -&gt; T '*' F: ['$2', '$1', '$3']. T -&gt; F : '$1'. F -&gt; '(' E ')' : '$2'. F -&gt; number : '$1'. </pre> <p>2. The same with operator precedence becomes simpler:</p> <pre class="sh_erlang"> Nonterminals E. Terminals '+' '*' '(' ')' number. Rootsymbol E. Left 100 '+'. Left 200 '*'. E -&gt; E '+' E : ['$2', '$1', '$3']. E -&gt; E '*' E : ['$2', '$1', '$3']. E -&gt; '(' E ')' : '$2'. E -&gt; number : '$1'. </pre> <p>3. An overloaded minus operator:</p> <pre class="sh_erlang"> Nonterminals E uminus. Terminals '*' '-' number. Rootsymbol E. Left 100 '-'. Left 200 '*'. Unary 300 uminus. E -&gt; E '-' E. E -&gt; E '*' E. E -&gt; uminus. E -&gt; number. uminus -&gt; '-' E. </pre> <p>4. The Yecc grammar that is used for parsing grammar files, including itself:</p> <pre class="sh_erlang"> Nonterminals grammar declaration rule head symbol symbols attached_code token tokens. Terminals atom float integer reserved_symbol reserved_word string char var '-&gt;' ':' dot. Rootsymbol grammar. Endsymbol '$end'. grammar -&gt; declaration : '$1'. grammar -&gt; rule : '$1'. declaration -&gt; symbol symbols dot: {'$1', '$2'}. rule -&gt; head '-&gt;' symbols attached_code dot: {rule, ['$1' | '$3'], '$4'}. head -&gt; symbol : '$1'. symbols -&gt; symbol : ['$1']. symbols -&gt; symbol symbols : ['$1' | '$2']. attached_code -&gt; ':' tokens : {erlang_code, '$2'}. attached_code -&gt; '$empty' : {erlang_code, [{atom, 0, '$undefined'}]}. tokens -&gt; token : ['$1']. tokens -&gt; token tokens : ['$1' | '$2']. symbol -&gt; var : value_of('$1'). symbol -&gt; atom : value_of('$1'). symbol -&gt; integer : value_of('$1'). symbol -&gt; reserved_word : value_of('$1'). token -&gt; var : '$1'. token -&gt; atom : '$1'. token -&gt; float : '$1'. token -&gt; integer : '$1'. token -&gt; string : '$1'. token -&gt; char : '$1'. token -&gt; reserved_symbol : {value_of('$1'), line_of('$1')}. token -&gt; reserved_word : {value_of('$1'), line_of('$1')}. token -&gt; '-&gt;' : {'-&gt;', line_of('$1')}. token -&gt; ':' : {':', line_of('$1')}. Erlang code. value_of(Token) -&gt; element(3, Token). line_of(Token) -&gt; element(2, Token). </pre> <div class="note"><h2>Note!</h2> <p>The symbols <code>'-&gt;'</code>, and <code>':'</code> have to be treated in a special way, as they are meta symbols of the grammar notation, as well as terminal symbols of the Yecc grammar.</p> </div> <p>5. The file <code>erl_parse.yrl</code> in the <code>lib/stdlib/src</code> directory contains the grammar for Erlang.</p> <div class="note"><h2>Note!</h2> <p>Syntactic tests are used in the code associated with some rules, and an error is thrown (and caught by the generated parser to produce an error message) when a test fails. The same effect can be achieved with a call to <code>return_error(Error_line, Message_string)</code>, which is defined in the <code>yeccpre.hrl</code> default header file.</p> </div> </div> <div class="section"> <h4>Files</h4> <pre class="sh_erlang"> lib/parsetools/include/yeccpre.hrl </pre> </div> <div class="section"> <h4>See Also</h4> <p>Aho & Johnson: 'LR Parsing', ACM Computing Surveys, vol. 6:2, 1974.</p> </div>
