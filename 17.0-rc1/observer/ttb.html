<h1>ttb</h1> <h2 class="modsummary">A base for building trace tools for distributed systems.</h2> <div class="description"> <p>The Trace Tool Builder <code>ttb</code> is a base for building trace tools for distributed systems. </p> <p>When using <code>ttb</code>, <code>dbg</code> shall not be used in parallel.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start_trace/4">start_trace(Nodes, Patterns, FlagSpec, Opts) -&gt; Result</h3> <ul class="type"> <li><code>Result = see p/2</code></li> <li><code>Nodes = see tracer/2</code></li> <li><code>Patterns = [tuple()]</code></li> <li><code>FlagSpec = {Procs, Flags}</code></li> <li><code>Proc = see p/2</code></li> <li><code>Flags = see p/2</code></li> <li><code>Opts = see tracer/2</code></li> </ul> <div class="description"> <p>This function is a shortcut allowing to start a trace with one command. Each tuple in <code>Patterns</code> is converted to list which is in turn passed to <code>ttb:tpl</code>. The call:<pre class="sh_erlang"> ttb:start_trace([Node, OtherNode], [{mod, foo, []}, {mod, bar, 2}], {all, call}, [{file, File}, {handler,{fun myhandler/4, S}}])</pre> is equivalent to <pre class="sh_erlang"> ttb:start_trace([Node, OtherNode], [{file, File}, {handler,{fun myhandler/4, S}}]), ttb:tpl(mod, foo, []), ttb:tpl(mod, bar, 2, []), ttb:p(all, call)</pre> </p> </div> </div> <div class="function"> <h3 id="tracer/0">tracer() -&gt; Result</h3> <div class="description"> <p>This is equivalent to <code>tracer(node())</code>.</p> </div> </div> <div class="function"> <h3 id="tracer/1">tracer(Shortcut) -&gt; Result</h3> <ul class="type"> <li><code>Shortcut = shell | dbg</code></li> </ul> <div class="description"> <p><code>shell</code> is equivalent to <code>tracer(node(),[{file, {local, "ttb"}}, shell])</code>.</p> <p><code>dbg</code> is equivalent to <code>tracer(node(),[{shell, only}])</code>.</p> </div> </div> <div class="function"> <h3 id="tracer/1-1">tracer(Nodes) -&gt; Result</h3> <div class="description"> <p>This is equivalent to <code>tracer(Nodes,[])</code>.</p> </div> </div> <div class="function"> <h3 id="tracer/2">tracer(Nodes,Opts) -&gt; Result</h3> <ul class="type"> <li><code>Result = {ok, ActivatedNodes} | {error,Reason}</code></li> <li><code>Nodes = atom() | [atom()] | all | existing | new</code></li> <li><code>Opts = Opt | [Opt]</code></li> <li><code>Opt = {file,Client} | {handler, FormatHandler} | {process_info,PI} | shell | {shell, ShellSpec} | {timer, TimerSpec} | {overload_check, {MSec, Module, Function}} | {flush, MSec} | resume | {resume, FetchTimeout}</code></li> <li><code>TimerSpec = MSec | {MSec, StopOpts}</code></li> <li><code>MSec = FetchTimeout = integer()</code></li> <li><code>Module = Function = atom() </code></li> <li><code>StopOpts = see stop/2</code></li> <li><code>Client = File | {local, File}</code></li> <li><code>File = Filename | Wrap</code></li> <li><code>Filename = string()</code></li> <li><code>Wrap = {wrap,Filename} | {wrap,Filename,Size,Count}</code></li> <li><code>FormatHandler = See format/2</code></li> <li><code>PI = true | false </code></li> <li><code>ShellSpec = true | false | only</code></li> </ul> <div class="description"> <p>This function starts a file trace port on all given nodes and also points the system tracer for sequential tracing to the same port. </p> <p>The given <code>Filename</code> will be prefixed with the node name. Default <code>Filename</code> is "ttb". </p> <p><code>File={wrap,Filename,Size,Count}</code> can be used if the size of the trace logs must be limited. Default values are <code>Size=128*1024</code> and <code>Count=8</code>. </p> <p>When tracing diskless nodes, <code>ttb</code> must be started from an external "trace control node" with disk access, and <code>Client</code> must be <code>{local, File}</code>. All trace information is then sent to the trace control node where it is written to file. </p> <p>The <code>process_info</code> option indicates if process information should be collected. If <code>PI = true</code> (which is default), each process identifier <code>Pid</code> is replaced by a tuple <code>{Pid,ProcessInfo,Node}</code>, where <code>ProcessInfo</code> is the process' registered name its globally registered name, or its initial function. It is possible to turn off this functionality by setting <code>PI = false</code>. </p> <p>The <code>{shell, ShellSpec}</code> option indicates that the trace messages should be printed on the console as they are received by the tracing process. This implies <code>{local, File}</code> trace client. If the ShellSpec is <code>only</code> (instead of <code>true</code>), no trace logs are stored. </p> <p>The <code>shell</code> option is a shortcut for <code>{shell, true}</code>.</p> <p>The <code>timer</code> option indicates that the trace should be automatically stopped after <code>MSec</code> milliseconds. <code>StopOpts</code> are passed to <code>ttb:stop/2</code> command if specified (default is <code>[]</code>). Note that the timing is approximate, as delays related to network communication are always present. The timer starts after <code>ttb:p/2</code> is issued, so you can set up your trace patterns before. </p> <p>The <code>overload_check</code> option allows to enable overload checking on the nodes under trace. <code>Module:Function(check)</code> is performed each <code>MSec</code> milliseconds. If the check returns <code>true</code>, the tracing is disabled on a given node.<br /> <code>Module:Function</code> should be able to handle at least three atoms: <code>init</code>, <code>check</code> and <code>stop</code>. <code>init</code> and <code>stop</code> give the user a possibility to initialize and clean up the check environment.<br /> When a node gets overloaded, it is not possible to issue <code>ttb:p</code> nor any command from the <code>ttb:tp</code> family, as it would lead to inconsistent tracing state (different trace specifications on different node). </p> <p>The <code>flush</code> option periodically flushes all file trace port clients (see <code>dbg:flush_trace_port/1</code>). When enabled, the buffers are freed each <code>MSec</code> milliseconds. This option is not allowed with <code>{file, {local, File}}</code> tracing. </p> <p><code>{resume, FetchTimeout}</code> enables the autoresume feature. Whenever enabled, remote nodes try to reconnect to the controlling node in case they were restarted. The feature requires <code>runtime_tools</code> application to be started (so it has to be present in the <code>.boot</code> scripts if the traced nodes run with embedded erlang). If this is not possible, resume may be performed manually by starting <code>runtime_tools</code> remotely using <code>rpc:call/4</code>.<br /> <code>ttb</code> tries to fetch all logs from a reconnecting node before reinitializing the trace. This has to finish within FetchTimeout milliseconds or is aborted<br /> By default, autostart information is stored in a file called <code>ttb_autostart.bin</code> on each node. If this is not desired (i.e. on diskless nodes), a custom module to handle autostart information storage and retrieval can be provided by specifying <code>ttb_autostart_module</code> environment variable for the <code>runtime_tools</code> application. The module has to respond to the following API: <taglist> <dt><code>write_config(Data) -&gt; ok</code></dt> <item>Store the provided data for further retrieval. It is important to realize that the data storage used must not be affected by the node crash.</item> <dt><code>read_config() -&gt; {ok, Data} | {error, Error}</code></dt> <item>Retrieve configuration stored with <code>write_config(Data)</code>.</item> <dt><code>delete_config() -&gt; ok</code></dt> <item>Delete configuration stored with <code>write_config(Data)</code>. Note that after this call any subsequent calls to <code>read_config</code> must return <code>{error, Error}</code>. </item> </taglist> </p> <p>The <code>resume</code> option implies the default <code>FetchTimeout</code>, which is 10 seconds</p> </div> </div> <div class="function"> <h3 id="p/2">p(Procs,Flags) -&gt; Return</h3> <ul class="type"> <li><code>Return = {ok,[{Procs,MatchDesc}]}</code></li> <li><code>Procs = Process | [Process] | all | new | existing</code></li> <li><code>Process = pid() | atom() | {global,atom()}</code></li> <li><code>Flags = Flag | [Flag]</code></li> </ul> <div class="description"> <p>This function sets the given trace flags on the given processes. The <code>timestamp</code> flag is always turned on. </p> <p>Please turn to the Reference manual for module <code>dbg</code> for details about the possible trace flags. The parameter <code>MatchDesc</code> is the same as returned from <code>dbg:p/2</code></p> <p>Processes can be given as registered names, globally registered names or process identifiers. If a registered name is given, the flags are set on processes with this name on all active nodes.</p> <p>Issuing this command starts the timer for this trace if <code>timer</code> option was specified with <code>tracer/2</code>. </p> </div> </div> <div class="function"> <div class="description"> <p>These functions should be used in combination with the <code>call</code> trace flag for setting and clearing trace patterns. When the <code>call</code> trace flag is set on a process, function calls will be traced on that process if a trace pattern has been set for the called function. Trace patterns specifies how to trace a function by using match specifications. Match specifications are described in the User's Guide for the erlang runtime system <code>erts</code>. </p> <p>These functions are equivalent to the corresponding functions in <code>dbg</code>, but all calls are stored in the history. The history buffer makes it easy to create config files so that the same trace environment can be setup several times, e.g. if you want to compare two test runs. It also reduces the amount of typing when using <code>ttb</code> from the erlang shell. </p> <taglist> <dt><code>tp</code></dt> <item>Set trace pattern on global function calls</item> <dt><code>tpl</code></dt> <item>Set trace pattern on local and global function calls</item> <dt><code>ctp</code></dt> <item>Clear trace pattern on local and global function calls</item> <dt><code>ctpl</code></dt> <item>Clear trace pattern on local function calls</item> <dt><code>ctpg</code></dt> <item>Clear trace pattern on global function calls</item> </taglist> <p>With <code>tp</code> and <code>tpl</code> one of match specification shortcuts may be used (example: <code>ttb:tp(foo_module, caller)</code>). The shortcuts are: <taglist> <item><code>return</code> - for <code>[{'_',[],[{return_trace}]}]</code> (report the return value)</item> <item><code>caller</code> - for <code>[{'_',[],[{message,{caller}}]}]</code> (report the calling function)</item> <item><code>{codestr, Str}</code> - for <code>dbg:fun2ms/1</code> arguments passed as strings (example: <code>"fun(_) -&gt; return_trace() end"</code>) </item> </taglist> </p> </div> </div> <div class="function"> <h3 id="list_history/0">list_history() -&gt; History</h3> <ul class="type"> <li><code>History = [{N,Func,Args}]</code></li> </ul> <div class="description"> <p>All calls to <code>ttb</code> is stored in the history. This function returns the current content of the history. Any entry can be re-executed with <code>run_history/1</code> or stored in a config file with <code>write_config/2/3</code>.</p> </div> </div> <div class="function"> <h3 id="run_history/1">run_history(N) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>N = integer() | [integer()]</code></li> </ul> <div class="description"> <p>Executes the given entry or entries from the history list. History can be listed with <code>list_history/0</code>.</p> </div> </div> <div class="function"> <h3 id="write_config/2">write_config(ConfigFile,Config)</h3> <div class="description"> <p>Equivalent to <code>write_config(ConfigFile,Config,[])</code>.</p> </div> </div> <div class="function"> <h3 id="write_config/3">write_config(ConfigFile,Config,Opts) -&gt; ok | {error,Reason}</h3> <ul class="type"> <li><code>ConfigFile = string()</code></li> <li><code>Config = all | [integer()] | [{Mod,Func,Args}]</code></li> <li><code>Mod = atom()</code></li> <li><code>Func = atom()</code></li> <li><code>Args = [term()]</code></li> <li><code>Opts = Opt | [Opt]</code></li> <li><code>Opt = append</code></li> </ul> <div class="description"> <p>This function creates or extends a config file which can be used for restoring a specific configuration later. </p> <p>The content of the config file can either be fetched from the history or given directly as a list of <code>{Mod,Func,Args}</code>. </p> <p>If the complete history is to be stored in the config file <code>Config</code> should be <code>all</code>. If only a selected number of entries from the history should be stored, <code>Config</code> should be a list of integers pointing out the entries to be stored. </p> <p>If <code>Opts</code> is not given or if it is <code>[]</code>, <code>ConfigFile</code> is deleted and a new file is created. If <code>Opts = [append]</code>, <code>ConfigFile</code> will not be deleted. The new information will be appended at the end of the file.</p> </div> </div> <div class="function"> <h3 id="run_config/1">run_config(ConfigFile) -&gt; ok | {error,Reason}</h3> <ul class="type"> <li><code>ConfigFile = string()</code></li> </ul> <div class="description"> <p>Executes all entries in the given config file. Note that the history of the last trace is always available in the file named <code>ttb_last_config</code>.</p> </div> </div> <div class="function"> <h3 id="run_config/2">run_config(ConfigFile,NumList) -&gt; ok | {error,Reason}</h3> <ul class="type"> <li><code>ConfigFile = string()</code></li> <li><code>NumList = [integer()]</code></li> </ul> <div class="description"> <p>Executes selected entries from the given config file. <code>NumList</code> is a list of integers pointing out the entries to be executed. </p> <p>The content of a config file can be listed with <code>list_config/1</code>.</p> <p> Note that the history of the last trace is always available in the file named <code>ttb_last_config</code>.</p> </div> </div> <div class="function"> <h3 id="list_config/1">list_config(ConfigFile) -&gt; Config | {error,Reason}</h3> <ul class="type"> <li><code>ConfigFile = string()</code></li> <li><code>Config = [{N,Func,Args}]</code></li> </ul> <div class="description"> <p>Lists all entries in the given config file.</p> </div> </div> <div class="function"> <h3 id="write_trace_info/2">write_trace_info(Key,Info) -&gt; ok</h3> <ul class="type"> <li><code>Key = term()</code></li> <li><code>Info = Data | fun() -&gt; Data</code></li> <li><code>Data = term()</code></li> </ul> <div class="description"> <p>The <code>.ti</code> file contains <code>{Key,ValueList}</code> tuples. This function adds <code>Data</code> to the ValueList associated with <code>Key</code>. All information written with this function will be included in the call to the format handler.</p> </div> </div> <div class="function"> <h3 id="seq_trigger_ms/0">seq_trigger_ms() -&gt; MatchSpec</h3> <div class="description"> <p>Equivalent to <code>seq_trigger_ms(all)</code></p> </div> </div> <div class="function"> <h3 id="seq_trigger_ms/1">seq_trigger_ms(Flags) -&gt; MatchSpec</h3> <ul class="type"> <li><code>MatchSpec = match_spec()</code></li> <li><code>Flags = all | SeqTraceFlag | [SeqTraceFlag]</code></li> <li><code>SeqTraceFlag = atom()</code></li> </ul> <div class="description"> <p>A match specification can turn on or off sequential tracing. This function returns a match specification which turns on sequential tracing with the given <code>Flags</code>. </p> <p>This match specification can be given as the last argument to <code>tp</code> or <code>tpl</code>. The activated <code>Item</code> will then become a <em>trigger</em> for sequential tracing. This means that if the item is called on a process with the <code>call</code> trace flag set, the process will be "contaminated" with the seq_trace token. </p> <p>If <code>Flags = all</code>, all possible flags are set. </p> <p>Please turn to the reference manual for the <em><code>seq_trace</code></em> module in the <em><code>kernel</code></em> application to see the possible values for <code>SeqTraceFlag</code>. For a description of the match_spec() syntax, please turn to the <em>User's guide</em> for the runtime system (<em>erts</em>). The chapter <em>Match Specification in Erlang</em> explains the general match specification "language". </p> <div class="note"><h2>Note!</h2> <p>The <em>system tracer</em> for sequential tracing is automatically initiated by <code>ttb</code> when a trace port is started with <code>ttb:tracer/0/1/2</code>.</p> </div> <p>Example of how to use the <code>seq_trigger_ms/0/1</code> function:</p> <pre class="sh_erlang"> (tiger@durin)5&gt; ttb:tracer(). {ok,[tiger@durin]} (tiger@durin)6&gt; ttb:p(all,call). {ok,{[all],[call]}} (tiger@durin)7&gt; ttb:tp(mod,func,ttb:seq_trigger_ms()). {ok,[{matched,1},{saved,1}]} (tiger@durin)8&gt; </pre> <p>Whenever <code>mod:func(...)</code> is called after this, the seq_trace token will be set on the executing process.</p> </div> </div> <div class="function"> <h3 id="stop/0">stop()</h3> <div class="description"> <p>Equivalent to <code>stop([])</code>.</p> </div> </div> <div class="function"> <h3 id="stop/1">stop(Opts) -&gt; stopped | {stopped, Dir}</h3> <ul class="type"> <li><code>Opts = Opt | [Opt]</code></li> <li><code>Opt = nofetch | {fetch_dir, Dir} | format | {format, FormatOpts} | return_fetch_dir</code></li> <li><code>Dir = string()</code></li> <li><code>FormatOpts = see format/2</code></li> </ul> <div class="description"> <p>Stops tracing on all nodes. Logs and trace information files are sent to the trace control node and stored in a directory named <code>ttb_upload_FileName-Timestamp</code>, where <code>Filename</code> is the one provided with <code>{file, File}</code> during trace setup and <code>Timestamp</code> is of the form <code>yyyymmdd-hhmmss</code>. Even logs from nodes on the same machine as the trace control node are moved to this directory. The history list is saved to a file named <code>ttb_last_config</code> for further reference (as it will be not longer accessible through history and configuration management functions (like <code>ttb:list_history/0</code>). </p> <p>The <code>nofetch</code> option indicates that trace logs shall not be collected after tracing is stopped. </p> <p>The <code>{fetch, Dir}</code> option allows to specify the directory to fetch the data to. If the directory already exists, an error is thrown. </p> <p>The <code>format</code> option indicates that the trace logs shall be formatted after tracing is stopped. All logs in the fetch directory will be merged. You may use <code>{format, FormatOpts}</code> to pass additional arguments to <code>format/2</code>.</p> <p>The <code>return_fetch_dir</code> option indicates that the return value should be <code>{stopped, Dir}</code> and not just <code>stopped</code>. This implies <code>fetch</code>. </p> </div> </div> <div class="function"> <h3 id="get_et_handler/0">get_et_handler()</h3> <div class="description"> <p>The <code>et</code> handler returned by the function may be used with <code>format/2</code> or <code>tracer/2</code>. Example: <code>ttb:format(Dir, [{handler, ttb:get_et_handler()}])</code>.</p> </div> </div> <div class="function"> <h3 id="format/1">format(File)</h3> <div class="description"> <p>Same as <code>format(File,[])</code>.</p> </div> </div> <div class="function"> <h3 id="format/2">format(File,Options) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>File = string() | [string()]</code></li> <d>This can be the name of a binary log, a list of such logs or the name of a directory containing one or more binary logs.</d> <li><code>Options = Opt | [Opt]</code></li> <li><code>Opt = {out,Out} | {handler,FormatHandler} | disable_sort</code></li> <li><code>Out = standard_io | string()</code></li> <li><code>FormatHandler = {Function, InitialState}</code></li> <li><code>Function = fun(Fd,Trace,TraceInfo,State) -&gt; State</code></li> <li><code>Fd = standard_io | FileDescriptor</code></li> <d>This is the file descriptor of the destination file <code>Out</code></d> <li><code>Trace = tuple()</code></li> <d>This is the trace message. Please turn to the Reference manual for the <code>erlang</code>module for details.</d> <li><code>TraceInfo = [{Key,ValueList}]</code></li> <d>This includes the keys <code>flags</code>, <code>client</code> and <code>node</code>, and if <code>handler</code> is given as option to the tracer function, this is also included. In addition all information written with the <code>write_trace_info/2</code>function is included. </d> </ul> <div class="description"> <p>Reads the given binary trace log(s). The logs are processed in the order of their timestamp as long as <code>disable_sort</code> option is not given. </p> <p>If <code>FormatHandler = {Function,InitialState}</code>, <code>Function</code> will be called for each trace message. If <code>FormatHandler = get_et_handler()</code>, <code>et_viewer</code> in the <em>Event Tracer</em> application (<code>et</code>) is used for presenting the trace log graphically. <code>ttb</code> provides a few different filters which can be selected from the Filter menu in the <code>et_viewer</code>. If <code>FormatHandler</code> is not given, a default handler is used which presents each trace message as a line of text. </p> <p>The state returned from each call of <code>Function</code> is passed to the next call, even if next call is to format a message from another log file. </p> <p>If <code>Out</code> is given, <code>FormatHandler</code> gets the file descriptor to <code>Out</code> as the first parameter. </p> <p><code>Out</code> is ignored if <code>et</code> format handler is used. </p> <p>Wrap logs can be formatted one by one or all in one go. To format one of the wrap logs in a set, give the exact name of the file. To format the whole set of wrap logs, give the name with '*' instead of the wrap count. See examples in the <code>ttb</code> User's Guide.</p> </div> </div> </div>
