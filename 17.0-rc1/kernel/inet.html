<h1>inet</h1> <h2 class="modsummary">Access to TCP/IP Protocols</h2> <div class="description"> <p>Provides access to TCP/IP protocols.</p> <p>See also <em>ERTS User's Guide, Inet configuration</em> for more information on how to configure an Erlang runtime system for IP communication.</p> <p>Two Kernel configuration parameters affect the behaviour of all sockets opened on an Erlang node: <code>inet_default_connect_options</code> can contain a list of default options used for all sockets returned when doing <code>connect</code>, and <code>inet_default_listen_options</code> can contain a list of default options used when issuing a <code>listen</code> call. When <code>accept</code> is issued, the values of the listensocket options are inherited, why no such application variable is needed for <code>accept</code>.</p> <p>Using the Kernel configuration parameters mentioned above, one can set default options for all TCP sockets on a node. This should be used with care, but options like <code>{delay_send,true}</code> might be specified in this way. An example of starting an Erlang node with all sockets using delayed send could look like this:</p> <pre class="sh_erlang"> $ <code>erl -sname test -kernel \</code> <code>inet_default_connect_options '[{delay_send,true}]' \</code> <code>inet_default_listen_options '[{delay_send,true}]'</code></pre> <p>Note that the default option <code>{active, true}</code> currently cannot be changed, for internal reasons.</p> <p>Addresses as inputs to functions can be either a string or a tuple. For instance, the IP address 150.236.20.73 can be passed to <code>gethostbyaddr/1</code> either as the string "150.236.20.73" or as the tuple <code>{150, 236, 20, 73}</code>.</p> <p>IPv4 address examples:</p> <pre class="sh_erlang"> Address ip_address() ------- ------------ 127.0.0.1 {127,0,0,1} 192.168.42.2 {192,168,42,2}</pre> <p>IPv6 address examples:</p> <pre class="sh_erlang"> Address ip_address() ------- ------------ ::1 {0,0,0,0,0,0,0,1} ::192.168.42.2 {0,0,0,0,0,0,(192 bsl 8) bor 168,(42 bsl 8) bor 2} FFFF::192.168.42.2 {16#FFFF,0,0,0,0,0,(192 bsl 8) bor 168,(42 bsl 8) bor 2} 3ffe:b80:1f8d:2:204:acff:fe17:bf38 {16#3ffe,16#b80,16#1f8d,16#2,16#204,16#acff,16#fe17,16#bf38} fe80::204:acff:fe17:bf38 {16#fe80,0,0,0,0,16#204,16#acff,16#fe17,16#bf38}</pre> <p>A function that may be useful is <a href="#parse_address/1" class="seealso">parse_address/1</a>:</p> <pre class="sh_erlang"> 1&gt; <code>inet:parse_address("192.168.42.2").</code> {ok,{192,168,42,2}} 2&gt; <code>inet:parse_address("FFFF::192.168.42.2").</code> {ok,{65535,0,0,0,0,0,49320,10754}}</pre> </div> <datatypes> <datatype> <name name="hostent" /> <div class="description"> <p>The record is defined in the Kernel include file "inet.hrl". See <a href="#setopts/2" class="seealso">setopts/2</a> for a list of available options.</p> <p>The number of elements in the returned <code><anno>OptionValues</anno></code> list does not necessarily correspond to the number of options asked for. If the operating system fails to support an option, it is simply left out in the returned list. An error tuple is only returned when getting options for the socket is impossible (i.e. the socket is closed or the buffer size in a raw request is too large). This behavior is kept for backward compatibility reasons.</p> <p>A raw option request <code>RawOptReq = {raw, Protocol, OptionNum, ValueSpec}</code> can be used to get information about socket options not (explicitly) supported by the emulator. The use of raw socket options makes the code non portable, but allows the Erlang programmer to take advantage of unusual features present on the current platform.</p> <p>The <code>RawOptReq</code> consists of the tag <code>raw</code> followed by the protocol level, the option number and either a binary or the size, in bytes, of the buffer in which the option value is to be stored. A binary should be used when the underlying <code>getsockopt</code> requires <em>input</em> in the argument field, in which case the size of the binary should correspond to the required buffer size of the return value. The supplied values in a <code>RawOptReq</code> correspond to the second, third and fourth/fifth parameters to the <code>getsockopt</code> call in the C socket API. The value stored in the buffer is returned as a binary <code>ValueBin</code> where all values are coded in the native endianess.</p> <p>Asking for and inspecting raw socket options require low level information about the current operating system and TCP stack.</p> <p>As an example, consider a Linux machine where the <code>TCP_INFO</code> option could be used to collect TCP statistics for a socket. Lets say we're interested in the <code>tcpi_sacked</code> field of the <code>struct tcp_info</code> filled in when asking for <code>TCP_INFO</code>. To be able to access this information, we need to know both the numeric value of the protocol level <code>IPPROTO_TCP</code>, the numeric value of the option <code>TCP_INFO</code>, the size of the <code>struct tcp_info</code> and the size and offset of the specific field. By inspecting the headers or writing a small C program, we found <code>IPPROTO_TCP</code> to be 6, <code>TCP_INFO</code> to be 11, the structure size to be 92 (bytes), the offset of <code>tcpi_sacked</code> to be 28 bytes and the actual value to be a 32 bit integer. We could use the following code to retrieve the value:</p> <pre class="sh_erlang"> get_tcpi_sacked(Sock) -&gt; {ok,[{raw,_,_,Info}]} = inet:getopts(Sock,[{raw,6,11,92}]), &lt;&lt;_:28/binary,TcpiSacked:32/native,_/binary&gt;&gt; = Info, TcpiSacked.</pre> <p>Preferably, you would check the machine type, the OS and the kernel version prior to executing anything similar to the code above.</p> </div> </div> <div class="function"> <h3 id="getstat/1">getstat/1</h3> <h3 id="getstat/2">getstat/2</h3> <type name="stat_option" /> <div class="description"> <p>Gets one or The following options are available:</p> <taglist> <dt><code>{active, true | false | once | N}</code></dt> <item> <p>If the value is <code>true</code>, which is the default, everything received from the socket will be sent as messages to the receiving process. If the value is <code>false</code> (passive mode), the process must explicitly receive incoming data by calling <a href="gen_tcp.html#recv/2" class="seealso"><code>gen_tcp:recv/2,3</code></a>, <a href="gen_udp.html#recv/2" class="seealso"><code>gen_udp:recv/2,3</code></a> or <a href="gen_sctp.html#recv/1" class="seealso"><code>gen_sctp:recv/1,2</code></a> (depending on the type of socket).</p> <p>If the value is <code>once</code> (<code>{active, once}</code>), <em>one</em> data message from the socket will be sent to the process. To receive one more message, <code>setopts/2</code> must be called again with the <code>{active, once}</code> option.</p> <p>If the value is an integer <code>N</code> in the range -32768 to 32767 (inclusive), the value is added to the socket's count of data messages sent to the controlling process. A socket's default message count is 0. If a negative value is specified and its magnitude is equal to or greater than the socket's current message count, the socket's message count is set to 0. Once the socket's message count reaches 0, either due to sending received data messages to the process or by being explicitly set, the process is then notified by a special message, specific to the type of socket, that the socket has entered passive mode. Once the socket enters passive mode, to receive more messages <code>setopts/2</code> must be called again to set the socket back into an active mode.</p> <p>When using <code>{active, once}</code> or <code>{active, N}</code>, the socket changes behaviour automatically when data is received. This can sometimes be confusing in combination with connection-oriented sockets (i.e. <code>gen_tcp</code>) as a socket with <code>{active, false}</code> behaviour reports closing differently than a socket with <code>{active, true}</code> behaviour. To make programming easier, a socket where the peer closed and this was detected while in <code>{active, false}</code> mode, will still generate the message <code>{tcp_closed,Socket}</code> when set to <code>{active, once}</code>, <code>{active, true}</code> or <code>{active, N}</code> mode. It is therefore safe to assume that the message <code>{tcp_closed,Socket}</code>, possibly followed by socket port termination (depending on the <code>exit_on_close</code> option) will eventually appear when a socket changes back and forth between <code>{active, true}</code> and <code>{active, false}</code> mode. However, <em>when</em> peer closing is detected is all up to the underlying TCP/IP stack and protocol.</p> <p>Note that <code>{active, true}</code> mode provides no flow control; a fast sender could easily overflow the receiver with incoming messages. The same is true of <code>{active, N}</code> mode while the message count is greater than zero. Use active mode only if your high-level protocol provides its own flow control (for instance, acknowledging received messages) or the amount of data exchanged is small. <code>{active, false}</code> mode, use of the <code>{active, once}</code> mode or <code>{active, N}</code> mode with values of <code>N</code> appropriate for the application provides flow control; the other side will not be able send faster than the receiver can read.</p> </item> <dt><code>{broadcast, Boolean}</code>(UDP sockets)</dt> <item> <p>Enable/disable permission to send broadcasts.</p> <span id="option-buffer"> </span> </item> <dt><code>{buffer, Size}</code></dt> <item> <p>Determines the size of the user-level software buffer used by the driver. Not to be confused with <code>sndbuf</code> and <code>recbuf</code> options which correspond to the kernel socket buffers. It is recommended to have <code>val(buffer) &gt;= max(val(sndbuf),val(recbuf))</code>. In fact, the <code>val(buffer)</code> is automatically set to the above maximum when <code>sndbuf</code> or <code>recbuf</code> values are set.</p> </item> <dt><code>{delay_send, Boolean}</code></dt> <item> <p>Normally, when an Erlang process sends to a socket, the driver will try to immediately send the data. If that fails, the driver will use any means available to queue up the message to be sent whenever the operating system says it can handle it. Setting <code>{delay_send, true}</code> will make <em>all</em> messages queue up. This makes the messages actually sent onto the network be larger but fewer. The option actually affects the scheduling of send requests versus Erlang processes instead of changing any real property of the socket. Needless to say it is an implementation specific option. Default is <code>false</code>.</p> </item> <dt><code>{deliver, port | term}</code></dt> <item> <p> When <code>{active, true}</code> delivers data on the forms <code>port</code> : <code>{S, {data, [H1,..Hsz | Data]}}</code> or <code>term</code> : <code>{tcp, S, [H1..Hsz | Data]}</code>. </p> </item> <dt><code>{dontroute, Boolean}</code></dt> <item> <p>Enable/disable routing bypass for outgoing messages.</p> </item> <dt><code>{exit_on_close, Boolean}</code></dt> <item> <p>By default this option is set to <code>true</code>.</p> <p>The only reason to set it to <code>false</code> is if you want to continue sending data to the socket after a close has been detected, for instance if the peer has used <a href="gen_tcp.html#shutdown/2" class="seealso">gen_tcp:shutdown/2</a> to shutdown the write side.</p> </item> <dt><code>{header, Size}</code></dt> <item> <p>This option is only meaningful if the <code>binary</code> option was specified when the socket was created. If the <code>header</code> option is specified, the first <code>Size</code> number bytes of data received from the socket will be elements of a list, and the rest of the data will be a binary given as the tail of the same list. If for example <code>Size == 2</code>, the data received will match <code>[Byte1,Byte2|Binary]</code>.</p> </item> <dt><code>{high_msgq_watermark, Size}</code></dt> <item> <p>The socket message queue will be set into a busy state when the amount of data queued on the message queue reaches this limit. Note that this limit only concerns data that have not yet reached the ERTS internal socket implementation. Default value used is 8 kB.</p> <p>Senders of data to the socket will be suspended if either the socket message queue is busy, or the socket itself is busy.</p> <p>For more information see the <code>low_msgq_watermark</code>, <code>high_watermark</code>, and <code>low_watermark</code> options.</p> <p>Note that distribution sockets will disable the use of <code>high_msgq_watermark</code> and <code>low_msgq_watermark</code>, and will instead use the <a href="../erts/erlang.html#system_info_dist_buf_busy_limit" class="seealso">distribution buffer busy limit</a> which is a similar feature.</p> </item> <dt><code>{high_watermark, Size}</code> (TCP/IP sockets)</dt> <item> <p>The socket will be set into a busy state when the amount of data queued internally by the ERTS socket implementation reaches this limit. Default value used is 8 kB.</p> <p>Senders of data to the socket will be suspended if either the socket message queue is busy, or the socket itself is busy.</p> <p>For more information see the <code>low_watermark</code>, <code>high_msgq_watermark</code>, and <code>low_msqg_watermark</code> options.</p> </item> <dt><code>{ipv6_v6only, Boolean}</code></dt> <item> <p> Restricts the socket to only use IPv6, prohibiting any IPv4 connections. This is only applicable for IPv6 sockets (option <code>inet6</code>). </p> <p> On most platforms this option has to be set on the socket before associating it to an address. Therefore it is only reasonable to give it when creating the socket and not to use it when calling the function (<a href="#setopts/2" class="seealso">setopts/2</a>) containing this description. </p> <p> The behaviour of a socket with this socket option set to <code>true</code> is becoming the only portable one. The original idea when IPv6 was new of using IPv6 for all traffic is now not recommended by FreeBSD (you can use <code>{ipv6_v6only,false}</code> to override the recommended system default value), forbidden by OpenBSD (the supported GENERIC kernel) and impossible on Windows (that has separate IPv4 and IPv6 protocol stacks). Most Linux distros still have a system default value of <code>false</code>. This policy shift among operating systems towards separating IPv6 from IPv4 traffic has evolved since it gradually proved hard and complicated to get a dual stack implementation correct and secure. </p> <p> On some platforms the only allowed value for this option is <code>true</code>, e.g. OpenBSD and Windows. Trying to set this option to <code>false</code> when creating the socket will in this case fail. </p> <p> Setting this option on platforms where it does not exist is ignored and getting this option with <a href="#getopts/2" class="seealso">getopts/2</a> returns no value i.e the returned list will not contain an <code>{ipv6_v6only,_}</code> tuple. On Windows the option acually does not exist, but it is emulated as being a read-only option with the value <code>true</code>. </p> <p> So it boils down to that setting this option to <code>true</code> when creating a socket will never fail except possibly (at the time of this writing) on a platform where you have customized the kernel to only allow <code>false</code>, which might be doable (but weird) on e.g. OpenBSD. </p> <p> If you read back the option value using <a href="#getopts/2" class="seealso">getopts/2</a> and get no value the option does not exist in the host OS and all bets are off regarding the behaviour of both an IPv6 and an IPv4 socket listening on the same port as well as for an IPv6 socket getting IPv4 traffic. </p> </item> <dt><code>{keepalive, Boolean}</code>(TCP/IP sockets)</dt> <item> <p>Enables/disables periodic transmission on a connected socket, when no other data is being exchanged. If the other end does not respond, the connection is considered broken and an error message will be sent to the controlling process. Default disabled.</p> <span id="option-linger"> </span> </item> <dt><code>{linger, {true|false, Seconds}}</code></dt> <item> <p>Determines the timeout in seconds for flushing unsent data in the <code>close/1</code> socket call. If the 1st component of the value tuple is <code>false</code>, the 2nd one is ignored, which means that <code>close/1</code> returns immediately not waiting for data to be flushed. Otherwise, the 2nd component is the flushing time-out in seconds.</p> </item> <dt><code>{low_msgq_watermark, Size}</code></dt> <item> <p>If the socket message queue is in a busy state, the socket message queue will be set in a not busy state when the amount of data queued in the message queue falls below this limit. Note that this limit only concerns data that have not yet reached the ERTS internal socket implementation. Default value used is 4 kB.</p> <p>Senders that have been suspended due to either a busy message queue or a busy socket, will be resumed when neither the socket message queue, nor the socket are busy.</p> <p>For more information see the <code>high_msgq_watermark</code>, <code>high_watermark</code>, and <code>low_watermark</code> options.</p> <p>Note that distribution sockets will disable the use of <code>high_msgq_watermark</code> and <code>low_msgq_watermark</code>, and will instead use the <a href="../erts/erlang.html#system_info_dist_buf_busy_limit" class="seealso">distribution buffer busy limit</a> which is a similar feature.</p> </item> <dt><code>{low_watermark, Size}</code> (TCP/IP sockets)</dt> <item> <p>If the socket is in a busy state, the socket will be set in a not busy state when the amount of data queued internally by the ERTS socket implementation falls below this limit. Default value used is 4 kB.</p> <p>Senders that have been suspended due to either a busy message queue or a busy socket, will be resumed when neither the socket message queue, nor the socket are busy.</p> <p>For more information see the <code>high_watermark</code>, <code>high_msgq_watermark</code>, and <code>low_msgq_watermark</code> options.</p> </item> <dt><code>{mode, Mode :: binary | list}</code></dt> <item> <p>Received <code>Packet</code> is delivered as defined by Mode.</p> </item> <dt><code>{netns, Namespace :: file:filename_all()}</code></dt> <item> <p>Set a network namespace for the socket. The <code>Namespace</code> parameter is a filename defining the namespace for example <code>"/var/run/netns/example"</code> typically created by the command <code>ip netns add example</code>. This option must be used in a function call that creates a socket i.e <a href="gen_tcp.html#connect/3" class="seealso"> gen_tcp:connect/3,4</a>, <a href="gen_tcp.html#listen/2" class="seealso"> gen_tcp:listen/2</a>, <a href="gen_udp.html#open/1" class="seealso"> gen_udp:open/1,2</a> or <a href="gen_sctp.html#open/0" class="seealso"> gen_sctp:open/0-2</a>. </p> <p>This option uses the Linux specific syscall <code>setns()</code> such as in Linux kernel 3.0 or later and therefore only exists when the runtime system has been compiled for such an operating system. </p> <p> The virtual machine also needs elevated privileges either running as superuser or (for Linux) having the capability <code>CAP_SYS_ADMIN</code> according to the documentation for setns(2). However, during testing also <code>CAP_SYS_PTRACE</code> and <code>CAP_DAC_READ_SEARCH</code> has proven to be necessary. Example:<code> setcap cap_sys_admin,cap_sys_ptrace,cap_dac_read_search+epi beam.smp </code> Note also that the filesystem containing the virtual machine executable (<code>beam.smp</code> in the example above) has to be local, mounted without the <code>nosetuid</code> flag, support extended attributes and that the kernel has to support file capabilities. All this runs out of the box on at least Ubuntu 12.04 LTS, except that SCTP sockets appears to not support network namespaces. </p> <p>The <code>Namespace</code> is a file name and is encoded and decoded as discussed in <a href="file.html" class="seealso">file</a> except that the emulator flag <code>+fnu</code> is ignored and <a href="#getopts/2" class="seealso">getopts/2</a> for this option will return a binary for the filename if the stored filename can not be decoded, which should only happen if you set the option using a binary that can not be decoded with the emulator's filename encoding: <a href="file.html#native_name_encoding/0" class="seealso"> file:native_name_encoding/0</a>. </p> </item> <dt><code>list</code></dt> <item> <p>Received <code>Packet</code> is delivered as a list.</p> </item> <dt><code>binary</code></dt> <item> <p>Received <code>Packet</code> is delivered as a binary.</p> </item> <dt><code>{nodelay, Boolean}</code>(TCP/IP sockets)</dt> <item> <p>If <code>Boolean == true</code>, the <code>TCP_NODELAY</code> option is turned on for the socket, which means that even small amounts of data will be sent immediately.</p> </item> <dt><code>{packet, PacketType}</code>(TCP/IP sockets)</dt> <item> <p>Defines the type of packets to use for a socket. The following values are valid:</p> <taglist> <dt><code>raw | 0</code></dt> <item> <p>No packaging is done.</p> </item> <dt><code>1 | 2 | 4</code></dt> <item> <p>Packets consist of a header specifying the number of bytes in the packet, followed by that number of bytes. The length of header can be one, two, or four bytes; containing an unsigned integer in big-endian byte order. Each send operation will generate the header, and the header will be stripped off on each receive operation.</p> <p>In current implementation the 4-byte header is limited to 2Gb.</p> </item> <dt><code>asn1 | cdr | sunrm | fcgi | tpkt | line</code></dt> <item> <p>These packet types only have effect on receiving. When sending a packet, it is the responsibility of the application to supply a correct header. On receiving, however, there will be one message sent to the controlling process for each complete packet received, and, similarly, each call to <code>gen_tcp:recv/2,3</code> returns one complete packet. The header is <em>not</em> stripped off.</p> <p>The meanings of the packet types are as follows: <br /> <code>asn1</code> - ASN.1 BER, <br /> <code>sunrm</code> - Sun's RPC encoding, <br /> <code>cdr</code> - CORBA (GIOP 1.1), <br /> <code>fcgi</code> - Fast CGI, <br /> <code>tpkt</code> - TPKT format [RFC1006], <br /> <code>line</code> - Line mode, a packet is a line terminated with newline, lines longer than the receive buffer are truncated.</p> </item> <dt><code>http | http_bin</code></dt> <item> <p>The Hypertext Transfer Protocol. The packets are returned with the format according to <code>HttpPacket</code> described in <a href="../erts/erlang.html#decode_packet/3" class="seealso"> erlang:decode_packet/3</a>. A socket in passive mode will return <code>{ok, HttpPacket}</code> from <code>gen_tcp:recv</code> while an active socket will send messages like <code>{http, Socket, HttpPacket}</code>.</p> </item> <dt><code>httph | httph_bin</code></dt> <item> <p>These two types are often not needed as the socket will automatically switch from <code>http</code>/<code>http_bin</code> to <code>httph</code>/<code>httph_bin</code> internally after the first line has been read. There might be occasions however when they are useful, such as parsing trailers from chunked encoding.</p> </item> </taglist> </item> <dt><code>{packet_size, Integer}</code>(TCP/IP sockets)</dt> <item> <p>Sets the max allowed length of the packet body. If the packet header indicates that the length of the packet is longer than the max allowed length, the packet is considered invalid. The same happens if the packet header is too big for the socket receive buffer.</p> <p>For line oriented protocols (<code>line</code>,<code>http*</code>), option <code>packet_size</code> also guarantees that lines up to the indicated length are accepted and not considered invalid due to internal buffer limitations.</p> </item> <dt><code>{priority, Priority}</code></dt> <item> <p>Set the protocol-defined priority for all packets to be sent on this socket.</p> </item> <dt><code>{raw, Protocol, OptionNum, ValueBin}</code></dt> <item> <p>See below.</p> </item> <dt><code>{read_packets, Integer}</code>(UDP sockets)</dt> <item> <p>Sets the max number of UDP packets to read without intervention from the socket when data is available. When this many packets have been read and delivered to the destination process, new packets are not read until a new notification of available data has arrived. The default is 5, and if this parameter is set too high the system can become unresponsive due to UDP packet flooding.</p> </item> <dt><code>{recbuf, Size}</code></dt> <item> <p>Gives the size of the receive buffer to use for the socket.</p> </item> <dt><code>{reuseaddr, Boolean}</code></dt> <item> <p>Allows or disallows local reuse of port numbers. By default, reuse is disallowed.</p> </item> <dt><code>{send_timeout, Integer}</code></dt> <item> <p>Only allowed for connection oriented sockets.</p> <p>Specifies a longest time to wait for a send operation to be accepted by the underlying TCP stack. When the limit is exceeded, the send operation will return <code>{error,timeout}</code>. How much of a packet that actually got sent is unknown, why the socket should be closed whenever a timeout has occurred (see <code>send_timeout_close</code>). Default is <code>infinity</code>.</p> </item> <dt><code>{send_timeout_close, Boolean}</code></dt> <item> <p>Only allowed for connection oriented sockets.</p> <p>Used together with <code>send_timeout</code> to specify whether the socket will be automatically closed when the send operation returns <code>{error,timeout}</code>. The recommended setting is <code>true</code> which will automatically close the socket. Default is <code>false</code> due to backward compatibility.</p> <span id="option-sndbuf"> </span> </item> <dt><code>{sndbuf, Size}</code></dt> <item> <p>Gives the size of the send buffer to use for the socket.</p> </item> <dt><code>{priority, Integer}</code></dt> <item> <p>Sets the SO_PRIORITY socket level option on platforms where this is implemented. The behaviour and allowed range varies on different systems. The option is ignored on platforms where the option is not implemented. Use with caution.</p> </item> <dt><code>{tos, Integer}</code></dt> <item> <p>Sets IP_TOS IP level options on platforms where this is implemented. The behaviour and allowed range varies on different systems. The option is ignored on platforms where the option is not implemented. Use with caution.</p> </item> </taglist> <p>In addition to the options mentioned above, <em>raw</em> option specifications can be used. The raw options are specified as a tuple of arity four, beginning with the tag <code>raw</code>, followed by the protocol level, the option number and the actual option value specified as a binary. This corresponds to the second, third and fourth argument to the <code>setsockopt</code> call in the C socket API. The option value needs to be coded in the native endianess of the platform and, if a structure is required, needs to follow the struct alignment conventions on the specific platform.</p> <p>Using raw socket options require detailed knowledge about the current operating system and TCP stack.</p> <p>As an example of the usage of raw options, consider a Linux system where you want to set the <code>TCP_LINGER2</code> option on the <code>IPPROTO_TCP</code> protocol level in the stack. You know that on this particular system it defaults to 60 (seconds), but you would like to lower it to 30 for a particular socket. The <code>TCP_LINGER2</code> option is not explicitly supported by inet, but you know that the protocol level translates to the number 6, the option number to the number 8 and the value is to be given as a 32 bit integer. You can use this line of code to set the option for the socket named <code>Sock</code>:</p> <pre class="sh_erlang"> inet:setopts(Sock,[{raw,6,8,&lt;&lt;30:32/native&gt;&gt;}]),</pre> <p>As many options are silently discarded by the stack if they are given out of range, it could be a good idea to check that a raw option really got accepted. This code places the value in the variable TcpLinger2:</p> <pre class="sh_erlang"> {ok,[{raw,6,8,&lt;&lt;TcpLinger2:32/native&gt;&gt;}]}=inet:getopts(Sock,[{raw,6,8,4}]),</pre> <p>Code such as the examples above is inherently non portable, even different versions of the same OS on the same platform may respond differently to this kind of option manipulation. Use with care.</p> <p>Note that the default options for TCP/IP sockets can be changed with the Kernel configuration parameters mentioned in the beginning of this document.</p> </div> </div> </div> <div class="section"> <span id="error_codes"> </span> <h4>POSIX Error Codes</h4> <list type="bulleted"> <item><code>e2big</code> -
