<h1>global</h1> <h2 class="modsummary">A Global Name Registration Facility</h2> <div class="description"> <p>This documentation describes the Global module which consists of the following functionalities:</p> <list type="bulleted"> <item>registration of global names;</item> <item>global locks;</item> <item>maintenance of the fully connected network.</item> </list> <p>These services are controlled via the process <code>global_name_server</code> which exists on every node. The global name server is started automatically when a node is started. With the term <em>global</em> is meant over a system consisting of several Erlang nodes.</p> <p>The ability to globally register names is a central concept in the programming of distributed Erlang systems. In this module, the equivalent of the <code>register/2</code> and <code>whereis/1</code> BIFs (for local name registration) are implemented, but for a network of Erlang nodes. A registered name is an alias for a process identifier (pid). The global name server monitors globally registered pids. If a process terminates, the name will also be globally unregistered.</p> <p>The registered names are stored in replica global name tables on every node. There is no central storage point. Thus, the translation of a name to a pid is fast, as it is always done locally. When any action in taken which results in a change to the global name table, all tables on other nodes are automatically updated.</p> <p>Global locks have lock identities and are set on a specific resource. For instance, the specified resource could be a pid. When a global lock is set, access to the locked resource is denied for all other resources other than the lock requester.</p> <p>Both the registration and lock functionalities are atomic. All nodes involved in these actions will have the same view of the information.</p> <p>The global name server also performs the critical task of continuously monitoring changes in node configuration: if a node which runs a globally registered process goes down, the name will be globally unregistered. To this end the global name server subscribes to <code>nodeup</code> and <code>nodedown</code> messages sent from the <code>net_kernel</code> module. Relevant Kernel application variables in this context are <code>net_setuptime</code>, <code>net_ticktime</code>, and <code>dist_auto_connect</code>. See also <a href="kernel_app.html#net_setuptime" class="seealso">kernel(6)</a>.</p> <p>The name server will also maintain a fully connected network. For example, if node <code>N1</code> connects to node <code>N2</code> (which is already connected to <code>N3</code>), the global name servers on the nodes <code>N1</code> and <code>N3</code> will make sure that also <code>N1</code> and <code>N3</code> are connected. If this is not desired, the command line flag <code>-connect_all false</code> can be used (see also <a href="../erts/erl.html#connect_all" class="seealso">erl(1)</a>). In this case the name registration facility cannot be used, but the lock mechanism will still work.</p> <p>If the global name server fails to connect nodes (<code>N1</code> and <code>N3</code> in the example above) a warning event is sent to the error logger. The presence of such an event does not exclude the possibility that the nodes will later connect--one can for example try the command <code>rpc:call(N1, net_adm, ping, [N2])</code> in the Erlang shell--but it indicates some kind of problem with the network.</p> <div class="note"><h2>Note!</h2> <p>If the fully connected network is not set up properly, the first thing to try is to increase the value of <code>net_setuptime</code>.</p> </div> </div> <datatypes> <datatype> <name name="id" /> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="del_lock/1">del_lock/1</h3> <h3 id="del_lock/2">del_lock/2</h3> <div class="description"> <p>Deletes the lock <code><anno>Id</anno></code> synchronously.</p> </div> </div> <div class="function"> <h3 id="notify_all_name/3">notify_all_name/3</h3> <div class="description"> <p>This function can be used as a name resolving function for <code>register_name/3</code> and <code>re_register_name/3</code>. The network is also informed of any global names in newly connected nodes. If any name clashes are discovered, the <code><anno>Resolve</anno></code> function is called. Its purpose is to decide which pid is correct. If the function crashes, or returns anything other than one of the pids, the name is unregistered. This function is called once for each name clash.</p> <div class="warning"><h2>Warning!</h2> <p>If you plan to change code without restarting your system, you must use an external fun (<code>fun Module:Function/Arity</code>) as the <code><anno>Resolve</anno></code> function; if you use a local fun you can never replace the code for the module that the fun belongs to. </p> </div> <p>There are three pre-defined resolve functions: <code>random_exit_name/3</code>, <code>random_notify_name/3</code>, and <code>notify_all_name/3</code>. If no <code><anno>Resolve</anno></code> function is defined, <code>random_exit_name</code> is used. This means that one of the two registered processes will be selected as correct while the other is killed.</p> <p>This function is completely synchronous. This means that when this function returns, the name is either registered on all nodes or none.</p> <p>The function returns <code>yes</code> if successful, <code>no</code> if it fails. For example, <code>no</code> is returned if an attempt is made to register an already registered process or to register a process with a name that is already in use.</p> <div class="note"><h2>Note!</h2> <p>Releases up to and including OTP R10 did not check if the process was already registered. As a consequence the global name table could become inconsistent. The old (buggy) behavior can be chosen by giving the Kernel application variable <code>global_multi_name_action</code> the value <code>allow</code>.</p> </div> <p>If a process with a registered name dies, or the node goes down, the name is unregistered on all nodes.</p> </div> </div> <div class="function"> <h3 id="registered_names/0">registered_names/0</h3> <div class="description"> <p>Returns a lists of all globally registered names.</p> </div> </div> <div class="function"> <h3 id="re_register_name/2">re_register_name/2</h3> <h3 id="re_register_name/3">re_register_name/3</h3> <type name="method" /> <type_desc name="method">{<code>Module</code>, <code>Function</code>} is also allowed </type_desc> <div class="description"> <p>Atomically changes the registered name <code><anno>Name</anno></code> on all nodes to refer to <code><anno>Pid</anno></code>.</p> <p>The <code><anno>Resolve</anno></code> function has the same behavior as in <code>register_name/2,3</code>.</p> </div> </div> <div class="function"> <h3 id="send/2">send/2</h3> <div class="description"> <p>Sends the message <code><anno>Msg</anno></code> to the pid globally registered as <code><anno>Name</anno></code>.</p> <p>Failure: If <code><anno>Name</anno></code> is not a globally registered name, the calling function will exit with reason <code>{badarg, {<anno>Name</anno>, <anno>Msg</anno>}}</code>.</p> </div> </div> <div class="function"> <h3 id="set_lock/1">set_lock/1</h3> <h3 id="set_lock/2">set_lock/2</h3> <h3 id="set_lock/3">set_lock/3</h3> <type name="id" /> <type name="retries" /> <div class="description"> <p>Sets a lock on the specified nodes (or on all nodes if none are specified) on <code><anno>ResourceId</anno></code> for <code><anno>LockRequesterId</anno></code>. If a lock already exists on <code><anno>ResourceId</anno></code> for another requester than <code><anno>LockRequesterId</anno></code>, and <code><anno>Retries</anno></code> is not equal to 0, the process sleeps for a while and will try to execute the action later. When <code><anno>Retries</anno></code> attempts have been made, <code>false</code> is returned, otherwise <code>true</code>. If <code><anno>Retries</anno></code> is <code>infinity</code>, <code>true</code> is eventually returned (unless the lock is never released).</p> <p>If no value for <code><anno>Retries</anno></code> is given, <code>infinity</code> is used.</p> <p>This function is completely synchronous.</p> <p>If a process which holds a lock dies, or the node goes down, the locks held by the process are deleted.</p> <p>The global name server keeps track of all processes sharing the same lock, that is, if two processes set the same lock, both processes must delete the lock.</p> <p>This function does not address the problem of a deadlock. A deadlock can never occur as long as processes only lock one resource at a time. But if some processes try to lock two or more resources, a deadlock may occur. It is up to the application to detect and rectify a deadlock.</p> <div class="note"><h2>Note!</h2> <p>Some values of <code><anno>ResourceId</anno></code> should be avoided or Erlang/OTP will not work properly.
