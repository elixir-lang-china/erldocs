<h1>error_handler</h1> <h2 class="modsummary">Default System Error Handler</h2> <div class="description"> <p>The error handler module defines what happens when certain types of errors occur.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="undefined_function/3">undefined_function/3</h3> <type_desc variable="Args"> A (possibly empty) list of arguments <code>Arg1,..,ArgN</code> </type_desc> <div class="description"> <p>This function is called by the run-time system if a call is made to <code><anno>Module</anno>:<anno>Function</anno>(Arg1,.., ArgN)</code> and <code><anno>Module</anno>:<anno>Function</anno>/N</code> is undefined. Note that <code>undefined_function/3</code> is evaluated inside the process making the original call.</p> <p>This function will first attempt to autoload <code><anno>Module</anno></code>. If that is not possible, an <code>undef</code> exception will be raised.</p> <p>If it was possible to load <code><anno>Module</anno></code> and the function <code><anno>Function</anno>/N</code> is exported, it will be called.</p> <p>Otherwise, if the function <code>'$handle_undefined_function'/2</code> is exported, it will be called as <code>'$handle_undefined_function'(</code><anno>Function</anno>, <anno>Args</anno>). </p> <div class="warning"><h2>Warning!</h2> <p>Defining <code>'$handle_undefined_function'/2</code> in ordinary application code is highly discouraged. It is very easy to make subtle errors that can take a long time to debug. Furthermore, none of the tools for static code analysis (such as Dialyzer and Xref) supports the use of <code>'$handle_undefined_function'/2</code> and no such support will be added. Only use this function after having carefully considered other, less dangerous, solutions. One example of potential legitimate use is creating stubs for other sub-systems during testing and debugging. </p> </div> <p>Otherwise an <code>undef</code> exception will be raised.</p> </div> </div> <div class="function"> <h3 id="raise_undef_exception/3">raise_undef_exception/3</h3> <type_desc variable="Args"> A (possibly empty) list of arguments <code>Arg1,..,ArgN</code> </type_desc> <div class="description"> <p>Raise an <code>undef</code> exception with a stacktrace indicating that <code><anno>Module</anno>:<anno>Function</anno>/N</code> is undefined. </p> </div> </div> <div class="function"> <h3 id="undefined_lambda/3">undefined_lambda/3</h3> <type_desc variable="Args"> A (possibly empty) list of arguments <code>Arg1,..,ArgN</code> </type_desc> <div class="description"> <p>This function is evaluated if a call is made to <code><anno>Fun</anno>(Arg1,.., ArgN)</code> when the module defining the fun is not loaded. The function is evaluated inside the process making the original call.</p> <p>If <code><anno>Module</anno></code> is interpreted, the interpreter is invoked and the return value of the interpreted <code><anno>Fun</anno>(Arg1,.., ArgN)</code> call is returned.</p> <p>Otherwise, it returns, if possible, the value of <code>apply(<anno>Fun</anno>, <anno>Args</anno>)</code> after an attempt has been made to autoload <code><anno>Module</anno></code>. If this is not possible, the call fails with exit reason <code>undef</code>.</p> </div> </div> </div> <div class="section"> <h4>Notes</h4> <p>The code in <code>error_handler</code> is complex and should not be changed without fully understanding the interaction between the error handler, the <code>init</code> process of the code server, and the I/O mechanism of the code.</p> <p>Changes in the code which may seem small can cause a deadlock as unforeseen consequences may occur. The use of <code>input</code> is dangerous in this type of code.</p> </div>
