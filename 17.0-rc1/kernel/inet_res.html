<h1>inet_res</h1> <h2 class="modsummary">A Rudimentary DNS Client</h2> <div class="description"> <p>Performs DNS name resolving towards recursive name servers</p> <p>See also <a href="../erts/inet_cfg.html" class="seealso"> ERTS User's Guide: Inet configuration </a> for more information on how to configure an Erlang runtime system for IP communication and how to enable this DNS client by defining <code>'dns'</code> as a lookup method. It then acts as a backend for the resolving functions in <a href="../kernel/inet.html" class="seealso">inet</a>.</p> <p>This DNS client can resolve DNS records even if it is not used for normal name resolving in the node.</p> <p>This is not a full-fledged resolver. It is just a DNS client that relies on asking trusted recursive nameservers.</p> </div> <div class="section"> <h4>Name Resolving</h4> <p>UDP queries are used unless resolver option <code>usevc</code> is <code>true</code>, which forces TCP queries. If the query is to large for UDP, TCP is used instead. For regular DNS queries 512 bytes is the size limit. When EDNS is enabled (resolver option <code>edns</code> is set to the EDNS version i.e <code>0</code> instead of <code>false</code>), resolver option <code>udp_payload_size</code> sets the limit. If a nameserver replies with the TC bit set (truncation), indicating the answer is incomplete, the query is retried to that nameserver using TCP. The resolver option <code>udp_payload_size</code> also sets the advertised size for the max allowed reply size, if EDNS is enabled, otherwise the nameserver uses the limit 512 byte. If the reply is larger it gets truncated, forcing a TCP re-query.</p> <p>For UDP queries, the resolver options <code>timeout</code> and <code>retry</code> control retransmission. Each nameserver in the <code>nameservers</code> list is tried with a timeout of <code>timeout</code> / <code>retry</code>. Then all nameservers are tried again doubling the timeout, for a total of <code>retry</code> times.</p> <p>For queries that not use the <code>search</code> list, if the query to all <code>nameservers</code> results in <code>{error,nxdomain}</code>or an empty answer, the same query is tried for the <code>alt_nameservers</code>.</p> </div> <datatypes> <p>Resolver types:</p> <datatype> <name name="res_option" /> </datatype> <datatype> <name name="nameserver" /> </datatype> <datatype> <name name="res_error" /> </datatype> <p><span id="dns_types"> </span>DNS types:</p> <datatype> <name name="dns_name" /> <div class="description"><p>A string with no adjacent dots.</p></div> </datatype> <datatype> <name name="rr_type" /> </datatype> <datatype> <name name="dns_class" /> </datatype> <datatype> <name name="dns_msg" /> <div class="description"> <p>This is the start of a hiearchy of opaque data structures that can be examined with access functions in inet_dns that return lists of {Field,Value} tuples. The arity 2 functions just return the value for a given field. <pre class="sh_erlang"> dns_msg() = DnsMsg inet_dns:msg(DnsMsg) -&gt; [ {header, dns_header()} | {qdlist, dns_query()} | {anlist, dns_rr()} | {nslist, dns_rr()} | {arlist, dns_rr()} ] inet_dns:msg(DnsMsg, header) -&gt; dns_header() % for example inet_dns:msg(DnsMsg, Field) -&gt; Value dns_header() = DnsHeader inet_dns:header(DnsHeader) -&gt; [ {id, integer()} | {qr, boolean()} | {opcode, 'query' | iquery | status | integer()} | {aa, boolean()} | {tc, boolean()} | {rd, boolean()} | {ra, boolean()} | {pr, boolean()} | {rcode, integer(0..16)} ] inet_dns:header(DnsHeader, Field) -&gt; Value query_type() = axfr | mailb | maila | any | rr_type() dns_query() = DnsQuery inet_dns:dns_query(DnsQuery) -&gt; [ {domain, dns_name()} | {type, query_type()} | {class, dns_class()} ] inet_dns:dns_query(DnsQuery, Field) -&gt; Value dns_rr() = DnsRr inet_dns:rr(DnsRr) -&gt; DnsRrFields | DnsRrOptFields DnsRrFields = [ {domain, dns_name()} | {type, rr_type()} | {class, dns_class()} | {ttl, integer()} | {data, dns_data()} ] DnsRrOptFields = [ {domain, dns_name()} | {type, opt} | {udp_payload_size, integer()} | {ext_rcode, integer()} | {version, integer()} | {z, integer()} | {data, dns_data()} ] inet_dns:rr(DnsRr, Field) -&gt; Value There is an info function for the types above: inet_dns:record_type(dns_msg()) -&gt; msg; inet_dns:record_type(dns_header()) -&gt; header; inet_dns:record_type(dns_query()) -&gt; dns_query; inet_dns:record_type(dns_rr()) -&gt; rr; inet_dns:record_type(_) -&gt; undefined. So; inet_dns:(inet_dns:record_type(X))(X) will convert any of these data structures into a {Field,Value} list.</pre></p> </div> </datatype> <datatype> <name name="dns_data" /> <div class="description"><p><code><anno>Regexp</anno></code> is a string with characters encoded in the UTF-8 coding standard.</p> </div> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="getbyname/2">getbyname/2</h3> <h3 id="getbyname/3">getbyname/3</h3> <div class="description"> <p>Resolve a DNS record of the given type for the given host, of class <code>in</code>. On success returns a <code>hostent()</code> record with <code>dns_data()</code> elements in the address list field. </p><p> This function uses the resolver option <code>search</code> that is a list of domain names. If the name to resolve contains no dots, it is prepended to each domain name in the search list, and they are tried in order. If the name contains dots, it is first tried as an absolute name and if that fails the search list is used. If the name has a trailing dot it is simply supposed to be an absolute name and the search list is not used. </p> </div> </div> <div class="function"> <h3 id="gethostbyaddr/1">gethostbyaddr/1</h3> <h3 id="gethostbyaddr/2">gethostbyaddr/2</h3> <div class="description"> <p>Backend functions used by <a href="../kernel/inet.html#gethostbyaddr/1" class="seealso"> inet:gethostbyaddr/1 </a>. </p> </div> </div> <div class="function"> <h3 id="gethostbyname/1">gethostbyname/1</h3> <h3 id="gethostbyname/2">gethostbyname/2</h3> <h3 id="gethostbyname/3">gethostbyname/3</h3> <div class="description"> <p>Backend functions used by <a href="../kernel/inet.html#gethostbyname/1" class="seealso"> inet:gethostbyname/1,2 </a>. </p><p> This function uses the resolver option <code>search</code> just like <a href="#getbyname/2" class="seealso">getbyname/2,3</a>. </p><p> If the resolver option <code>inet6</code> is <code>true</code>, an IPv6 address is looked up, and if that fails the IPv4 address is looked up and returned on IPv6 mapped IPv4 format. </p> </div> </div> <div class="function"> <h3 id="lookup/3">lookup/3</h3> <h3 id="lookup/4">lookup/4</h3> <h3 id="lookup/5">lookup/5</h3> <div class="description"> <p>Resolve the DNS data for the record of the given type and class for the given name. On success filters out the answer records with the correct <code><anno>Class</anno></code> and <code><anno>Type</anno></code> and returns a list of their data fields. So a lookup for type <code>any</code> will give an empty answer since the answer records have specific types that are not <code>any</code>. An empty answer as well as a failed lookup returns an empty list. </p><p> Calls <a href="#resolve/3" class="seealso">resolve/2..4</a> with the same arguments and filters the result, so <code><anno>Opts</anno></code> is explained there. </p> </div> </div> <div class="function"> <h3 id="resolve/3">resolve/3</h3> <h3 id="resolve/4">resolve/4</h3> <h3 id="resolve/5">resolve/5</h3> <div class="description"> <p>Resolve a DNS record of the given type and class for the given name. The returned <code>dns_msg()</code> can be examined using access functions in <code>inet_db</code> as described in <a href="#dns_types" class="seealso">DNS types</a>. </p><p> If <code><anno>Name</anno></code> is an <code>ip_address()</code>, the domain name to query for is generated as the standard reverse ".IN-ADDR.ARPA." name for an IPv4 address, or the ".IP6.ARPA." name for an IPv6 address. In this case you most probably want to use <code><anno>Class</anno> = in</code> and <code><anno>Type</anno> = ptr</code> but it is not done automatically. </p><p> <code><anno>Opts</anno></code> override the corresponding resolver options. If the option <code>nameservers</code> is given, it is also assumed that it is the complete list of nameserves, so the resolver option <code>alt_nameserves</code> is ignored. Of course, if that option is also given to this function, it is used. </p><p> The <code>verbose</code> option (or rather <code>{verbose,true}</code>), causes diagnostics printout through <a href="../stdlib/io.html#format/3" class="seealso">io:format/2</a> of queries, replies retransmissions, etc, similar to from utilities like <code>dig</code>, <code>nslookup</code> et.al. </p><p> If <code><anno>Opt</anno></code> is an arbitrary atom it is interpreted as <code>{<anno>Opt</anno>,true}</code> unless the atom string starts with <code>"no"</code> making the interpretation <code>{<anno>Opt</anno>,false}</code>. For example: <code>usevc</code> is an alias for <code>{usevc,true}</code>, and <code>nousevc</code> an alias for <code>{usevc,false}</code>. </p><p> The <code>inet6</code> option currently has no effect on this function. You probably want to use <code><anno>Type</anno> = a | aaaa</code> instead. </p> </div> </div> </div> <div class="section"> <h4>Examples</h4> <p>Access functions example: how <a href="#lookup/3" class="seealso">lookup/3</a> could have been implemented using <a href="#resolve/3" class="seealso">resolve/3</a> from outside the module. </p><pre class="sh_erlang"> example_lookup(Name, Class, Type) -&gt; case inet_res:resolve(Name, Class, Type) of {ok,Msg} -&gt; [inet_dns:rr(RR, data) || RR &lt;- inet_dns:msg(Msg, anlist), inet_dns:rr(RR, type) =:= Type, inet_dns:rr(RR, class) =:= Class]; {error,_} -&gt; [] end.</pre> </div> <div class="section"> <h4>Legacy Functions</h4> <p>These have been deprecated due to the annoying double meaning of the nameservers/timeout argument, and because they had no decent place for a resolver options list.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="nslookup/3">nslookup/3</h3> <name name="nslookup" arity="4" clause_i="1" /> <name name="nslookup" arity="4" clause_i="2" /> <type variable="Name" /> <type variable="Class" /> <type variable="Type" /> <type variable="Timeout" name_i="2" /> <type variable="Nameservers" /> <type variable="Reason" /> <div class="description"> <p>Resolve a DNS record of the given type and class for the given name. </p> </div> </div> <div class="function"> <h3 id="nnslookup/4">nnslookup/4</h3> <h3 id="nnslookup/5">nnslookup/5</h3> <div class="description"> <p>Resolve a DNS record of the given type and class for the given name. </p> </div> </div> </div>
