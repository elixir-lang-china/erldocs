<h1>os</h1> <h2 class="modsummary">Operating System Specific Functions</h2> <div class="description"> <p>The functions in this module are operating system specific. Careless use of these functions will result in programs that will only run on a specific platform. On the other hand, with careful use these functions can be of help in enabling a program to run on most platforms.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="cmd/1">cmd/1</h3> <div class="description"> <p>Executes <code><anno>Command</anno></code> in a command shell of the target OS, captures the standard output of the command and returns this result as a string. This function is a replacement of the previous <code>unix:cmd/1</code>; on a Unix platform they are equivalent.</p> <p>Examples:</p> <pre class="sh_erlang"> LsOut = os:cmd("ls"), % on unix platform DirOut = os:cmd("dir"), % on Win32 platform</pre> <p>Note that in some cases, standard output of a command when called from another program (for example, <code>os:cmd/1</code>) may differ, compared to the standard output of the command when called directly from an OS command shell.</p> </div> </div> <div class="function"> <h3 id="find_executable/1">find_executable/1</h3> <h3 id="find_executable/2">find_executable/2</h3> <div class="description"> <p>These two functions look up an executable program given its name and a search path, in the same way as the underlying operating system. <code>find_executable/1</code> uses the current execution path (that is, the environment variable PATH on Unix and Windows).</p> <p><code><anno>Path</anno></code>, if given, should conform to the syntax of execution paths on the operating system. The absolute filename of the executable program <code><anno>Name</anno></code> is returned, or <code>false</code> if the program was not found.</p> </div> </div> <div class="function"> <h3 id="getenv/0">getenv/0</h3> <div class="description"> <p>Returns a list of all environment variables. Each environment variable is given as a single string on the format <code>"VarName=Value"</code>, where <code>VarName</code> is the name of the variable and <code>Value</code> its value.</p> <p>If Unicode file name encoding is in effect (see the <a href="../erts/erl.html#file_name_encoding" class="seealso">erl manual page</a>), the strings may contain characters with codepoints &gt; 255.</p> </div> </div> <div class="function"> <h3 id="getenv/1">getenv/1</h3> <div class="description"> <p>Returns the <code><anno>Value</anno></code> of the environment variable <code><anno>VarName</anno></code>, or <code>false</code> if the environment variable is undefined.</p> <p>If Unicode file name encoding is in effect (see the <a href="../erts/erl.html#file_name_encoding" class="seealso">erl manual page</a>), the strings (both <code><anno>VarName</anno></code> and <code><anno>Value</anno></code>) may contain characters with codepoints &gt; 255.</p> </div> </div> <div class="function"> <h3 id="getpid/0">getpid/0</h3> <div class="description"> <p>Returns the process identifier of the current Erlang emulator in the format most commonly used by the operating system environment. <code><anno>Value</anno></code> is returned as a string containing the (usually) numerical identifier for a process. On Unix, this is typically the return value of the <code>getpid()</code> system call. On Windows, the process id as returned by the <code>GetCurrentProcessId()</code> system call is used.</p> </div> </div> <div class="function"> <h3 id="putenv/2">putenv/2</h3> <div class="description"> <p>Sets a new <code><anno>Value</anno></code> for the environment variable <code><anno>VarName</anno></code>.</p> <p>If Unicode filename encoding is in effect (see the <a href="../erts/erl.html#file_name_encoding" class="seealso">erl manual page</a>), the strings (both <code><anno>VarName</anno></code> and <code><anno>Value</anno></code>) may contain characters with codepoints &gt; 255.</p> <p>On Unix platforms, the environment will be set using UTF-8 encoding if Unicode file name translation is in effect. On Windows the environment is set using wide character interfaces.</p> </div> </div> <div class="function"> <h3 id="timestamp/0">timestamp/0</h3> <div class="type_desc"><code>Timestamp = Timestamp = {MegaSecs, Secs, MicroSecs}</code></div> <div class="description"> <p>Returns a tuple in the same format as <a href="../erts/erlang.html#now/0" class="seealso">erlang:now/0</a>. The difference is that this function returns what the operating system thinks (a.k.a. the wall clock time) without any attempts at time correction. The result of two different calls to this function is <em>not</em> guaranteed to be different.</p> <p>The most obvious use for this function is logging. The tuple can be used together with the function <a href="../stdlib/calendar.html#now_to_universal_time/1" class="seealso">calendar:now_to_universal_time/1</a> or <a href="../stdlib/calendar.html#now_to_local_time/1" class="seealso">calendar:now_to_local_time/1</a> to get calendar time. Using the calendar time together with the <code>MicroSecs</code> part of the return tuple from this function allows you to log timestamps in high resolution and consistent with the time in the rest of the operating system.</p> <p>Example of code formatting a string in the format "DD Mon YYYY HH:MM:SS.mmmmmm", where DD is the day of month, Mon is the textual month name, YYYY is the year, HH:MM:SS is the time and mmmmmm is the microseconds in six positions:</p> <code> -module(print_time). -export([format_utc_timestamp/0]). format_utc_timestamp() -&gt; TS = {_,_,Micro} = os:timestamp(), {{Year,Month,Day},{Hour,Minute,Second}} = calendar:now_to_universal_time(TS), Mstr = element(Month,{"Jan","Feb","Mar","Apr","May","Jun","Jul", "Aug","Sep","Oct","Nov","Dec"}), io_lib:format("~2w ~s ~4w ~2w:~2..0w:~2..0w.~6..0w", [Day,Mstr,Year,Hour,Minute,Second,Micro]). </code> <p>The module above could be used in the following way:</p> <pre class="sh_erlang"> 1&gt; <code>io:format("~s~n",[print_time:format_utc_timestamp()]).</code> 29 Apr 2009 9:55:30.051711 </pre> </div> </div> <div class="function"> <h3 id="type/0">type/0</h3> <div class="description"> <p>Returns the <code><anno>Osfamily</anno></code> and, in some cases, <code><anno>Osname</anno></code> of the current operating system.</p> <p>On Unix, <code><anno>Osname</anno></code> will have same value as <code>uname -s</code> returns, but in lower case. For example, on Solaris 1 and 2, it will be <code>sunos</code>.</p> <p>In Windows, <code><anno>Osname</anno></code> will be either <code>nt</code> (on Windows NT), or <code>windows</code> (on Windows 95).</p> <div class="note"><h2>Note!</h2> <p>Think twice before using this function. Use the <code>filename</code> module if you want to inspect or build file names in a portable way. Avoid matching on the <code><anno>Osname</anno></code> atom.</p> </div> </div> </div> <div class="function"> <h3 id="unsetenv/1">unsetenv/1</h3> <div class="description"> <p>Deletes the environment variable <code><anno>VarName</anno></code>.</p> <p>If Unicode filename encoding is in effect (see the <a href="../erts/erl.html#file_name_encoding" class="seealso">erl manual page</a>), the string (<code><anno>VarName</anno></code>) may contain characters with codepoints &gt; 255.</p> </div> </div> <div class="function"> <h3 id="version/0">version/0</h3> <div class="description"> <p>Returns the operating system version. On most systems, this function returns a tuple, but a string will be returned instead if the system has versions which cannot be expressed as three numbers.</p> <div class="note"><h2>Note!</h2> <p>Think twice before using this function. If you still need to use it, always <code>call os:type()</code> first.</p> </div> </div> </div> </div>
