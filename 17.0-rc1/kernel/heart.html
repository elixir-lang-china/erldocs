<h1>heart</h1> <h2 class="modsummary">Heartbeat Monitoring of an Erlang Runtime System</h2> <div class="description"> <p>This modules contains the interface to the <code>heart</code> process. <code>heart</code> sends periodic heartbeats to an external port program, which is also named <code>heart</code>. The purpose of the heart port program is to check that the Erlang runtime system it is supervising is still running. If the port program has not received any heartbeats within <code>HEART_BEAT_TIMEOUT</code> seconds (default is 60 seconds), the system can be rebooted. Also, if the system is equipped with a hardware watchdog timer and is running Solaris, the watchdog can be used to supervise the entire system.</p> <p>An Erlang runtime system to be monitored by a heart program, should be started with the command line flag <code>-heart</code> (see also <a href="../erts/erl.html" class="seealso">erl(1)</a>). The <code>heart</code> process is then started automatically:</p> <pre class="sh_erlang"> % <code>erl -heart ...</code></pre> <p>If the system should be rebooted because of missing heart-beats, or a terminated Erlang runtime system, the environment variable <code>HEART_COMMAND</code> has to be set before the system is started. If this variable is not set, a warning text will be printed but the system will not reboot. However, if the hardware watchdog is used, it will trigger a reboot <code>HEART_BEAT_BOOT_DELAY</code> seconds later nevertheless (default is 60).</p> <p>To reboot on the WINDOWS platform <code>HEART_COMMAND</code> can be set to <code>heart -shutdown</code> (included in the Erlang delivery) or of course to any other suitable program which can activate a reboot.</p> <p>The hardware watchdog will not be started under Solaris if the environment variable <code>HW_WD_DISABLE</code> is set.</p> <p>The <code>HEART_BEAT_TIMEOUT</code> and <code>HEART_BEAT_BOOT_DELAY</code> environment variables can be used to configure the heart timeouts, they can be set in the operating system shell before Erlang is started or be specified at the command line:</p> <pre class="sh_erlang"> % <code>erl -heart -env HEART_BEAT_TIMEOUT 30 ...</code></pre> <p>The value (in seconds) must be in the range 10 &lt; X &lt;= 65535.</p> <p>It should be noted that if the system clock is adjusted with more than <code>HEART_BEAT_TIMEOUT</code> seconds, <code>heart</code> will timeout and try to reboot the system. This can happen, for example, if the system clock is adjusted automatically by use of NTP (Network Time Protocol).</p> <p> If a crash occurs, an <code>erl_crash.dump</code> will <em>not</em> be written unless the environment variable <code>ERL_CRASH_DUMP_SECONDS</code> is set. </p> <pre class="sh_erlang"> % <code>erl -heart -env ERL_CRASH_DUMP_SECONDS 10 ...</code></pre> <p> Furthermore, <code>ERL_CRASH_DUMP_SECONDS</code> has the following behaviour on <code>heart</code>: </p> <taglist> <dt><code>ERL_CRASH_DUMP_SECONDS=0</code></dt> <item><p> Suppresses the writing a crash dump file entirely, thus rebooting the runtime system immediately. This is the same as not setting the environment variable. </p> </item> <dt><code>ERL_CRASH_DUMP_SECONDS=-1</code></dt> <item><p> Setting the environment variable to a negative value will not reboot the runtime system until the crash dump file has been completly written. </p> </item> <dt><code>ERL_CRASH_DUMP_SECONDS=S</code></dt> <item><p> Heart will wait for <code>S</code> seconds to let the crash dump file be written. After <code>S</code> seconds <code>heart</code> will reboot the runtime system regardless of the crash dump file has been written or not. </p> </item> </taglist> <p>In the following descriptions, all function fails with reason <code>badarg</code> if <code>heart</code> is not started.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="set_cmd/1">set_cmd/1</h3> <div class="description"> <p>Sets a temporary reboot command. This command is used if a <code>HEART_COMMAND</code> other than the one specified with the environment variable should be used in order to reboot the system. The new Erlang runtime system will (if it misbehaves) use the environment variable <code>HEART_COMMAND</code> to reboot.</p> <p>Limitations: The <code><anno>Cmd</anno></code> command string will be sent to the heart program as a ISO-latin-1 or UTF-8 encoded binary depending on the file name encoding mode of the emulator (see <a href="../kernel/file.html#native_name_encoding/0" class="seealso"><code>file:native_name_encoding/0</code></a>). The size of the encoded binary must be less than 2047 bytes.</p> </div> </div> <div class="function"> <h3 id="clear_cmd/0">clear_cmd/0</h3> <div class="description"> <p>Clears the temporary boot command. If the system terminates, the normal <code>HEART_COMMAND</code> is used to reboot.</p> </div> </div> <div class="function"> <h3 id="get_cmd/0">get_cmd/0</h3> <div class="description"> <p>Get the temporary reboot command. If the command is cleared, the empty string will be returned.</p> </div> </div> </div>
