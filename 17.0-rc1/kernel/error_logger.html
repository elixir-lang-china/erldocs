<h1>error_logger</h1> <h2 class="modsummary">Erlang Error Logger</h2> <div class="description"> <p>The Erlang <em>error logger</em> is an event manager (see <a href="../doc/design_principles/des_princ.html" class="seealso">OTP Design Principles</a> and <a href="../stdlib/gen_event.html" class="seealso">gen_event(3)</a>), registered as <code>error_logger</code>. Error, warning and info events are sent to the error logger from the Erlang runtime system and the different Erlang/OTP applications. The events are, by default, logged to tty. Note that an event from a process <code>P</code> is logged at the node of the group leader of <code>P</code>. This means that log output is directed to the node from which a process was created, which not necessarily is the same node as where it is executing.</p> <p>Initially, <code>error_logger</code> only has a primitive event handler, which buffers and prints the raw event messages. During system startup, the application Kernel replaces this with a <em>standard event handler</em>, by default one which writes nicely formatted output to tty. Kernel can also be configured so that events are logged to file instead, or not logged at all, see <a href="kernel_app.html" class="seealso">kernel(6)</a>.</p> <p>Also the SASL application, if started, adds its own event handler, which by default writes supervisor, crash and progress reports to tty. See <a href="../sasl/sasl_app.html" class="seealso">sasl(6)</a>.</p> <p>It is recommended that user defined applications should report errors through the error logger, in order to get uniform reports. User defined event handlers can be added to handle application specific events. (<code>add_report_handler/1,2</code>). Also, there is a useful event handler in STDLIB for multi-file logging of events, see <code>log_mf_h(3)</code>.</p> <p>Warning events was introduced in Erlang/OTP R9C. To retain backwards compatibility, these are by default tagged as errors, thus showing up as error reports in the logs. By using the command line flag <code>+W &lt;w | i&gt;</code>, they can instead be tagged as warnings or info. Tagging them as warnings may require rewriting existing user defined event handlers.</p> </div> <datatypes> <datatype> <name name="report" /> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="error_msg/1">error_msg/1</h3> <h3 id="error_msg/2">error_msg/2</h3> <h3 id="format/2">format/2</h3> <div class="description"> <p>Sends a standard error event to the error logger. added to the error logger must handle the following events. <code>Gleader</code> is the group leader pid of the process which sent the event, and <code>Pid</code> is the process which sent the event.</p> <taglist> <dt><code>{error, Gleader, {Pid, Format, Data}}</code></dt> <item> <p>Generated when <code>error_msg/1,2</code> or <code>format</code> is called.</p> </item> <dt><code>{error_report, Gleader, {Pid, std_error, Report}}</code></dt> <item> <p>Generated when <code>error_report/1</code> is called.</p> </item> <dt><code>{error_report, Gleader, {Pid, Type, Report}}</code></dt> <item> <p>Generated when <code>error_report/2</code> is called.</p> </item> <dt><code>{warning_msg, Gleader, {Pid, Format, Data}}</code></dt> <item> <p>Generated when <code>warning_msg/1,2</code> is called, provided that warnings are set to be tagged as warnings.</p> </item> <dt><code>{warning_report, Gleader, {Pid, std_warning, Report}}</code></dt> <item> <p>Generated when <code>warning_report/1</code> is called, provided that warnings are set to be tagged as warnings.</p> </item> <dt><code>{warning_report, Gleader, {Pid, Type, Report}}</code></dt> <item> <p>Generated when <code>warning_report/2</code> is called, provided that warnings are set to be tagged as warnings.</p> </item> <dt><code>{info_msg, Gleader, {Pid, Format, Data}}</code></dt> <item> <p>Generated when <code>info_msg/1,2</code> is called.</p> </item> <dt><code>{info_report, Gleader, {Pid, std_info, Report}}</code></dt> <item> <p>Generated when <code>info_report/1</code> is called.</p> </item> <dt><code>{info_report, Gleader, {Pid, Type, Report}}</code></dt> <item> <p>Generated when <code>info_report/2</code> is called.</p> </item> </taglist> <p>Note that also a number of system internal events may be received, a catch-all clause last in the definition of the event handler callback function <code>Module:handle_event/2</code> is necessary. This also holds true for <code>Module:handle_info/2</code>, as there are a number of system internal messages the event handler must take care of as well.</p> </div> <div class="section"> <h4>SEE ALSO</h4> <p>gen_event(3), log_mf_h(3), kernel(6), sasl(6)</p> </div>
