<h1>pg2</h1> <h2 class="modsummary">Distributed Named Process Groups</h2> <div class="description"> <p>This module implements process groups. The groups in this module differ from the groups in the module <code>pg</code> in several ways. In <code>pg</code>, each message is sent to all members in the group. In this module, each message may be sent to one, some, or all members. </p> <p>A group of processes can be accessed by a common name. For example, if there is a group named <code>foobar</code>, there can be a set of processes (which can be located on different nodes) which are all members of the group <code>foobar</code>. There are no special functions for sending a message to the group. Instead, client functions should be written with the functions <code>get_members/1</code> and <code>get_local_members/1</code> to find out which processes are members of the group. Then the message can be sent to one or more members of the group. </p> <p>If a member terminates, it is automatically removed from the group. </p> <div class="warning"><h2>Warning!</h2> <p>This module is used by the <code>disk_log</code> module for managing distributed disk logs. The disk log names are used as group names, which means that some action may need to be taken to avoid name clashes.</p> </div> </div> <datatypes> <datatype> <name name="name" /> <div class="description"><p>The name of a process group.</p></div> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="create/1">create/1</h3> <div class="description"> <p>Creates a new, empty process group. The group is globally visible on all nodes. If the group exists, nothing happens. </p> </div> </div> <div class="function"> <h3 id="delete/1">delete/1</h3> <div class="description"> <p>Deletes a process group. </p> </div> </div> <div class="function"> <h3 id="get_closest_pid/1">get_closest_pid/1</h3> <div class="description"> <p>This is a useful dispatch function which can be used from client functions. It returns a process on the local node, if such a process exist. Otherwise, it chooses one randomly. </p> </div> </div> <div class="function"> <h3 id="get_members/1">get_members/1</h3> <div class="description"> <p>Returns all processes in the group <code>Name</code>. This function should be used from within a client function that accesses the group. It is therefore optimized for speed. </p> </div> </div> <div class="function"> <h3 id="get_local_members/1">get_local_members/1</h3> <div class="description"> <p>Returns all processes running on the local node in the group <code>Name</code>. This function should to be used from within a client function that accesses the group. It is therefore optimized for speed. </p> </div> </div> <div class="function"> <h3 id="join/2">join/2</h3> <div class="description"> <p>Joins the process <code>Pid</code> to the group <code>Name</code>. A process can join a group several times; it must then leave the group the same number of times. </p> </div> </div> <div class="function"> <h3 id="leave/2">leave/2</h3> <div class="description"> <p>Makes the process <code>Pid</code> leave the group <code>Name</code>. If the process is not a member of the group, <code>ok</code> is returned. </p> </div> </div> <div class="function"> <h3 id="which_groups/0">which_groups/0</h3> <div class="description"> <p>Returns a list of all known groups. </p> </div> </div> <div class="function"> <h3 id="start/0">start/0</h3> <h3 id="start_link/0">start_link/0</h3> <div class="description"> <p>Starts the pg2 server. Normally, the server does not need to be started explicitly, as it is started dynamically if it is needed. This is useful during development, but in a target system the server should be started explicitly. Use configuration parameters for <code>kernel</code> for this. </p> </div> </div> </div> <div class="section"> <h4>See Also</h4> <p><a href="kernel_app.html" class="seealso">kernel(6)</a>, <a href="../stdlib/pg.html" class="seealso">pg(3)</a></p> </div>
