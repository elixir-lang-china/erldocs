<h1>compile</h1> <h2 class="modsummary">Erlang Compiler</h2> <div class="description"> <p>This module provides an interface to the standard Erlang compiler. It can generate either a new file which contains the object code, or return a binary which can be loaded directly. </p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="file/1">file(File)</h3> <div class="description"> <p>Is the same as <code>file(File, [verbose,report_errors,report_warnings])</code>. </p> </div> </div> <div class="function"> <h3 id="file/2">file(File, Options) -&gt; CompRet</h3> <ul class="type"> <li><code>CompRet = ModRet | BinRet | ErrRet</code></li> <li><code>ModRet = {ok,ModuleName} | {ok,ModuleName,Warnings}</code></li> <li><code>BinRet = {ok,ModuleName,Binary} | {ok,ModuleName,Binary,Warnings}</code></li> <li><code>ErrRet = error | {error,Errors,Warnings}</code></li> </ul> <div class="description"> <p>Compiles the code in the file <code>File</code>, which is an Erlang source code file without the <code>.erl</code> extension. <code>Options</code> determine the behavior of the compiler.</p> <p>Returns <code>{ok,ModuleName}</code> if successful, or <code>error</code> if there are errors. An object code file is created if the compilation succeeds with no errors. It is considered to be an error if the module name in the source code is not the same as the basename of the output file.</p> <p><span id="type-option"> </span>Here follows first all elements of <code>Options</code> that in some way control the behavior of the compiler.</p> <taglist> <dt><code>basic_validation</code></dt> <item> <p>This option is fast way to test whether a module will compile successfully (mainly useful for code generators that want to verify the code they emit). No code will generated. If warnings are enabled, warnings generated by the <code>erl_lint</code> module (such as warnings for unused variables and functions) will be returned too.</p> <p>Use the <code>strong_validation</code> option to generate all warnings that the compiler would generate.</p> </item> <dt><code>strong_validation</code></dt> <item> <p>Similar to the <code>basic_validation</code> option, no code will be generated, but more compiler passes will be run to ensure also warnings generated by the optimization passes are generated (such as clauses that will not match or expressions that are guaranteed to fail with an exception at run-time).</p> </item> <dt><code>binary</code></dt> <item> <p>Causes the compiler to return the object code in a binary instead of creating an object file. If successful, the compiler returns <code>{ok,ModuleName,Binary}</code>.</p> </item> <dt><code>bin_opt_info</code></dt> <item> <p>The compiler will emit informational warnings about binary matching optimizations (both successful and unsuccessful). See the <em>Efficiency Guide</em> for further information.</p> </item> <dt><code>compressed</code></dt> <item> <p>The compiler will compress the generated object code, which can be useful for embedded systems.</p> </item> <dt><code>debug_info</code></dt> <item> <span id="debug_info"> </span> <p>Include debug information in the form of abstract code (see <a href="../erts/absform.html" class="seealso">The Abstract Format</a> in ERTS User's Guide) in the compiled beam module. Tools such as Debugger, Xref and Cover require the debug information to be included.</p> <p><em>Warning</em>: Source code can be reconstructed from the debug information. Use encrypted debug information (see below) to prevent this.</p> <p>See <a href="../stdlib/beam_lib.html#debug_info" class="seealso">beam_lib(3)</a> for details.</p> </item> <dt><code>{debug_info_key,KeyString}</code></dt> <item /> <dt><code>{debug_info_key,{Mode,KeyString}}</code></dt> <item> <span id="debug_info_key"> </span> <p>Include debug information, but encrypt it, so that it cannot be accessed without supplying the key. (To give the <code>debug_info</code> option as well is allowed, but is not necessary.) Using this option is a good way to always have the debug information available during testing, yet protect the source code.</p> <p><code>Mode</code> is the type of crypto algorithm to be used for encrypting the debug information. The default type -- and currently the only type -- is <code>des3_cbc</code>.</p> <p>See <a href="../stdlib/beam_lib.html#debug_info" class="seealso">beam_lib(3)</a> for details.</p> </item> <dt><code>encrypt_debug_info</code></dt> <item> <span id="encrypt_debug_info"> </span> <p>Like the <code>debug_info_key</code> option above, except that the key will be read from an <code>.erlang.crypt</code> file. </p> <p>See <a href="../stdlib/beam_lib.html#debug_info" class="seealso">beam_lib(3)</a> for details.</p> </item> <dt><code>makedep</code></dt> <item> <p>Produce a Makefile rule to track headers dependencies. No object file is produced. </p> <p>By default, this rule is written to <code>&lt;File&gt;.Pbeam</code>. No object file is produced. </p> </item> <dt><code>'E'</code></dt> <item> <p>Produces a listing of the code after all source code transformations have been performed, in the file <code>&lt;File&gt;.E</code>. The current directory is used for output, except when a directory has been specified with this option.</p> </item> <dt><code>export_all</code></dt> <item> <p>Causes all functions in the module to be exported.</p> </item> <dt><code>{i,Dir}</code></dt> <item> <p>Add <code>Dir</code> to the list of directories to be searched when including a file. The directory specified last is searched first.</p> </item> </list> </item> <dt><code>{d,Macro}</code></dt> <item /> <dt><code>{d,Macro,Value}</code></dt> <item> <p>Defines a macro <code>Macro</code> to have the value <code>Value</code>. <code>Macro</code> is of type atom, and <code>Value</code> can be any term. The default <code>Value</code> is <code>true</code>.</p> </item> <dt><code>{parse_transform,Module}</code></dt> <item> <p>Causes the parse transformation function <code>Module:parse_transform/2</code> to be applied to the parsed code before the code is checked for errors.</p> </item> <dt><code>from_asm</code></dt> <item> <p>The input file is expected to be assembler code (default file suffix ".S"). Note that the format of assembler files is not documented, and may change between releases.</p> </item> <dt><code>from_core</code></dt> <item> <p>The input file is expected to be core code (default file suffix ".core"). Note that the format of core files is not documented, and may change between releases.</p> </item> <dt><code>no_strict_record_tests</code></dt> <item> <p>This option is not recommended.</p> <p>By default, the generated code for the <code>Record#record_tag.field</code> operation verifies that the tuple <code>Record</code> is of the correct size for the record and that the first element is the tag <code>record_tag</code>. Use this option to omit the verification code.</p> </item> <dt><code>no_error_module_mismatch</code></dt> <item> <p>Normally the compiler verifies that the module name given in the source code is the same as the base name of the output file and refuses to generate an output file if there is a mismatch. If you have a good reason (or other reason) for having a module name unrelated to the name of the output file, this option disables that verification (there will not even be a warning if there is a mismatch).</p> </item> <dt><code>{no_auto_import,[{F,A}, ...]}</code></dt> <item> <p>Makes the function <code>F/A</code> no longer being auto-imported from the module <code>erlang</code>, which resolves BIF name clashes. This option has to be used to resolve name clashes with BIFs auto-imported before R14A, if one wants to call the local function with the same name as an auto-imported BIF without module prefix.</p> <div class="note"><h2>Note!</h2> <p>From R14A and forward, the compiler resolves calls without module prefix to local or imported functions before trying auto-imported BIFs. If the BIF is to be called, use the <code>erlang</code> module prefix in the call, not <code>{ no_auto_import,[{F,A}, ...]}</code></p> </div> <p>If this option is written in the source code, as a <code>-compile</code> directive, the syntax <code>F/A</code> can be used instead of <code>{F,A}</code>. Example:</p> <code>-compile({no_auto_import,[error/1]}).</code> </item> <dt><code>no_line_info</code></dt> <item> <p>Omit line number information in order to produce a slightly smaller output file. </p> </item> </taglist> <p>If warnings are turned on (the <code>report_warnings</code> option described above), the following options control what type of warnings that will be generated. <span id="erl_lint_options"> </span> With the exception of <code>{warn_format,Verbosity}</code> all options below have two forms; one <code>warn_xxx</code> form to turn on the warning and one <code>nowarn_xxx</code> form to turn off the warning. In the description that follows, the form that is used to change the default value is listed.</p> <taglist> <dt><code>{warn_format, Verbosity}</code></dt> <item> <p>Causes warnings to be emitted for malformed format strings as arguments to <code>io:format</code> and similar functions. <code>Verbosity</code> selects the amount of warnings: 0 = no warnings; 1 = warnings for invalid format strings and incorrect number of arguments; 2 = warnings also when the validity could not be checked (for example, when the format string argument is a variable). The default verbosity is 1. Verbosity 0 can also be selected by the option <code>nowarn_format</code>.</p> </item> <dt><code>nowarn_bif_clash</code></dt> <item> <p>This option is removed, it will generate a fatal error if used.</p> <div class="warning"><h2>Warning!</h2> <p>Beginning with R14A, the compiler no longer calls the auto-imported BIF if the name clashes with a local or explicitly imported function and a call without explicit module name is issued. Instead the local or imported function is called. Still accepting <code>nowarn_bif_clash</code> would makes a module calling functions clashing with autoimported BIFs compile with both the old and new compilers, but with completely different semantics, why the option was removed.</p> <p>The use of this option has always been strongly discouraged. From OTP R14A and forward it's an error to use it.</p> <p>To resolve BIF clashes, use explicit module names or the <code>{no_auto_import,[F/A]}</code> compiler directive.</p> </div> </item> <dt><code>{nowarn_bif_clash, FAs}</code></dt> <item> <p>This option is removed, it will generate a fatal error if used.</p> <div class="warning"><h2>Warning!</h2> <p>The use of this option has always been strongly discouraged. From OTP R14A and forward it's an error to use it.</p> <p>To resolve BIF clashes, use explicit module names or the <code>{no_auto_import,[F/A]}</code> compiler directive.</p> </div> </item> <dt><code>warn_export_all</code></dt> <item> <p>Causes a warning to be emitted if the <code>export_all</code> option has also been given.</p> </item> <dt><code>warn_export_vars</code></dt> <item> <p>Causes warnings to be emitted for all implicitly exported variables referred to after the primitives where they were first defined. No warnings for exported variables unless they are referred to in some pattern, which is the default, can be selected by the option <code>nowarn_export_vars</code>.</p> </item> <dt><code>warn_shadow_vars</code></dt> <item> <p>Causes warnings to be emitted for "fresh" variables in functional objects or list comprehensions with the same name as some already defined variable. By default (<code>warn_unused_function</code>), warnings are emitted for all local functions that are not called directly or indirectly by an exported function. The compiler does not include unused local functions in the generated beam file, but the warning is still useful to keep the source code cleaner.</p> </item> <dt><code>{nowarn_unused_function, FAs}</code></dt> <item> <p>Turns off warnings for unused local functions as <code>nowarn_unused_function</code> but only for the mentioned local functions. <code>FAs</code> is a tuple <code>{Name,Arity}</code> or a list of such tuples.</p> </item> <dt><code>nowarn_deprecated_function</code></dt> <item> <p>Turns off warnings for calls to deprecated functions. See also <a href="../tools/xref.html#deprecated_function" class="seealso">xref(3)</a> and the function <a href="../tools/xref.html#m/1" class="seealso">xref:m/1</a> also accessible through the <a href="../stdlib/c.html#xm/1" class="seealso">c:xm/1</a> function.</p> </item> <dt><code>{nowarn_deprecated_function, MFAs}</code></dt> <item> <p>Turns off warnings for calls to deprecated functions as <code>nowarn_deprecated_function</code> but only for the mentioned functions. <code>MFAs</code> is a tuple <code>{Module,Name,Arity}</code> or a list of such tuples.</p> </item> <dt><code>warn_obsolete_guard</code></dt> <item> <p>Causes warnings to be emitted for calls to old type testing BIFs such as <code>pid/1</code> and <code>list/1</code>. No warnings for unused imported functions, which is the default, can be selected by the option <code>nowarn_unused_import</code>. </p> </item> <dt><code>nowarn_unused_vars</code></dt> <item> <p>By default, warnings are emitted for variables which are not used, with the exception of variables beginning with an underscore ("Prolog style warnings"). Use this option to turn off this kind of warnings.</p> </item> <dt><code>nowarn_unused_record</code></dt> <item> <p>Turns off warnings for unused record types. By default (<code>warn_unused_records</code>), warnings are emitted for unused locally defined record types.</p> </item> </taglist> <p>Another class of warnings is generated by the compiler during optimization and code generation. They warn about patterns that will never match (such as <code>a=b</code>), guards that will always evaluate to false, and expressions that will always fail (such as <code>atom+42</code>).</p> <p>Note that the compiler does not warn for expressions that it does not attempt to optimize. For instance, the compiler tries to evaluate <code>1/0</code>, notices that it will cause an exception and emits a warning. On the other hand, the compiler is silent about the similar expression <code>X/0</code>; because of the variable in it, the compiler does not even try to evaluate and therefore it emits no warnings. </p> <p>Currently, those warnings cannot be disabled (except by disabling all warnings).</p> <div class="warning"><h2>Warning!</h2> <p>Obviously, the absence of warnings does not mean that there are no remaining errors in the code.</p> </div> <p>Note that all the options except the include path (<code>{i,Dir}</code>) can also be given in the file with a <code>-compile([Option,...])</code>. attribute. The <code>-compile()</code> attribute is allowed after function definitions.</p> <p>Note also that the <code>{nowarn_unused_function, FAs}</code>, <code>{nowarn_bif_clash, FAs}</code>, and <code>{nowarn_deprecated_function, MFAs}</code> options are only recognized when given in files. They are not affected by the <code>warn_unused_function</code>, <code>warn_bif_clash</code>, or <code>warn_deprecated_function</code> options.</p> <p>For debugging of the compiler, or for pure curiosity, the intermediate code generated by each compiler pass can be inspected. A complete list of the options to produce list files can be printed by typing <code>compile:options()</code> at the Erlang shell prompt. The options will be printed in order that the passes are executed. If more than one listing option is used, the one representing the earliest pass takes effect.</p> <p><em>Unrecognized options are ignored.</em></p> <p>Both <code>WarningList</code> and <code>ErrorList</code> have the following format:</p> <code> [{FileName,[ErrorInfo]}]. </code> <p><code>ErrorInfo</code> is described below. The file name has been included here as the compiler uses the Erlang pre-processor <code>epp</code>, which allows the code to be included in other files. For this reason, it is important to know to <em>which</em> file an error or warning line number refers. </p> </div> </div> <div class="function"> <h3 id="forms/1">forms(Forms)</h3> <div class="description"> <p>Is the same as <code>forms(File, [verbose,report_errors,report_warnings])</code>. </p> </div> </div> <div class="function"> <h3 id="forms/2">forms(Forms, Options) -&gt; CompRet</h3> <ul class="type"> <li><code>Forms = [Form]</code></li> <li><code>CompRet = BinRet | ErrRet</code></li> <li><code>BinRet = {ok,ModuleName,BinaryOrCode} | {ok,ModuleName,BinaryOrCode,Warnings}</code></li> <li><code>BinaryOrCode = binary() | term()</code></li> <li><code>ErrRet = error | {error,Errors,Warnings}</code></li> </ul> <div class="description"> <p>Analogous to <code>file/1</code>, but takes a list of forms (in the Erlang abstract format representation) as first argument. Its value must be a valid Erlang term. If the value is a list, it will be used as is. If it is not a list, it will be put into a list.</p> <p>The list will be appended to any options given to <a href="#file/2" class="seealso">file/2</a>, <a href="#forms/2" class="seealso">forms/2</a>, and <a href="#output_generated/1" class="seealso">output_generated/2</a>. Use the alternative functions <a href="#noenv_file/2" class="seealso">noenv_file/2</a>, <a href="#noenv_forms/2" class="seealso">noenv_forms/2</a>, or <a href="#noenv_output_generated/1" class="seealso">noenv_output_generated/2</a> if you don't want the environment variable to be consulted (for instance, if you are calling the compiler recursively from inside a parse transform).</p> </div> <div class="section"> <h4>Inlining</h4> <p>The compiler can do function inlining within an Erlang module. Inlining means that a call to a function is replaced with the function body with the arguments replaced with the actual values. The semantics are preserved, except if exceptions are generated in the inlined code. Exceptions will be reported as occurring in the function the body was inlined into. Also, <code>function_clause</code> exceptions will be converted to similar <code>case_clause</code> exceptions.</p> <p>When a function is inlined, the original function will be kept if it is exported (either by an explicit export or if the <code>export_all</code> option was given) or if not all calls to the function were inlined.</p> <p>Inlining does not necessarily improve running time. For instance, inlining may increase Beam stack usage which will probably be detrimental to performance for recursive functions. </p> <p>Inlining is never default; it must be explicitly enabled with a compiler option or a <code>-compile()</code> attribute in the source module.</p> <p>To enable inlining, either use the <code>inline</code> option to let the compiler decide which functions to inline or <code>{inline,[{Name,Arity},...]}</code> to have the compiler inline all calls to the given functions. If the option is given inside a <code>compile</code> directive in an Erlang module, <code>{Name,Arity}</code> may be written as <code>Name/Arity</code>.</p> <p>Example of explicit inlining:</p> <pre class="sh_erlang"> -compile({inline,[pi/0]}). pi() -&gt; 3.1416. </pre> <p>Example of implicit inlining:</p> <pre class="sh_erlang"> -compile(inline). </pre> <p>The <code>{inline_size,Size}</code> option controls how large functions that are allowed to be inlined. Default is <code>24</code>, which will keep the size of the inlined code roughly the same as the un-inlined version (only relatively small functions will be inlined).</p> <p>Example:</p> <pre class="sh_erlang"> %% Aggressive inlining - will increase code size. -compile(inline). -compile({inline_size,100}). </pre> </div> <div class="section"> <h4>Inlining of list functions</h4> <p>The compiler can also inline a variety of list manipulation functions from the stdlib's lists module.</p> <p>This feature must be explicitly enabled with a compiler option or a <code>-compile()</code> attribute in the source module.</p> <p>To enable inlining of list functions, use the <code>inline_list_funcs</code> option.</p> <p>The following functions are inlined:</p> <list type="bulleted"> <item><a href="../stdlib/lists.html#all/2" class="seealso">lists:all/2</a></item> <item><a href="../stdlib/lists.html#any/2" class="seealso">lists:any/2</a></item> <item><a href="../stdlib/lists.html#foreach/2" class="seealso">lists:foreach/2</a></item> <item><a href="../stdlib/lists.html#map/2" class="seealso">lists:map/2</a></item> <item><a href="../stdlib/lists.html#flatmap/2" class="seealso">lists:flatmap/2</a></item> <item><a href="../stdlib/lists.html#filter/2" class="seealso">lists:filter/2</a></item> <item><a href="../stdlib/lists.html#foldl/3" class="seealso">lists:foldl/3</a></item> <item><a href="../stdlib/lists.html#foldr/3" class="seealso">lists:foldr/3</a></item> <item><a href="../stdlib/lists.html#mapfoldl/3" class="seealso">lists:mapfoldl/3</a></item> <item><a href="../stdlib/lists.html#mapfoldr/3" class="seealso">lists:mapfoldr/3</a></item> </list> </div> <div class="section"> <h4>Parse Transformations</h4> <p>Parse transformations are used when a programmer wants to use Erlang syntax but with different semantics. The original Erlang code is then transformed into other Erlang code.</p> </div> <div class="section"> <h4>Error Information</h4> <p>The <code>ErrorInfo</code> mentioned above is the standard <code>ErrorInfo</code> structure which is returned from all IO modules. It has the following format:</p> <code> {ErrorLine, Module, ErrorDescriptor} </code> <p><code>ErrorLine</code> will be the atom <code>none</code> if the error does not correspond to a specific line (e.g. if the source file does not exist).</p> <p>A string describing the error is obtained with the following call:</p> <code> Module:format_error(ErrorDescriptor) </code> </div> <div class="section"> <h4>See Also</h4> <p> <a href="../stdlib/epp.html" class="seealso">epp(3)</a>, <a href="../stdlib/erl_id_trans.html" class="seealso">erl_id_trans(3)</a>, <a href="../stdlib/erl_lint.html" class="seealso">erl_lint(3)</a>, <a href="../stdlib/beam_lib.html" class="seealso">beam_lib(3)</a> </p> </div>
