<h1>dialyzer</h1> <h2 class="modsummary">The Dialyzer, a DIscrepancy AnalYZer for ERlang programs</h2> <div class="description"> <p>The Dialyzer is a static analysis tool that identifies software discrepancies such as definite type errors, code which has become dead or unreachable due to some programming error, unnecessary tests, etc. in single Erlang modules or entire (sets of) applications. Dialyzer starts its analysis from either debug-compiled BEAM bytecode or from Erlang source code. The file and line number of a discrepancy is reported along with an indication of what the discrepancy is about. Dialyzer bases its analysis on the concept of success typings which allows for sound warnings (no false positives).</p> <p>Read more about Dialyzer and about how to use it from the GUI in <a href="dialyzer_chapter.html" class="seealso">Dialyzer User's Guide</a>.</p> </div> <div class="section"> <h4>Using the Dialyzer from the command line</h4> <p>Dialyzer also has a command line version for automated use. Below is a brief description of the list of its options. The same information can be obtained by writing</p> <pre class="sh_erlang"> dialyzer --help </pre> <p>in a shell. Please refer to the GUI description for more details on the operation of Dialyzer.</p> <p>The exit status of the command line version is:</p> <pre class="sh_erlang"> 0 - No problems were encountered during the analysis and no warnings were emitted. 1 - Problems were encountered during the analysis. 2 - No problems were encountered, but warnings were emitted. </pre> <p>Usage:</p> <pre class="sh_erlang"> dialyzer [--help] [--version] [--shell] [--quiet] [--verbose] [-pa dir]* [--plt plt] [--plts plt*] [-Ddefine]* [-I include_dir]* [--output_plt file] [-Wwarn]* [--src] [--gui] [files_or_dirs] [-r dirs] [--apps applications] [-o outfile] [--build_plt] [--add_to_plt] [--remove_from_plt] [--check_plt] [--no_check_plt] [--plt_info] [--get_warnings] [--no_native] [--fullpath] </pre> <p>Options:</p> <taglist> <dt><code>files_or_dirs</code> (for backwards compatibility also as: <code>-c files_or_dirs</code></dt> <item>Use Dialyzer from the command line to detect defects in the specified files or directories containing <code>.erl</code> or <code>.beam</code> files, depending on the type of the analysis.</item> <dt><code>-r dirs</code></dt> <item>Same as the previous but the specified directories are searched recursively for subdirectories containing <code>.erl</code> or <code>.beam</code> files in them, depending on the type of analysis.</item> <dt><code>--apps applications</code></dt> <item>Option typically used when building or modifying a plt as in: <pre class="sh_erlang"> dialyzer --build_plt --apps erts kernel stdlib mnesia ... </pre> to conveniently refer to library applications corresponding to the Erlang/OTP installation. However, the option is general and can also be used during analysis in order to refer to Erlang/OTP applications. In addition, file or directory names can also be included, as in: <pre class="sh_erlang"> dialyzer --apps inets ssl ./ebin ../other_lib/ebin/my_module.beam </pre></item> <dt><code>-o outfile</code> (or <code>--output outfile</code>)</dt> <item>When using Dialyzer from the command line, send the analysis results to the specified outfile rather than to stdout.</item> <dt><code>--raw</code></dt> <item>When using Dialyzer from the command line, output the raw analysis results (Erlang terms) instead of the formatted result. The raw format is easier to post-process (for instance, to filter warnings or to output HTML pages).</item> <dt><code>--src</code></dt> <item>Override the default, which is to analyze BEAM files, and analyze starting from Erlang source code instead.</item> <dt><code>-Dname</code> (or <code>-Dname=value</code>)</dt> <item>When analyzing from source, pass the define to Dialyzer. (**)</item> <dt><code>-I include_dir</code></dt> <item>When analyzing from source, pass the <code>include_dir</code> to Dialyzer. (**)</item> <dt><code>-pa dir</code></dt> <item>Include <code>dir</code> in the path for Erlang (useful when analyzing files that have <code>'-include_lib()'</code> directives).</item> <dt><code>--output_plt file</code></dt> <item>Store the plt at the specified file after building it.</item> <dt><code>--plt plt</code></dt> <item>Use the specified plt as the initial plt (if the plt was built during setup the files will be checked for consistency).</item> <dt><code>--plts plt*</code></dt> <item>Merge the specified plts to create the initial plt -- requires that the plts are disjoint (i.e., do not have any module appearing in more than one plt). The plts are created in the usual way: <pre class="sh_erlang"> dialyzer --build_plt --output_plt plt_1 files_to_include ... dialyzer --build_plt --output_plt plt_n files_to_include </pre> and then can be used in either of the following ways: <pre class="sh_erlang"> dialyzer files_to_analyze --plts plt_1 ... plt_n </pre> or: <pre class="sh_erlang"> dialyzer --plts plt_1 ... plt_n -- files_to_analyze </pre> (Note the -- delimiter in the second case)</item> <dt><code>-Wwarn</code></dt> <item>A family of options which selectively turn on/off warnings (for help on the names of warnings use <code>dialyzer -Whelp</code>).</item> <dt><code>--shell</code></dt> <item>Do not disable the Erlang shell while running the GUI.</item> <dt><code>--version</code> (or <code>-v</code>)</dt> <item>Print the Dialyzer version and some more information and exit.</item> <dt><code>--help</code> (or <code>-h</code>)</dt> <item>Print this message and exit.</item> <dt><code>--quiet</code> (or <code>-q</code>)</dt> <item>Make Dialyzer a bit more quiet.</item> <dt><code>--verbose</code></dt> <item>Make Dialyzer a bit more verbose.</item> <dt><code>--statistics</code></dt> <item>Prints information about the progress of execution (analysis phases, time spent in each and size of the relative input).</item> <dt><code>--build_plt</code></dt> <item>The analysis starts from an empty plt and creates a new one from the files specified with <code>-c</code> and <code>-r</code>. Only works for beam files. Use <code>--plt</code> or <code>--output_plt</code> to override the default plt location.</item> <dt><code>--add_to_plt</code></dt> <item>The plt is extended to also include the files specified with <code>-c</code> and <code>-r</code>. Use <code>--plt</code> to specify which plt to start from, and <code>--output_plt</code> to specify where to put the plt. Note that the analysis might include files from the plt if they depend on the new files. This option only works with beam files.</item> <dt><code>--remove_from_plt</code></dt> <item>The information from the files specified with <code>-c</code> and <code>-r</code> is removed from the plt. Note that this may cause a re-analysis of the remaining dependent files.</item> <dt><code>--check_plt</code></dt> <item>Check the plt for consistency and rebuild it if it is not up-to-date.</item> <dt><code>--no_check_plt</code></dt> <item>Skip the plt check when running Dialyzer. Useful when working with installed plts that never change.</item> <dt><code>--plt_info</code></dt> <item>Make Dialyzer print information about the plt and then quit. The plt can be specified with <code>--plt(s)</code>.</item> <dt><code>--get_warnings</code></dt> <item>Make Dialyzer emit warnings even when manipulating the plt. Warnings are only emitted for files that are actually analyzed.</item> <dt><code>--dump_callgraph file</code></dt> <item>Dump the call graph into the specified file whose format is determined by the file name extension. Supported extensions are: raw, dot, and ps. If something else is used as file name extension, default format '.raw' will be used.</item> <dt><code>--no_native</code> (or <code>-nn</code>)</dt> <item>Bypass the native code compilation of some key files that Dialyzer heuristically performs when dialyzing many files; this avoids the compilation time but it may result in (much) longer analysis time.</item> <dt><code>--fullpath</code></dt> <item>Display the full path names of files for which warnings are emitted.</item> <dt><code>--gui</code></dt> <item>Use the GUI.</item> </taglist> <div class="note"><h2>Note!</h2> <p>* denotes that multiple occurrences of these options are possible.</p> <p>** options <code>-D</code> and <code>-I</code> work both from command-line and in the Dialyzer GUI; the syntax of defines and includes is the same as that used by <code>erlc</code>.</p> </div> <p>Warning options:</p> <taglist> <dt><code>-Wno_return</code></dt> <item>Suppress warnings for functions that will never return a value.</item> <dt><code>-Wno_unused</code></dt> <item>Suppress warnings for unused functions.</item> <dt><code>-Wno_improper_lists</code></dt> <item>Suppress warnings for construction of improper lists.</item> <dt><code>-Wno_tuple_as_fun</code></dt> <item>Suppress warnings for using tuples instead of funs.</item> <dt><code>-Wno_fun_app</code></dt> <item>Suppress warnings for fun applications that will fail.</item> <dt><code>-Wno_match</code></dt> <item>Suppress warnings for patterns that are unused or cannot match.</item> <dt><code>-Wno_opaque</code></dt> <item>Suppress warnings for violations of opaqueness of data types.</item> <dt><code>-Wno_behaviours</code>***</dt> <item>Suppress warnings about behaviour callbacks which drift from the published recommended interfaces.</item> <dt><code>-Wunmatched_returns</code>***</dt> <item>Include warnings for function calls which ignore a structured return value or do not match against one of many possible return value(s).</item> <dt><code>-Werror_handling</code>***</dt> <item>Include warnings for functions that only return by means of an exception.</item> <dt><code>-Wrace_conditions</code>***</dt> <item>Include warnings for possible race conditions.</item> <dt><code>-Wunderspecs</code>***</dt> <item>Warn about underspecified functions (the -spec is strictly more allowing than the success typing).</item> </taglist> <p>The following options are also available but their use is not recommended: (they are mostly for Dialyzer developers and internal debugging)</p> <taglist> <dt><code>-Woverspecs</code>***</dt> <item>Warn about overspecified functions (the -spec is strictly less allowing than the success typing).</item> <dt><code>-Wspecdiffs</code>***</dt> <item>Warn when the -spec is different than the success typing.</item> </taglist> <div class="note"><h2>Note!</h2> <p>*** Identifies options that turn on warnings rather than turning them off.</p> </div> </div> <div class="section"> <h4>Using the Dialyzer
