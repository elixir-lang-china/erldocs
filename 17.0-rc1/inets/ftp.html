<h1>ftp</h1> <h2 class="modsummary">A File Transfer Protocol client</h2> <div class="description"> <p>The <code>ftp</code> module implements a client for file transfer according to a subset of the File Transfer Protocol (see RFC959). </p> <p>Starting from inets version 4.4.1 the ftp client will always try to use passive ftp mode and only resort to active ftp mode if this fails. There is a start option <a href="#mode" class="seealso">mode</a> where this default behavior may be changed. </p> <span id="two_start"> </span> <p>There are two ways to start an ftp client. One is using the <a href="#service_start" class="seealso">Inets service framework</a> and the other is to start it directy as a standalone process using the <a href="#open" class="seealso">open</a> function. </p> <p>For a simple example of an ftp session see <a href="ftp_client.html" class="seealso">Inets User's Guide.</a></p> <p>In addition to the ordinary functions for receiving and sending files (see <code>recv/2</code>, <code>recv/3</code>, <code>send/2</code> and <code>send/3</code>) there are functions for receiving remote files as binaries (see <code>recv_bin/2</code>) and for sending binaries to to be stored as remote files (see <code>send_bin/3</code>).</p> <p>There is also a set of functions for sending and receiving contiguous parts of a file to be stored in a remote file (for send see <code>send_chunk_start/2</code>, <code>send_chunk/2</code> and <code>send_chunk_end/1</code> and for receive see <code>recv_chunk_start/2</code> and <code>recv_chunk/</code>).</p> <p>The particular return values of the functions below depend very much on the implementation of the FTP server at the remote host. In particular the results from <code>ls</code> and <code>nlist</code> varies. Often real errors are not reported as errors by <code>ls</code>, even if for instance a file or directory does not exist. <code>nlist</code> is usually more strict, but some implementations have the peculiar behaviour of responding with an error, if the request is a listing of the contents of directory which exists but is empty.</p> <span id="service_start"> </span> </div> <div class="section"> <h4>FTP CLIENT SERVICE START/STOP </h4> <p>The FTP client can be started and stopped dynamically in runtime by calling the Inets application API <code>inets:start(ftpc, ServiceConfig)</code>, or <code>inets:start(ftpc, ServiceConfig, How)</code>, and <code>inets:stop(ftpc, Pid)</code>. See <a href="inets.html" class="seealso">inets(3)</a> for more info. </p> <p>Below follows a description of the available configuration options.</p> <taglist> <dt>{host, Host}</dt> <item> <span id="host"> </span> <p>Host = <code>string() | ip_address()</code> </p> </item> <dt>{port, Port}</dt> <item> <span id="port"> </span> <p>Port = <code>integer() &gt; 0</code> </p> <p>Default is 21.</p> </item> <dt>{mode, Mode}</dt> <item> <span id="mode"> </span> <p>Mode = <code>active | passive</code> </p> <p>Default is <code>passive</code>. </p> </item> <dt>{verbose, Verbose}</dt> <item> <span id="verbose"> </span> <p>Verbose = <code>boolean()</code> </p> <p>This determines if the FTP communication should be verbose or not. </p> <p>Default is <code>false</code>. </p> </item> <dt>{debug, Debug}</dt> <item> <span id="debug"> </span> <p>Debug = <code>trace | debug | disable</code> </p> <p>Debugging using the dbg toolkit. </p> <p>Default is <code>disable</code>. </p> </item> <dt>{ipfamily, IpFamily}</dt> <item> <span id="ipfamily"> </span> <p>IpFamily = <code>inet | inet6 | inet6fb4</code> </p> <p>With <code>inet6fb4</code> the client behaves as before (it tries to use IPv6 and only if that does not work, it uses IPv4). </p> <p>Default is <code>inet</code> (IPv4). </p> </item> <dt>{timeout, Timeout}</dt> <item> <span id="timeout"> </span> <p>Timeout = <code>non_neg_integer()</code> </p> <p>Connection timeout. </p> <p>Default is 60000 (milliseconds). </p> </item> <dt>{dtimeout, DTimeout}</dt> <item> <span id="dtimeout"> </span> <p>DTimeout = <code>non_neg_integer() | infinity</code> </p> <p>Data Connect timeout. The time the client will wait for the server to connect to the data socket. </p> <p>Default is infinity. </p> </item> <dt>{progress, Progress}</dt> <item> <span id="progress"> </span> <p>Progress = <code>ignore | {CBModule, CBFunction, InitProgress}</code></p> <p>CBModule = <code>atom()</code>, CBFunction = <code>atom()</code> </p> <p>InitProgress = <code>term()</code> </p> <p>Default is <code>ignore</code>. </p> </item> </taglist> <p>The progress option is intended to be used by applications that want to create some type of progress report such as a progress bar in a GUI. The default value for the progress option is ignore e.i. the option is not used. When the progress option is specified the following will happen when ftp:send/[3,4] or ftp:recv/[3,4] are called.</p> <list type="bulleted"> <item> <p>Before a file is transfered the following call will be made to indicate the start of the file transfer and how big the file is. The return value of the callback function should be a new value for the UserProgressTerm that will bu used as input next time the callback function is called.</p> <br /> <p><code> CBModule:CBFunction(InitProgress, File, {file_size, FileSize}) </code></p> <br /> </item> <item> <p>Every time a chunk of bytes is transfered the following call will be made:</p> <br /> <p><code> CBModule:CBFunction(UserProgressTerm, File, {transfer_size, TransferSize}) </code></p> <br /> </item> <item> <p>At the end of the file the following call will be made to indicate the end of the transfer.</p> <br /> <p><code> CBModule:CBFunction(UserProgressTerm, File, {transfer_size, 0}) </code></p> <br /> </item> </list> <p>The callback function should be defined as </p> <p><code> CBModule:CBFunction(UserProgressTerm, File, Size) -&gt; UserProgressTerm </code></p> <p><code> CBModule = CBFunction = atom() </code></p> <p><code> UserProgressTerm = term() </code></p> <p><code> File = string() </code></p> <p><code> Size = {transfer_size, integer()} | {file_size, integer()} | {file_size, unknown} </code></p> <p>Alas for remote files it is not possible for ftp to determine the file size in a platform independent way. In this case the size will be <code>unknown</code> and it is left to the application to find out the size. </p> <div class="note"><h2>Note!</h2> <p>The callback is made by a middleman process, hence the file transfer will not be affected by the code in the progress callback function. If the callback should crash this will be detected by the ftp connection process that will print an info-report and then go one as if the progress option was set to ignore. </p> </div> <p>The file transfer type is set to the default of the FTP server when the session is opened. This is usually ASCCI-mode. </p> <p>The current local working directory (cf. <code>lpwd/1</code>) is set to the value reported by <code>file:get_cwd/1</code>. the wanted local directory. </p> <p>The return value <code>Pid</code> is used as a reference to the newly created ftp client in all other functions, and they should be called by the process that created the connection.
