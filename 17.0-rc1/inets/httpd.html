<h1>httpd</h1> <h2 class="modsummary">An implementation of an HTTP 1.1 compliant Web server, as defined in RFC 2616. </h2> <div class="description"> <p>Documents the HTTP server start options, some administrative functions and also specifies the Erlang Web server callback API</p> </div> <div class="section"> <h4>COMMON DATA TYPES </h4> <p>Type definitions that are used more than once in this module:</p> <p><code>boolean() = true | false </code></p> <p><code>string() = list of ASCII characters</code></p> <p><code>path() = string() - representing a file or directory path.</code></p> <p><code> ip_address() = {N1,N2,N3,N4} % IPv4 | {K1,K2,K3,K4,K5,K6,K7,K8} % IPv6</code></p> <p><code>hostname() = string() - representing a host ex "foo.bar.com"</code></p> <p><code>property() = atom()</code></p> </div> <div class="section"> <h4>ERLANG HTTP SERVER SERVICE START/STOP </h4> <p>A web server can be configured to start when starting the inets application or started dynamically in runtime by calling the Inets application API <code>inets:start(httpd, ServiceConfig)</code>, or <code>inets:start(httpd, ServiceConfig, How)</code>, see <a href="inets.html" class="seealso">inets(3)</a> Below follows a description of the available configuration options, also called properties.</p> <span id="props_file"> </span> <p><em>File properties</em></p> <p>When the web server is started at application start time the properties should be fetched from a configuration file that could consist of a regular erlang property list, e.i. <code>[{Option, Value}] </code> where <code> Option = property() </code> and <code>Value = term()</code>, followed by a full stop, or for backwards compatibility an Apache like configuration file. If the web server is started dynamically at runtime you may still specify a file but you could also just specify the complete property list.</p> <taglist> <span id="prop_proplist_file"> </span> <dt>{proplist_file, path()}</dt> <item> <p>If this property is defined inets will expect to find all other properties defined in this file. Note that the file must include all properties listed under mandatory properties. </p> </item> <span id="prop_file"> </span> <dt>{file, path()}</dt> <item> <p>If this property is defined inets will expect to find all other properties defined in this file, that uses Apache like syntax. Note that the file must include all properties listed under mandatory properties. The Apache like syntax is the property, written as one word where each new word begins with a capital, followed by a white-space followed by the value followed by a new line. Ex: </p> <code> {server_root, "/urs/local/www"} -&gt; ServerRoot /usr/local/www </code> <p>With a few exceptions, that are documented for each property that behaves differently, and the special case {directory, {path(), PropertyList}} and {security_directory, {Dir, PropertyList}} that are represented as:</p> <pre class="sh_erlang"> &lt;Directory Dir&gt; &lt;Properties handled as described above&gt; &lt;/Directory&gt; </pre> </item> </taglist> <div class="note"><h2>Note!</h2> <p>The properties proplist_file and file are mutually exclusive.</p> </div> <span id="props_mand"> </span> <p><em>Mandatory properties</em></p> <taglist> <span id="prop_port"> </span> <dt>{port, integer()} </dt> <item> <p>The port that the HTTP server shall listen on. If zero is specified as port, an arbitrary available port will be picked and you can use the httpd:info/2 function to find out which port was picked. </p> </item> <span id="prop_server_name"> </span> <dt>{server_name, string()} </dt> <item> <p>The name of your server, normally a fully qualified domain name. </p> </item> <span id="prop_server_root"> </span> <dt>{server_root, path()} </dt> <item> <p>Defines the servers home directory where log files etc can be stored. Relative paths specified in other properties refer to this directory. </p> </item> <span id="prop_doc_root"> </span> <dt>{document_root, path()}</dt> <item> Defines the top directory for the documents that are available on the HTTP server. </item> </taglist> <span id="props_comm"> </span> <p><em>Communication properties</em> </p> <taglist> <span id="prop_bind_address"> </span> <dt>{bind_address, ip_address() | hostname() | any} </dt> <item> <p>Defaults to <code>any</code>. Note that <code>any</code> is denoted <em>*</em> in the apache like configuration file. </p> </item> <span id="prop_socket_type"> </span> <dt>{socket_type, ip_comm | {essl, Config::proplist()}}</dt> <item> <p> For ssl configuration options see <a href="../ssl/ssl.html#listen-2" class="seealso">ssl:listen/2</a> </p> <p>Defaults to <code>ip_comm</code>. </p> </item> <span id="prop_ipfamily"> </span> <dt>{ipfamily, inet | inet6 | inet6fb4}</dt> <item> <p>Defaults to <code>inet6fb4. </code> </p> <p>Note that this option is only used when the option <code>socket_type</code> has the value <code>ip_comm</code>. </p> </item> <span id="prop_minimum_bytes_per_second"> </span> <dt>{minimum_bytes_per_second, integer()}</dt> <item> <p>If given, sets a minimum bytes per second value for connections.</p> <p>If the value is not reached, the socket will close for that connection.</p> <p>The option is good for reducing the risk of "slow dos" attacks.</p> </item> </taglist> <span id="props_api_modules"> </span> <p><em>Erlang Web server API modules</em> </p> <taglist> <span id="prop_modules"> </span> <dt>{modules, [atom()]} </dt> <item> <p>Defines which modules the HTTP server will use to handle requests. Defaults to: <code>[mod_alias, mod_auth, mod_esi, mod_actions, mod_cgi, mod_dir, mod_get, mod_head, mod_log, mod_disk_log] </code> Note that some mod-modules are dependent on others, so the order can not be entirely arbitrary. See the <a href="http_server.html" class="seealso"> Inets Web server Modules in the Users guide</a> for more information. </p> </item> </taglist> <span id="props_limit"> </span> <p><em>Limit properties</em> </p> <taglist> <span id="prop_disable_chunked_encoding"> </span> <dt>{disable_chunked_transfer_encoding_send, boolean()}</dt> <item> <p>This property allows you to disable chunked transfer-encoding when sending a response to a HTTP/1.1 client, by default this is false. </p> </item> <span id="prop_keep_alive"> </span> <dt>{keep_alive, boolean()}</dt> <item> <p>Instructs the server whether or not to use persistent connections when the client claims to be HTTP/1.1 compliant, default is true. </p> </item> <span id="prop_keep_alive_timeout"> </span> <dt>{keep_alive_timeout, integer()}</dt> <item> <p>The number of seconds the server will wait for a subsequent request from the client before closing the connection. Default is 150. </p> </item> <span id="prop_max_body_size"> </span> <dt>{max_body_size, integer()}</dt> <item> <p>Limits the size of the message body of HTTP request. By the default there is no limit. </p> </item> <span id="prop_max_clients"> </span> <dt>{max_clients, integer()}</dt> <item> <p>Limits the number of simultaneous requests that can be supported. Defaults to 150. </p> </item> <span id="prop_max_header_size"> </span> <dt>{max_header_size, integer()}</dt> <item> <p>Limits the size of the message header of HTTP request. By default there is no limit. </p> </item> <span id="prop_max_keep_alive_req"> </span> <dt>{max_keep_alive_request, integer()}</dt> <item> <p>The number of request that a client can do on one connection. When the server has responded to the number of requests defined by max_keep_alive_requests the server close the connection. The server will close it even if there are queued request. Defaults to no limit. </p> </item> </taglist> <span id="props_admin"> </span> <p><em>Administrative properties</em></p> <taglist> <span id="prop_mime_types"> </span> <dt>{mime_types, [{MimeType, Extension}] | path()}</dt> <item> <p>Where MimeType = string() and Extension = string(). Files delivered to the client are MIME typed according to RFC 1590. File suffixes are mapped to MIME types before file delivery. The mapping between file suffixes and MIME types can be specified as an Apache like file as well as directly in the property list. Such a file may look like:</p> <pre class="sh_erlang"> # MIME type Extension text/html html htm text/plain asc txt </pre> <p>Defaults to [{"html","text/html"},{"htm","text/html"}]</p> </item> <span id="prop_mime_type"> </span> <dt>{mime_type, string()}</dt> <item> <p>When the server is asked to provide a document type which cannot be determined by the MIME Type Settings, the server will use this default type. </p> </item> <span id="prop_server_admin"> </span> <dt>{server_admin, string()}</dt> <item> <p>ServerAdmin defines the email-address of the server administrator, to be included in any error messages returned by the server. </p> </item> <span id="prop_server_tokens"> </span> <dt>{server_tokens, prod|major|minor|minimal|os|full|{private, string()}}</dt> <item> <p>ServerTokens defines how the value of the server header should look. </p> <p>Example: Assuming the version of inets is 5.8.1, here is what the server header string could look like for the different values of server-tokens: </p> <pre class="sh_erlang"> prod "inets" major "inets/5" minor "inets/5.8" minimal "inets/5.8.1" os "inets/5.8.1 (unix)" full "inets/5.8.1 (unix/linux) OTP/R15B" {private, "foo/bar"} "foo/bar" </pre> <p>By default, the value is as before, which is <code>minimal</code>. </p> </item> <span id="prop_log_format"> </span> <dt>{log_format, common | combined}</dt> <item> <p>Defines if access logs should be written according to the common log format or to the extended common log format. The <code>common</code> format is one line that looks like this: <code>remotehost rfc931 authuser [date] "request" status bytes</code></p> <pre class="sh_erlang"> remotehost Remote rfc931 The client's remote username (RFC 931). authuser The username with which the user authenticated himself. [date] Date and time of the request (RFC 1123). "request" The request line exactly as it came from the client (RFC 1945). status The HTTP status code returned to the client (RFC 1945). bytes The content-length of the document transferred. </pre> <p>The <code>combined</code> format is on line that look like this: <code>remotehost rfc931 authuser [date] "request" status bytes "referer" "user_agent" </code></p> <pre class="sh_erlang"> "referer" The url the client was on before requesting your url. (If it could not be determined a minus sign will be placed in this field) "user_agent" The software the client claims to be using. (If it could not be determined a minus sign will be placed in this field) </pre> <p>This affects the access logs written by mod_log and mod_disk_log. </p> </item> <span id="prop_elog_format"> </span> <dt>{error_log_format, pretty | compact}</dt> <item> <p>Defaults to pretty. If the error log is meant to be read directly by a human <code>pretty</code> will be the best option. <code>pretty</code> has the format corresponding to: </p> <code>io:format("[~s] ~s, reason: ~n ~p ~n~n", [Date, Msg, Reason]). </code> <p><code>compact</code> has the format corresponding to:</p> <code>io:format("[~s] ~s, reason: ~w ~n", [Date, Msg, Reason]). </code> <p>This affects the error logs written by mod_log and mod_disk_log. </p> </item> </taglist> <span id="props_alias"> </span> <p><em>URL aliasing properties - requires mod_alias</em></p> <taglist> <span id="prop_alias"> </span> <dt>{alias, {Alias, RealName}}</dt> <item> <p>Where Alias = string() and RealName = string(). The Alias property allows documents to be stored in the local file system instead of the document_root location. URLs with a path that begins with url-path is mapped to local files that begins with directory-filename, for example: <code>{alias, {"/image", "/ftp/pub/image"}}</code> and an access to http://your.server.org/image/foo.gif would refer to the file /ftp/pub/image/foo.gif. </p> </item> <span id="prop_re_write"> </span> <dt>{re_write, {Re, Replacement}}</dt> <item> <p>Where Re = string() and Replacement = string(). The ReWrite property allows documents to be stored in the local file system instead of the document_root location. URLs are rewritten by re:replace/3 to produce a path in the local filesystem. For example: <code>{re_write, {"^/[~]([^/]+)(.*)$", "/home/\\1/public\\2"}}</code> and an access to http://your.server.org/~bob/foo.gif would refer to the file /home/bob/public/foo.gif. In an Apache like configuration file the Re is separated from Replacement with one single space, and as expected backslashes do not need to be backslash escaped so the same example would become: <code>ReWrite ^/[~]([^/]+)(.*)$ /home/\1/public\2</code> Beware of trailing space in Replacement that will be used. If you must have a space in Re use e.g the character encoding <code>\040</code> see <a href="../stdlib/re.html" class="seealso">re(3)</a>. </p> </item> <span id="prop_dir_idx"> </span> <dt>{directory_index, [string()]}</dt> <item> <p>DirectoryIndex specifies a list of resources to look for if a client requests a directory using a / at the end of the directory name. file depicts the name of a file in the directory. Several files may be given, in which case the server will return the first it finds, for example: <code>{directory_index, ["index.hml", "welcome.html"]}</code> and access to http://your.server.org/docs/ would return http://your.server.org/docs/index.html or http://your.server.org/docs/welcome.html if index.html do not exist. </p> </item> </taglist> <span id="props_cgi"> </span> <p><em>CGI properties - requires mod_cgi</em></p> <taglist> <span id="prop_script_alias"> </span> <dt>{script_alias, {Alias, RealName}}</dt> <item> <p>Where Alias = string() and RealName = string(). Has the same behavior as the Alias property, except that it also marks the target directory as containing CGI scripts. URLs with a path beginning with url-path are mapped to scripts beginning with directory-filename, for example: <code>{script_alias, {"/cgi-bin/", "/web/cgi-bin/"}}</code> and an access to http://your.server.org/cgi-bin/foo would cause the server to run the script /web/cgi-bin/foo. </p> </item> <span id="prop_script_re_write"> </span> <dt>{script_re_write, {Re, Replacement}}</dt> <item> <p>Where Re = string() and Replacement = string(). URLs with a path beginning with url-path are mapped to scripts beginning with directory-filename, for example: <code>{script_re_write, {"^/cgi-bin/(\\d+)/", "/web/\\1/cgi-bin/"}}</code> and an access to http://your.server.org/cgi-bin/17/foo would cause the server to run the script /web/17/cgi-bin/foo. </p> </item> <span id="prop_script_nocache"> </span> <dt>{script_nocache, boolean()}</dt> <item> <p>If ScriptNoCache is set to true the HTTP server will by default add the header fields necessary to prevent proxies from caching the page. Generally this is something you want. Defaults to false. </p> </item> <span id="prop_script_timeout"> </span> <dt>{script_timeout, integer()}</dt> <item> <p>The time in seconds the web server will wait between each chunk of data from the script. Defaults to 15. </p> </item> <span id="prop_action"> </span> <dt>{action, {MimeType, CgiScript}} - requires mod_action</dt> <item> <p>Where MimeType = string() and CgiScript = string(). It propagates the URL and file path of the requested document using the standard CGI PATH_INFO and PATH_TRANSLATED environment variables. <code>{action, {"text/plain", "/cgi-bin/log_and_deliver_text"}}</code> </p> </item> <span id="prop_script"> </span> <dt>{script, {Method, CgiScript}} - requires mod_action</dt> <item> <p>Where Method = string() and CgiScript = string(). It propagates the URL and file path of the requested document using the standard CGI PATH_INFO and PATH_TRANSLATED environment variables. <code>{script, {"PUT", "/cgi-bin/put"}}</code> </p> </item> </taglist> <span id="props_esi"> </span> <p><em>ESI properties - requires mod_esi</em></p> <taglist> <span id="prop_esi_alias"> </span> <dt>{erl_script_alias, {URLPath, [AllowedModule]}}</dt> <item> <p>Where URLPath = string() and AllowedModule = atom(). erl_script_alias marks all URLs matching url-path as erl scheme scripts.
