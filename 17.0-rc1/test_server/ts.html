<h1>ts</h1> <h2 class="modsummary">Test Server Framework for testing OTP</h2> <div class="description"> <p>This is a framework for testing OTP. The <code>ts</code> module implements the interface to all the functionality in the framework. </p> <p>The framework is built on top of the Test Server Controller, <code>test_server_ctrl</code>, and provides a high level operator interface. The main features added by the framework are: </p> <list type="bulleted"> <item>Automatic compilation of test suites and data directories </item> <item>Collection of files in central directories and creation of additional HTML pages for better overview. </item> <item>Single command interface for running all available tests </item> <item>Spawns a new node with correct parameters before starting the test server </item> <item>Atomatically creates the parameter file needed when running tests on remote target </item> </list> <p>More information about the Test Server Framework and how to run test cases can be found in the Test Server User's Guide. </p> <p>For writing you own test server framework, please turn to the reference manual for the Test Server Controller and chapter named "Writing your own test server framework" in the Test Server User's Guide. </p> <p>SETUP</p> <p>To be able to run <code>ts</code>, you must first `install' <code>ts</code> for the current environment. This is done by calling <code>ts:install/0/1/2</code>. A file called `variables' is created and used by <code>ts</code> when running test suites. It is not recommended to edit this file, but it is possible to alter if <code>ts</code> gets the wrong idea about your environment. </p> <p><code>ts:install/0</code> is used if the target platform is the same as the controller host, i.e. if you run on "local target" and no options are needed. Then running <code>ts:install/0</code> <code>ts</code> will run an autoconf script for your current environment and set up the necessary variables needed by the test suites. </p> <p><code>ts:install/1</code> or <code>ts:install/2</code> is used if the target platform is different from the controller host, i.e. if you run on "remote target" or if special options are required for your system. </p> <p>See the reference manual for detailed information about <code>ts:install/0/1/2</code>. </p> <p>Some of the common variables in the 'variables' file are described below. Do not make any assumptions as of what is found in this file, as it may change at any time. </p> <list> <item><code>longnames</code><br /> Set to true if the system is using fully qualified nodenames. </item> <item><code>platform_id</code><br /> This is the currently installed platform identification string. </item> <item><code>platform_filename</code><br /> This is the name used to create the final save directory for test runs. </item> <item><code>platform_label</code><br /> This is the string presented in the generated test results index page. </item> <item><code>rsh_name</code><br /> This is the rsh program to use when starting slave or peer nodes on a remote host. </item> <item><code>erl_flags</code><br /> Compile time flags used when compiling test suites. </item> <item><code>erl_release</code><br /> The Erlang/OTP release being tested. </item> <item><code>'EMULATOR'</code><br /> The emulator being tested (e.g. beam) </item> <item><code>'CPU'</code><br /> The CPU in the machine running the tests, e.g. sparc. </item> <item><code>target_host</code><br /> The target host name </item> <item><code>os</code><br /> The target operating system, e.g. solaris2.8 </item> <item><code>target</code><br /> The current target platform, e.g. sparc-sun-solaris2.8 </item> </list> <p>RUNNING TESTS</p> <p>After installing <code>ts</code>, you can run your test with the <code>ts:run/0/1/2/3/4</code> functions. These functions, however, require a special directory structure to be able to find your test suites. Both the test server and all tests must be located under your $TESTROOT directory. The test server implementation shall be located in the directory <code>$TESTROOT/test_server</code> and for each application there must be a directory named <code>$TESTROOT/&lt;application&gt;_test</code> containing the .spec file and all test suites and data directories for the application. Note that there shall only be one .spec file for each application. </p> <p><code>$TESTROOT/test_server</code> must be the current directory when calling the <code>ts:run/*</code> function. </p> <p>All available tests can be found with <code>ts:tests()</code>. This will list all applications for which a test specification file <code>../&lt;application&gt;_test/&lt;application&gt;.spec</code> can be found. </p> <p>To run all these tests, use <code>ts:run()</code>. </p> <p>To run one or some of the tests, use <code>ts:run(Tests)</code>, where <code>Tests</code> is the name of the application you want to test, or a list of such names. </p> <p>To run one test suite within a test, use <code>ts:run(Test,Suite)</code>. </p> <p>To run one test case within a suite, use <code>ts:run(Test,Suite,Case)</code></p> <p>To all these functions, you can also add a list of options. Please turn to the reference manual for the <code>ts</code> module to see the valid options to use. </p> <p>The function <code>ts:help()</code> displays some simple help for the functions in <code>ts</code>. Use this for quick reference. </p> <p>LOG FILES</p> <p>As the execution of the test suites go on, events are logged in four different ways: </p> <list type="bulleted"> <item>Text to the operator's console.</item> <item>Suite related information is sent to the major log file.</item> <item>Case related information is sent to the minor log file.</item> <item>The HTML log file gets updated with test results.</item> </list> <p>Typically the operator, who may run hundreds or thousands of test cases, doesn't want to fill the screen with details about/from the specific test cases. By default, the operator will only see: </p> <list type="bulleted"> <item>A confirmation that the test has started. </item> <item>A small note about each failed test case. </item> <item>A summary of all the run test cases. </item> <item>A confirmation that the test run is complete </item> <item>Some special information like error reports and progress reports, printouts written with erlang:display/1 or io:format/3 specifically addressed to somewhere other than <code>standard_io</code>.</item> </list> <p>This is enough for the operator to know, and if he wants to dig in deeper into a specific test case result, he can do so by following the links in the HTML presentation to take a look in the major or minor log files. </p> <p>A detailed report of the entire test suite is stored in the major logfile, the exact reason for failure, time spent etc. </p> <p>The HTML log file is a summary of the major log file, but gives a much better overview of the test run. It also has links to every test case's log file for quick viewing with a HTML browser. </p> <p>The minor log file contain full details of every single test case, each one in a separate file. This way the files should be easy to compare with previous test runs, even if the set of test cases change. </p> <p>Which information that goes where is user configurable via the test server controller. Three threshold values determine what comes out on screen, and in the major or minor log files. The contents that goes to the HTML log file is fixed, and cannot be altered. </p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="install/0">install() -&gt; ok | {error, Reason}</h3> <h3 id="install/1">install(TargetSystem) -&gt; ok | {error, Reason}</h3> <h3 id="install/1-1">install(Opts) -&gt; ok | {error, Reason}</h3> <h3 id="install/2">install(TargetSystem,Opts) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>TargetSystem = {Architecture, TargetHost}</code></li> <li><code>Architecture = atom() or string()</code></li> <d>e.g. "ose" or "vxworks_ppc603"</d> <li><code>TargetHost = atom() or string()</code></li> <d>The name of the target host</d> <li><code>Opts = list()</code></li> </ul> <div class="description"> <p>Installs and configures the Test Server Framework for running test suites. If a remote host is to be used, the <code>TargetSystem</code> argument must be given so that "cross installation" can be done. Installation is required for any of the functions in <code>ts</code> to work. </p> <p>Opts may be one or more of </p> <list> <item><code>{longnames, Bool}</code><br /> Use fully qualified hostnames for test_server and slave nodes. Bool is <code>true</code> or <code>false</code> (default). </item> <item><code>{verbose, Level}</code><br /> Verbosity level for test server output, set to 0, 1 or 2, where 0 is quiet(default). </item> <item><code>{hosts, Hosts}</code><br /> This is a list of available hosts on which to start slave nodes. It is used when the <code>{remote, true}</code> option is given to the <code>test_server:start_node/3</code> function. Also, if <code>{require_nodenames, Num}</code> is contained in a test specification file, the generated nodenames will be spread over all hosts given in this <code>Hosts</code> list. The hostnames are given as atoms or strings. </item> <item><code>{slavetargets, SlaveTarges}</code><br /> For VxWorks only. This is a list of available hosts where slave nodes can be started. This is necessary because only one node can run per host in the VxWorks environment. This is not the same as <code>{hosts, Hosts}</code> because it is used for all slave nodes - not only the ones started with <code>{remote, true}</code>. The hostnames are given as atoms or strings. </item> <item><code>{crossroot, TargetErlRoot}</code><br /> Erlang root directory on target host <br /> This option is mandatory for remote targets </item> <item><code>{master, {MasterHost, MasterCookie}}</code><br /> If target is remote and the target node is started as a slave node, this option indicates which master and cookie to use. The given master will also be used as master for slave nodes started with <code>test_server:start_node/3</code>. It is expected that the <code>erl_boot_server</code> is started on the master node before the test is run. If this option is not given, the test server controller node is used as master and the <code>erl_boot_server</code> is automatically started. </item> <item><code>{erl_start_args, ArgString}</code><br /> Additional arguments to be used when starting the test server controller node. <code>ArgString</code> will be appended to the command line when starting the erlang node. Note that this will only affect the startup of the <em>controller node</em>, i.e. not the target node or any slave nodes startet from a test case. </item> <item><code>{ipv6_hosts, HostList}</code><br /> This option will be inserted in the <code>Config</code> parameter for each test case. <code>HostList</code> is a list of hosts supporting IPv6. </item> </list> </div> </div> <div class="function"> <h3 id="help/0">help() -&gt; ok</h3> <div class="description"> <p>Presents simple help on the functions in <code>ts</code>. Useful for quick reference.</p> </div> </div> <div class="function"> <h3 id="tests/0">tests() -&gt; Tests</h3> <div class="description"> <p>Returns the list of available tests. This is actually just a list of all test specification files found by looking up "../*_test/*.spec". </p> <p>In each ../Name_test/ directory there should be one test specification file named Name.spec.</p> </div> </div> <div class="function"> <h3 id="run/0">run() -&gt; ok | {error, Reason}</h3> <h3 id="run/1">run([all_tests|Opts])</h3> <h3 id="run/1-1">run(Specs)</h3> <h3 id="run/2">run(Specs, Opts)</h3> <h3 id="run/2-1">run(Spec, Module)</h3> <h3 id="run/3">run(Spec, Module, Opts)</h3> <h3 id="run/3-1">run(Spec, Module, Case)</h3> <h3 id="run/4">run(Spec, Module, Case, Opts)</h3> <ul class="type"> <li><code>Specs = Spec | [Spec]</code></li> <li><code>Spec = atom()</code></li> <li><code>Module = atom()</code></li> <li><code>Case = atom()</code></li> <li><code>Opts = [Opt]</code></li> <li><code>Opt = batch | verbose | {verbose, Level} | {vars, Vars} | keep_topcase | cover | cover_details |{cover,CoverFile} | {cover_details,CoverFile} | {trace, TraceSpec}</code></li> <li><code>Level = integer(); 0 means silent</code></li> <li><code>Vars = list() of key-value tuples</code></li> <li><code>CoverFile = string(); name of file listing modules to exclude from or include in cover compilation. The name must include full path to the file.</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>This function runs test suite(s)/case(s). To be able to run any tests, ts:install must first be called to create the <code>variables</code> file needed. To run a whole test specification, only specify the name of the test specification, and all test suite modules belonging to that test spec will be run. To run a single module in a test specification, use the Module argument to specify the name of the module to run and all test cases in that module will be run, and to run a specified test case, specify the name of the test case using the Case argument. If called with no argument, all test specifications available will be run. Use ts:tests/0 to see the available test specifications. </p> <p>If the <code>batch</code> option is not given, a new xterm is started (unix) when <code>ts:run</code> is called. </p> <p>The <code>verbose</code> option sets the verbosity level for test server output. This has the same effect as if given to <code>ts:install/1/2</code></p> <p>The <code>vars</code> option can be used for adding configuration variables that are not in the <code>variables</code> file generated during installation. Can be any of the <code>Opts</code> valid for <code>ts:install/1/2</code>. </p> <p>The <code>keep_topcase</code> option forces <code>ts</code> to keep the topcase in your test specification file as is. This option can only be used if you don't give the <code>Module</code> or <code>Case</code> parameters to <code>ts:run</code>. The <code>keep_topcase</code> option is necessary if your topcase contains anything other than <code>{dir,"../&lt;Name&gt;_test"}</code>. If the option is not used, <code>ts</code> will modify your topcase. </p> <p>The <code>cover</code> and <code>cover_details</code> options indicates that the test shall be run with code coverage analysis. <code>cover_details</code> means that analysis shall be done on the most detailed level. If the test is run with a remote target, this option creates a list of uncovered lines in each cover compiled module. If the test is run with a local target, each cover compiled module will be analysed with <code>cover:analyse_to_file/1</code>. The <code>cover</code> options will only create an overview of all cover compiled modules with the number of covered and not covered lines. </p> <p>The <code>CoverFile</code> which can be given with the <code>cover</code> and <code>cover_details</code> options must be the filename of a file listing modules to be excluded from or included in the cover compilation. By default, <code>ts</code> believes that <code>Spec</code> is the name of an OTP application and that all modules in this application shall be cover compiled. The <code>CoverFile</code> can exclude modules that belong to the application and add modules that don't belong to the application. The file can have the following entries:</p> <pre class="sh_erlang"> {exclude, all | ExcludeModuleList}. {include, IncludeModuleList}. </pre> <p>Note that each line must end with a full stop. <code>ExcludeModuleList</code> and <code>IncludeModuleList</code> are lists of atoms, where each atom is a module name. </p> <p>If the <code>cover</code> or <code>cover_details</code> options are given on their own, the directory <code>../&lt;Spec&gt;_test</code> is searched for a <code>CoverFile</code> named <code>&lt;Spec&gt;.cover</code>. If this file is not found, <code>Spec</code> is assumed to be the name of an OTP application, and all modules in the <code>ebin</code> directory for the application are cover compiled. The <code>ebin</code> directory is found by adding <code>ebin</code> to <code>code:lib_dir(Spec)</code>. </p> <p>The same cover compiled code will be loaded on all slave or peer nodes started with <code>test_server:start_node/3</code>. The exception is nodes that run an old version of OTP. If the loading fails, the node will simply not be a part of the coverage analysis. Note that slave and peer nodes must be stopped with <code>test_server:stop_node/1</code> for the node to be part of the coverage analysis, else the test server will not be able to fetch coverage data from the node. </p> <p>The <code>trace</code> option is used to turn on call trace on target and on slave or peer nodes started with <code>test_server:start_node/3</code>. <code>TraceSpec</code> can be the name of a trace information file, or a list of elements like the ones in a trace information file. Please turn to the reference manual for <code>test_server_ctrl:trc/1</code> for details about the trace information file. </p> </div> </div> <div class="function"> <h3 id="cross_cover_analyse/1">cross_cover_analyse(Level) -&gt; ok</h3> <h3 id="cross_cover_analyse/1-1">cross_cover_analyse([Level]) -&gt; ok</h3> <div class="description"> <p>Analyse cover data collected from all tests. </p> <p>See test_server_ctrl:cross_cover_analyse/2 </p> </div> </div> <div class="function"> <h3 id="r/0">r() -&gt; ok</h3> <h3 id="r/1">r(Opts) -&gt; ok</h3> <h3 id="r/1-1">r(SpecOrSuite) -&gt; ok</h3> <h3 id="r/2">r(SpecOrSuite,Opts) -&gt; ok</h3> <h3 id="r/2-1">r(Suite,Case) -&gt; ok</h3> <h3 id="r/3">r(Suite,Case,Opts) -&gt; ok</h3> <ul class="type"> <li><code>SpecOrSuite = Spec | Suite</code></li> <li><code>Spec = string()</code></li> <d>"Name.spec" or "Name.spec.OsType", where OsType is vxworks</d> <li><code>Suite = atom()</code></li> <li><code>Case = atom()</code></li> <li><code>Opts = [Opt]</code></li> <li><code>Opt = {Cover,AppOrCoverFile} | {Cover,Application,CoverFile}</code></li> <li><code>Cover = cover | cover_details</code></li> <li><code>AppOrCoverFile = Application | CoverFile</code></li> <li><code>Application = atom()</code></li> <d>OTP application to cover compile</d> <li><code>CoverFile = string()</code></li> <d>Name of file listing modules to exclude from or include in cover compilation</d> </ul> <div class="description"> <p>This function can be used to run a test suites or test cases directly, without any of the additional features added by the test server framework. It is a shortcut for running the test suite <code>estone_SUITE</code> in the <code>kernel</code> application. </p> <p><code>Opts</code> is the same as the <code>Opts</code> argument for the <code>ts:run</code> functions.</p> </div> </div> </div> <div class="section"> <h4>Makfile.src in Data Directory</h4> <p>If a data directory contains code which must be compiled before the test suite is run, a makefile source called <code>Makefile.src</code> can be placed in the data directory. This file will be converted to a valid makefile by <code>ts:run/0/1/2/3/4</code>. </p> <p>The reason for generating the makefile is that you can use variables from the <code>variables</code> file which was generated by <code>ts:install/0/1/2</code>. All occurrences of <code>@Key@</code> in <code>Makefile.src</code> is substituted by the <code>Value</code> from <code>{Key,Value}</code> found in the <code>variables</code> file. Example: </p> <p>Cut from <code>variables</code>:</p> <pre class="sh_erlang"> ... {'EMULATOR',"beam"}. {'CFLAGS',"-g -O2"}. {'LD',"$(CC) $(CFLAGS)"}. {'CC',"gcc"}. ... </pre> <p><code>Makefile.src</code> for compiling erlang code could look something like this:</p> <pre class="sh_erlang"> EFLAGS=+debug_info all: ordsets1.@EMULATOR@ ordsets1.@EMULATOR@: ordsets1.erl erlc $(EFLAGS) ordsets1.erl </pre> <p><code>Makefile.src</code> for compiling c code could look something like this:</p> <pre class="sh_erlang"> CC = @CC@ LD = @LD@ CFLAGS = @CFLAGS@ -I@erl_include@ @DEFS@ CROSSLDFLAGS = @CROSSLDFLAGS@ PROGS = nfs_check@exe@ all: $(PROGS) nfs_check@exe@: nfs_check@obj@ $(LD) $(CROSSLDFLAGS) -o nfs_check nfs_check@obj@ @LIBS@ nfs_check@obj@: nfs_check.c $(CC) -c -o nfs_check@obj@ $(CFLAGS) nfs_check.c </pre> </div>
