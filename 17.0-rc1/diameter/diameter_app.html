<h1>diameter_app</h1> <h2 class="modsummary"> Callback module of a Diameter application.</h2> <div class="description"> <p> A diameter service as started by <a href="diameter.html#start_service-2" class="seealso">diameter:start_service/2</a> configures one of more Diameter applications, each of whose configuration specifies a callback that handles messages specific to the application. The messages and AVPs of the application are defined in a dictionary file whose format is documented in <a href="diameter_dict.html" class="seealso">diameter_dict(4)</a> while the callback module is documented here. The callback module implements the Diameter application-specific functionality of a service.</p> <p> A callback module must export all of the functions documented below. The functions themselves are of three distinct flavours:</p> <list> <item> <p> <a href="../#Mod/peer_up-3.html" class="seealso">peer_up/3</a> and <a href="../#Mod/peer_down-3.html" class="seealso">peer_up/3</a> signal the attainment or loss of connectivity with a Diameter peer.</p> </item> <item> <p> <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a>, <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a>, <a href="../#Mod/prepare_retransmit-3.html" class="seealso">prepare_retransmit/3</a>, <a href="../#Mod/handle_answer-4.html" class="seealso">handle_answer/4</a> and <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> are (or may be) called as a consequence of a call to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> to send an outgoing Diameter request message.</p> </item> <item> <p> <a href="../#Mod/handle_request-3.html" class="seealso">handle_request/3</a> is called in response to an incoming Diameter request message.</p> </item> </list> </div> <div class="note"><h2>Note!</h2> <p> The arities given for the the callback functions here assume no extra arguments. All functions will also be passed any extra arguments configured with the callback module itself when calling <a href="diameter.html#start_service-2" class="seealso">diameter:start_service/2</a> and, for the call-specific callbacks, any extra arguments passed to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a>.</p> </div> <div class="section"> <h4>DATA TYPES</h4> <taglist> <span id="capabilities"> </span> <dt><code>capabilities() = #diameter_caps{}</code></dt> <item> <p> A record containing the identities of the local Diameter node and the remote Diameter peer having an established transport connection, as well as the capabilities as determined by capabilities exchange. Each field of the record is a 2-tuple consisting of values for the (local) host and (remote) peer. Optional or possibly multiple values are encoded as lists of values, mandatory values as the bare value.</p> </item> <span id="message"> </span> <dt><code>message() = <a href="diameter_codec.html#message" class="seealso">diameter_codec:message()</a></code></dt> <item> <p> The representation of a Diameter message as passed to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> or returned from a <a href="../#Mod/handle_request-3.html" class="seealso">handle_request/3</a> callback.</p> </item> <span id="packet"> </span> <dt><code>packet() = <a href="diameter_codec.html#packet" class="seealso">diameter_codec:packet()</a></code></dt> <item> <p> A container for incoming and outgoing Diameter messages that's passed through encode/decode and transport. Fields should not be set in return values except as documented.</p> </item> <span id="peer_ref"> </span> <dt><code>peer_ref() = term()</code></dt> <item> <p> A term identifying a transport connection with a Diameter peer.</p> </item> <span id="peer"> </span> <dt><code>peer() = {<a href="#peer_ref" class="seealso">peer_ref()</a>, <a href="#capabilities" class="seealso">capabilities()</a>}</code></dt> <item> <p> A tuple representing a Diameter peer connection.</p> </item> <span id="state"> </span> <dt><code>state() = term()</code></dt> <item> <p> The state maintained by the application callback functions <a href="../#Mod/peer_up-3.html" class="seealso">peer_up/3</a>, <a href="../#Mod/peer_down-3.html" class="seealso">peer_up/3</a> and (optionally) <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a>. The initial state is configured in the call to <a href="diameter.html#start_service-2" class="seealso">diameter:start_service/2</a> that configures the application on a service. Callback functions returning a state are evaluated in a common service-specific process while those not returning state are evaluated in a request-specific process.</p> </item> </taglist> <span id="peer_up"> </span> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="peer_up/3">Mod:peer_up(SvcName, Peer, State) -&gt; NewState</h3> <ul class="type"> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>State = NewState = <a href="#state" class="seealso">state()</a></code></li> </ul> <div class="description"> <p> Invoked to signal the availability of a peer connection on the local Erlang node. In particular, capabilities exchange with the peer has indicated support for the application in question, the RFC 3539 watchdog state machine for the connection has reached state <code>OKAY</code> and Diameter messages can be both sent and received.</p> <div class="note"><h2>Note!</h2> <p> A watchdog state machine can reach state <code>OKAY</code> from state <code>SUSPECT</code> without a new capabilities exchange taking place. A new transport connection (and capabilities exchange) results in a new peer_ref().</p> </div> <div class="note"><h2>Note!</h2> <p> There is no requirement that a callback return before incoming requests are received: <a href="../#Mod/handle_request-3.html" class="seealso">handle_request/3</a> callbacks must be handled independently of <a href="../#Mod/peer_up-3.html" class="seealso">peer_up/3</a> and <a href="../#Mod/peer_down-3.html" class="seealso">peer_up/3</a>.</p> </div> </div> </div> <div class="function"> <h3 id="peer_down/3">Mod:peer_down(SvcName, Peer, State) -&gt; NewState</h3> <ul class="type"> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>State = NewState = <a href="#state" class="seealso">state()</a></code></li> </ul> <div class="description"> <p> Invoked to signal that a peer connection on the local Erlang node is no longer available following a previous call to <a href="../#Mod/peer_up-3.html" class="seealso">peer_up/3</a>. In particular, that the RFC 3539 watchdog state machine for the connection has left state <code>OKAY</code> and the peer will no longer be a candidate in <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a> callbacks.</p> </div> </div> <div class="function"> <h3 id="pick_peer/4">Mod:pick_peer(LocalCandidates, RemoteCandidates, SvcName, State) -&gt; Selection | false</h3> <ul class="type"> <li><code>LocalCandidates = RemoteCandidates = [<a href="#peer" class="seealso">peer()</a>]</code></li> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>State = NewState = <a href="#state" class="seealso">state()</a></code></li> <li><code>Selection = {ok, Peer} | {Peer, NewState}</code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a> | false</code></li> </ul> <div class="description"> <p> Invoked as a consequence of a call to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> to select a destination peer for an outgoing request. The return value indicates the selected peer.</p> <p> The candidate lists contain only those peers that have advertised support for the Diameter application in question during capabilities exchange, that have not be excluded by a <code>filter</code> option in the call to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> and whose watchdog state machine is in the <code>OKAY</code> state. The order of the elements is unspecified except that any peers whose Origin-Host and Origin-Realm matches that of the outgoing request (in the sense of a <code>{filter, {all, [host, realm]}}</code> option to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a>) will be placed at the head of the list. <code>LocalCandidates</code> contains peers whose transport process resides on the local Erlang node while <code>RemoteCandidates</code> contains peers that have been communicated from other nodes by services of the same name.</p> <p> A callback that returns a peer() will be followed by a <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a> callback and, if the latter indicates that the request should be sent, by either <a href="../#Mod/handle_answer-4.html" class="seealso">handle_answer/4</a> or <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> depending on whether or not an answer message is received from the peer. If the transport becomes unavailable after <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a> then a new <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a> callback may take place to failover to an alternate peer, after which <a href="../#Mod/prepare_retransmit-3.html" class="seealso">prepare_retransmit/3</a> takes the place of <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a> in resending the request. There is no guarantee that a <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a> callback to select an alternate peer will be followed by any additional callbacks since a retransmission to an alternate peer is abandoned if an answer is received from a previously selected peer.</p> <p> The return values <code>false</code> and <code>{false, State}</code> (that is, <code>NewState = State</code>) are equivalent, as are <code>{ok, Peer}</code> and <code>{Peer, State}</code>.</p> <div class="note"><h2>Note!</h2> <p> The <a href="diameter.html#service_opt" class="seealso">diameter:service_opt()</a> <code>use_shared_peers</code> determines whether or not a service uses peers shared from other nodes. If not then <code>RemoteCandidates</code> is the empty list.</p> </div> <div class="warning"><h2>Warning!</h2> <p> The return value <code>{Peer, NewState}</code> is only allowed if the Diameter application in question was configured with the <a href="diameter.html#application_opt" class="seealso">diameter:application_opt()</a> <code>{call_mutates_state, true}</code>. Otherwise, the <code>State</code> argument is always the initial value as configured on the application, not any subsequent value returned by a <a href="../#Mod/peer_up-3.html" class="seealso">peer_up/3</a> or <a href="../#Mod/peer_down-3.html" class="seealso">peer_up/3</a> callback.</p> </div> </div> </div> <div class="function"> <h3 id="prepare_request/3">Mod:prepare_request(Packet, SvcName, Peer) -&gt; Action</h3> <ul class="type"> <li><code>Packet = <a href="#packet" class="seealso">packet()</a></code></li> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>Action = Send | Discard | {eval_packet, Action, PostF}</code></li> <li><code>Send = {send, <a href="#packet" class="seealso">packet()</a> | <a href="#message" class="seealso">message()</a>}</code></li> <li><code>Discard = {discard, Reason} | discard</code></li> <li><code>PostF = <a href="diameter.html#evaluable" class="seealso">diameter:evaluable()</a>}</code></li> </ul> <div class="description"> <p> Invoked to return a request for encoding and transport. Allows the sender to use the selected peer's capabilities to modify the outgoing request. Many implementations may simply want to return <code>{send, Packet}</code></p> <p> A returned <a href="#packet" class="seealso">packet()</a> should set the request to be encoded in its <code>msg</code> field and can set the <code>transport_data</code> field in order to pass information to the transport process. Extra arguments passed to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> can be used to communicate transport (or any other) data to the callback.</p> <p> A returned <a href="#packet" class="seealso">packet()</a> can set the <code>header</code> field to a <code>#diameter_header{}</code> to specify values that should be preserved in the outgoing request, values otherwise being those in the header record contained in <code>Packet</code>. A returned <code>length</code>, <code>cmd_code</code> or <code>application_id</code> is ignored.</p> <p> A returned <code>PostF</code> will be evaluated on any encoded <code>#diameter_packet{}</code> prior to transmission, the <code>bin</code> field containing the encoded binary. The return value is ignored.</p> <p> Returning <code>{discard, Reason}</code> causes the request to be aborted and the <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> for which the callback has taken place to return <code>{error, Reason}</code>. Returning <code>discard</code> is equivalent to returning <code>{discard, discarded}</code>.</p> </div> </div> <div class="function"> <h3 id="prepare_retransmit/3">Mod:prepare_retransmit(Packet, SvcName, Peer) -&gt; Action</h3> <ul class="type"> <li><code>Packet = <a href="#packet" class="seealso">packet()</a></code></li> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>Action = Send | Discard | {eval_packet, Action, PostF}</code></li> <li><code>Send = {send, <a href="#packet" class="seealso">packet()</a> | <a href="#message" class="seealso">message()</a>}</code></li> <li><code>Discard = {discard, Reason} | discard</code></li> <li><code>PostF = <a href="diameter.html#evaluable" class="seealso">diameter:evaluable()</a>}</code></li> </ul> <div class="description"> <p> Invoked to return a request for encoding and retransmission. Has the same role as <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a> in the case that a peer connection is lost an an alternate peer selected but the argument <a href="#packet" class="seealso">packet()</a> is as returned by the initial <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a>.</p> <p> Returning <code>{discard, Reason}</code> causes the request to be aborted and a <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> callback to take place with <code>Reason</code> as initial argument. Returning <code>discard</code> is equivalent to returning <code>{discard, discarded}</code>.</p> </div> </div> <div class="function"> <h3 id="handle_answer/4">Mod:handle_answer(Packet, Request, SvcName, Peer) -&gt; Result</h3> <ul class="type"> <li><code>Packet = <a href="#packet" class="seealso">packet()</a></code></li> <li><code>Request = <a href="#message" class="seealso">message()</a></code></li> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>Result = term()</code></li> </ul> <div class="description"> <p> Invoked when an answer message is received from a peer. The return value is returned from <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> unless the <code>detach</code> option was specified.</p> <p> The decoded answer record and undecoded binary are in the <code>msg</code> and <code>bin</code> fields of the argument <a href="#packet" class="seealso">packet()</a> respectively. <code>Request</code> is the outgoing request message as was returned from <a href="../#Mod/prepare_request-3.html" class="seealso">prepare_request/3</a> or <a href="../#Mod/prepare_retransmit-3.html" class="seealso">prepare_retransmit/3</a>.</p> <p> For any given call to <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> there is at most one <a href="../#Mod/handle_answer-4.html" class="seealso">handle_answer/4</a> callback: any duplicate answer (due to retransmission or otherwise) is discarded. Similarly, only one of <a href="../#Mod/handle_answer-4.html" class="seealso">handle_answer/4</a> or <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> is called.</p> <p> By default, an incoming answer message that cannot be successfully decoded causes the request process to fail, causing <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> to return <code>{error, failure}</code> unless the <code>detach</code> option was specified. In particular, there is no <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> callback in this case. The <a href="diameter.html#application_opt" class="seealso">diameter:application_opt()</a> <code>answer_errors</code> can be set to change this behaviour.</p> </div> </div> <div class="function"> <h3 id="handle_error/4">Mod:handle_error(Reason, Request, SvcName, Peer) -&gt; Result</h3> <ul class="type"> <li><code>Reason = timeout | failover | term()</code></li> <li><code>Request = <a href="#message" class="seealso">message()</a></code></li> <li><code>SvcName = <a href="diameter.html#service_name" class="seealso">diameter:service_name()</a></code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>Result = term()</code></li> </ul> <div class="description"> <p> Invoked when an error occurs before an answer message is received in response to an outgoing request. The return value is returned from <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> unless the <code>detach</code> option was specified.</p> <p> Reason <code>timeout</code> indicates that an answer message has not been received within the time specified with the corresponding <a href="diameter.html#call_opt" class="seealso">diameter:call_opt()</a>. Reason <code>failover</code> indicates that the transport connection to the peer to which the request has been sent has become unavailable and that not alternate peer was not selected.</p> </div> </div> <div class="function"> <h3 id="handle_request/3">Mod:handle_request(Packet, SvcName, Peer) -&gt; Action</h3> <ul class="type"> <li><code>Packet = <a href="#packet" class="seealso">packet()</a></code></li> <li><code>SvcName = term()</code></li> <li><code>Peer = <a href="#peer" class="seealso">peer()</a></code></li> <li><code>Action = Reply | {relay, [Opt]} | discard | {eval|eval_packet, Action, PostF}</code></li> <li><code>Reply = {reply, <a href="#packet" class="seealso">packet()</a> | <a href="#message" class="seealso">message()</a>} | {answer_message, 3000..3999|5000..5999} | {protocol_error, 3000..3999}</code></li> <li><code>Opt = <a href="diameter.html#call_opt" class="seealso">diameter:call_opt()</a></code></li> <li><code>PostF = <a href="diameter.html#evaluable" class="seealso">diameter:evaluable()</a></code></li> </ul> <div class="description"> <p> Invoked when a request message is received from a peer. The application in which the callback takes place (that is, the callback module as configured with <a href="diameter.html#start_service-2" class="seealso">diameter:start_service/2</a>) is determined by the Application Identifier in the header of the incoming request message, the selected module being the one whose corresponding dictionary declares itself as defining either the application in question or the Relay application.</p> <p> The argument <a href="#packet" class="seealso">packet()</a> has the following signature.</p> <pre class="sh_erlang"> #diameter_packet{header = #diameter_header{}, avps = [#diameter_avp{}], msg = record() | undefined, errors = [<a href="diameter_dict.html#DATA_TYPES" class="seealso">Unsigned32()</a> | {<a href="diameter_dict.html#DATA_TYPES" class="seealso">Unsigned32()</a>, #diameter_avp{}}], bin = binary(), transport_data = term()} </pre> <p> The <code>msg</code> field will be <code>undefined</code> in case the request has been received in the relay application. Otherwise it contains the record representing the request as outlined in <a href="diameter_dict.html#MESSAGE_RECORDS" class="seealso">diameter_dict(4)</a>.</p> <p> The <code>errors</code> field specifies any results codes identifying errors found while decoding the request. This is used to set Result-Code and/or Failed-AVP in a returned answer unless the callback returns a <code>#diameter_packet{}</code> whose <code>errors</code> field is set to either a non-empty list of its own, in which case this list is used instead, or the atom <code>false</code> to disable any setting of Result-Code and Failed-AVP. Note that the errors detected by diameter are of the 3xxx and 5xxx series, Protocol Errors and Permanent Failures respectively. The <code>errors</code> list is empty if the request has been received in the relay application.</p> <p> The <code>transport_data</code> field contains an arbitrary term passed into diameter from the transport module in question, or the atom <code>undefined</code> if the transport specified no data. The term is preserved if a <a href="#message" class="seealso">message()</a> is returned but must be set explicitly in a returned <a href="#packet" class="seealso">packet()</a>.</p> <p> The semantics of each of the possible return values are as follows.</p> <taglist> <dt><code>{reply, <a href="#packet" class="seealso">packet()</a> | <a href="#message" class="seealso">message()</a>}</code></dt> <item> <p> Send the specified answer message to the peer. In the case of a <a href="#packet" class="seealso">packet()</a>, the message to be sent must be set in the <code>msg</code> field and the <code>header</code> field can be set to a <code>#diameter_header{}</code> to specify values that should be preserved in the outgoing answer, appropriate values otherwise being set by diameter.</p> </item> <dt><code>{answer_message, 3000..3999|5000..5999}</code></dt> <item> <p> Send an answer message to the peer containing the specified Result-Code. Equivalent to</p> <pre class="sh_erlang"> {reply, ['answer-message' | Avps] </pre> <p> where <code>Avps</code> sets the Origin-Host, Origin-Realm, the specified Result-Code and (if the request contained one) Session-Id AVPs, and possibly Failed-AVP as described below.</p> <p> Returning a value other than 3xxx or 5xxx will cause the request process in question to fail, as will returning a 5xxx value if the peer connection in question has been configured with the RFC 3588 common dictionary <code>diameter_gen_base_rfc3588</code>. (Since RFC 3588 only allows 3xxx values in an answer-message.)</p> <p> When returning 5xxx, Failed-AVP will be populated with the AVP of the first matching Result-Code/AVP pair in the <code>errors</code> field of the argument <a href="#packet" class="seealso">packet()</a>, if found. If this is not appropriate then an answer-message should be constructed explicitly and returned in a <code>reply</code> tuple instead.</p> </item> <dt><code>{relay, Opts}</code></dt> <item> <p> Relay a request to another peer in the role of a Diameter relay agent. If a routing loop is detected then the request is answered with 3005 (DIAMETER_LOOP_DETECTED). Otherwise a Route-Record AVP (containing the sending peer's Origin-Host) is added to the request and <a href="../#Mod/pick_peer-4.html" class="seealso">pick_peer/4</a> and subsequent callbacks take place just as if <a href="diameter.html#call-4" class="seealso">diameter:call/4</a> had been called explicitly. The End-to-End Identifier of the incoming request is preserved in the header of the relayed request.</p> <p> The returned <code>Opts</code> should not specify <code>detach</code>. A subsequent <a href="../#Mod/handle_answer-4.html" class="seealso">handle_answer/4</a> callback for the relayed request must return its first argument, the <a href="#packet" class="seealso">packet()</a> containing the answer message. Note that the <code>extra</code> option can be specified to supply arguments that can distinguish the relay case from others if so desired. Any other return value (for example, from a <a href="../#Mod/handle_error-4.html" class="seealso">handle_error/4</a> callback) causes the request to be answered with 3002 (DIAMETER_UNABLE_TO_DELIVER).</p> </item> <dt><code>discard</code></dt> <item> <p> Discard the request. No answer message is sent to the peer.</p> </item> <dt><code>{eval, Action, PostF}</code></dt> <item> <p> Handle the request as if <code>Action</code> has been returned and then evaluate <code>PostF</code> in the request process. The return value is ignored.</p> </item> <dt><code>{eval_packet, Action, PostF}</code></dt> <item> <p> Like <code>eval</code> but evaluate <code>PostF</code> on any encoded <code>#diameter_packet{}</code> prior to transmission, the <code>bin</code> field containing the encoded binary. The return value is ignored.</p> </item> <dt><code>{protocol_error, 3000..3999}</code></dt> <item> <p> Equivalent to <code>{answer_message, 3000..3999}</code>.</p> </item> </taglist> <div class="note"><h2>Note!</h2> <p> Requests containing errors may be answered by diameter, without a callback taking place, depending on the value of the <a href="diameter.html#application_opt" class="seealso">diameter:application_opt()</a> <code>request_errors</code>.</p> </div> </div> </div> </div>
