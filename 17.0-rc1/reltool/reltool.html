<h1>reltool</h1> <h2 class="modsummary">Main API of the Reltool application</h2> <div class="description"> <p>This is an interface module for the Reltool application.</p> <p><em>Reltool</em> is a release management tool. It analyses a given Erlang/OTP installation and determines various dependencies between applications. The <code>graphical</code> frontend depicts the dependencies and enables interactive customization of a target system. The backend provides a <code>batch</code> interface for generation of customized target systems.</p> <p>The tool uses an installed Erlang/OTP system as input. <code>root_dir</code> is the root directory of the analysed system and it defaults to the system executing <code>reltool</code>. Applications may also be located outside <code>root_dir</code>. <code>lib_dirs</code> defines library directories where additional applications may reside and it defaults to the directories listed by the operating system environment variable <code>ERL_LIBS</code>. See the module <code>code</code> for more info.</p> <p>An application directory <code>AppDir</code> under a library directory is recognized by the existence of an <code>AppDir/ebin</code> directory. If this does not exist, <code>reltool</code> will not consider <code>AppDir</code> at all when looking for applications.</p> <p>It is recommended that application directories are named as the application, possibly followed by a dash and the version number. For example <code>myapp</code> or <code>myapp-1.1</code>.</p> <p>Finally single modules and entire applications may be read from Escripts.</p> <p>Some configuration parameters control the behavior of Reltool on system (<code>sys</code>) level. Others provide control on application (<code>app</code>) level and yet others are on module (<code>mod</code>) level. Module level parameters override application level parameters and application level parameters override system level parameters. Escript <code>escript</code> level parameters override system level parameters.</p> <p>The following top level <code>options</code> are supported:</p> <taglist> <dt><code>config</code></dt> <item> <p>This is the main option and it controls the configuration of <code>reltool</code>. It can either be a <code>sys</code> tuple or a name of a <code>file</code> containing a sys tuple.</p> </item> <dt><code>trap_exit</code></dt> <item> <p>This option controls the error handling behavior of <code>reltool</code>. By default the window processes traps exit, but this behavior can altered by setting <code>trap_exit</code> to <code>false</code>.</p> </item> <dt><code>wx_debug</code></dt> <item> <p>This option controls the debug level of <code>wx</code>. As its name indicates it is only useful for debugging. See <code>wx:debug/1</code> for more info.</p> </item> </taglist> <p>Besides the already mentioned source parameters <code>root_dir</code> and <code>lib_dirs</code>, the following system (<code>sys</code>) level options are supported:</p> <taglist> <dt><code>erts</code></dt> <item> <p>Erts specific configuration. See application level options below.</p> </item> <dt><code>escript</code></dt> <item> <p>Escript specific configuration. An escript has a mandatory file name and escript level options that are described below.</p> </item> <dt><code>app</code></dt> <item> <p>Application specific configuration. An application has a mandatory name and application level options that are described below.</p> </item> <dt><code>mod_cond</code></dt> <item> <p>This parameter controls the module inclusion policy. It defaults to <code>all</code> which means that if an application is included (either explicitly or implicitly) all modules in that application will be included. This implies that both modules that exist in the <code>ebin</code> directory of the application, as well as modules that are named in the <code>app</code> file will be included. If the parameter is set to <code>ebin</code>, both modules in the <code>ebin</code> directory and derived modules are included. If the parameter is set to <code>app</code>, both modules in the <code>app</code> file and derived modules are included. <code>derived</code> means that only modules that are used by other included modules are included. The <code>mod_cond</code> setting on system level is used as default for all applications.</p> </item> <dt><code>incl_cond</code></dt> <item> <p>This parameter controls the application and escript inclusion policy. It defaults to <code>derived</code> which means that the applications that do not have any explicit <code>incl_cond</code> setting, will only be included if any other (explicitly or implicitly included) application uses it. The value <code>include</code> implies that all applications and escripts that do not have any explicit <code>incl_cond</code> setting will be included. <code>exclude</code> implies that all applications and escripts that do not have any explicit <code>incl_cond</code> setting will be excluded.</p> </item> <dt><code>boot_rel</code></dt> <item> <p>A target system may have several releases but the one given as <code>boot_rel</code> will be used as default when the system is booting up.</p> </item> <dt><code>rel</code></dt> <item> <p>Release specific configuration. Each release maps to a <code>rel</code>, <code>script</code> and <code>boot</code> file. See the module <code>systools</code> for more info about the details. Each release has a name, a version and a set of applications with a few release specific parameters such as type and included applications.</p> </item> <dt><code>relocatable</code></dt> <item> <p>This parameter controls whether the <code>erl</code> executable in the target system should automatically determine where it is installed or if it should use a hardcoded path to the installation. In the latter case the target system must be installed with <code>reltool:install/2</code> before it can be used. If the system is relocatable, the file tree containing the target system can be moved to another location without re-installation. The default is <code>true</code>.</p> </item> <dt><code>profile</code></dt> <item> <p>The creation of the specification for a target system is performed in two steps. In the first step a complete specification is generated. It will likely contain much more files than you are interested in in your customized target system. In the second step the specification will be filtered according to your filters. There you have the ability to specify filters per application as well as system wide filters. You can also select a <code>profile</code> for your system. Depending on the <code>profile</code>, different default filters will be used. There are three different profiles to choose from: <code>development</code>, <code>embedded</code> and <code>standalone</code>. <code>development</code> is default. The parameters that are affected by the <code>profile</code> are: <code>incl_sys_filters</code>, <code>excl_sys_filters</code>, <code>incl_app_filters</code> and <code>excl_app_filters</code>.</p> </item> <dt><code>app_file</code></dt> <item> <p>This parameter controls the default handling of the <code>app</code> files when a target system is generated. It defaults to <code>keep</code> which means that <code>app</code> files are copied to the target system and their contents are kept as they are. <code>strip</code> means that a new <code>app</code> file is generated from the contents of the original <code>app</code> file where the non included modules are removed from the file. <code>all</code> does also imply that a new <code>app</code> file is generated from the contents of the original <code>app</code> file, with the difference that all included modules are added to the file. If the application does not have any <code>app</code> file a file will be created for <code>all</code> but not for <code>keep</code> and <code>strip</code>.</p> </item> <dt><code>debug_info</code></dt> <item> <p>The <code>debug_info</code> parameter controls whether the debug information in the beam file should be kept (<code>keep</code>) or stripped <code>strip</code> when the file is copied to the target system.</p> </item> <dt><code>excl_lib</code></dt> <item> <div class="warning"><h2>Warning!</h2><p>This option is experimental.</p></div> <p>If the <code>excl_lib</code> option is set to <code>otp_root</code> then reltool will not copy anything from the Erlang/OTP installation ($OTP_ROOT) into the target structure. The goal is to create a "slim" release which can be used together with an existing Erlang/OTP installation. The target structure will therefore only contain a <code>lib</code> directory with the applications that were found outside of $OTP_ROOT (typically your own applications), and a <code>releases</code> directory with the generated <code>.rel,</code> <code>.script</code> and <code>.boot</code> files.</p> <p>When starting this release, three things must be specified:</p> <taglist> <dt><b>Which <code>releases</code> directory to use</b></dt> <item>Tell the release handler to use the <code>releases</code> directory in our target structure instead of <code>$OTP_ROOT/releases</code>. This is done by setting the SASL environment variable <code>releases_dir</code>, either from the command line (<code>-sasl releases_dir &lt;target-dir&gt;/releases</code>) or in <code>sys.config</code>.</item> <dt><b>Which boot file to use</b></dt> <item>The default boot file is <code>$OTP_ROOT/bin/start</code>, but in this case we need to specify a boot file from our target structure, typically <code>&lt;target-dir&gt;/releases/&lt;vsn&gt;/&lt;RelName&gt;</code>. This is done with the <code>-boot</code> command line option to <code>erl</code></item> <dt><b>The location of our applications</b></dt> <item>The generated .script (and .boot) file uses the environment variable <code>$RELTOOL_EXT_LIB</code> as prefix for the paths to all applications. The <code>-boot_var</code> option to <code>erl</code> can be used for specifying the value of this variable, typically <code>-boot_var RELTOOL_EXT_LIB &lt;target-dir&gt;/lib</code>.</item> </taglist> <p>Example:</p> <p><code>erl -sasl releases_dir \"mytarget/releases\" -boot mytarget/releases/1.0/myrel\ -boot_var RELTOOL_EXT_LIB mytarget/lib</code></p> </item> <dt><code>incl_sys_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which files in the system should be included. Each file in the target system must match at least one of the listed regular expressions in order to be included. Further the files may not match any filter in <code>excl_sys_filters</code> in order to be included. Which application files should be included is controlled with the parameters <code>incl_app_filters</code> and <code>excl_app_filters</code>. This parameter defaults to <code>[".*"]</code>.</p> </item> <dt><code>excl_sys_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which files in the system should not be included in the target system. In order to be included, a file must match some filter in <code>incl_sys_filters</code> but not any filter in <code>excl_sys_filters</code>. This parameter defaults to <code>[]</code>.</p> </item> <dt><code>incl_app_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which application specific files that should be included. Each file in the application must match at least one of the listed regular expressions in order to be included. Further the files may not match any filter in <code>excl_app_filters</code> in order to be included. This parameter defaults to <code>[".*"]</code>.</p> </item> <dt><code>excl_app_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which application specific files should not be included in the target system. In order to be included, a file must match some filter in <code>incl_app_filters</code> but not any filter in <code>excl_app_filters</code>. This parameter defaults to <code>[]</code>.</p> </item> <dt><code>incl_archive_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which top level directories in an application should be included in an archive file (as opposed to being included as a regular directory outside the archive). Each top directory in the application must match at least one of the listed regular expressions in order to be included. Further the files may not match any filter in <code>excl_app_filters</code> in order to be included. This parameter defaults to <code>[".*"]</code>.</p> </item> <dt><code>excl_archive_filters</code></dt> <item> <p>This parameter normally contains a list of regular expressions that controls which top level directories in an application should not be included in an archive file. In order to be included in the application archive, a top directory must match some filter in <code>incl_archive_filters</code> but not any filter in <code>excl_archive_filters</code>. This parameter defaults to <code>["^include$","^priv$"]</code>.</p> </item> <dt><code>archive_opts</code></dt> <item> <p>This parameter contains a list of options that are given to <code>zip:create/3</code> when application specific files are packaged into an archive. Only a subset of the options are supported. The most useful options in this context are the ones that control which types of files should be compressed. This parameter defaults to <code>[]</code>.</p> </item> </taglist> <p>On application (<code>escript</code>) level, the following options are supported:</p> <taglist> <dt><code>incl_cond</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> </taglist> <p>On application (<code>app</code>) level, the following options are supported:</p> <taglist> <dt><code>vsn</code></dt> <item> <p>The version of the application. In an installed system there may exist several versions of an application. The <code>vsn</code> parameter controls which version of the application will be chosen.</p> <p>This parameter is mutual exclusive with <code>lib_dir</code>. If <code>vsn</code> and <code>lib_dir</code> are both omitted, the latest version will be chosen.</p> <p>Note that in order for reltool to sort application versions and thereby be able to select the latest, it is required that the version id for the application consits of integers and dots only, for example <code>1</code>, <code>2.0</code> or <code>3.17.1</code>.</p> </item> <dt><code>lib_dir</code></dt> <item> <p>The directory to read the application from. This parameter can be used to point out a specific location to fetch the application from. This is useful for instance if the parent directory for some reason is no good as a library directory on system level.</p> <p>This parameter is mutual exclusive with <code>vsn</code>. If <code>vsn</code> and <code>lib_dir</code> are both omitted, the latest version will be chosen.</p> <p>Note that in order for reltool to sort application versions and thereby be able to select the latest, it is required that the version id for the application consits of integers and dots only, for example <code>1</code>, <code>2.0</code> or <code>3.17.1</code>.</p> </item> <dt><code>mod</code></dt> <item> <p>Module specific configuration. A module has a mandatory name and module level options that are described below.</p> </item> <dt><code>mod_cond</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>incl_cond</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>app_file</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>debug_info</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>incl_app_filters</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>excl_app_filters</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>incl_archive_filters</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>excl_archive_filters</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> <dt><code>archive_opts</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on system level.</p> </item> </taglist> <p>On module (<code>mod</code>) level, the following options are supported:</p> <taglist> <dt><code>incl_cond</code></dt> <item> <p>This parameter controls whether the module is included or not. By default the <code>mod_cond</code> parameter on application and system level will be used to control whether the module is included or not. The value of <code>incl_cond</code> overrides the module inclusion policy. <code>include</code> implies that the module is included, while <code>exclude</code> implies that the module is not included. <code>derived</code> implies that the module is included if it is used by any other included module.</p> </item> <dt><code>debug_info</code></dt> <item> <p>The value of this parameter overrides the parameter with the same name on application level.</p> </item> </taglist> </div> <div class="section"> <h4>DATA TYPES</h4> <pre class="sh_erlang"> options() = [option()] option() = {config, config() | server()</code></li> <li><code>TargetDir = target_dir()</code></li> <li><code>Reason = reason()</code></li> </ul> <div class="description"><p>Create a target system. Gives the same result as <code>{ok,TargetSpec}=reltool:get_target_spec(Server)</code> and <code>reltool:eval_target_spec(TargetSpec,RootDir,TargetDir)</code>.</p></div> </div> <div class="function"> <h3 id="eval_target_spec/3">eval_target_spec(TargetSpec, RootDir, TargetDir) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>TargetSpec = target_spec()</code></li> <li><code>RootDir = root_dir()</code></li> <li><code>TargetDir = target_dir()</code></li> <li><code>Reason = reason()</code></li> </ul> <div class="description"><p>Create the actual target system from a specification generated by <code>reltool:get_target_spec/1</code>. The creation of the specification for a target system is performed in two steps. In the first step a complete specification will be generated. It will likely contain much more files than you are interested in in your target system. In the second step the specification will be filtered according to your filters. There you have the ability to specify filters per application as well as system wide filters. You can also select a <code>profile</code> for your system. The <code>lib</code> directory contains the applications. Which applications are included and if they should be customized (archived, stripped from debug info etc.) is specified with various configuration parameters. The files in the <code>bin</code> directory are copied from the <code>erts-vsn/bin</code> directory, but only those files that were originally included in the <code>bin</code> directory of the source system.</p> <p>If the configuration parameter <code>relocatable</code> was set to <code>true</code> there is no need to install the target system with <code>reltool:install/2</code> before it can be started. In that case the file tree containing the target system can be moved without re-installation.</p> <p>In most cases, the <code>RootDir</code> parameter should be set to the same as the <code>root_dir</code> configuration parameter used in the call to <code>reltool:get_target_spec/1</code> (or <code>code:root_dir()</code> if the configuration parameter is not set).
