<h1>asn1ct</h1> <h2 class="modsummary">ASN.1 compiler and compile-time support functions</h2> <div class="description"> <p>The ASN.1 compiler takes an ASN.1 module as input and generates a corresponding Erlang module which can encode and decode the data-types specified. Alternatively the compiler takes a specification module (se below) specifying all input modules and generates one module with encode/decode functions. There are also some generic functions which can be used in during development of applications which handles ASN.1 data (encoded as BER or PER).</p> <div class="note"><h2>Note!</h2> <p>By default in OTP 17, the representation of the BIT STRING and OCTET STRING types as Erlang terms have changed. BIT STRING values are now Erlang bitstrings and OCTET STRING values are binaries. For details see <a href="asn1_ug.html#BIT STRING" class="seealso">BIT STRING</a> and <a href="asn1_ug.html#OCTET STRING" class="seealso">OCTET STRING</a> in User's Guide.</p> <p>To revert to the old representation of the types, use the <code>legacy_erlang_types</code> option.</p> </div> <div class="note"><h2>Note!</h2> <p>In R16, the options have been simplified. The back-end is chosen using one of the options <code>ber</code>, <code>per</code>, or <code>uper</code>. The options <code>optimize</code>, <code>nif</code>, and <code>driver</code> options are no longer necessary (and the ASN.1 compiler will print a warning if they are used). The options <code>ber_bin</code>, <code>per_bin</code>, and <code>uper_bin</code> options will still work, but will print a warning. </p> <p>Another change in R16 is that the generated <code>encode/2</code> function (and <code>asn1rt:encode/3</code>) always returns a binary. The <code>encode/2</code> function for the BER back-end used to return an iolist.</p> </div> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="compile/1">compile(Asn1module) -&gt; ok | {error, Reason}</h3> <h3 id="compile/2">compile(Asn1module, Options) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>Asn1module = atom() | string()</code></li> <li><code>Options = [Option| OldOption]</code></li> <li><code>Option = ber | per | uper | der | compact_bit_string | legacy_bit_string | legacy_erlang_types | noobj | {n2n, EnumTypeName} |{outdir, Dir} | {i, IncludeDir} | asn1config | undec_rest | no_ok_wrapper | {macro_name_prefix, Prefix} | {record_name_prefix, Prefix} | verbose | warnings_as_errors</code></li> <li><code>OldOption = ber | per</code></li> <li><code>Reason = term()</code></li> <li><code>Prefix = string()</code></li> </ul> <div class="description"> <p>Compiles the ASN.1 module <code>Asn1module</code> and generates an Erlang module <code>Asn1module.erl</code> with encode and decode functions for the types defined in <code>Asn1module</code>. For each ASN.1 value defined in the module an Erlang function which returns the value in Erlang representation is generated.</p> <p>If <code>Asn1module</code> is a filename without extension first <code>".asn1"</code> is assumed, then <code>".asn"</code> and finally <code>".py"</code> (to be compatible with the old ASN.1 compiler). Of course <code>Asn1module</code> can be a full pathname (relative or absolute) including filename with (or without) extension. <span id="asn1set"> </span> </p> <p>If one wishes to compile a set of Asn1 modules into one Erlang file with encode/decode functions one has to list all involved files in a configuration file. This configuration file must have a double extension ".set.asn", (".asn" can alternatively be ".asn1" or ".py"). The input files' names must be listed, within quotation marks (""), one at each row in the file. If the input files are <code>File1.asn</code>, <code>File2.asn</code> and <code>File3.asn</code> the configuration file shall look like:</p> <pre class="sh_erlang"> File1.asn File2.asn File3.asn </pre> <p>The output files will in this case get their names from the configuration file. If the configuration file has the name <code>SetOfFiles.set.asn</code> the name of the output files will be <code>SetOfFiles.hrl, SetOfFiles.erl and SetOfFiles.asn1db</code>.</p> <p>Sometimes in a system of ASN.1 modules there are different default tag modes, e.g. AUTOMATIC, IMPLICIT or EXPLICIT. The multi file compilation resolves the default tagging as if the modules were compiled separately.</p> <p>Another unwanted effect that may occur in multi file compilation is name collisions. The compiler solves this problem in two ways: If the definitions are identical then the output module keeps only one definition with the original name. But if definitions only have same name and differs in the definition, then they will be renamed. The new names will be the definition name and the original module name concatenated.</p> <p>If any name collision have occurred the compiler reports a "NOTICE: ..." message that tells if a definition was renamed, and the new name that must be used to encode/decode data.</p> <p> <code>Options</code> is a list with options specific for the asn1 compiler and options that are applied to the Erlang compiler. The latter are those that not is recognized as asn1 specific. Available options are: </p> <taglist> <dt><code>ber | per | uper</code></dt> <item> <p> The encoding rule to be used. The supported encoding rules are BER (Basic Encoding Rules), PER aligned (Packed Encoding Rules) and PER unaligned. If the encoding rule option is omitted <code>ber</code> is the default. </p> <p> The generated Erlang module always gets the same name as the ASN.1 module and as a consequence of this only one encoding rule per ASN.1 module can be used at runtime. </p> </item> <dt><code>der</code></dt> <item> <p> By this option the Distinguished Encoding Rules (DER) is chosen. DER is regarded as a specialized variant of the BER encoding rule, therefore the <code>der</code> option only makes sense together with the <code>ber</code> option. This option sometimes adds sorting and value checks when encoding, which implies a slower encoding. The decoding routines are the same as for <code>ber</code>. </p> </item> <dt><code>compact_bit_string</code></dt> <item> <p> The BIT STRING type will be decoded to the "compact notation". <em>This option is not recommended for new code.</em> </p> <p>For details see <a href="asn1_ug.html#BIT STRING" class="seealso"> BIT STRING type section in the Users Guide </a>. </p> <p>This option implies the <code>legacy_erlang_types</code> option.</p> </item> <dt><code>legacy_bit_string</code></dt> <item> <p> The BIT STRING type will be decoded to the legacy format, i.e. a list of zeroes and ones. <em>This option is not recommended for new code.</em> </p> <p>For details see <a href="asn1_ug.html#BIT STRING" class="seealso"> BIT STRING type section in the Users Guide </a>. <p>This option implies the <code>legacy_erlang_types</code> option.</p> </p> </item> <dt><code>legacy_erlang_types</code></dt> <item> <p>Use the same Erlang types to represent BIT STRING and OCTET STRING as in R16. For details see <a href="asn1_ug.html#BIT STRING" class="seealso">BIT STRING</a> and <a href="asn1_ug.html#OCTET STRING" class="seealso">OCTET STRING</a> in User's Guide.</p> <p><em>This option is not recommended for new code.</em></p> </item> <dt><code>{n2n, EnumTypeName}</code></dt> <item> <p> Tells the compiler to generate functions for conversion between names (as atoms) and numbers and vice versa for the EnumTypeName specified. There can be multiple occurrences of this option in order to specify several type names. The type names must be declared as ENUMERATIONS in the ASN.1 spec. If the EnumTypeName does not exist in the ASN.1 spec the compilation will stop with an error code. The generated conversion functions are named <code>name2num_EnumTypeName/1</code> and <code>num2name_EnumTypeName/1</code>. </p> </item> <dt><code>noobj</code></dt> <item> <p>Do not compile (i.e do not produce object code) the generated <code>.erl</code> file. If this option is omitted the generated Erlang module will be compiled.</p> </item> <dt><code>{i, IncludeDir}</code></dt> <item> <p>Adds <code>IncludeDir</code> to the search-path for <code>.asn1db</code> and asn1 source files. The compiler tries to open a <code>.asn1db</code> file when a module imports definitions from another ASN.1 module. If no <code>.asn1db</code> file is found the asn1 source file is parsed. Several <code>{i, IncludeDir}</code> can be given. </p> </item> <dt><code>{outdir, Dir}</code></dt> <item> <p>Specifies the directory <code>Dir</code> where all generated files shall be placed. If omitted the files are placed in the current directory.</p> </item> <dt><code>asn1config</code></dt> <item> <p>When one of the specialized decodes, exclusive or selective decode, is wanted one has to give instructions in a configuration file. The option <code>asn1config</code> enables specialized decodes and takes the configuration file, which has the same name as the ASN.1 spec but with extension <code>.asn1config</code>, in concern. </p> <p>The instructions for exclusive decode must follow the <a href="asn1_spec.html#Exclusive Instruction" class="seealso">instruction and grammar in the User's Guide</a>. </p> <p>You can also find the instructions for selective decode in the <a href="asn1_spec.html#Selective Instruction" class="seealso">User's Guide</a>. </p> </item> <dt><code>undec_rest</code></dt> <item> <p>A buffer that holds a message, being decoded may also have some following bytes. Now it is possible to get those following bytes returned together with the decoded value. If an asn1 spec is compiled with this option a tuple <code>{ok, Value, Rest}</code> is returned. <code>Rest</code> may be a list or a binary. Earlier versions of the compiler ignored those following bytes.</p> </item> <dt><code>no_ok_wrapper</code></dt> <item> <p>If this option is given, the generated <code>encode/2</code> and <code>decode/2</code> functions will not wrap a successful return value in an <code>{ok,...}</code> tuple. If any error occurs, there will be an exception.</p> </item> <dt><code>{macro_name_prefix, Prefix}</code></dt> <item> <p>All macro names generated by the compiler are prefixed with <code>Prefix</code>. This is useful when multiple protocols that contains macros with identical names are included in a single module.</p> </item> <dt><code>{record_name_prefix, Prefix}</code></dt> <item> <p>All record names generated by the compiler are prefixed with <code>Prefix</code>. This is useful when multiple protocols that contains records with identical names are included in a single module.</p> </item> <dt><code>verbose</code></dt> <item> <p>Causes more verbose information from the compiler describing what it is doing.</p> </item> <dt><code>warnings_as_errors</code></dt> <item> <p>Causes warnings to be treated as errors.</p> </item> </taglist> <p>Any additional option that is applied will be passed to the final step when the generated .erl file is compiled. </p> <p>The compiler generates the following files:</p> <list type="bulleted"> <item> <p><code>Asn1module.hrl</code> (if any SET or SEQUENCE is defined)</p> </item> <item> <p><code>Asn1module.erl</code> the Erlang module with encode, decode and value functions.</p> </item> <item> <p><code>Asn1module.asn1db</code> intermediate format used by the compiler when modules IMPORTS definitions from each other.</p> </item> </list> </div> </div> <div class="function"> <h3 id="encode/3">encode(Module, Type, Value)-&gt; {ok, Bytes} | {error, Reason}</h3> <ul class="type"> <li><code>Module = Type = atom()</code></li> <li><code>Value = term()</code></li> <li><code>Bytes = binary()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Encodes <code>Value</code> of <code>Type</code> defined in the ASN.1 module <code>Module</code>. To get as fast execution as possible the encode function only performs rudimentary tests that the input <code>Value</code> is a correct instance of <code>Type</code>. The length of strings is for example not always checked. Returns <code>{ok, Bytes}</code> if successful or <code>{error, Reason}</code> if an error occurred. </p> </div> </div> <div class="function"> <h3 id="decode/3">decode(Module, Type, Bytes) -&gt; {ok, Value} | {error, Reason}</h3> <ul class="type"> <li><code>Module = Type = atom()</code></li> <li><code>Value = Reason = term()</code></li> <li><code>Bytes = binary()</code></li> </ul> <div class="description"> <p>Decodes <code>Type</code> from <code>Module</code> from the binary <code>Bytes</code>. Returns <code>{ok, Value}</code> if successful.</p> </div> </div> <div class="function"> <h3 id="value/2">value(Module, Type) -&gt; {ok, Value} | {error, Reason}</h3> <ul class="type"> <li><code>Module = Type = atom()</code></li> <li><code>Value = term()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns an Erlang term which is an example of a valid Erlang representation of a value of the ASN.1 type <code>Type</code>. The value is a random value and subsequent calls to this function will for most types return different values.</p> </div> </div> <div class="function"> <h3 id="test/1">test(Module) -&gt; ok | {error, Reason}</h3> <h3 id="test/2">test(Module, Type | Options) -&gt; ok | {error, Reason}</h3> <h3 id="test/3">test(Module, Type, Value | Options) -&gt; ok | {error, Reason}</h3> <ul class="type"> <li><code>Module = Type = atom()</code></li> <li><code>Value = term()</code></li> <li><code>Options = [{i, IncludeDir}]</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Performs a test of encode and decode of types in <code>Module</code>. The generated functions are called by this function. This function is useful during test to secure that the generated encode and decode functions and the general runtime support work as expected.</p> <list type="bulleted"> <item> <p><code>test/1</code> iterates over all types in <code>Module</code>.</p> </item> <item> <p><code>test/2</code> tests type <code>Type</code> with a random value.</p> </item> <item> <p><code>test/3</code> tests type <code>Type</code> with <code>Value</code>.</p> </item> </list> <p>Schematically the following happens for each type in the module:</p> <pre class="sh_erlang"> {ok, Value} = asn1ct:value(Module, Type), {ok, Bytes} = asn1ct:encode(Module, Type, Value), {ok, Value} = asn1ct:decode(Module, Type, Bytes).</pre> <p>The <code>test</code> functions utilizes the <code>*.asn1db</code> files for all included modules. If they are located in a different directory than the current working directory, use the include option to add paths. This is only needed when automatically generating values. For static values using <code>Value</code> no options are needed.</p> </div> </div> </div>
