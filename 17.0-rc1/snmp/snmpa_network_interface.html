<h1>snmpa_network_interface</h1> <h2 class="modsummary">Behaviour module for the SNMP agent network interface.</h2> <div class="description"> <p>This module defines the behaviour of the agent network interface. A <code>snmpa_network_interface</code> compliant module must export the following functions: </p> <list type="bulleted"> <item> <p><a href="#start_link" class="seealso">start_link/4</a></p> </item> <item> <p><a href="#info" class="seealso">info/1</a></p> </item> <item> <p><a href="#get_log_type" class="seealso">get_log_type/1</a></p> </item> <item> <p><a href="#set_log_type" class="seealso">set_log_type/2</a></p> </item> <item> <p><a href="#verbosity" class="seealso">verbosity/2</a></p> </item> </list> <p>The semantics of them and their exact signatures are explained below. </p> <p>But this is not enough. There is also a set of <em>mandatory</em> messages which the network interface entity must be able to receive and be able to send. This is described in chapter <a href="snmp_agent_netif.html" class="seealso">snmp_agent_netif</a>. </p> <span id="start_link"> </span> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start_link/4">start_link(Prio, NoteStore, MasterAgent, Opts) -&gt; {ok, Pid} | {error, Reason}</h3> <ul class="type"> <li><code>Prio = priority()</code></li> <li><code>NoteStore = pid()</code></li> <li><code>MasterAgent = pid()</code></li> <li><code>Opts = [opt()]</code></li> <li><code>opt() = {verbosity, verbosity()} | {versions, versions()} | term()</code></li> <li><code>versions() = [version()]</code></li> <li><code>version() = v1 | v2 | v3</code></li> </ul> <div class="description"> <p>Start-link the network interface process.</p> <p><code>NoteStore</code> is the pid of the note-store process and <code>MasterAgent</code> is the pid of the master-agent process.</p> <p><code>Opts</code> is an (basically) implementation dependent list of options to the network interface process. There are however a number of options which <em>must</em> be handled: <code>versions</code> and <code>verbosity</code>.</p> <span id="info"> </span> </div> </div> <div class="function"> <h3 id="info/1">info(Pid) -&gt; [{Key, Value}]</h3> <ul class="type"> <li><code>Pid = pid()</code></li> </ul> <div class="description"> <p>The info returned is basically up to the implementer to decide. This implementation provided by the application provides info about memory allocation and various socket information.</p> <p>The info returned by this function is returned together with other info collected by the agent when the <a href="snmpa.html#info" class="seealso">info</a> function is called (tagged with with the key <code>net_if</code>).</p> <span id="verbosity"> </span> </div> </div> <div class="function"> <h3 id="verbosity/2">verbosity(Pid, Verbosity) -&gt; void()</h3> <ul class="type"> <li><code>Pid = pid()</code></li> <li><code>Verbosity = verbosity()</code></li> </ul> <div class="description"> <p>Change the verbosity of a running network interface process.</p> <span id="get_log_type"> </span> </div> </div> <div class="function"> <h3 id="get_log_type/1">get_log_type(Pid) -&gt; {ok, LogType} | {error, Reason}</h3> <ul class="type"> <li><code>Pid = pid()</code></li> <li><code>LogType = atl_type()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>The Audit Trail Log is managed by the network interface process. So, it is this process that has to retrieve the actual log-type. </p> <span id="set_log_type"> </span> </div> </div> <div class="function"> <h3 id="set_log_type/2">set_log_type(Pid, NewType) -&gt; {ok, OldType} | {error, Reason}</h3> <ul class="type"> <li><code>Pid = pid()</code></li> <li><code>NewType = OldType = atl_type()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>The Audit Trail Log is managed by the network interface process. So, it is this process that has to do the actual changing of the type. </p> <p>See <a href="snmpa.html#set_log_type" class="seealso">set_log_type</a> for more info. </p> </div> </div> </div>
