<h1>ssh_connection</h1> <h2 class="modsummary">This module provides API functions to send <url href="http://www.ietf.org/rfc/rfc4254.txt"> SSH Connection Protocol </url> events to the other side of an SSH channel. </h2> <div class="description"> <p>The SSH Connection Protocol is used by clients and servers (i.e. SSH channels) to communicate over the SSH connection. The API functions in this module sends SSH Connection Protocol events that are received as messages by the remote channel. In the case that the receiving channel is an Erlang process the message will be on the following format <code>{ssh_cm, ssh_connection_ref(), ssh_event_msg()}</code>. If the <a href="ssh_channel.html" class="seealso">ssh_channel</a> behavior is used to implement the channel process these will be handled by <a href="../ssh_channel#Module/handle_ssh_msg-2.html" class="seealso">handle_ssh_msg/2 </a>.</p> </div> <div class="section"> <h4>DATA TYPES </h4> <p>Type definitions that are used more than once in this module and/or abstractions to indicate the intended use of the data type:</p> <p><code>boolean() = true | false </code></p> <p><code>string() = list of ASCII characters</code></p> <p><code>timeout() = infinity | integer() - in milliseconds.</code></p> <p><code>ssh_connection_ref() - opaque to the user returned by ssh:connect/3 or sent to an SSH channel processes</code></p> <p><code>ssh_channel_id() = integer() </code></p> <p><code>ssh_data_type_code() = 1 ("stderr") | 0 ("normal") are currently valid values see</code> <url href="http://www.ietf.org/rfc/rfc4254.txt">RFC 4254 </url> section 5.2.</p> <p><code>ssh_request_status() = success | failure</code></p> <p><code>event() = {ssh_cm, ssh_connection_ref(), ssh_event_msg()} </code></p> <p><code>ssh_event_msg() = data_events() | status_events() | terminal_events() </code></p> <taglist> <dt><b>data_events()</b></dt> <item> <taglist> <dt><code>{data, ssh_channel_id(), ssh_data_type_code(), binary() = Data}</code></dt> <item> Data has arrived on the channel. This event is sent as result of calling <a href="ssh_connection.html#send-3" class="seealso"> ssh_connection:send/[3,4,5] </a></item> <dt><code>{eof, ssh_channel_id()}</code></dt> <item>Indicates that the other side will not send any more data. This event is sent as result of calling <a href="ssh_connection.html#send_eof-2" class="seealso"> ssh_connection:send_eof/2</a> </item> </taglist> </item> <dt><b>status_events()</b></dt> <item> <taglist> <dt><code>{signal, ssh_channel_id(), ssh_signal()}</code></dt> <item>A signal can be delivered to the remote process/service using the following message. Some systems will not support signals, in which case they should ignore this message. There is currently no funtion to generate this event as the signals refered to are on OS-level and not something generated by an Erlang program.</item> <dt><code>{exit_signal, ssh_channel_id(), string() = ExitSignal, string() = ErrorMsg, string() = LanguageString}</code></dt> <item>A remote execution may terminate violently due to a signal then this message may be received. For details on valid string values see <url href="http://www.ietf.org/rfc/rfc4254.txt">RFC 4254</url> section 6.10. Special case of the signals mentioned above.</item> <dt><code>{exit_status, ssh_channel_id(), integer() = ExitStatus}</code></dt> <item> When the command running at the other end terminates, the following message can be sent to return the exit status of the command. A zero 'exit_status' usually means that the command terminated successfully. This event is sent as result of calling <a href="ssh_connection.html#exit_status-3" class="seealso"> ssh_connection:exit_status/3</a></item> <dt><code>{closed, ssh_channel_id()}</code></dt> <item> This event is sent as result of calling <a href="ssh_connection.html#close-2" class="seealso">ssh_connection:close/2</a> Both the handling of this event and sending of it will be taken care of by the <a href="ssh_channel.html" class="seealso">ssh_channel</a> behavior.</item> </taglist> </item> <dt><b>terminal_events()</b></dt> <item> <p> Channels implementing a shell and command execution on the server side should handle the following messages that may be sent by client channel processes. </p> <div class="note"><h2>Note!</h2> <p>Events that includes a <code> WantReply</code> expects the event handling process to call <a href="ssh_connection.html#reply_request-4" class="seealso">ssh_connection:reply_request/4</a> with the boolean value of <code> WantReply</code> as the second argument. </p></div> <taglist> <dt><code>{env, ssh_channel_id(), boolean() = WantReply, string() = Var, string() = Value}</code></dt> <item> Environment variables may be passed to the shell/command to be started later. This event is sent as result of calling <a href="ssh_connection.html#setenv-5" class="seealso"> ssh_connection:setenv/5</a> </item> <dt><code>{pty, ssh_channel_id(), boolean() = WantReply, {string() = Terminal, integer() = CharWidth, integer() = RowHeight, integer() = PixelWidth, integer() = PixelHight, [{atom() | integer() = Opcode, integer() = Value}] = TerminalModes}}</code></dt> <item>A pseudo-terminal has been requested for the session. Terminal is the value of the TERM environment variable value (e.g., vt100). Zero dimension parameters must be ignored. The character/row dimensions override the pixel dimensions (when nonzero). Pixel dimensions refer to the drawable area of the window. The <code>Opcode</code> in the <code>TerminalModes</code> list is the mnemonic name, represented as an lowercase erlang atom, defined in <url href="http://www.ietf.org/rfc/rfc4254.txt">RFC 4254 </url> section 8, or the opcode if the mnemonic name is not listed in the RFC. Example <code>OP code: 53, mnemonic name ECHO erlang atom: echo</code>. There is currently no API function to generate this event.</item> <dt><code>{shell, boolean() = WantReply}</code></dt> <item> This message will request that the user's default shell be started at the other end. This event is sent as result of calling <a href="ssh_connection.html#shell-2" class="seealso"> ssh_connection:shell/2</a> </item> <dt><code>{window_change, ssh_channel_id(), integer() = CharWidth, integer() = RowHeight, integer() = PixWidth, integer() = PixHeight}</code></dt> <item> When the window (terminal) size changes on the client side, it MAY send a message to the server side to inform it of the new dimensions. There is currently no API function to generate this event.</item> <dt><code>{exec, ssh_channel_id(), boolean() = WantReply, string() = Cmd}</code></dt> <item> This message will request that the server starts execution of the given command. This event is sent as result of calling <a href="ssh_connection.html#exec-4" class="seealso">ssh_connection:exec/4 </a> </item> </taglist> </item> </taglist> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="adjust_window/3">adjust_window(ConnectionRef, ChannelId, NumOfBytes) -&gt; ok</h3> <ul class="type"> <li><code> ConnectionRef = ssh_connection_ref() </code></li> <li><code> ChannelId = ssh_channel_id() </code></li> <li><code> NumOfBytes = integer()</code></li> </ul> <div class="description"> <p>Adjusts the SSH flowcontrol window.
