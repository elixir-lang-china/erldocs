<h1>supervisor_bridge</h1> <h2 class="modsummary">Generic Supervisor Bridge Behaviour.</h2> <div class="description"> <p>A behaviour module for implementing a supervisor_bridge, a process which connects a subsystem not designed according to the OTP design principles to a supervision tree. The supervisor_bridge sits between a supervisor and the subsystem. It behaves like a real supervisor to its own supervisor, but has a different interface than a real supervisor to the subsystem. Refer to <em>OTP Design Principles</em> for more information.</p> <p>A supervisor_bridge assumes the functions for starting and stopping the subsystem to be located in a callback module exporting a pre-defined set of functions.</p> <p>The <code>sys</code> module can be used for debugging a supervisor_bridge.</p> <p>Unless otherwise stated, all functions in this module will fail if the specified supervisor_bridge does not exist or if bad arguments are given.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start_link/2">start_link/2</h3> <h3 id="start_link/3">start_link/3</h3> <div class="description"> <p>Creates a supervisor_bridge process, linked to the calling process, which calls <code><anno>Module</anno>:init/1</code> to start the subsystem. To ensure a synchronized start-up procedure, this function does not return until <code><anno>Module</anno>:init/1</code> has returned.</p> <p>If <code><anno>SupBridgeName</anno>={local,<anno>Name</anno>}</code> the supervisor_bridge is registered locally as <code><anno>Name</anno></code> using <code>register/2</code>. If <code><anno>SupBridgeName</anno>={global,<anno>Name</anno>}</code> the supervisor_bridge is registered globally as <code><anno>Name</anno></code> using <code>global:register_name/2</code>. If <code><anno>SupBridgeName</anno>={via,<anno>Module</anno>,<anno>Name</anno>}</code> the supervisor_bridge is registered as <code><anno>Name</anno></code> using a registry represented by <anno>Module</anno>. The <code>Module</code> callback should export the functions <code>register_name/2</code>, <code>unregister_name/1</code> and <code>send/2</code>, which should behave like the corresponding functions in <code>global</code>. Thus, <code>{via,global,GlobalName}</code> is a valid reference. If no name is provided, the supervisor_bridge is not registered. If there already exists a process with the specified <code><anno>SupBridgeName</anno></code> the function returns <code>{error,{already_started,<anno>Pid</anno>}}</code>, where <code><anno>Pid</anno></code> is the pid of that process.</p> <p><code><anno>Module</anno></code> is the name of the callback module.</p> <p><code><anno>Args</anno></code> is an arbitrary term which is passed as the argument to <code><anno>Module</anno>:init/1</code>.</p> <p>If the supervisor_bridge and the subsystem are successfully started the function returns <code>{ok,<anno>Pid</anno>}</code>, where <code><anno>Pid</anno></code> is is the pid of the supervisor_bridge.</p> <p>If <code><anno>Module</anno>:init/1</code> returns <code>ignore</code>, this function returns <code>ignore</code> as well and the supervisor_bridge terminates with reason <code>normal</code>. If <code><anno>Module</anno>:init/1</code> fails or returns an error tuple or an incorrect value, this function returns <code>{error,<anno>Error</anno>r}</code> where <code><anno>Error</anno></code> is a term with information about the error, and the supervisor_bridge terminates with reason <code><anno>Error</anno></code>.</p> </div> </div> </div> <div class="section"> <h4>CALLBACK FUNCTIONS</h4> <p>The following functions should be exported from a <code>supervisor_bridge</code> callback module.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="init/1">Module:init(Args) -&gt; Result</h3> <ul class="type"> <li><code>Args = term()</code></li> <li><code>Result = {ok,Pid,State} | ignore | {error,Error}</code></li> <li><code>&nbsp;Pid = pid()</code></li> <li><code>&nbsp;State = term()</code></li> <li><code>&nbsp;Error = term()</code></li> </ul> <div class="description"> <p>Whenever a supervisor_bridge is started using <code>supervisor_bridge:start_link/2,3</code>, this function is called by the new process to start the subsystem and initialize.</p> <p><code>Args</code> is the <code>Args</code> argument provided to the start function.</p> <p>The function should return <code>{ok,Pid,State}</code> where <code>Pid</code> is the pid of the main process in the subsystem and <code>State</code> is any term.</p> <p>If later <code>Pid</code> terminates with a reason <code>Reason</code>, the supervisor bridge will terminate with reason <code>Reason</code> as well. If later the supervisor_bridge is stopped by its supervisor with reason <code>Reason</code>, it will call <code>Module:terminate(Reason,State)</code> to terminate.</p> <p>If something goes wrong during the initialization the function should return <code>{error,Error}</code> where <code>Error</code> is any term, or <code>ignore</code>.</p> </div> </div> <div class="function"> <h3 id="terminate/2">Module:terminate(Reason, State)</h3> <ul class="type"> <li><code>Reason = shutdown | term()</code></li> <li><code>State = term()</code></li> </ul> <div class="description"> <p>This function is called by the supervisor_bridge when it is about to terminate. It should be the opposite of <code>Module:init/1</code> and stop the subsystem and do any necessary cleaning up. The return value is ignored.</p> <p><code>Reason</code> is <code>shutdown</code> if the supervisor_bridge is terminated by its supervisor. If the supervisor_bridge terminates because a a linked process (apart from the main process of the subsystem) has terminated with reason <code>Term</code>, <code>Reason</code> will be <code>Term</code>.</p> <p><code>State</code> is taken from the return value of <code>Module:init/1</code>.</p> </div> </div> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="supervisor.html" class="seealso">supervisor(3)</a>, <a href="sys.html" class="seealso">sys(3)</a></p> </div>
