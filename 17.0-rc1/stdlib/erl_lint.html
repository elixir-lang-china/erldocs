<h1>erl_lint</h1> <h2 class="modsummary">The Erlang Code Linter</h2> <div class="description"> <p>This module is used to check Erlang code for illegal syntax and other bugs. It also warns against coding practices which are not recommended. </p> <p>The errors detected include:</p> <list type="bulleted"> <item>redefined and undefined functions</item> <item>unbound and unsafe variables</item> <item>illegal record usage.</item> </list> <p>Warnings include:</p> <list type="bulleted"> <item>unused functions and imports</item> <item>unused variables</item> <item>variables imported into matches</item> <item>variables exported from <code>if</code>/<code>case</code>/<code>receive</code></item> <item>variables shadowed in lambdas and list comprehensions.</item> </list> <p>Some of the warnings are optional, and can be turned on by giving the appropriate option, described below.</p> <p>The functions in this module are invoked automatically by the Erlang compiler and there is no reason to invoke these functions separately unless you have written your own Erlang compiler.</p> </div> <datatypes> <datatype> <name name="error_info" /> </datatype> <datatype> <name name="error_description" /> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="module/1">module/1</h3> <h3 id="module/2">module/2</h3> <h3 id="module/3">module/3</h3> <div class="description"> <p>This function checks all the forms in a module for errors. It returns: </p> <taglist> <dt><code>{ok,<anno>Warnings</anno>}</code></dt> <item> <p>There were no errors in the module.</p> </item> <dt><code>{error,<anno>Errors</anno>,<anno>Warnings</anno>}</code></dt> <item> <p>There were errors in the module.</p> </item> </taglist> <p>Since this module is of interest only to the maintainers of the compiler, and to avoid having the same description in two places to avoid the usual maintenance nightmare, the elements of <code>Options</code> that control the warnings are only described in <a href="../compiler/compile.html#erl_lint_options" class="seealso">compile(3)</a>. </p> <p>The <code><anno>AbsForms</anno></code> of a module which comes from a file that is read through <code>epp</code>, the Erlang pre-processor, can come from many files. This means that any references to errors must include the file name (see <a href="epp.html" class="seealso">epp(3)</a>, or parser <a href="erl_parse.html" class="seealso">erl_parse(3)</a>). The warnings and errors returned have the following format: </p> <pre class="sh_erlang"> [{<anno>FileName2</anno>,[<anno>ErrorInfo</anno>]}] </pre> <p>The errors and warnings are listed in the order in which they are encountered in the forms. This means that the errors from one file may be split into different entries in the list of errors.</p> </div> </div> <div class="function"> <h3 id="is_guard_test/1">is_guard_test/1</h3> <div class="description"> <p>This function tests if <code><anno>Expr</anno></code> is a legal guard test. <code><anno>Expr</anno></code> is an Erlang term representing the abstract form for the expression. <code>erl_parse:parse_exprs(Tokens)</code> can be used to generate a list of <code><anno>Expr</anno></code>.</p> </div> </div> <div class="function"> <h3 id="format_error/1">format_error/1</h3> <div class="description"> <p>Takes an <code><anno>ErrorDescriptor</anno></code> and returns a string which describes the error or warning. It has the following format: </p> <pre class="sh_erlang"> {ErrorLine, Module, ErrorDescriptor} </pre> <p>A string which describes the error is obtained with the following call: </p> <pre class="sh_erlang"> Module:format_error(ErrorDescriptor) </pre> </div> <div class="section"> <h4>See Also</h4> <p><a href="erl_parse.html" class="seealso">erl_parse(3)</a>, <a href="epp.html" class="seealso">epp(3)</a></p> </div>
