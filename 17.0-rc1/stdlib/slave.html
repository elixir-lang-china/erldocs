<h1>slave</h1> <h2 class="modsummary">Functions to Starting and Controlling Slave Nodes</h2> <div class="description"> <p>This module provides functions for starting Erlang slave nodes. All slave nodes which are started by a master will terminate automatically when the master terminates. All TTY output produced at the slave will be sent back to the master node. File I/O is done via the master.</p> <p>Slave nodes on other hosts than the current one are started with the program <code>rsh</code>. The user must be allowed to <code>rsh</code> to the remote hosts without being prompted for a password. This can be arranged in a number of ways (refer to the <code>rsh</code> documentation for details). A slave node started on the same host as the master inherits certain environment values from the master, such as the current directory and the environment variables. For what can be assumed about the environment when a slave is started on another host, read the documentation for the <code>rsh</code> program.</p> <p>An alternative to the <code>rsh</code> program can be specified on the command line to <code>erl</code> as follows: <code>-rsh Program</code>.</p> <p>The slave node should use the same file system at the master. At least, Erlang/OTP should be installed in the same place on both computers and the same version of Erlang should be used.</p> <p>Currently, a node running on Windows NT can only start slave nodes on the host on which it is running.</p> <p>The master node must be alive.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start/1">start/1</h3> <h3 id="start/2">start/2</h3> <h3 id="start/3">start/3</h3> <div class="description"> <p>Starts a slave node on the host <code><anno>Host</anno></code>. Host names need not necessarily be specified as fully qualified names; short names can also be used. This is the same condition that applies to names of distributed Erlang nodes.</p> <p>The name of the started node will be <code><anno>Name</anno>@<anno>Host</anno></code>. If no name is provided, the name will be the same as the node which executes the call (with the exception of the host name part of the node name).</p> <p>The slave node resets its <code>user</code> process so that all terminal I/O which is produced at the slave is automatically relayed to the master. Also, the file process will be relayed to the master.</p> <p>The <code><anno>Args</anno></code> argument is used to set <code>erl</code> command line arguments. If provided, it is passed to the new node and can be used for a variety of purposes. See <a href="../erts/erl.html#erl" class="seealso">erl(1)</a></p> <p>As an example, suppose that we want to start a slave node at host <code>H</code> with the node name <code>Name@H</code>, and we also want the slave node to have the following properties:</p> <list type="bulleted"> <item> <p>directory <code>Dir</code> should be added to the code path;</p> </item> <item> <p>the Mnesia directory should be set to <code>M</code>;</p> </item> <item> <p>the unix <code>DISPLAY</code> environment variable should be set to the display of the master node.</p> </item> </list> <p>The following code is executed to achieve this:</p> <pre class="sh_erlang"> E = " -env DISPLAY " ++ net_adm:localhost() ++ ":0 ", Arg = "-mnesia_dir " ++ M ++ " -pa " ++ Dir ++ E, slave:start(H, Name, Arg).</pre> <p>If successful, the function returns <code>{ok, <anno>Node</anno>}</code>, where <code><anno>Node</anno></code> is the name of the new node. Otherwise it returns <code>{error, <anno>Reason</anno>}</code>, where <code><anno>Reason</anno></code> can be one of:</p> <taglist> <dt><code>timeout</code></dt> <item> <p>The master node failed to get in contact with the slave node. This can happen in a number of circumstances:</p> <list type="bulleted"> <item>Erlang/OTP is not installed on the remote host</item> <item>the file system on the other host has a different structure to the the master</item> <item>the Erlang nodes have different cookies.</item> </list> </item> <dt><code>no_rsh</code></dt> <item> <p>There is no <code>rsh</code> program on the computer.</p> </item> <dt><code>{already_running, <anno>Node</anno>}</code></dt> <item> <p>A node with the name <code><anno>Name</anno>@<anno>Host</anno></code> already exists.</p> </item> </taglist> </div> </div> <div class="function"> <h3 id="start_link/1">start_link/1</h3> <h3 id="start_link/2">start_link/2</h3> <h3 id="start_link/3">start_link/3</h3> <div class="description"> <p>Starts a slave node in the same way as <code>start/1,2,3</code>, except that the slave node is linked to the currently executing process. If that process terminates, the slave node also terminates.</p> <p>See <code>start/1,2,3</code> for a description of arguments and return values.</p> </div> </div> <div class="function"> <h3 id="stop/1">stop/1</h3> <div class="description"> <p>Stops (kills) a node.</p> </div> </div> <div class="function"> <h3 id="pseudo/1">pseudo([Master | ServerList]) -&gt; ok</h3> <ul class="type"> <li><code>Master = node()</code></li> <li><code>ServerList = [atom()]</code></li> </ul> <div class="description"> <p>Calls <code>pseudo(Master, ServerList)</code>. If we want to start a node from the command line and set up a number of pseudo servers, an Erlang runtime system can be started as follows:</p> <pre class="sh_erlang"> % erl -name abc -s slave pseudo klacke@super x --</pre> </div> </div> <div class="function"> <h3 id="pseudo/2">pseudo/2</h3> <div class="description"> <p>Starts a number of pseudo servers. A pseudo server is a server with a registered name which does absolutely nothing but pass on all message to the real server which executes at a master node. A pseudo server is an intermediary which only has the same registered name as the real server.</p> <p>For example, if we have started a slave node <code>N</code> and want to execute <code>pxw</code> graphics code on this node, we can start the server <code>pxw_server</code> as a pseudo server at the slave node. The following code illustrates:</p> <pre class="sh_erlang"> rpc:call(N, slave, pseudo, [node(), [pxw_server]]).</pre> </div> </div> <div class="function"> <h3 id="relay/1">relay/1</h3> <div class="description"> <p>Runs a pseudo server. This function never returns any value and the process which executes the function will receive messages. All messages received will simply be passed on to <code><anno>Pid</anno></code>.</p> </div> </div> </div>
