<h1>proc_lib</h1> <h2 class="modsummary">Functions for asynchronous and synchronous start of processes adhering to the OTP design principles.</h2> <div class="description"> <p>This module is used to start processes adhering to the <a href="../doc/design_principles/des_princ.html" class="seealso">OTP Design Principles</a>. Specifically, the functions in this module are used by the OTP standard behaviors (<code>gen_server</code>, <code>gen_fsm</code>, ...) when starting new processes. The functions can also be used to start <em>special processes</em>, user defined processes which comply to the OTP design principles. See <a href="../doc/design_principles/spec_proc.html" class="seealso">Sys and Proc_Lib</a> in OTP Design Principles for an example.</p> <p>Some useful information is initialized when a process starts. The registered names, or the process identifiers, of the parent process, and the parent ancestors, are stored together with information about the function initially called in the process.</p> <p>While in "plain Erlang" a process is said to terminate normally only for the exit reason <code>normal</code>, a process started using <code>proc_lib</code> is also said to terminate normally if it exits with reason <code>shutdown</code> or <code>{shutdown,Term}</code>. <code>shutdown</code> is the reason used when an application (supervision tree) is stopped.</p> <p>When a process started using <code>proc_lib</code> terminates abnormally -- that is, with another exit reason than <code>normal</code>, <code>shutdown</code>, or <code>{shutdown,Term}</code> -- a <em>crash report</em> is generated, which is written to terminal by the default SASL event handler. That is, the crash report is normally only visible if the SASL application is started. See <a href="../sasl/sasl_app.html" class="seealso">sasl(6)</a> and <a href="../sasl/error_logging.html" class="seealso">SASL User's Guide</a>.</p> <p>The crash report contains the previously stored information such as ancestors and initial function, the termination reason, and information regarding other processes which terminate as a result of this process terminating.</p> </div> <datatypes> <datatype> <name name="spawn_option" /> <div class="description"> <p>See <a href="../erts/erlang.html#spawn_opt/4" class="seealso"> erlang:spawn_opt/2,3,4,5</a>.</p> </div> </datatype> <datatype> <name name="priority_level" /> </datatype> <datatype> <name name="dict_or_pid" /> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="spawn/1">spawn/1</h3> <h3 id="spawn/2">spawn/2</h3> <h3 id="spawn/3">spawn/3</h3> <h3 id="spawn/4">spawn/4</h3> <type variable="Node" /> <type variable="Fun" name_i="1" /> <type variable="Module" /> <type variable="Function" /> <type variable="Args" /> <div class="description"> <p>Spawns a new process and initializes it as described above. The process is spawned using the <a href="../erts/erlang.html#spawn_link/1" class="seealso">spawn_link</a> BIFs.</p> </div> </div> <div class="function"> <h3 id="spawn_opt/2">spawn_opt/2</h3> <h3 id="spawn_opt/3">spawn_opt/3</h3> <h3 id="spawn_opt/4">spawn_opt/4</h3> <h3 id="spawn_opt/5">spawn_opt/5</h3> <type variable="Node" /> <type variable="Fun" name_i="1" /> <type variable="Module" /> <type variable="Function" /> <type variable="Args" /> <type variable="SpawnOpts" /> <div class="description"> <p>Spawns a new process and initializes it as described above. The process is spawned using the <a href="../erts/erlang.html#spawn_opt/2" class="seealso">spawn_opt</a> BIFs.</p> <div class="note"><h2>Note!</h2> <p>Using the spawn option <code>monitor</code> is currently not allowed, but will cause the function to fail with reason <code>badarg</code>.</p> </div> </div> </div> <div class="function"> <h3 id="start/3">start/3</h3> <h3 id="start/4">start/4</h3> <h3 id="start/5">start/5</h3> <h3 id="start_link/3">start_link/3</h3> <h3 id="start_link/4">start_link/4</h3> <h3 id="start_link/5">start_link/5</h3> <div class="description"> <p>Starts a new process synchronously. Spawns the process and waits for it to start. When the process has started, it <em>must</em> call <a href="#init_ack/2" class="seealso">init_ack(Parent,Ret)</a> or <a href="#init_ack/1" class="seealso">init_ack(Ret)</a>, where <code>Parent</code> is the process that evaluates this function. At this time, <code>Ret</code> is returned.</p> <p>If the <code>start_link/3,4,5</code> function is used and the process crashes before it has called <code>init_ack/1,2</code>, <code>{error, <anno>Reason</anno>}</code> is returned if the calling process traps exits.</p> <p>If <code><anno>Time</anno></code> is specified as an integer, this function waits for <code><anno>Time</anno></code> milliseconds for the new process to call <code>init_ack</code>, or <code>{error, timeout}</code> is returned, and the process is killed.</p> <p>The <code><anno>SpawnOpts</anno></code> argument, if given, will be passed as the last argument to the <code>spawn_opt/2,3,4,5</code> BIF.</p> <div class="note"><h2>Note!</h2> <p>Using the spawn option <code>monitor</code> is currently not allowed, but will cause the function to fail with reason <code>badarg</code>.</p> </div> </div> </div> <div class="function"> <h3 id="init_ack/1">init_ack/1</h3> <h3 id="init_ack/2">init_ack/2</h3> <div class="description"> <p>This function must used by a process that has been started by a <a href="#start/3" class="seealso">start[_link]/3,4,5</a> function. It tells <code><anno>Parent</anno></code> that the process has initialized itself, has started, or has failed to initialize itself.</p> <p>The <code>init_ack/1</code> function uses the parent value previously stored by the start function used.</p> <p>If this function is not called, the start function will return an error tuple (if a link and/or a timeout is used) or hang otherwise.</p> <p>The following example illustrates how this function and <code>proc_lib:start_link/3</code> are used.</p> <pre class="sh_erlang"> -module(my_proc). -export([start_link/0]). -export([init/1]). start_link() -&gt; proc_lib:start_link(my_proc, init, [self()]). init(Parent) -&gt; case do_initialization() of ok -&gt; proc_lib:init_ack(Parent, {ok, self()}); {error, Reason} -&gt; exit(Reason) end, loop(). ...</pre> </div> </div> <div class="function"> <h3 id="format/1">format/1</h3> <div class="description"> <p>Equivalent to <code>format(<anno>CrashReport</anno>, latin1)</code>.</p> </div> </div> <div class="function"> <h3 id="format/2">format/2</h3> <div class="description"> <p>This function can be used by a user defined event handler to format a crash report. The crash report is sent using <code>error_logger:error_report(crash_report, <anno>CrashReport</anno>)</code>. That is, the event to be handled is of the format <code>{error_report, GL, {Pid, crash_report, <anno>CrashReport</anno>}}</code> where <code>GL</code> is the group leader pid of the process <code>Pid</code> which sent the crash report.</p> </div> </div> <div class="function"> <h3 id="initial_call/1">initial_call/1</h3> <div class="description"> <p>Extracts the initial call of a process that was started using one of the spawn or start functions described above. <code><anno>Process</anno></code> can either be a pid, an integer tuple (from which a pid can be created), or the process information of a process <code>Pid</code> fetched through an <code>erlang:process_info(Pid)</code> function call.</p> <div class="note"><h2>Note!</h2><p>The list <code><anno>Args</anno></code> no longer contains the actual arguments, but the same number of atoms as the number of arguments; the first atom is always <code>'Argument__1'</code>, the second <code>'Argument__2'</code>, and so on. The reason is that the argument list could waste a significant amount of memory, and if the argument list contained funs, it could be impossible to upgrade the code for the module.</p> <p>If the process was spawned using a fun, <code>initial_call/1</code> no longer returns the actual fun, but the module, function for the local function implementing the fun, and the arity, for instance <code>{some_module,-work/3-fun-0-,0}</code> (meaning that the fun was created in the function <code>some_module:work/3</code>). The reason is that keeping the fun would prevent code upgrade for the module, and that a significant amount of memory could be wasted.</p> </div> </div> </div> <div class="function"> <h3 id="translate_initial_call/1">translate_initial_call/1</h3> <div class="description"> <p>This function is used by the <code>c:i/0</code> and <code>c:regs/0</code> functions in order to present process information.</p> <p>Extracts the initial call of a process that was started using one of the spawn or start functions described above, and translates it to more useful information. <code><anno>Process</anno></code> can either be a pid, an integer tuple (from which a pid can be created), or the process information of a process <code>Pid</code> fetched through an <code>erlang:process_info(Pid)</code> function call.</p> <p>If the initial call is to one of the system defined behaviors such as <code>gen_server</code> or <code>gen_event</code>, it is translated to more useful information. If a <code>gen_server</code> is spawned, the returned <code><anno>Module</anno></code> is the name of the callback module and <code><anno>Function</anno></code> is <code>init</code> (the function that initiates the new server).</p> <p>A <code>supervisor</code> and a <code>supervisor_bridge</code> are also <code>gen_server</code> processes. In order to return information that this process is a supervisor and the name of the call-back module, <code><anno>Module</anno></code> is <code>supervisor</code> and <code><anno>Function</anno></code> is the name of the supervisor callback module. <code><anno>Arity</anno></code> is <code>1</code> since the <code>init/1</code> function is called initially in the callback module.</p> <p>By default, <code>{proc_lib,init_p,5}</code> is returned if no information about the initial call can be found. It is assumed that the caller knows that the process has been spawned with the <code>proc_lib</code> module.</p> </div> </div> <div class="function"> <h3 id="hibernate/3">hibernate/3</h3> <div class="description"> <p>This function does the same as (and does call) the BIF <a href="../erts/erlang.html#erlanghibernate/3" class="seealso">hibernate/3</a>, but ensures that exception handling and logging continues to work as expected when the process wakes up. Always use this function instead of the BIF for processes started using <code>proc_lib</code> functions.</p> </div> </div> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="../kernel/error_logger.html" class="seealso">error_logger(3)</a></p> </div>
