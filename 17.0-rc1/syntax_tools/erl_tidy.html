<h1>erl_tidy</h1> <h2 class="modsummary">Tidies and pretty-prints Erlang source code, removing unused functions, updating obsolete constructs and function calls, etc.</h2> <div class="description"> <p>Tidies and pretty-prints Erlang source code, removing unused functions, updating obsolete constructs and function calls, etc.</p> <p>Caveats: It is possible that in some intricate uses of macros, the automatic addition or removal of parentheses around uses or arguments could cause the resulting program to be rejected by the compiler; however, we have found no such case in existing code. Programs defining strange macros can usually not be read by this program, and in those cases, no changes will be made.</p> <p>If you really, really want to, you may call it "Inga".</p> <p>Disclaimer: The author accepts no responsibility for errors introduced in code that has been processed by the program. It has been reasonably well tested, but the possibility of errors remains. Keep backups of your original code safely stored, until you feel confident that the new, modified code can be trusted.</p></div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="dir/0">dir() -&gt; ok</h3> <div class="description"><span id="dir-0"> </span> <p>Equivalent to <a href="#dir-1" class="seealso">dir("")</a>.</p> </div></div> <div class="function"> <h3 id="dir/1">dir(Dir) -&gt; ok</h3> <div class="description"><span id="dir-1"> </span> <p>Equivalent to <a href="#dir-2" class="seealso">dir(Dir, [])</a>.</p> </div></div> <div class="function"> <h3 id="dir/1-1">dir(Directory::filename(), Options::[term()]) -&gt; ok</h3> <ul class="type"> <li><code>filename() (see module file)</code></li></ul> <div class="description"><span id="dir-2"> </span> <p>Tidies Erlang source files in a directory and its subdirectories.</p> <p>Available options: </p><taglist> <dt>{follow_links, boolean()}</dt> <item><p>If the value is <code>true</code>, symbolic directory links will be followed. The default value is <code>false</code>.</p></item> <dt>{recursive, boolean()}</dt> <item><p>If the value is <code>true</code>, subdirectories will be visited recursively. The default value is <code>true</code>.</p></item> <dt>{regexp, string()}</dt> <item><p>The value denotes a regular expression (see module <code>re</code>). Tidying will only be applied to those regular files whose names match this pattern. The default value is <code>".*\\.erl$"</code>, which matches normal Erlang source file names.</p></item> <dt>{test, boolean()}</dt> <item><p>If the value is <code>true</code>, no files will be modified. The default value is <code>false</code>.</p></item> <dt>{verbose, boolean()}</dt> <item><p>If the value is <code>true</code>, progress messages will be output while the program is running, unless the <code>quiet</code> option is <code>true</code>. The default value when calling <a href="#dir-2" class="seealso">dir/2</a> is <code>true</code>.</p></item> </taglist> <p>See the function <a href="#file-2" class="seealso">file/2</a> for further options. </p> <p><em>See also:</em> <a href="../http///www.erlang.org/edoc/doc/stdlib/doc/re.xml.html" class="seealso">//stdlib/re</a>, <a href="#file-2" class="seealso">file/2</a>.</p> </div></div> <div class="function"> <h3 id="file/1">file(Name) -&gt; ok</h3> <div class="description"><span id="file-1"> </span> <p>Equivalent to <a href="#file-2" class="seealso">file(Name, [])</a>.</p> </div></div> <div class="function"> <h3 id="file/1-1">file(Name::filename(), Options::[term()]) -&gt; ok</h3> <div class="description"><span id="file-2"> </span> <p>Tidies an Erlang source code file.</p> <p>Available options are: </p><taglist> <dt>{backup_suffix, string()}</dt> <item><p>Specifies the file name suffix to be used when a backup file is created; the default value is <code>".bak"</code> (cf. the <code>backups</code> option).</p></item> <dt>{backups, boolean()}</dt> <item><p>If the value is <code>true</code>, existing files will be renamed before new files are opened for writing. The new names are formed by appending the string given by the <code>backup_suffix</code> option to the original name. The default value is <code>true</code>.</p></item> <dt>{dir, filename()}</dt> <item><p>Specifies the name of the directory in which the output file is to be written. By default, the current directory is used. If the value is an empty string, the current directory is used. </p></item> <dt>{outfile, filename()}</dt> <item><p>Specifies the name of the file (without suffix) to which the resulting source code is to be written. If this option is not specified, the <code>Name</code> argument is used.</p></item> <dt>{printer, Function}</dt> <item><list> <item><p><code>Function = (syntaxTree()) -&gt; string()</code></p></item> </list><p> Specifies a function for prettyprinting Erlang syntax trees. This is used for outputting the resulting module definition. The function is assumed to return formatted text for the given syntax tree, and should raise an exception if an error occurs. The default formatting function calls <code>erl_prettypr:format/2</code>.</p></item> <dt>{test, boolean()}</dt> <item><p>If the value is <code>true</code>, no files will be modified; this is typically most useful if the <code>verbose</code> flag is enabled, to generate reports about the program files without affecting them. The default value is <code>false</code>.</p></item> <dt>{stdout, boolean()}</dt> <item><p>If the value is <code>true</code>, instead of the file being written to disk it will be printed to stdout. The default value is <code>false</code>.</p></item> </taglist> <p>See the function <code>module/2</code> for further options. </p> <p><em>See also:</em> <a href="#module-2" class="seealso">module/2</a>, <a href="erl_prettypr.html#format-2" class="seealso">erl_prettypr:format/2</a>.</p> </div></div> <div class="function"> <h3 id="module/1">module(Forms) -&gt; syntaxTree()</h3> <div class="description"><span id="module-1"> </span> <p>Equivalent to <a href="#module-2" class="seealso">module(Forms, [])</a>.</p> </div></div> <div class="function"> <h3 id="module/2">module(Forms, Options::[term()]) -&gt; syntaxTree()</h3> <ul class="type"> <li><code>Forms = syntaxTree() | [syntaxTree()]</code></li><li><code>syntaxTree() (see module erl_syntax)</code></li></ul> <div class="description"><span id="module-2"> </span> <p>Tidies a syntax tree representation of a module definition. The given <code>Forms</code> may be either a single syntax tree of type <code>form_list</code>, or a list of syntax trees representing "program forms". In either case, <code>Forms</code> must represent a single complete module definition. The returned syntax tree has type <code>form_list</code> and represents a tidied-up version of the same source code.</p> <p>Available options are: </p><taglist> <dt>{auto_export_vars, boolean()}</dt> <item><p>If the value is <code>true</code>, all matches "<code>{V1, ..., Vn} = E</code>" where <code>E</code> is a case-, if- or receive-expression whose branches all return n-tuples (or explicitly throw exceptions) will be rewritten to bind and export the variables <code>V1</code>, ..., <code>Vn</code> directly. The default value is <code>false</code>.</p> <p>For example: </p><pre class="sh_erlang"> {X, Y} = case ... of ... -&gt; {17, foo()}; ... -&gt; {42, bar()} end </pre><p> will be rewritten to: </p><pre class="sh_erlang"> case ... of ... -&gt; X = 17, Y = foo(), {X, Y}; ... -&gt; X = 42, Y = bar(), {X, Y} end </pre></item> <dt>{auto_list_comp, boolean()}</dt> <item><p>If the value is <code>true</code>, calls to <code>lists:map/2</code> and <code>lists:filter/2</code> will be rewritten using list comprehensions. The default value is <code>true</code>.</p></item> <dt>{file, string()}</dt> <item><p>Specifies the name of the file from which the source code was taken. This is only used for generation of error reports. The default value is the empty string.</p></item> <dt>{idem, boolean()}</dt> <item><p>If the value is <code>true</code>, all options that affect how the code is modified are set to "no changes". For example, to only update guard tests, and nothing else, use the options <code>[new_guard_tests, idem]</code>. (Recall that options closer to the beginning of the list have higher precedence.)</p></item> <dt>{keep_unused, boolean()}</dt> <item><p>If the value is <code>true</code>, unused functions will not be removed from the code. The default value is <code>false</code>.</p></item> <dt>{new_guard_tests, boolean()}</dt> <item><p>If the value is <code>true</code>, guard tests will be updated to use the new names, e.g. "<code>is_integer(X)</code>" instead of "<code>integer(X)</code>". The default value is <code>true</code>. See also <code>old_guard_tests</code>.</p></item> <dt>{no_imports, boolean()}</dt> <item><p>If the value is <code>true</code>, all import statements will be removed and calls to imported functions will be expanded to explicit remote calls. The default value is <code>false</code>.</p></item> <dt>{old_guard_tests, boolean()}</dt> <item><p>If the value is <code>true</code>, guard tests will be changed to use the old names instead of the new ones, e.g. "<code>integer(X)</code>" instead of "<code>is_integer(X)</code>". The default value is <code>false</code>. This option overrides the <code>new_guard_tests</code> option.</p></item> <dt>{quiet, boolean()}</dt> <item><p>If the value is <code>true</code>, all information messages and warning messages will be suppressed. The default value is <code>false</code>.</p></item> <dt>{rename, [{{atom(), atom(), integer()}, {atom(), atom()}}]}</dt> <item><p>The value is a list of pairs, associating tuples <code>{Module, Name, Arity}</code> with tuples <code>{NewModule, NewName}</code>, specifying renamings of calls to remote functions. By default, the value is the empty list.</p> <p>The renaming affects only remote calls (also when disguised by import declarations); local calls within a module are not affected, and no function definitions are renamed. Since the arity cannot change, the new name is represented by <code>{NewModule, NewName}</code> only. Only calls matching the specified arity will match; multiple entries are necessary for renaming calls to functions that have the same module and function name, but different arities.</p> <p>This option can also be used to override the default renaming of calls which use obsolete function names.</p></item> <dt>{verbose, boolean()}</dt> <item><p>If the value is <code>true</code>, progress messages will be output while the program is running, unless the <code>quiet</code> option is <code>true</code>. The default value is <code>false</code>.</p></item> </taglist> </div></div></div> <authors> <aname>Richard Carlsson</aname> <email>carlsson.richard@gmail.com</email></authors>
