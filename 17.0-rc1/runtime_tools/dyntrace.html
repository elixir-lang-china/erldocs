<h1>dyntrace</h1> <h2 class="modsummary">Interface to dynamic tracing</h2> <div class="description"> <p>This module implements interfaces to dynamic tracing, should such be compiled into the virtual machine. For a standard and/or commercial build, no dynamic tracing is available, in which case none of the functions in this module is usable or give any effect.</p> <p>Should dynamic tracing be enabled in the current build, either by configuring with <code>./configure --with-dynamic-trace=dtrace</code> or with <code>./configure --with-dynamic-trace=systemtap</code>, the module can be used for two things:</p> <list type="bulleted"> <item>Trigger the user-probe <code>user_trace_i4s4</code> in the NIF library <code>dyntrace.so</code> by calling <code>dyntrace:p/{1,2,3,4,5,6,7,8}</code>.</item> <item>Set a user specified tag that will be present in the trace messages of both the <code>efile_drv</code> and the user-probe mentioned above.</item> </list> <p>Both building with dynamic trace probes and using them is experimental and unsupported by Erlang/OTP. It is included as an option for the developer to trace and debug performance issues in their systems.</p> <p>The original implementation is mostly done by Scott Lystiger Fritchie as an Open Source Contribution and it should be viewed as such even though the source for dynamic tracing as well as this module is included in the main distribution. However, the ability to use dynamic tracing of the virtual machine is a very valuable contribution which OTP has every intention to maintain as a tool for the developer.</p> <p>How to write <code>d</code> programs or <code>systemtap</code> scripts can be learned from books and from a lot of pages on the Internet. This manual page does not include any documentation about using the dynamic trace tools of respective platform. The <code>examples</code> directory of the <code>runtime_tools</code> application however contains comprehensive examples of both <code>d</code> and <code>systemtap</code> programs that will help you get started. Another source of information is the <a href="DTRACE.html" class="seealso">dtrace</a> and <a href="SYSTEMTAP.html" class="seealso">systemtap</a> chapters in the Runtime Tools Users' Guide.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="available/0">available() -&gt; boolean()</h3> <div class="description"> <p>This function uses the NIF library to determine if dynamic tracing is available. For an explanation of how user tags can be spread together with messages, see <a href="#spread_tag/1" class="seealso">spread_tag/1</a>. If no tag is found or dynamic tracing is not available, it returns <code>undefined</code></p> </div> </div> <div class="function"> <h3 id="put_tag/1">put_tag(Item) -&gt; binary() | undefined </h3> <ul class="type"> <li><code>Item = iodata()</code></li> </ul> <div class="description"> <p>This function sets the user tag of the current process. The user tag is a binary(), but can be specified as any iodata(), which is automatically converted to a binary by this function.</p> <p>The user tag is provided to the user probes triggered by calls top <code>dyntrace:p/{1,2,3,4,5,6,7,8}</code> as well as probes in the efile_driver. In the future, user tags might be added to more probes.</p> <p>The old user tag (if any) is returned, or <code>undefined</code> if no user tag was present or dynamic tracing is not enabled.</p> </div> </div> <div class="function"> <h3 id="spread_tag/1">spread_tag(boolean()) -&gt; TagData</h3> <ul class="type"> <li><code>TagData = opaque data that can be used as parameter to <a href="#restore_tag/1" class="seealso">restore_tag/1</a></code></li> </ul> <div class="description"> <p>This function controls if user tags are to be spread to other processes with the next message. Spreading of user tags work like spreading of sequential trace tokens, so that a received user tag will be active in the process until the next message arrives (if that message does not also contain the user tag.</p> <p>This functionality is used when a client process communicates with a file i/o-server to spread the user tag to the I/O-server and then down to the efile_drv driver. By using <code>spread_tag/1</code> and <code>restore_tag/1</code>, one can enable or disable spreading of user tags to other processes and then restore the previous state of the user tag. The TagData returned from this call contains all previous information so the state (including any previously spread user tags) will be completely restored by a later call to <code>restore_tag/1</code>.</p> <p>The <a href="../kernel/file.html" class="seealso">file</a> module already spread's tags, so there is noo need to manually call these function to get user tags spread to the efile driver through that module.</p> <p>The most use of this function would be if one for example uses the <a href="../stdlib/io.html" class="seealso">io</a> module to communicate with an I/O-server for a regular file, like in the following example:</p> <pre class="sh_erlang"> f() -&gt; {ok, F} = file:open("test.tst",[write]), Saved = dyntrace:spread_tag(true), io:format(F,"Hello world!",[]), dyntrace:restore_tag(Saved), file:close(F). </pre> <p>In this example, any user tag set in the calling process will be spread to the I/O-server when the io:format call is done.</p> </div> </div> <div class="function"> <h3 id="restore_tag/1">restore_tag(TagData) -&gt; true</h3> <ul class="type"> <li><code>TagData = opaque data returned by <a href="#spread_tag/1" class="seealso">spread_tag/1</a></code></li> </ul> <div class="description"> <p>Restores the previous state of user tags and their spreading as it was before a call to <a href="#spread_tag/1" class="seealso">spread_tag/1</a>. Note that the restoring is not limited to the same process, one can utilize this to turn off spreding in one process and restore it in a newly created, the one that actually is going to send messages:</p> <pre class="sh_erlang"> f() -&gt; TagData=dyntrace:spread_tag(false), spawn(fun() -&gt; dyntrace:restore_tag(TagData), do_something() end), do_something_else(), dyntrace:restore_tag(TagData). </pre> <p>Correctly handling user tags and their spreading might take some effort, as Erlang programs tend to send and receive messages so that sometimes the user tag gets lost due to various things, like double receives or communication with a port (ports do not handle user tags, in the same way as they do not handle regular sequential trace tokens).</p> </div> </div> </div>
