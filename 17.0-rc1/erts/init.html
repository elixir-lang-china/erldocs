<h1>init</h1> <h2 class="modsummary">Coordination of System Startup</h2> <div class="description"> <p>The <code>init</code> module is pre-loaded and contains the code for the <code>init</code> system process which coordinates the start-up of the system. The first function evaluated at start-up is <code>boot(BootArgs)</code>, where <code>BootArgs</code> is a list of command line arguments supplied to the Erlang runtime system from the local operating system. See <a href="../erts/erl.html" class="seealso">erl(1)</a>.</p> <p><code>init</code> reads the boot script which contains instructions on how to initiate the system. See <a href="../sasl/script.html" class="seealso">script(4)</a> for more information about boot scripts.</p> <p><code>init</code> also contains functions to restart, reboot, and stop the system.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="boot/1">boot/1</h3> <div class="description"> <p>Starts the Erlang runtime system. This function is called when the emulator is started and coordinates system start-up.</p> <p><code><anno>BootArgs</anno></code> are all command line arguments except the emulator flags, that is, flags and plain arguments. See <a href="../erts/erl.html" class="seealso">erl(1)</a>.</p> <p><code>init</code> itself interprets some of the flags, see <a href="#flags" class="seealso">Command Line Flags</a> below. The remaining flags ("user flags") and plain arguments are passed to the <code>init</code> loop and can be retrieved by calling <code>get_arguments/0</code> and <code>get_plain_arguments/0</code>, respectively.</p> </div> </div> <div class="function"> <h3 id="get_argument/1">get_argument/1</h3> <div class="description"> <p>Returns all values associated with the command line user flag <code><anno>Flag</anno></code>. If <code><anno>Flag</anno></code> is provided several times, each <code><anno>Values</anno></code> is returned in preserved order.</p> <pre class="sh_erlang"> % <code>erl -a b c -a d</code> ... 1&gt; <code>init:get_argument(a).</code> {ok,[["b","c"],["d"]]}</pre> <p>There are also a number of flags, which are defined automatically and can be retrieved using this function:</p> <taglist> <dt><code>root</code></dt> <item> <p>The installation directory of Erlang/OTP, <code>$ROOT</code>.</p> <pre class="sh_erlang"> 2&gt; <code>init:get_argument(root).</code> {ok,[["/usr/local/otp/releases/otp_beam_solaris8_r10b_patched"]]}</pre> </item> <dt><code>progname</code></dt> <item> <p>The name of the program which started Erlang.</p> <pre class="sh_erlang"> 3&gt; <code>init:get_argument(progname).</code> {ok,[["erl"]]}</pre> </item> <dt><code>home</code></dt> <item> <p>The home directory.</p> <pre class="sh_erlang"> 4&gt; <code>init:get_argument(home).</code> {ok,[["/home/harry"]]}</pre> </item> </taglist> <p>Returns <code>error</code> if there is no value associated with <code>Flag</code>.</p> </div> </div> <div class="function"> <h3 id="get_arguments/0">get_arguments/0</h3> <div class="description"> <p>Returns all command line flags, as well as the system defined flags, see <code>get_argument/1</code>.</p> </div> </div> <div class="function"> <h3 id="get_plain_arguments/0">get_plain_arguments/0</h3> <div class="description"> <p>Returns any plain command line arguments as a list of strings (possibly empty).</p> </div> </div> <div class="function"> <h3 id="get_status/0">get_status/0</h3> <type name="internal_status" /> <div class="description"> <p>The current status of the <code>init</code> process can be inspected. During system startup (initialization), <code><anno>InternalStatus</anno></code> is <code>starting</code>, and <code><anno>ProvidedStatus</anno></code> indicates how far the boot script has been interpreted. Each <code>{progress, Info}</code> term interpreted in the boot script affects <code><anno>ProvidedStatus</anno></code>, that is, <code><anno>ProvidedStatus</anno></code> gets the value of <code>Info</code>.</p> </div> </div> <div class="function"> <h3 id="reboot/0">reboot/0</h3> <div class="description"> <p>All applications are taken down smoothly, all code is unloaded, and all ports are closed before the system terminates. If the <code>-heart</code> command line flag was given, the <code>heart</code> program will try to reboot the system. Refer to <code>heart(3)</code> for more information.</p> <p>To limit the shutdown time, the time <code>init</code> is allowed to spend taking down applications, the <code>-shutdown_time</code> command line flag should be used.</p> </div> </div> <div class="function"> <h3 id="restart/0">restart/0</h3> <div class="description"> <p>The system is restarted <em>inside</em> the running Erlang node, which means that the emulator is not restarted. All applications are taken down smoothly, all code is unloaded, and all ports are closed before the system is booted again in the same way as initially started. The same <code>BootArgs</code> are used again.</p> <p>To limit the shutdown time, the time <code>init</code> is allowed to spend taking down applications, the <code>-shutdown_time</code> command line flag should be used.</p> </div> </div> <div class="function"> <h3 id="script_id/0">script_id/0</h3> <div class="description"> <p>Get the identity of the boot script used to boot the system. <code><anno>Id</anno></code> can be any Erlang term. In the delivered boot scripts, <code><anno>Id</anno></code> is <code>{Name, Vsn}</code>. <code>Name</code> and <code>Vsn</code> are strings.</p> </div> </div> <div class="function"> <h3 id="stop/0">stop/0</h3> <div class="description"> <p>All applications are taken down smoothly, all code is unloaded, and all ports are closed before the system terminates. If the <code>-heart</code> command line flag was given, the <code>heart</code> program is terminated before the Erlang node terminates. Refer to <code>heart(3)</code> for more information.</p> <p>To limit the shutdown time, the time <code>init</code> is allowed to spend taking down applications, the <code>-shutdown_time</code> command line flag should be used.</p> </div> </div> <div class="function"> <h3 id="stop/1">stop/1</h3> <div class="description"> <p>All applications are taken down smoothly, all code is unloaded, and all ports are closed before the system terminates by calling <code>halt(<anno>Status</anno>)</code>. If the <code>-heart</code> command line flag was given, the <code>heart</code> program is terminated before the Erlang node terminates. Refer to <code>heart(3)</code> for more information.</p> <p>To limit the shutdown time, the time <code>init</code> is allowed to spend taking down applications, the <code>-shutdown_time</code> command line flag should be used.</p> </div> </div> </div> <div class="section"> <span id="flags"> </span> <h4>Command Line Flags</h4> <div class="warning"><h2>Warning!</h2><p>The support for loading of code from archive files is experimental. The sole purpose of releasing it before it is ready is to obtain early feedback. The file format, semantics, interfaces etc. may be changed in a future release. The <code>-code_path_choice</code> flag is also experimental.</p></div> <p>The <code>init</code> module interprets the following command line flags:</p> <taglist> <dt><code>--</code></dt> <item> <p>Everything following <code>--</code> up to the next flag is considered plain arguments and can be retrieved using <code>get_plain_arguments/0</code>.</p> </item> <dt><code>-code_path_choice Choice</code></dt> <item> <p>This flag can be set to <code>strict</code> or <code>relaxed</code>. It controls whether each directory in the code path should be interpreted strictly as it appears in the <code>boot script</code> or if <code>init</code> should be more relaxed and try to find a suitable directory if it can choose from a regular ebin directory and an ebin directory in an archive file. This flag is particular useful when you want to elaborate with code loading from archives without editing the <code>boot script</code>. See <a href="../sasl/script.html" class="seealso">script(4)</a> for more information about interpretation of boot scripts. The flag does also have a similar affect on how the code server works. See <a href="../kernel/code.html" class="seealso">code(3)</a>.</p> </item> <dt><code>-eval Expr</code></dt> <item> <p>Scans, parses and evaluates an arbitrary expression <code>Expr</code> during system initialization. If any of these steps fail (syntax error, parse error or exception during evaluation), Erlang stops with an error message. Here is an example that seeds the random number generator:</p> <pre class="sh_erlang"> % <code>erl -eval '{X,Y,Z}' = now(), random:seed(X,Y,Z).'</code></pre> <p>This example uses Erlang as a hexadecimal calculator:</p> <pre class="sh_erlang"> % <code>erl -noshell -eval 'R = 16#1F+16#A0, io:format("~.16B~n", [R])' \\</code> <code>-s erlang halt</code> BF</pre> <p>If multiple <code>-eval</code> expressions are specified, they are evaluated sequentially in the order specified. <code>-eval</code> expressions are evaluated sequentially with <code>-s</code> and <code>-run</code> function calls (this also in the order specified). As with <code>-s</code> and <code>-run</code>, an evaluation that does not terminate, blocks the system initialization process.</p> </item> <dt><code>-extra</code></dt> <item> <p>Everything following <code>-extra</code> is considered plain arguments and can be retrieved using <code>get_plain_arguments/0</code>.</p> </item> <dt><code>-run Mod [Func [Arg1, Arg2, ...]]</code></dt> <item> <p>Evaluates the specified function call during system initialization. <code>Func</code> defaults to <code>start</code>. If no arguments are provided, the function is assumed to be of arity 0. Otherwise it is assumed to be of arity 1, taking the list <code>[Arg1,Arg2,...]</code> as argument. All arguments are passed as strings. If an exception is raised, Erlang stops with an error message.</p> <p>Example:</p> <pre class="sh_erlang"> % <code>erl -run foo -run foo bar -run foo bar baz 1 2</code></pre> <p>This starts the Erlang runtime system and evaluates the following functions:</p> <pre class="sh_erlang"> foo:start() foo:bar() foo:bar(["baz", "1", "2"]).</pre> <p>The functions are executed sequentially in an initialization process, which then terminates normally and passes control to the user. This means that a <code>-run</code> call which does not return will block further processing; to avoid this, use some variant of <code>spawn</code> in such cases.</p> </item> <dt><code>-s Mod [Func [Arg1, Arg2, ...]]</code></dt> <item> <p>Evaluates the specified function call during system initialization. <code>Func</code> defaults to <code>start</code>. If no arguments are provided, the function is assumed to be of arity 0. Otherwise it is assumed to be of arity 1, taking the list <code>[Arg1,Arg2,...]</code> as argument. All arguments are passed as atoms. If an exception is raised, Erlang stops with an error message.</p> <p>Example:</p> <pre class="sh_erlang"> % <code>erl -s foo -s foo bar -s foo bar baz 1 2</code></pre> <p>This starts the Erlang runtime system and evaluates the following functions:</p> <pre class="sh_erlang"> foo:start() foo:bar() foo:bar([baz, '1', '2']).</pre> <p>The functions are executed sequentially in an initialization process, which then terminates normally and passes control to the user. This means that a <code>-s</code> call which does not return will block further processing; to avoid this, use some variant of <code>spawn</code> in such cases.</p> <p>Due to the limited length of atoms, it is recommended that <code>-run</code> be used instead.</p> </item> </taglist> </div> <div class="section"> <h4>Example</h4> <pre class="sh_erlang"> % <code>erl -- a b -children thomas claire -ages 7 3 -- x y</code> ... 1&gt; <code>init:get_plain_arguments().</code> ["a","b","x","y"] 2&gt; <code>init:get_argument(children).</code> {ok,[["thomas","claire"]]} 3&gt; <code>init:get_argument(ages).</code> {ok, [["7","3"]]} 4&gt; <code>init:get_argument(silly).</code> error</pre> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="erl_prim_loader.html" class="seealso">erl_prim_loader(3)</a>, <a href="../kernel/heart.html" class="seealso">heart(3)</a></p> </div>
