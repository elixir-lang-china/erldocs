<h1>erl_prim_loader</h1> <h2 class="modsummary">Low Level Erlang Loader</h2> <div class="description"> <p><code>erl_prim_loader</code> is used to load all Erlang modules into the system. The start script is also fetched with this low level loader.</p> <p><code>erl_prim_loader</code> knows about the environment and how to fetch modules. The loader could, for example, fetch files using the file system (with absolute file names as input), or a database (where the binary format of a module is stored).</p> <p>The <code>-loader Loader</code> command line flag can be used to choose the method used by the <code>erl_prim_loader</code>. Two <code>Loader</code> methods are supported by the Erlang runtime system: <code>efile</code> and <code>inet</code>. If another loader is required, then it has to be implemented by the user. The <code>Loader</code> provided by the user must fulfill the protocol defined below, and it is started with the <code>erl_prim_loader</code> by evaluating <code>open_port({spawn,Loader},[binary])</code>.</p> <div class="warning"><h2>Warning!</h2><p>The support for loading of code from archive files is experimental. The sole purpose of releasing it before it is ready is to obtain early feedback. The file format, semantics, interfaces etc. may be changed in a future release. The functions <code>list_dir/1</code> and <code>read_file_info/1</code> as well as the flag <code>-loader_debug</code> are also experimental</p></div> </div> <datatypes> <datatype> <name name="host" /> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="start/3">start/3</h3> <div class="description"> <p>Starts the Erlang low level loader. This function is called by the <code>init</code> process (and module). The <code>init</code> process reads the command line flags <code>-id <anno>Id</anno></code>, <code>-loader <anno>Loader</anno></code>, and <code>-hosts <anno>Hosts</anno></code>. These are the arguments supplied to the <code>start/3</code> function.</p> <p>If <code>-loader</code> is not given, the default loader is <code>efile</code> which tells the system to read from the file system.</p> <p>If <code>-loader</code> is <code>inet</code>, the <code>-id <anno>Id</anno></code>, <code>-hosts <anno>Hosts</anno></code>, and <code>-setcookie Cookie</code> flags must also be supplied. <code><anno>Hosts</anno></code> identifies hosts which this node can contact in order to load modules. One Erlang runtime system with a <code>erl_boot_server</code> process must be started on each of hosts given in <code><anno>Hosts</anno></code> in order to answer the requests. See <a href="../kernel/erl_boot_server.html" class="seealso">erl_boot_server(3)</a>.</p> <p>If <code>-loader</code> is something else, the given port program is started. The port program is supposed to follow the protocol specified below.</p> </div> </div> <div class="function"> <h3 id="get_file/1">get_file/1</h3> <div class="description"> <p>This function fetches a file using the low level loader. <code><anno>Filename</anno></code> is either an absolute file name or just the name of the file, for example <code>"lists.beam"</code>. If an internal path is set to the loader, this path is used to find the file. If a user supplied loader is used, the path can be stripped off if it is obsolete, and the loader does not use a path. <code><anno>FullName</anno></code> is the complete name of the fetched file. <code><anno>Bin</anno></code> is the contents of the file as a binary.</p> <p>The <code><anno>Filename</anno></code> can also be a file in an archive. For example <code>$OTPROOT/lib/</code><code>mnesia-4.4.7.ez/mnesia-4.4.7/ebin/</code><code>mnesia.beam</code>. See <a href="../kernel/code.html" class="seealso">code(3)</a> about archive files.</p> </div> </div> <div class="function"> <h3 id="get_path/0">get_path/0</h3> <div class="description"> <p>This function gets the path set in the loader. The path is set by the <code>init</code> process according to information found in the start script.</p> </div> </div> <div class="function"> <h3 id="list_dir/1">list_dir/1</h3> <div class="description"> <p>Lists all the files in a directory. Returns <code>{ok, <anno>Filenames</anno>}</code> if successful. Otherwise, it returns <code>error</code>. <code><anno>Filenames</anno></code> is a list of the names of all the files in the directory. The names are not sorted.</p> <p>The <code><anno>Dir</anno></code> can also be a directory in an archive. For example <code>$OTPROOT/lib/</code><code>mnesia-4.4.7.ez/mnesia-4.4.7/ebin</code>. See <a href="../kernel/code.html" class="seealso">code(3)</a> about archive files.</p> </div> </div> <div class="function"> <h3 id="read_file_info/1">read_file_info/1</h3> <div class="description"> <p>Retrieves information about a file. Returns <code>{ok, <anno>FileInfo</anno>}</code> if successful, otherwise <code>error</code>. <code><anno>FileInfo</anno></code> is a record <code>file_info</code>, defined in the Kernel include file <code>file.hrl</code>. Include the following directive in the module from which the function is called:</p> <pre class="sh_erlang"> -include_lib("kernel/include/file.hrl").</pre> <p>See <a href="../kernel/file.html" class="seealso">file(3)</a> for more info about the record <code>file_info</code>.</p> <p>The <code><anno>Filename</anno></code> can also be a file in an archive. For example <code>$OTPROOT/lib/</code><code>mnesia-4.4.7.ez/mnesia-4.4.7/ebin/</code><code>mnesia</code>. See <a href="../kernel/code.html" class="seealso">code(3)</a> about archive files.</p> </div> </div> <div class="function"> <h3 id="set_path/1">set_path/1</h3> <div class="description"> <p>This function sets the path of the loader if <code>init</code> interprets a <code>path</code> command in the start script.</p> </div> </div> </div> <div class="section"> <h4>Protocol</h4> <p>The following protocol must be followed if a user provided loader port program is used. The <code>Loader</code> port program is started with the command <code>open_port({spawn,Loader},[binary])</code>. The protocol is as follows:</p> <pre class="sh_erlang"> Function Send Receive ------------------------------------------------------------- get_file [102 | FileName] [121 | BinaryFile] (on success) [122] (failure) stop eof terminate</pre> </div> <div class="section"> <h4>Command Line Flags</h4> <p>The <code>erl_prim_loader</code> module interprets the following command line flags:</p> <taglist> <dt><code>-loader Loader</code></dt> <item> <p>Specifies the name of the loader used by <code>erl_prim_loader</code>. <code>Loader</code> can be <code>efile</code> (use the local file system), or <code>inet</code> (load using the <code>boot_server</code> on another Erlang node). If <code>Loader</code> is user defined, the defined <code>Loader</code> port program is started.</p> <p>If the <code>-loader</code> flag is omitted, it defaults to <code>efile</code>.</p> </item> <dt><code>-loader_debug</code></dt> <item> <p>Makes the <code>efile</code> loader write some debug information, such as the reason for failures, while it handles files.</p> </item> <dt><code>-hosts Hosts</code></dt> <item> <p>Specifies which other Erlang nodes the <code>inet</code> loader can use. This flag is mandatory if the <code>-loader inet</code> flag is present. On each host, there must be on Erlang node with the <code>erl_boot_server</code> which handles the load requests. <code>Hosts</code> is a list of IP addresses (hostnames are not acceptable).</p> </item> <dt><code>-id Id</code></dt> <item> <p>Specifies the identity of the Erlang runtime system. If the system runs as a distributed node, <code>Id</code> must be identical to the name supplied with the <code>-sname</code> or <code>-name</code> distribution flags.</p> </item> <dt><code>-setcookie Cookie</code></dt> <item> <p>Specifies the cookie of the Erlang runtime system. This flag is mandatory if the <code>-loader inet</code> flag is present.</p> </item> </taglist> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="init.html" class="seealso">init(3)</a>, <a href="../kernel/erl_boot_server.html" class="seealso">erl_boot_server(3)</a></p> </div>
