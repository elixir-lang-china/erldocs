<h1>erts_alloc</h1> <libsummary>An Erlang Run-Time System internal memory allocator library.</libsummary> <div class="description"> <p><code>erts_alloc</code> is an Erlang Run-Time System internal memory allocator library. <code>erts_alloc</code> provides the Erlang Run-Time System with a number of memory allocators.</p> </div> <div class="section"> <h4>Allocators</h4> <span id="allocators"> </span> <p>Currently the following allocators are present:</p> <taglist> <dt><code>temp_alloc</code></dt> <item>Allocator used for temporary allocations.</item> <dt><code>eheap_alloc</code></dt> <item>Allocator used for Erlang heap data, such as Erlang process heaps.</item> <dt><code>binary_alloc</code></dt> <item>Allocator used for Erlang binary data.</item> <dt><code>ets_alloc</code></dt> <item>Allocator used for ETS data.</item> <dt><code>driver_alloc</code></dt> <item>Allocator used for driver data.</item> <dt><code>sl_alloc</code></dt> <item>Allocator used for memory blocks that are expected to be short-lived.</item> <dt><code>ll_alloc</code></dt> <item>Allocator used for memory blocks that are expected to be long-lived, for example Erlang code.</item> <dt><code>fix_alloc</code></dt> <item>A fast allocator used for some frequently used fixed size data types.</item> <dt><code>std_alloc</code></dt> <item>Allocator used for most memory blocks not allocated via any of the other allocators described above.</item> <dt><code>sys_alloc</code></dt> <item>This is normally the default <code>malloc</code> implementation used on the specific OS.</item> <dt><code>mseg_alloc</code></dt> <item>A memory segment allocator. <code>mseg_alloc</code> is used by other allocators for allocating memory segments and is currently only available on systems that have the <code>mmap</code> system call. Memory segments that are deallocated are kept for a while in a segment cache before they are destroyed. When segments are allocated, cached segments are used if possible instead of creating new segments. This in order to reduce the number of system calls made.</item> </taglist> <p><code>sys_alloc</code> is always enabled and cannot be disabled. <code>mseg_alloc</code> is always enabled if it is available and an allocator that uses it is enabled. All other allocators can be <a href="#M_e" class="seealso">enabled or disabled</a>. By default all allocators are enabled. When an allocator is disabled, <code>sys_alloc</code> is used instead of the disabled allocator.</p> <p>The main idea with the <code>erts_alloc</code> library is to separate memory blocks that are used differently into different memory areas, and by this achieving less memory fragmentation. By putting less effort in finding a good fit for memory blocks that are frequently allocated than for those less frequently allocated, a performance gain can be achieved.</p> </div> <div class="section"> <span id="alloc_util"> </span> <h4>The alloc_util framework</h4> <p>Internally a framework called <code>alloc_util</code> is used for implementing allocators. <code>sys_alloc</code>, and <code>mseg_alloc</code> do not use this framework; hence, the following does <em>not</em> apply to them.</p> <p>An allocator manages multiple areas, called carriers, in which memory blocks are placed. A carrier is either placed in a separate memory segment (allocated via <code>mseg_alloc</code>), or in the heap segment (allocated via <code>sys_alloc</code>). Multiblock carriers are used for storage of several blocks. Singleblock carriers are used for storage of one block. Blocks that are larger than the value of the singleblock carrier threshold (<a href="#M_sbct" class="seealso">sbct</a>) parameter are placed in singleblock carriers. Blocks that are smaller than the value of the <code>sbct</code> parameter are placed in multiblock carriers. Normally an allocator creates a "main multiblock carrier". Main multiblock carriers are never deallocated. The size of the main multiblock carrier is determined by the value of the <a href="#M_mmbcs" class="seealso">mmbcs</a> parameter.</p> <p><span id="mseg_mbc_sizes"> </span>Sizes of multiblock carriers allocated via <code>mseg_alloc</code> are decided based on the values of the largest multiblock carrier size (<a href="#M_lmbcs" class="seealso">lmbcs</a>), the smallest multiblock carrier size (<a href="#M_smbcs" class="seealso">smbcs</a>), and the multiblock carrier growth stages (<a href="#M_mbcgs" class="seealso">mbcgs</a>) parameters. If <code>nc</code> is the current number of multiblock carriers (the main multiblock carrier excluded) managed by an allocator, the size of the next <code>mseg_alloc</code> multiblock carrier allocated by this allocator will roughly be <code>smbcs+nc*(lmbcs-smbcs)/mbcgs</code> when <code>nc &lt;= mbcgs</code>, and <code>lmbcs</code> when <code>nc &gt; mbcgs</code>. If the value of the <code>sbct</code> parameter should be larger than the value of the <code>lmbcs</code> parameter, the allocator may have to create multiblock carriers that are larger than the value of the <code>lmbcs</code> parameter, though. Singleblock carriers allocated via <code>mseg_alloc</code> are sized to whole pages.</p> <p>Sizes of carriers allocated via <code>sys_alloc</code> are decided based on the value of the <code>sys_alloc</code> carrier size (<a href="#Muycs" class="seealso">ycs</a>) parameter. The size of a carrier is the least number of multiples of the value of the <code>ycs</code> parameter that satisfies the request.</p> <p>Coalescing of free blocks are always performed immediately. Boundary tags (headers and footers) in free blocks are used which makes the time complexity for coalescing constant.</p> <p><span id="strategy"> </span>The memory allocation strategy used for multiblock carriers by an allocator is configurable via the <a href="#M_as" class="seealso">as</a> parameter. Currently the following strategies are available:</p> <taglist> <dt>Best fit</dt> <item> <p>Strategy: Find the smallest block that satisfies the requested block size.</p> <p>Implementation: A balanced binary search tree is used. The time complexity is proportional to log N, where N is the number of sizes of free blocks.</p> </item> <dt>Address order best fit</dt> <item> <p>Strategy: Find the smallest block that satisfies the requested block size. If multiple blocks are found, choose the one with the lowest address.</p> <p>Implementation: A balanced binary search tree is used. The time complexity is proportional to log N, where N is the number of free blocks.</p> </item> <dt>Address order first fit</dt> <item> <p>Strategy: Find the block with the lowest address that satisfies the requested block size.</p> <p>Implementation: A balanced binary search tree is used. The time complexity is proportional to log N, where N is the number of free blocks.</p> </item> <dt>Address order first fit carrier best fit</dt> <item> <p>Strategy: Find the <em>carrier</em> with the lowest address that can satisfy the requested block size, then find a block within that carrier using the "best fit" strategy.</p> <p>Implementation: Balanced binary search trees are used. The time complexity is proportional to log N, where N is the number of free blocks.</p> </item> <dt>Address order first fit carrier address order best fit</dt> <item> <p>Strategy: Find the <em>carrier</em> with the lowest address that can satisfy the requested block size, then find a block within that carrier using the "adress order best fit" strategy.</p> <p>Implementation: Balanced binary search trees are used. The time complexity is proportional to log N, where N is the number of free blocks.</p> </item> <dt>Good fit</dt> <item> <p>Strategy: Try to find the best fit, but settle for the best fit found during a limited search.</p> <p>Implementation: The implementation uses segregated free lists with a maximum block search depth (in each list) in order to find a good fit fast. When the maximum block search depth is small (by default 3) this implementation has a time complexity that is constant. The maximum block search depth is configurable via the <a href="#M_mbsd" class="seealso">mbsd</a> parameter.</p> </item> <dt>A fit</dt> <item> <p>Strategy: Do not search for a fit, inspect only one free block to see if it satisfies the request. This strategy is only intended to be used for temporary allocations.</p> <p>Implementation: Inspect the first block in a free-list. If it satisfies the request, it is used; otherwise, a new carrier is created. The implementation has a time complexity that is constant.</p> <p>As of erts version 5.6.1 the emulator will refuse to use this strategy on other allocators than <code>temp_alloc</code>. This since it will only cause problems for other allocators.</p> </item> </taglist> <p>Apart from the ordinary allocators described above a number of pre-allocators are used for some specific data types. These pre-allocators pre-allocate a fixed amount of memory for certain data types when the run-time system starts. As long as pre-allocated memory is available, it will be used. When no pre-allocated memory is available, memory will be allocated in ordinary allocators. These pre-allocators are typically much faster than the ordinary allocators, but can only satisfy a limited amount of requests.</p> </div> <div class="section"> <span id="flags"> </span> <h4>System Flags Effecting erts_alloc</h4> <div class="warning"><h2>Warning!</h2> <p>Only use these flags if you are absolutely sure what you are doing. Unsuitable settings may cause serious performance degradation and even a system crash at any time during operation.</p> </div> <p>Memory allocator system flags have the following syntax: <code>+M&lt;S&gt;&lt;P&gt; &lt;V&gt;</code> where <code>&lt;S&gt;</code> is a letter identifying a subsystem, <code>&lt;P&gt;</code> is a parameter, and <code>&lt;V&gt;</code> is the value to use. The flags can be passed to the Erlang emulator (<a href="erl.html" class="seealso">erl</a>) as command line arguments.</p> <p>System flags effecting specific allocators have an upper-case letter as <code>&lt;S&gt;</code>. The following letters are used for the currently present allocators:</p> <list type="bulleted"> <item><code>B: binary_alloc</code></item> <item><code>D: std_alloc</code></item> <item><code>E: ets_alloc</code></item> <item><code>F: fix_alloc</code></item> <item><code>H: eheap_alloc</code></item> <item><code>L: ll_alloc</code></item> <item><code>M: mseg_alloc</code></item> <item><code>R: driver_alloc</code></item> <item><code>S: sl_alloc</code></item> <item><code>T: temp_alloc</code></item> <item><code>Y: sys_alloc</code></item> </list> <p>The following flags are available for configuration of <code>mseg_alloc</code>:</p> <taglist> <dt><marker id="MMamcbf"><code>+MMamcbf &lt;size&gt;</code></marker></dt> <item> Absolute max cache bad fit (in kilobytes). A segment in the memory segment cache is not reused if its size exceeds the requested size with more than the value of this parameter. Default value is 4096. </item> <dt><marker id="MMrmcbf"><code>+MMrmcbf &lt;ratio&gt;</code></marker></dt> <item> Relative max cache bad fit (in percent). A segment in the memory segment cache is not reused if its size exceeds the requested size with more than relative max cache bad fit percent of the requested size. Default value is 20.</item> <dt><marker id="MMsco"><code>+MMsco true|false</code></marker></dt> <item> Set <a href="#MMscs" class="seealso">super carrier</a> only flag. This flag defaults to <code>true</code>. When a super carrier is used and this flag is <code>true</code>, <code>mseg_alloc</code> will only create carriers in the super carrier. Note that the <code>alloc_util</code> framework may create <code>sys_alloc</code> carriers, so if you want all carriers to be created in the super carrier, you therefore want to disable use of <code>sys_alloc</code> carriers by also passing <a href="#Musac" class="seealso"><code>+Musac false</code></a>. When the flag is <code>false</code>, <code>mseg_alloc</code> will try to create carriers outside of the super carrier when the super carrier is full. <br /><br /> <em>NOTE</em>: Setting this flag to <code>false</code> may not be supported on all systems. This flag will in that case be ignored. <br /><br /> <em>NOTE</em>: The super carrier cannot be enabled nor disabled on halfword heap systems. This flag will be ignored on halfword heap systems. </item> <dt><marker id="MMscrfsd"><code>+MMscrfsd &lt;amount&gt;</code></marker></dt> <item> Set <a href="#MMscs" class="seealso">super carrier</a> reserved free segment descriptors. This parameter defaults to <code>65536</code>. This parameter determines the amount of memory to reserve for free segment descriptors used by the super carrier. If the system runs out of reserved memory for free segment descriptors, other memory will be used. This may however cause fragmentation issues, so you want to ensure that this never happens. The maximum amount of free segment descriptors used can be retrieved from the <code>erts_mmap</code> tuple part of the result from calling <a href="../erts/erlang.html#system_info_allocator_tuple" class="seealso">erlang:system_info({allocator, mseg_alloc})</a>. </item> <dt><marker id="MMscrpm"><code>+MMscrpm true|false</code></marker></dt> <item> Set <a href="#MMscs" class="seealso">super carrier</a> reserve physical memory flag. This flag defaults to <code>true</code>. When this flag is <code>true</code>, physical memory will be reserved for the whole super carrier at once when it is created. The reservation will after that be left unchanged. When this flag is set to <code>false</code> only virtual address space will be reserved for the super carrier upon creation. The system will attempt to reserve physical memory upon carrier creations in the super carrier, and attempt to unreserve physical memory upon carrier destructions in the super carrier. <br /><br /> <em>NOTE</em>: What reservation of physical memory actually means highly depends on the operating system, and how it is configured. For example, different memory overcommit settings on Linux drastically change the behaviour. Also note, setting this flag to <code>false</code> may not be supported on all systems. This flag will in that case be ignored. <br /><br /> <em>NOTE</em>: The super carrier cannot be enabled nor disabled on halfword heap systems. This flag will be ignored on halfword heap systems. </item> <dt><marker id="MMscs"><code>+MMscs &lt;size in MB&gt;</code></marker></dt> <item> Set super carrier size (in MB). The super carrier size defaults to zero; i.e, the super carrier is by default disabled. The super carrier is a large continuous area in the virtual address space. <code>mseg_alloc</code> will always try to create new carriers in the super carrier if it exists. Note that the <code>alloc_util</code> framework may create <code>sys_alloc</code> carriers. For more information on this, see the documentation of the <a href="#MMsco" class="seealso"><code>+MMsco</code></a> flag. <br /><br /> <em>NOTE</em>: The super carrier cannot be enabled nor disabled on halfword heap systems. This flag will be ignored on halfword heap systems. </item> <dt><marker id="MMmcs"><code>+MMmcs &lt;amount&gt;</code></marker></dt> <item> Max cached segments. The maximum number of memory segments stored in the memory segment cache. Valid range is 0-30. Default value is 10.</item> </taglist> <p>The following flags are available for configuration of <code>sys_alloc</code>:</p> <taglist> <dt><marker id="MYe"><code>+MYe true</code></marker></dt> <item> Enable <code>sys_alloc</code>. Note: <code>sys_alloc</code> cannot be disabled.</item> <dt><marker id="MYm"><code>+MYm libc</code></marker></dt> <item> <code>malloc</code> library to use. Currently only <code>libc</code> is available. <code>libc</code> enables the standard <code>libc</code> malloc implementation. By default <code>libc</code> is used.</item> <dt><marker id="MYtt"><code>+MYtt &lt;size&gt;</code></marker></dt> <item> Trim threshold size (in kilobytes). This is the maximum amount of free memory at the top of the heap (allocated by <code>sbrk</code>) that will be kept by <code>malloc</code> (not released to the operating system). When the amount of free memory at the top of the heap exceeds the trim threshold, <code>malloc</code> will release it (by calling <code>sbrk</code>). Trim threshold is given in kilobytes. Default trim threshold is 128. <em>Note:</em> This flag will only have any effect when the emulator has been linked with the GNU C library, and uses its <code>malloc</code> implementation.</item> <dt><marker id="MYtp"><code>+MYtp &lt;size&gt;</code></marker></dt> <item> Top pad size (in kilobytes). This is the amount of extra memory that will be allocated by <code>malloc</code> when <code>sbrk</code> is called to get more memory from the operating system. Default top pad size is 0. <em>Note:</em> This flag will only have any effect when the emulator has been linked with the GNU C library, and uses its <code>malloc</code> implementation.</item> </taglist> <p>The following flags are available for configuration of allocators based on <code>alloc_util</code>. If <code>u</code> is used as subsystem identifier (i.e., <code>&lt;S&gt; = u</code>) all allocators based on <code>alloc_util</code> will be effected. If <code>B</code>, <code>D</code>, <code>E</code>, <code>F</code>, <code>H</code>, <code>L</code>, <code>R</code>, <code>S</code>, or <code>T</code> is used as subsystem identifier, only the specific allocator identified will be effected:</p> <taglist> <dt><marker id="M_acul"><code>+M&lt;S&gt;acul &lt;utilization&gt;|de</code></marker></dt> <item> Abandon carrier utilization limit. A valid <code>&lt;utilization&gt;</code> is an integer in the range <code>[0, 100]</code> representing utilization in percent. When a utilization value larger than zero is used, allocator instances are allowed to abandon multiblock carriers. If <code>de</code> (default enabled) is passed instead of a <code>&lt;utilization&gt;</code>, a recomended non zero utilization value will be used. The actual value chosen depend on allocator type and may be changed between ERTS versions. Currently the default equals <code>de</code>, but this may be changed in the future. Carriers will be abandoned when memory utilization in the allocator instance falls below the utilization value used. Once a carrier has been abandoned, no new allocations will be made in it. When an allocator instance gets an increased multiblock carrier need, it will first try to fetch an abandoned carrier from an allocator instances of the same allocator type. If no abandoned carrier could be fetched, it will create a new empty carrier. When an abandoned carrier has been fetched it will function as an ordinary carrier. This feature has special requirements on the <a href="#M_as" class="seealso">allocation strategy</a> used. Currently only the strategies <code>aoff</code>, <code>aoffcbf</code> and <code>aoffcaobf</code> support abandoned carriers. This feature also requires <a href="#M_t" class="seealso">multiple thread specific instances</a> to be enabled. When enabling this feature, multiple thread specific instances will be enabled if not already enabled, and the <code>aoffcbf</code> strategy will be enabled if current strategy does not support abandoned carriers. This feature can be enabled on all allocators based on the <code>alloc_util</code> framework with the exception of <code>temp_alloc</code> (which would be pointless). </item> <dt><marker id="M_as"><code>+M&lt;S&gt;as bf|aobf|aoff|aoffcbf|aoffcaobf|gf|af</code></marker></dt> <item> Allocation strategy. Valid strategies are <code>bf</code> (best fit), <code>aobf</code> (address order best fit), <code>aoff</code> (address order first fit), <code>aoffcbf</code> (address order first fit carrier best fit), <code>aoffcaobf</code> (address order first fit carrier address order best fit), <code>gf</code> (good fit), and <code>af</code> (a fit). See <a href="#strategy" class="seealso">the description of allocation strategies</a> in "the <code>alloc_util</code> framework" section.</item> <dt><marker id="M_asbcst"><code>+M&lt;S&gt;asbcst &lt;size&gt;</code></marker></dt> <item> Absolute singleblock carrier shrink threshold (in kilobytes). When a block located in an <code>mseg_alloc</code> singleblock carrier is shrunk, the carrier will be left unchanged if the amount of unused memory is less than this threshold; otherwise, the carrier will be shrunk. See also <a href="#M_rsbcst" class="seealso">rsbcst</a>.</item> <dt><marker id="M_e"><code>+M&lt;S&gt;e true|false</code></marker></dt> <item> Enable allocator <code>&lt;S&gt;</code>.</item> <dt><marker id="M_lmbcs"><code>+M&lt;S&gt;lmbcs &lt;size&gt;</code></marker></dt> <item> Largest (<code>mseg_alloc</code>) multiblock carrier size (in kilobytes). See <a href="#mseg_mbc_sizes" class="seealso">the description on how sizes for mseg_alloc multiblock carriers are decided</a> in "the <code>alloc_util</code> framework" section. On 32-bit Unix style OS this limit can not be set higher than 128 megabyte.</item> <dt><marker id="M_mbcgs"><code>+M&lt;S&gt;mbcgs &lt;ratio&gt;</code></marker></dt> <item> (<code>mseg_alloc</code>) multiblock carrier growth stages. See <a href="#mseg_mbc_sizes" class="seealso">the description on how sizes for mseg_alloc multiblock carriers are decided</a> in "the <code>alloc_util</code> framework" section.</item> <dt><marker id="M_mbsd"><code>+M&lt;S&gt;mbsd &lt;depth&gt;</code></marker></dt> <item> Max block search depth. This flag has effect only if the good fit strategy has been selected for allocator <code>&lt;S&gt;</code>. When the good fit strategy is used, free blocks are placed in segregated free-lists. Each free list contains blocks of sizes in a specific range. The max block search depth sets a limit on the maximum number of blocks to inspect in a free list during a search for suitable block satisfying the request.</item> <dt><marker id="M_mmbcs"><code>+M&lt;S&gt;mmbcs &lt;size&gt;</code></marker></dt> <item> Main multiblock carrier size. Sets the size of the main multiblock carrier for allocator <code>&lt;S&gt;</code>. The main multiblock carrier is allocated via <code>sys_alloc</code> and is never deallocated.</item> <dt><marker id="M_mmmbc"><code>+M&lt;S&gt;mmmbc &lt;amount&gt;</code></marker></dt> <item> Max <code>mseg_alloc</code> multiblock carriers. Maximum number of multiblock carriers allocated via <code>mseg_alloc</code> by allocator <code>&lt;S&gt;</code>. When this limit has been reached, new multiblock carriers will be allocated via <code>sys_alloc</code>.</item> <dt><marker id="M_mmsbc"><code>+M&lt;S&gt;mmsbc &lt;amount&gt;</code></marker></dt> <item> Max <code>mseg_alloc</code> singleblock carriers. Maximum number of singleblock carriers allocated via <code>mseg_alloc</code> by allocator <code>&lt;S&gt;</code>. When this limit has been reached, new singleblock carriers will be allocated via <code>sys_alloc</code>.</item> <dt><marker id="M_ramv"><code>+M&lt;S&gt;ramv &lt;bool&gt;</code></marker></dt> <item> Realloc always moves. When enabled, reallocate operations will more or less be translated into an allocate, copy, free sequence. This often reduce memory fragmentation, but costs performance. </item> <dt><marker id="M_rmbcmt"><code>+M&lt;S&gt;rmbcmt &lt;ratio&gt;</code></marker></dt> <item> Relative multiblock carrier move threshold (in percent). When a block located in a multiblock carrier is shrunk, the block will be moved if the ratio of the size of the returned memory compared to the previous size is more than this threshold; otherwise, the block will be shrunk at current location.</item> <dt><marker id="M_rsbcmt"><code>+M&lt;S&gt;rsbcmt &lt;ratio&gt;</code></marker></dt> <item> Relative singleblock carrier move threshold (in percent). When a block located in a singleblock carrier is shrunk to a size smaller than the value of the <a href="#M_sbct" class="seealso">sbct</a> parameter, the block will be left unchanged in the singleblock carrier if the ratio of unused memory is less than this threshold; otherwise, it will be moved into a multiblock carrier. </item> <dt><marker id="M_rsbcst"><code>+M&lt;S&gt;rsbcst &lt;ratio&gt;</code></marker></dt> <item> Relative singleblock carrier shrink threshold (in percent). When a block located in an <code>mseg_alloc</code> singleblock carrier is shrunk, the carrier will be left unchanged if the ratio of unused memory is less than this threshold; otherwise, the carrier will be shrunk. See also <a href="#M_asbcst" class="seealso">asbcst</a>.</item> <dt><marker id="M_sbct"><code>+M&lt;S&gt;sbct &lt;size&gt;</code></marker></dt> <item> Singleblock carrier threshold. Blocks larger than this threshold will be placed in singleblock carriers. Blocks smaller than this threshold will be placed in multiblock carriers. On 32-bit Unix style OS this threshold can not be set higher than 8 megabytes.</item> <dt><marker id="M_smbcs"><code>+M&lt;S&gt;smbcs &lt;size&gt;</code></marker></dt> <item> Smallest (<code>mseg_alloc</code>) multiblock carrier size (in kilobytes). See <a href="#mseg_mbc_sizes" class="seealso">the description on how sizes for mseg_alloc multiblock carriers are decided</a> in "the <code>alloc_util</code> framework" section.</item> <dt><marker id="M_t"><code>+M&lt;S&gt;t true|false</code></marker></dt> <item> <p>Multiple, thread specific instances of the allocator. This option will only have any effect on the runtime system with SMP support. Default behaviour on the runtime system with SMP support:</p> <taglist> <dt><code>ll_alloc</code></dt> <item><code>1</code> instance.</item> <dt>Other allocators</dt> <item><code>NoSchedulers+1</code> instances. Each scheduler will use a lock-free instance of its own and other threads will use a common instance.</item> </taglist> <p>It was previously (before ERTS version 5.9) possible to configure a smaller amount of thread specific instances than schedulers. This is, however, not possible any more.</p> </item> </taglist> <p>Currently the following flags are available for configuration of <code>alloc_util</code>, i.e. all allocators based on <code>alloc_util</code> will be effected:</p> <taglist> <dt><marker id="Muycs"><code>+Muycs &lt;size&gt;</code></marker></dt> <item> <code>sys_alloc</code> carrier size. Carriers allocated via <code>sys_alloc</code> will be allocated in sizes which are multiples of the <code>sys_alloc</code> carrier size. This is not true for main multiblock carriers and carriers allocated during a memory shortage, though.</item> <dt><marker id="Mummc"><code>+Mummc &lt;amount&gt;</code></marker></dt> <item> Max <code>mseg_alloc</code> carriers. Maximum number of carriers placed in separate memory segments. When this limit has been reached, new carriers will be placed in memory retrieved from <code>sys_alloc</code>.</item> <dt><marker id="Musac"><code>+Musac &lt;bool&gt;</code></marker></dt> <item> Allow <code>sys_alloc</code> carriers. By default <code>true</code>. If set to <code>false</code>, <code>sys_alloc</code> carriers will never be created by allocators using the <code>alloc_util</code> framework.</item> </taglist> <p>Instrumentation flags:</p> <taglist> <dt><marker id="Mim"><code>+Mim true|false</code></marker></dt> <item> A map over current allocations is kept by the emulator. The allocation map can be retrieved via the <code>instrument</code> module. <code>+Mim true</code> implies <code>+Mis true</code>. <code>+Mim true</code> is the same as <a href="erl.html#instr" class="seealso">-instr</a>.</item> <dt><marker id="Mis"><code>+Mis true|false</code></marker></dt> <item> Status over allocated memory is kept by the emulator. The allocation status can be retrieved via the <code>instrument</code> module.</item> <dt><marker id="Mit"><code>+Mit X</code></marker></dt> <item> Reserved for future use. Do <em>not</em> use this flag.</item> </taglist> <div class="note"><h2>Note!</h2> <p>When instrumentation of the emulator is enabled, the emulator uses more memory and runs slower.</p> </div> <p>Other flags:</p> <taglist> <dt><marker id="Mea"><code>+Mea min|max|r9c|r10b|r11b|config</code></marker></dt> <item> <taglist> <dt><code>min</code></dt> <item> Disables all allocators that can be disabled. </item> <dt><code>max</code></dt> <item> Enables all allocators (currently default). </item> <dt><code>r9c|r10b|r11b</code></dt> <item> Configures all allocators as they were configured in respective OTP release. These will eventually be removed. </item> <dt><code>config</code></dt> <item> Disables features that cannot be enabled while creating an allocator configuration with <a href="../runtime_tools/erts_alloc_config.html" class="seealso">erts_alloc_config(3)</a>. Note, this option should only be used while running <code>erts_alloc_config</code>, <em>not</em> when using the created configuration. </item> </taglist> </item> <dt><marker id="Mlpm"><code>+Mlpm all|no</code></marker></dt> <item>Lock physical memory. The default value is <code>no</code>, i.e., no physical memory will be locked. If set to <code>all</code>, all memory mappings made by the runtime system, will be locked into physical memory. If set to <code>all</code>, the runtime system will fail to start if this feature is not supported, the user has not got enough privileges, or the user is not allowed to lock enough physical memory. The runtime system will also fail with an out of memory condition if the user limit on the amount of locked memory is reached. </item> </taglist> <p>Only some default values have been presented here. <a href="../erts/erlang.html#system_info_allocator" class="seealso">erlang:system_info(allocator)</a>, and <a href="../erts/erlang.html#system_info_allocator_tuple" class="seealso">erlang:system_info({allocator, Alloc})</a> can be used in order to obtain currently used settings and current status of the allocators.</p> <div class="note"><h2>Note!</h2> <p>Most of these flags are highly implementation dependent, and they may be changed or removed without prior notice.</p> <p><code>erts_alloc</code> is not obliged to strictly use the settings that have been passed to it (it may even ignore them).</p> </div> <p><a href="../runtime_tools/erts_alloc_config.html" class="seealso">erts_alloc_config(3)</a> is a tool that can be used to aid creation of an <code>erts_alloc</code> configuration that is suitable for a limited number of runtime scenarios.</p> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="../runtime_tools/erts_alloc_config.html" class="seealso">erts_alloc_config(3)</a>, <a href="erl.html" class="seealso">erl(1)</a>, <a href="../tools/instrument.html" class="seealso">instrument(3)</a>, <a href="../erts/erlang.html" class="seealso">erlang(3)</a></p> </div>
