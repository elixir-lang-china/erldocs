<h1>driver_entry</h1> <libsummary>The driver-entry structure used by erlang drivers.</libsummary> <div class="description"> <span id="WARNING"> </span> <div class="warning"><h2>Warning!</h2><p><em>Use this functionality with extreme care!</em></p> <p>A driver callback is executed as a direct extension of the native code of the VM. Execution is not made in a safe environment. The VM can <em>not</em> provide the same services as provided when executing Erlang code, such as preemptive scheduling or memory protection. If the driver callback function doesn't behave well, the whole VM will misbehave.</p> <list> <item><p>A driver callback that crash will crash the whole VM.</p></item> <item><p>An erroneously implemented driver callback might cause a VM internal state inconsistency which may cause a crash of the VM, or miscellaneous misbehaviors of the VM at any point after the call to the driver callback.</p></item> <item><p>A driver callback that do <a href="erl_driver.html#lengthy_work" class="seealso">lengthy work</a> before returning will degrade responsiveness of the VM, and may cause miscellaneous strange behaviors. Such strange behaviors include, but are not limited to, extreme memory usage, and bad load balancing between schedulers. Strange behaviors that might occur due to lengthy work may also vary between OTP releases.</p></item> </list> </div> <p> As of erts version 5.9 (OTP release R15B) the driver interface has been changed with larger types for the callbacks <a href="#output" class="seealso">output</a>, <a href="#control" class="seealso">control</a> and <a href="#call" class="seealso">call</a>. See driver <a href="erl_driver.html#version_management" class="seealso"> version management</a> in <a href="erl_driver.html" class="seealso">erl_driver</a>. </p> <div class="note"><h2>Note!</h2> <p>Old drivers (compiled with an <code>erl_driver.h</code> from an earlier erts version than 5.9) have to be updated and have to use the extended interface (with <a href="erl_driver.html#version_management" class="seealso">version management </a>).</p> </div> <p>The <code>driver_entry</code> structure is a C struct that all erlang drivers define. It contains entry points for the erlang driver that are called by the erlang emulator when erlang code accesses the driver.</p> <p> <span id="emulator"> </span> The <a href="erl_driver.html" class="seealso">erl_driver</a> driver API functions need a port handle that identifies the driver instance (and the port in the emulator). This is only passed to the <code>start</code> function, but not to the other functions. The <code>start</code> function returns a driver-defined handle that is passed to the other functions. A common practice is to have the <code>start</code> function allocate some application-defined structure and stash the <code>port</code> handle in it, to use it later with the driver API functions.</p> <p>The driver call-back functions are called synchronously from the erlang emulator. If they take too long before completing, they can cause timeouts in the emulator. Use the queue or asynchronous calls if necessary, since the emulator must be responsive.</p> <p>The driver structure contains the name of the driver and some 15 function pointers. These pointers are called at different times by the emulator.</p> <p>The only exported function from the driver is <code>driver_init</code>. This function returns the <code>driver_entry</code> structure that points to the other functions in the driver. The <code>driver_init</code> function is declared with a macro <code>DRIVER_INIT(drivername)</code>. (This is because different OS's have different names for it.)</p> <p>When writing a driver in C++, the driver entry should be of <code>"C"</code> linkage. One way to do this is to put this line somewhere before the driver entry: <code>extern "C" DRIVER_INIT(drivername);</code>.</p> <p>When the driver has passed the <code>driver_entry</code> over to the emulator, the driver is <em>not</em> allowed to modify the <code>driver_entry</code>.</p> <p>If compiling a driver for static inclusion via --enable-static-drivers you have to define STATIC_ERLANG_DRIVER before the DRIVER_INIT declaration.</p> <div class="note"><h2>Note!</h2> <p>Do <em>not</em> declare the <code>driver_entry</code> <code>const</code>. This since the emulator needs to modify the <code>handle</code>, and the <code>handle2</code> fields. A statically allocated, and <code>const</code> declared <code>driver_entry</code> may be located in read only memory which will cause the emulator to crash.</p> </div> </div> <div class="section"> <h4>DATA TYPES</h4> <taglist> <dt><b>ErlDrvEntry</b></dt> <item> <p /> <pre class="sh_erlang"> typedef struct erl_drv_entry { int (*init)(void); /* called at system start up for statically linked drivers, and after loading for dynamically loaded drivers */ #ifndef ERL_SYS_DRV ErlDrvData (*start)(ErlDrvPort port, char *command); /* called when open_port/2 is invoked. return value -1 means failure. */ #else ErlDrvData (*start)(ErlDrvPort port, char *command, SysDriverOpts* opts); /* special options, only for system driver */ #endif void (*stop)(ErlDrvData drv_data); /* called when port is closed, and when the emulator is halted. */ void (*output)(ErlDrvData drv_data, char *buf, ErlDrvSizeT len); /* called when we have output from erlang to the port */ void (*ready_input)(ErlDrvData drv_data, ErlDrvEvent event); /* called when we have input from one of the driver's handles */ void (*ready_output)(ErlDrvData drv_data, ErlDrvEvent event); /* called when output is possible to one of the driver's handles */ char *driver_name; /* name supplied as command in open_port XXX ? */ void (*finish)(void); /* called before unloading the driver - DYNAMIC DRIVERS ONLY */ void *handle; /* Reserved -- Used by emulator internally */ ErlDrvSSizeT (*control)(ErlDrvData drv_data, unsigned int command, char *buf, ErlDrvSizeT len, char **rbuf, ErlDrvSizeT rlen); /* "ioctl" for drivers - invoked by port_control/3 */ void (*timeout)(ErlDrvData drv_data); /* Handling of timeout in driver */ void (*outputv)(ErlDrvData drv_data, ErlIOVec *ev); /* called when we have output from erlang to the port */ void (*ready_async)(ErlDrvData drv_data, ErlDrvThreadData thread_data); void (*flush)(ErlDrvData drv_data); /* called when the port is about to be closed, and there is data in the driver queue that needs to be flushed before 'stop' can be called */ ErlDrvSSizeT (*call)(ErlDrvData drv_data, unsigned int command, char *buf, ErlDrvSizeT len, char **rbuf, ErlDrvSizeT rlen, unsigned int *flags); /* Works mostly like 'control', a synchronous call into the driver. */ void (*event)(ErlDrvData drv_data, ErlDrvEvent event, ErlDrvEventData event_data); /* Called when an event selected by driver_event() has occurred */ int extended_marker; /* ERL_DRV_EXTENDED_MARKER */ int major_version; /* ERL_DRV_EXTENDED_MAJOR_VERSION */ int minor_version; /* ERL_DRV_EXTENDED_MINOR_VERSION */ int driver_flags; /* ERL_DRV_FLAGs */ void *handle2; /* Reserved -- Used by emulator internally */ void (*process_exit)(ErlDrvData drv_data, ErlDrvMonitor *monitor); /* Called when a process monitor fires */ void (*stop_select)(ErlDrvEvent event, void* reserved); /* Called to close an event object */ } ErlDrvEntry; </pre> <p /> <taglist> <dt><span id="init"> </span>int (*init)(void)</dt> <item> <p>This is called directly after the driver has been loaded by <code>erl_ddll:load_driver/2</code>. (Actually when the driver is added to the driver list.) The driver should return 0, or if the driver can't initialize, -1.</p> </item> <dt><span id="start"> </span>ErlDrvData (*start)(ErlDrvPort port, char* command)</dt> <item> <p>This is called when the driver is instantiated, when <code>open_port/2</code> is called. The driver should return a number &gt;= 0 or a pointer, or if the driver can't be started, one of three error codes should be returned:</p> <p>ERL_DRV_ERROR_GENERAL - general error, no error code</p> <p>ERL_DRV_ERROR_ERRNO - error with error code in erl_errno</p> <p>ERL_DRV_ERROR_BADARG - error, badarg</p> <p>If an error code is returned, the port isn't started.</p> </item> <dt><span id="stop"> </span>void (*stop)(ErlDrvData drv_data)</dt> <item> <p>This is called when the port is closed, with <code>port_close/1</code> or <code>Port ! {self(), close}</code>. Note that terminating the port owner process also closes the port. If <code>drv_data</code> is a pointer to memory allocated in <code>start</code>, then <code>stop</code> is the place to deallocate that memory.</p> </item> <dt><span id="output"> </span>void (*output)(ErlDrvData drv_data, char *buf, ErlDrvSizeT len)</dt> <item> <p>This is called when an erlang process has sent data to the port. The data is pointed to by <code>buf</code>, and is <code>len</code> bytes. Data is sent to the port with <code>Port ! {self(), {command, Data}}</code>, or with <code>port_command/2</code>. Depending on how the port was opened, it should be either a list of integers 0...255 or a binary. See <code>open_port/3</code> and <code>port_command/2</code>.</p> </item> <dt><span id="ready_input"> </span>void (*ready_input)(ErlDrvData drv_data, ErlDrvEvent event)</dt> <dt><span id="ready_output"> </span>void (*ready_output)(ErlDrvData drv_data, ErlDrvEvent event)</dt> <item> <p>This is called when a driver event (given in the <code>event</code> parameter) is signaled. This is used to help asynchronous drivers "wake up" when something happens.</p> <p>On unix the <code>event</code> is a pipe or socket handle (or something that the <code>select</code> system call understands).</p> <p>On Windows the <code>event</code> is an Event or Semaphore (or something that the <code>WaitForMultipleObjects</code> API function understands). (Some trickery in the emulator allows more than the built-in limit of 64 <code>Events</code> to be used.)</p> <p>To use this with threads and asynchronous routines, create a pipe on unix and an Event on Windows. When the routine completes, write to the pipe (use <code>SetEvent</code> on Windows), this will make the emulator call <code>ready_input</code> or <code>ready_output</code>.</p> <p>Spurious events may happen. That is, calls to <code>ready_input</code> or <code>ready_output</code> even though no real events are signaled. In reality it should be rare (and OS dependant), but a robust driver must nevertheless be able to handle such cases.</p> </item> <dt><span id="driver_name"> </span>char *driver_name</dt> <item> <p>This is the name of the driver, it must correspond to the atom used in <code>open_port</code>, and the name of the driver library file (without the extension).</p> </item> <dt><span id="finish"> </span>void (*finish)(void)</dt> <item> <p>This function is called by the <code>erl_ddll</code> driver when the driver is unloaded. (It is only called in dynamic drivers.)</p> <p>The driver is only unloaded as a result of calling <code>unload_driver/1</code>, or when the emulator halts.</p> </item> <dt>void *handle</dt> <item> <p>This field is reserved for the emulator's internal use. The emulator will modify this field; therefore, it is important that the <code>driver_entry</code> isn't declared <code>const</code>.</p> </item> <dt><span id="control"> </span>ErlDrvSSizeT (*control)(ErlDrvData drv_data, unsigned int command, char *buf, ErlDrvSizeT len, char **rbuf, ErlDrvSizeT rlen)</dt> <item> <p>This is a special routine invoked with the erlang function <code>port_control/3</code>. It works a little like an "ioctl" for erlang drivers. The data given to <code>port_control/3</code> arrives in <code>buf</code> and <code>len</code>. The driver may send data back, using <code>*rbuf</code> and <code>rlen</code>.</p> <p>This is the fastest way of calling a driver and get a response. It won't make any context switch in the erlang emulator, and requires no message passing. It is suitable for calling C function to get faster execution, when erlang is too slow.</p> <p>If the driver wants to return data, it should return it in <code>rbuf</code>. When <code>control</code> is called, <code>*rbuf</code> points to a default buffer of <code>rlen</code> bytes, which can be used to return data. Data is returned different depending on the port control flags (those that are set with <a href="erl_driver.html#set_port_control_flags" class="seealso">set_port_control_flags</a>). </p> <p>If the flag is set to <code>PORT_CONTROL_FLAG_BINARY</code>, a binary will be returned. Small binaries can be returned by writing the raw data into the default buffer. A binary can also be returned by setting <code>*rbuf</code> to point to a binary allocated with <a href="erl_driver.html#driver_alloc_binary" class="seealso">driver_alloc_binary</a>. This binary will be freed automatically after <code>control</code> has returned. The driver can retain the binary for <em>read only</em> access with <a href="erl_driver.html#driver_binary_inc_refc" class="seealso">driver_binary_inc_refc</a> to be freed later with <a href="erl_driver.html#driver_free_binary" class="seealso">driver_free_binary</a>. It is never allowed to alter the binary after <code>control</code> has returned. If <code>*rbuf</code> is set to NULL, an empty list will be returned. </p> <p>If the flag is set to <code>0</code>, data is returned as a list of integers. Either use the default buffer or set <code>*rbuf</code> to point to a larger buffer allocated with <a href="erl_driver.html#driver_alloc" class="seealso">driver_alloc</a>. The buffer will be freed automatically after <code>control</code> has returned.</p> <p>Using binaries is faster if more than a few bytes are returned.</p> <p>The return value is the number of bytes returned in <code>*rbuf</code>.</p> </item> <dt><span id="timeout"> </span>void (*timeout)(ErlDrvData drv_data)</dt> <item> <p>This function is called any time after the driver's timer reaches 0. The timer is activated with <code>driver_set_timer</code>. There are no priorities or ordering among drivers, so if several drivers time out at the same time, any one of them is called first.</p> </item> <dt><span id="outputv"> </span>void (*outputv)(ErlDrvData drv_data, ErlIOVec *ev)</dt> <item> <p>This function is called whenever the port is written to. If it is <code>NULL</code>, the <code>output</code> function is called instead. This function is faster than <code>output</code>, because it takes an <code>ErlIOVec</code> directly, which requires no copying of the data. The port should be in binary mode, see <code>open_port/2</code>.</p> <p>The <code>ErlIOVec</code> contains both a <code>SysIOVec</code>, suitable for <code>writev</code>, and one or more binaries. If these binaries should be retained, when the driver returns from <code>outputv</code>, they can be queued (using <a href="erl_driver.html#driver_enq_bin" class="seealso">driver_enq_bin</a> for instance), or if they are kept in a static or global variable, the reference counter can be incremented.</p> </item> <dt><span id="ready_async"> </span>void (*ready_async)(ErlDrvData drv_data, ErlDrvThreadData thread_data)</dt> <item> <p>This function is called after an asynchronous call has completed. The asynchronous call is started with <a href="erl_driver.html#driver_async" class="seealso">driver_async</a>. This function is called from the erlang emulator thread, as opposed to the asynchronous function, which is called in some thread (if multithreading is enabled).</p> </item> <dt><span id="call"> </span>ErlDrvSSizeT (*call)(ErlDrvData drv_data, unsigned int command, char *buf, ErlDrvSizeT len, char **rbuf, ErlDrvSizeT rlen, unsigned int *flags)</dt> <item> <p>This function is called from <code>erlang:port_call/3</code>. It works a lot like the <code>control</code> call-back, but uses the external term format for input and output.</p> <p><code>command</code> is an integer, obtained from the call from erlang (the second argument to <code>erlang:port_call/3</code>).</p> <p><code>buf</code> and <code>len</code> provide the arguments to the call (the third argument to <code>erlang:port_call/3</code>). They can be decoded using <code>ei</code> functions.</p> <p><code>rbuf</code> points to a return buffer, <code>rlen</code> bytes long. The return data should be a valid erlang term in the external (binary) format. This is converted to an erlang term and returned by <code>erlang:port_call/3</code> to the caller. If more space than <code>rlen</code> bytes is needed to return data, <code>*rbuf</code> can be set to memory allocated with <code>driver_alloc</code>. This memory will be freed automatically after <code>call</code> has returned.</p> <p>The return value is the number of bytes returned in <code>*rbuf</code>. If <code>ERL_DRV_ERROR_GENERAL</code> is returned (or in fact, anything &lt; 0), <code>erlang:port_call/3</code> will throw a <code>BAD_ARG</code>.</p> </item> <dt>void (*event)(ErlDrvData drv_data, ErlDrvEvent event, ErlDrvEventData event_data)</dt> <item> <p>Intentionally left undocumented.</p> </item> <dt><span id="extended_marker"> </span>int extended_marker</dt> <item> <p> This field should either be equal to <code>ERL_DRV_EXTENDED_MARKER</code> or <code>0</code>. An old driver (not aware of the extended driver interface) should set this field to <code>0</code>. If this field is equal to <code>0</code>, all the fields following this field also <em>have</em> to be <code>0</code>, or <code>NULL</code> in case it is a pointer field. </p> </item> <dt>int major_version</dt> <item> <p>This field should equal <code>ERL_DRV_EXTENDED_MAJOR_VERSION</code> if the <code>extended_marker</code> field equals <code>ERL_DRV_EXTENDED_MARKER</code>.</p> </item> <dt>int minor_version</dt> <item> <p> This field should equal <code>ERL_DRV_EXTENDED_MINOR_VERSION</code> if the <code>extended_marker</code> field equals <code>ERL_DRV_EXTENDED_MARKER</code>. </p> </item> <dt><span id="driver_flags"> </span>int driver_flags</dt> <item> <p>This field is used to pass driver capability and other information to the runtime system. If the <code>extended_marker</code> field equals <code>ERL_DRV_EXTENDED_MARKER</code>, it should contain <code>0</code> or driver flags (<code>ERL_DRV_FLAG_*</code>) ored bitwise. Currently the following driver flags exist: </p> <taglist> <dt><code>ERL_DRV_FLAG_USE_PORT_LOCKING</code></dt> <item> The runtime system will use port level locking on all ports executing this driver instead of driver level locking when the driver is run in a runtime system with SMP support. For more information see the <a href="erl_driver.html#smp_support" class="seealso">erl_driver</a> documentation. </item> <dt><code>ERL_DRV_FLAG_SOFT_BUSY</code></dt> <item> Marks that driver instances can handle being called in the <a href="#output" class="seealso">output</a> and/or <a href="#outputv" class="seealso">outputv</a> callbacks even though a driver instance has marked itself as busy (see <a href="erl_driver.html#set_busy_port" class="seealso">set_busy_port()</a>). Since erts version 5.7.4 this flag is required for drivers used by the Erlang distribution (the behaviour has always been required by drivers used by the distribution). </item> <dt><code>ERL_DRV_FLAG_NO_BUSY_MSGQ</code></dt> <item>Disable busy port message queue functionality. For more information, see the documentation of the <a href="erl_driver.html#erl_drv_busy_msgq_limits" class="seealso">erl_drv_busy_msgq_limits()</a> function. </item> </taglist> </item> <dt>void *handle2</dt> <item> <p> This field is reserved for the emulator's internal use. The emulator will modify this field; therefore, it is important that the <code>driver_entry</code> isn't declared <code>const</code>. </p> </item> <dt><span id="process_exit"> </span>void (*process_exit)(ErlDrvData drv_data, ErlDrvMonitor *monitor)</dt> <item> <p>This callback is called when a monitored process exits. The <code>drv_data</code> is the data associated with the port for which the process is monitored (using <a href="erl_driver.html#driver_monitor_process" class="seealso">driver_monitor_process</a>) and the <code>monitor</code> corresponds to the <code>ErlDrvMonitor</code> structure filled in when creating the monitor. The driver interface function <a href="erl_driver.html#driver_get_monitored_process" class="seealso">driver_get_monitored_process</a> can be used to retrieve the process id of the exiting process as an <code>ErlDrvTermData</code>.</p> </item> <dt><span id="stop_select"> </span>void (*stop_select)(ErlDrvEvent event, void* reserved)</dt> <item> <p>This function is called on behalf of <a href="erl_driver.html#driver_select" class="seealso">driver_select</a> when it is safe to close an event object.</p> <p>A typical implementation on Unix is to do <code>close((int)event)</code>.</p> <p>Argument <code>reserved</code> is intended for future use and should be ignored.</p> <p>In contrast to most of the other call-back functions, <code>stop_select</code> is called independent of any port. No <code>ErlDrvData</code> argument is passed to the function. No driver lock or port lock is guaranteed to be held. The port that called <code>driver_select</code> might even be closed at the time <code>stop_select</code> is called. But it could also be the case that <code>stop_select</code> is called directly by <code>driver_select</code>.</p> <p>It is not allowed to call any functions in the <a href="erl_driver.html" class="seealso">driver API</a> from <code>stop_select</code>. This strict limitation is due to the volatile context that <code>stop_select</code> may be called.</p> </item> </taglist> </item> </taglist> </div> <div class="section"> <h4>SEE ALSO</h4> <p><a href="erl_driver.html" class="seealso">erl_driver(3)</a>, <a href="../kernel/erl_ddll.html" class="seealso">erl_ddll(3)</a>, <a href="erlang.html" class="seealso">erlang(3)</a>, kernel(3)</p> </div>
