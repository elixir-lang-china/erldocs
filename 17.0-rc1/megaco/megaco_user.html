<h1>megaco_user</h1> <h2 class="modsummary">Callback module for users of the Megaco application</h2> <div class="description"> <p>This module defines the callback behaviour of Megaco users. A megaco_user compliant callback module must export the following functions: </p> <list type="bulleted"> <item> <p><a href="#connect" class="seealso">handle_connect/2,3</a></p> </item> <item> <p><a href="#disconnect" class="seealso">handle_disconnect/3</a></p> </item> <item> <p><a href="#syntax_error" class="seealso">handle_syntax_error/3,4</a></p> </item> <item> <p><a href="#message_error" class="seealso">handle_message_error/3,4</a></p> </item> <item> <p><a href="#trans_request" class="seealso">handle_trans_request/3,4</a></p> </item> <item> <p><a href="#trans_long_request" class="seealso">handle_trans_long_request/3,4</a></p> </item> <item> <p><a href="#trans_reply" class="seealso">handle_trans_reply/4,5</a></p> </item> <item> <p><a href="#trans_ack" class="seealso">handle_trans_ack/4,5</a></p> </item> <item> <p><a href="#unexpected_trans" class="seealso">handle_unexpected_trans/3,4</a></p> </item> <item> <p><a href="#request_abort" class="seealso">handle_trans_request_abort/4,5</a></p> </item> <item> <p><a href="#segment_reply" class="seealso">handle_segment_reply/5,6</a></p> </item> </list> <p>The semantics of them and their exact signatures are explained below. </p> <p>The <code>user_args</code> configuration parameter which may be used to extend the argument list of the callback functions. For example, the handle_connect function takes by default two arguments:</p> <pre class="sh_erlang"> handle_connect(Handle, Version) </pre> <p>but if the <code>user_args</code> parameter is set to a longer list, such as <code>[SomePid,SomeTableRef]</code>, the callback function is expected to have these (in this case two) extra arguments last in the argument list:</p> <pre class="sh_erlang"> handle_connect(Handle, Version, SomePid, SomeTableRef) </pre> <span id="extra_argument"> </span> <div class="note"><h2>Note!</h2> <p>Must of the functions below has an optional <code>Extra</code> argument (e.g. <a href="#unexpected_trans" class="seealso">handle_unexpected_trans/4</a>). The functions which takes this argument will be called if and only if one of the functions <a href="megaco.html#receive_message" class="seealso">receive_message/5</a> or <a href="megaco.html#process_received_message" class="seealso">process_received_message/5</a> was called with the <code>Extra</code> argument different than <code>ignore_extra</code>. </p> </div> </div> <div class="section"> <h4>DATA TYPES</h4> <pre class="sh_erlang"> action_request() = #'ActionRequest'{} action_reply() = #'ActionReply'{} error_desc() = #'ErrorDescriptor'{} segment_no() = integer() </pre> <pre class="sh_erlang"> conn_handle() = #megaco_conn_handle{} </pre> <p>The record initially returned by <code>megaco:connect/4,5</code>. It identifies a "virtual" connection and may be reused after a reconnect (disconnect + connect).</p> <pre class="sh_erlang"> protocol_version() = integer() </pre> <p>Is the actual protocol version. In most cases the protocol version is retrieved from the processed message, but there are exceptions:</p> <p /> <list type="bulleted"> <item> <p>When <code>handle_connect/2,3</code> is triggered by an explicit call to <code>megaco:connect/4,5</code>.</p> </item> <item> <p><code>handle_disconnect/3</code></p> </item> <item> <p><code>handle_syntax_error/3</code></p> </item> </list> <p>In these cases, the ProtocolVersion default version is obtained from the static connection configuration:</p> <list type="bulleted"> <item> <p><code>megaco:conn_info(ConnHandle, protocol_version)</code>.</p> </item> </list> <span id="connect"> </span> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="handle_connect/2">handle_connect(ConnHandle, ProtocolVersion) -&gt;
