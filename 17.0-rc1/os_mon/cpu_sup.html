<h1>cpu_sup</h1> <h2 class="modsummary">A CPU Load and CPU Utilization Supervisor Process</h2> <div class="description"> <p><code>cpu_sup</code> is a process which supervises the CPU load and CPU utilization. It is part of the OS_Mon application, see <a href="os_mon_app.html" class="seealso">os_mon(6)</a>. Available for Unix, although CPU utilization values (<code>util/0,1</code>) are only available for Solaris and Linux.</p> <p>The load values are proportional to how long time a runnable Unix process has to spend in the run queue before it is scheduled. Accordingly, higher values mean more system load. The returned value divided by 256 produces the figure displayed by <code>rup</code> and <code>top</code>. What is displayed as 2.00 in <code>rup</code>, is displayed as load up to the second mark in <code>xload</code>.</p> <p>For example, <code>rup</code> displays a load of 128 as 0.50, and 512 as 2.00.</p> <p>If the user wants to view load values as percentage of machine capacity, then this way of measuring presents a problem, because the load values are not restricted to a fixed interval. In this case, the following simple mathematical transformation can produce the load value as a percentage:</p> <pre class="sh_erlang"> PercentLoad = 100 * (1 - D/(D + Load)) </pre> <p><code>D</code> determines which load value should be associated with which percentage. Choosing <code>D</code> = 50 means that 128 is 60% load, 256 is 80%, 512 is 90%, and so on.</p> <p>Another way of measuring system load is to divide the number of busy CPU cycles by the total number of CPU cycles. This produces values in the 0-100 range immediately. However, this method hides the fact that a machine can be more or less saturated. CPU utilization is therefore a better name than system load for this measure.</p> <p>A server which receives just enough requests to never become idle will score a CPU utilization of 100%. If the server receives 50% more requests, it will still scores 100%. When the system load is calculated with the percentage formula shown previously, the load will increase from 80% to 87%.</p> <p>The <code>avg1/0</code>, <code>avg5/0</code>, and <code>avg15/0</code> functions can be used for retrieving system load values, and the <code>util/0</code> and <code>util/1</code> functions can be used for retrieving CPU utilization values.</p> <p>When run on Linux, <code>cpu_sup</code> assumes that the <code>/proc</code> file system is present and accessible by <code>cpu_sup</code>. If it is not, <code>cpu_sup</code> will terminate.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="nprocs/0">nprocs() -&gt; UnixProcesses | {error, Reason}</h3> <ul class="type"> <li><code>UnixProcesses = int()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns the number of UNIX processes running on this machine. This is a crude way of measuring the system load, but it may be of interest in some cases.</p> <p>Returns 0 if <code>cpu_sup</code> is not available.</p> </div> </div> <div class="function"> <h3 id="avg1/0">avg1() -&gt; SystemLoad | {error, Reason}</h3> <ul class="type"> <li><code>SystemLoad = int()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns the average system load in the last minute, as described above. 0 represents no load, 256 represents the load reported as 1.00 by <code>rup</code>.</p> <p>Returns 0 if <code>cpu_sup</code> is not available.</p> </div> </div> <div class="function"> <h3 id="avg5/0">avg5() -&gt; SystemLoad | {error, Reason}</h3> <ul class="type"> <li><code>SystemLoad = int()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns the average system load in the last five minutes, as described above. 0 represents no load, 256 represents the load reported as 1.00 by <code>rup</code>.</p> <p>Returns 0 if <code>cpu_sup</code> is not available.</p> </div> </div> <div class="function"> <h3 id="avg15/0">avg15() -&gt; SystemLoad | {error, Reason}</h3> <ul class="type"> <li><code>SystemLoad = int()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns the average system load in the last 15 minutes, as described above. 0 represents no load, 256 represents the load reported as 1.00 by <code>rup</code>.</p> <p>Returns 0 if <code>cpu_sup</code> is not available.</p> </div> </div> <div class="function"> <h3 id="util/0">util() -&gt; CpuUtil | {error, Reason}</h3> <ul class="type"> <li><code>CpuUtil = float()</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns CPU utilization since the last call to <code>util/0</code> or <code>util/1</code> by the calling process.</p> <div class="note"><h2>Note!</h2> <p>The returned value of the first call to <code>util/0</code> or <code>util/1</code> by a process will on most systems be the CPU utilization since system boot, but this is not guaranteed and the value should therefore be regarded as garbage. This also applies to the first call after a restart of <code>cpu_sup</code>.</p> </div> <p>The CPU utilization is defined as the sum of the percentage shares of the CPU cycles spent in all busy processor states (see <code>util/1</code> below) in average on all CPUs.</p> <p>Returns 0 if <code>cpu_sup</code> is not available.</p> </div> </div> <div class="function"> <h3 id="util/1">util(Opts) -&gt; UtilSpec | {error, Reason}</h3> <ul class="type"> <li><code>Opts = [detailed | per_cpu]</code></li> <li><code>UtilSpec = UtilDesc | [UtilDesc]</code></li> <li><code>&nbsp;UtilDesc = {Cpus, Busy, NonBusy, Misc}</code></li> <li><code>&nbsp;&nbsp;Cpus = all | int() | [int()]()</code></li> <li><code>&nbsp;&nbsp;Busy = NonBusy = {State, Share} | Share</code></li> <li><code>&nbsp;&nbsp;&nbsp;State = user | nice_user | kernel</code></li> <li><code>&nbsp;&nbsp;&nbsp;&nbsp;| wait | idle | atom()</code></li> <li><code>&nbsp;&nbsp;&nbsp;Share = float()</code></li> <li><code>&nbsp;&nbsp;Misc = []</code></li> <li><code>Reason = term()</code></li> </ul> <div class="description"> <p>Returns CPU utilization since the last call to <code>util/0</code> or <code>util/1</code> by the calling process, in more detail than <code>util/0</code>.</p> <div class="note"><h2>Note!</h2> <p>The returned value of the first call to <code>util/0</code> or <code>util/1</code> by a process will on most systems be the CPU utilization since system boot, but this is not guaranteed and the value should therefore be regarded as garbage. This also applies to the first call after a restart of <code>cpu_sup</code>.</p> </div> <p>Currently recognized options:</p> <taglist> <dt><code>detailed</code></dt> <item> <p>The returned <code>UtilDesc</code>(s) will be even more detailed.</p> </item> <dt><code>per_cpu</code></dt> <item> <p>Each CPU will be specified separately (assuming this information can be retrieved from the operating system), that is, a list with one <code>UtilDesc</code> per CPU will be returned.</p> </item> </taglist> <p>Description of <code>UtilDesc = {Cpus, Busy, NonBusy, Misc}</code>:</p> <taglist> <dt><code>Cpus</code></dt> <item> <p>If the <code>detailed</code> and/or <code>per_cpu</code> option is given, this is the CPU number, or a list of the CPU numbers.</p> <p>If not, this is the atom <code>all</code> which implies that the <code>UtilDesc</code> contains information about all CPUs.</p> </item> <dt><code>Busy</code></dt> <item> <p>If the <code>detailed</code> option is given, this is a list of <code>{State, Share}</code> tuples, where each tuple contains information about a processor state that has been identified as a busy processor state (see below). The atom <code>State</code> is the name of the state, and the float <code>Share</code> represents the percentage share of the CPU cycles spent in this state since the last call to <code>util/0</code> or <code>util/1</code>.</p> <p>If not, this is the sum of the percentage shares of the CPU cycles spent in all states identified as busy.</p> <p>If the <code>per_cpu</code> is not given, the value(s) presented are the average of all CPUs.</p> </item> <dt><code>NonBusy</code></dt> <item> <p>Similar to <code>Busy</code>, but for processor states that have been identified as non-busy (see below).</p> </item> <dt><code>Misc</code></dt> <item> <p>Currently unused; reserved for future use.</p> </item> </taglist> <p>Currently these processor states are identified as busy:</p> <taglist> <dt><code>user</code></dt> <item> <p>Executing code in user mode.</p> </item> <dt><code>nice_user</code></dt> <item> <p>Executing code in low priority (nice) user mode. This state is currently only identified on Linux.</p> </item> <dt><code>kernel</code></dt> <item> <p>Executing code in kernel mode.</p> </item> </taglist> <p>Currently these processor states are identified as non-busy:</p> <taglist> <dt><code>wait</code></dt> <item> <p>Waiting. This state is currently only identified on Solaris.</p> </item> <dt><code>idle</code></dt> <item> <p>Idle.</p> </item> </taglist> <div class="note"><h2>Note!</h2> <p>Identified processor states may be different on different operating systems and may change between different versions of <code>cpu_sup</code> on the same operating system. The sum of the percentage shares of the CPU cycles spent in all busy and all non-busy processor states will always add up to 100%, though.</p> </div> <p>Returns <code>{all,0,0,[]}</code> if <code>cpu_sup</code> is not available.</p> </div> </div> </div> <div class="section"> <h4>See Also</h4> <p><a href="os_mon_app.html" class="seealso">os_mon(3)</a></p> </div>
