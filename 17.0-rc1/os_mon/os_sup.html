<h1>os_sup</h1> <h2 class="modsummary">Interface to OS System Messages</h2> <div class="description"> <p><code>os_sup</code> is a process providing a message passing service from the operating system to the error logger in the Erlang runtime system. It is part of the OS_Mon application, see <a href="os_mon_app.html" class="seealso">os_mon(6)</a>. Available for Solaris and Windows.</p> <p>Messages received from the operating system results in an user defined callback function being called. This function can do whatever filtering and formatting is necessary and then deploy any type of logging suitable for the user's application.</p> </div> <div class="section"> <h4>Solaris Operation</h4> <p>The Solaris (SunOS 5.x) messages are retrieved from the syslog-daemon, <code>syslogd</code>.</p> <p>Enabling the service includes actions which require root privileges, such as change of ownership and file privileges of an executable binary file, and creating a modified copy of the configuration file for <code>syslogd</code>. When <code>os_sup</code> is terminated, the service must be disabled, meaning the original configuration must be restored. Enabling/disabling can be done either outside or inside <code>os_sup</code>, see <a href="#config" class="seealso">Configuration</a> below.</p> <div class="warning"><h2>Warning!</h2> <p>This process cannot run in multiple instances on the same hardware. OS_Mon must be configured to start <code>os_sup</code> on one node only if two or more Erlang nodes execute on the same machine.</p> </div> <p>The format of received events is not defined.</p> </div> <div class="section"> <h4>Windows Operation</h4> <p>The Windows messages are retrieved from the eventlog file.</p> <p>The <code>nteventlog</code> module is used to implement <code>os_sup</code>. See <a href="nteventlog.html" class="seealso">nteventlog(3)</a>. Note that the start functions of <code>nteventlog</code> does not need to be used, in this case the process is started automatically as part of the OS_Mon supervision tree.</p> <p>OS messages are formatted as a tuple <code>{Time, Category, Facility, Severity, Message}</code>:</p> <taglist> <dt><code>Time = {MegaSecs, Secs, MicroSecs}</code></dt> <item> <p>A time stamp as returned by the BIF <code>now()</code>.</p> </item> <dt><code>Category = string()</code></dt> <item> <p>Usually one of <code>"System"</code>, <code>"Application"</code> or <code>"Security"</code>. Note that the NT eventlog viewer has another notion of category, which in most cases is totally meaningless and therefore not imported into Erlang. What is called a category here is one of the main three types of events occurring in a normal NT system.</p> </item> <dt><code>Facility = string()</code></dt> <item> <p>The source of the message, usually the name of the application that generated it. This could be almost any string. When matching messages from certain applications, the version number of the application may have to be accounted for. This is what the NT event viewer calls "source".</p> </item> <dt><code>Severity = string()</code></dt> <item> <p>One of <code>"Error"</code>, <code>"Warning"</code>, <code>"Informational"</code>, <code>"Audit_Success"</code>, <code>"Audit_Faulure"</code> or, in case of a currently unknown Windows NT version <code>"Severity_Unknown"</code>.</p> </item> <dt><code>Message = string()</code></dt> <item> <p>Formatted exactly as it would be in the NT eventlog viewer. Binary data is not imported into Erlang.</p> </item> </taglist> </div> <div class="section"> <span id="config"> </span> <h4>Configuration</h4> <taglist> <dt><code>os_sup_mfa = {Module, Function, Args}</code></dt> <item> <p>The callback function to use. <code>Module</code> and <code>Function</code> are atoms and <code>Args</code> is a list of terms. When an OS message <code>Msg</code> is received, this function is called as <code>apply(Module, Function, [Msg | Args])</code>.</p> <p>Default is <code>{os_sup, error_report, [Tag]}</code> which will send the event to the error logger using <a href="../kernel/error_logger.html#error_report/2" class="seealso">error_logger:error_report(Tag, Msg)</a>. <code>Tag</code> is the value of <code>os_sup_errortag</code>, see below.</p> </item> <dt><code>os_sup_errortag = atom()</code></dt> <item> <p>This parameter defines the error report type used when messages are sent to error logger using the default callback function. Default is <code>std_error</code>, which means the events are handled by the standard event handler.</p> </item> <dt><code>os_sup_enable = bool()</code></dt> <item> <p>Solaris only. Defines if the service should be enabled (and disabled) inside (<code>true</code>) or outside (<code>false</code>) <code>os_sup</code>. For backwards compatibility reasons, the default is <code>true</code>. The recommended value is <code>false</code>, as the Erlang emulator should normally not be run with <code>root</code> privileges, as is required for enabling the service.</p> </item> <dt><code>os_sup_own = string()</code></dt> <item> <p>Solaris only. Defines the directory which contains the backup copy and the Erlang specific configuration files for <code>syslogd</code>, and a named pipe to receive the messages from <code>syslogd</code>. Default is <code>"/etc"</code>.</p> </item> <dt><code>os_sup_syslogconf = string()</code></dt> <item> <p>Solaris only. Defines the full name of the configuration file for <code>syslogd</code>. Default is <code>"/etc/syslog.conf"</code>.</p> </item> </taglist> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="enable/0">enable() -&gt; ok | {error, Res}</h3> <h3 id="enable/2">enable(Dir, Conf) -&gt; ok | {error, Error}</h3> <ul class="type"> <li><code>Dir = Conf = Res = string()</code></li> </ul> <div class="description"> <p>Enables the <code>os_sup</code> service. Needed on Solaris only.</p> <p>If the configuration parameter <code>os_sup_enable</code> is <code>false</code>, this function is called automatically by <code>os_sup</code>, using the values of <code>os_sup_own</code> and <code>os_sup_syslogconf</code> as arguments.</p> <p>If <code>os_sup_enable</code> is <code>true</code>, this function must be called <em>before</em> OS_Mon/<code>os_sup</code> is started. <code>Dir</code> defines the directory which contains the backup copy and the Erlang specific configuration files for <code>syslogd</code>, and a named pipe to receive the messages from <code>syslogd</code>. Defaults to <code>"/etc"</code>. <code>Conf</code> defines the full name of the configuration file for <code>syslogd</code>. Default is <code>"/etc/syslog.conf"</code>.</p> <p>Results in a OS call to:</p> <pre class="sh_erlang"> &lt;PRIVDIR&gt;/bin/mod_syslog otp Dir Conf </pre> <p>where <code>&lt;PRIVDIR&gt;</code> is the <code>priv</code> directory of OS_Mon, <code>code:priv_dir(os_mon)</code>.</p> <p>Returns <code>ok</code> if this yields the expected result <code>"0"</code>, and <code>{error, Res}</code> if it yields anything else.</p> <div class="note"><h2>Note!</h2> <p>This function requires root privileges to succeed.</p> </div> </div> </div> <div class="function"> <h3 id="disable/0">disable() -&gt; ok | {error, Res}</h3> <h3 id="disable/2">disable(Dir, Conf) -&gt; ok | {error, Error}</h3> <ul class="type"> <li><code>Dir = Conf = Res = string()</code></li> </ul> <div class="description"> <p>Disables the <code>os_sup</code> service. Needed on Solaris only.</p> <p>If the configuration parameter <code>os_sup_enable</code> is <code>false</code>, this function is called automatically by <code>os_sup</code>, using the same arguments as when <code>enable/2</code> was called.</p> <p>If <code>os_sup_enable</code> is <code>true</code>, this function must be called <em>after</em> OS_Mon/<code>os_sup</code> is stopped. <code>Dir</code> defines the directory which contains the backup copy and the Erlang specific configuration files for <code>syslogd</code>, and a named pipe to receive the messages from <code>syslogd</code>. Defaults to <code>"/etc"</code>. <code>Conf</code> defines the full name of the configuration file for <code>syslogd</code>. Default is <code>"/etc/syslog.conf"</code>.</p> <p>Results in a OS call to:</p> <pre class="sh_erlang"> &lt;PRIVDIR&gt;/bin/mod_syslog nootp Dir Conf </pre> <p>where <code>&lt;PRIVDIR&gt;</code> is the <code>priv</code> directory of OS_Mon, <code>code:priv_dir(os_mon)</code>.</p> <p>Returns <code>ok</code> if this yields the expected result <code>"0"</code>, and <code>{error, Res}</code> if it yields anything else.</p> <div class="note"><h2>Note!</h2> <p>This function requires root privileges to succeed.</p> </div> </div> </div> </div> <div class="section"> <h4>See also</h4> <p><a href="../kernel/error_logger.html" class="seealso">error_logger(3)</a>, <a href="os_mon_app.html" class="seealso">os_mon(3)</a></p> <p><code>syslogd(1M)</code>, <code>syslog.conf(4)</code> in the Solaris documentation.</p> <p /> </div>
