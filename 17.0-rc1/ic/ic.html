<h1>ic</h1> <h2 class="modsummary">The Erlang IDL Compiler</h2> <div class="description"> <p>The ic module is an Erlang implementation of an OMG IDL compiler. Depending on the choice of back-end the code will map to Erlang, C, or Java. The compiler generates client stubs and server skeletons.</p> <p>Two kinds of files are generated for each scope: Ordinary code files and header files. The latter are used for defining record definitions, while the ordinary files contain the object interface functions.</p> </div> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> [Error]}</code></li> <li> </li> <li><code>Option = [ GeneralOption | CodeOption | WarningOption | BackendOption]</code></li> <li> </li> <li><code>GeneralOption = </code></li> <li><code>{outdir, <code>Option</code> for boolean values.</p> <p>The <code>ic:gen/2</code> function can be called from the command line as follows:</p> <p><code>erlc "+Option" ... File.idl</code></p> <p>Example:</p> <p><code>erlc "+{be,c_client}" '+{outdir, "../out"}' File.idl</code></p> </div> </div> </div> <div class="section"> <h4>General options</h4> <taglist> <dt><em>outdir</em></dt> <item> <p>Places all output files in the directory given by the option. The directory will be created if it does not already exist.</p> <p>Example option: <code>{outdir, "output/generated"}</code>.</p> </item> <dt><em>cfgfile</em></dt> <item> <p>Uses <em>FileName</em> as configuration file. Options will override compiler defaults but can be overridden by command line options. Default value is <code>".ic_config"</code>.</p> <p>Example option: <code>{cfgfile, "special.cfg"}</code>.</p> </item> <dt><em>use_preproc</em></dt> <item> <p>Uses a preprocessor. Default value is true.</p> </item> <dt><em>preproc_cmd</em></dt> <item> <p>Command string to invoke the preprocessor. The actual command will be built as <code>preproc_cmd++preproc_flags++FileName</code></p> <p>Example option: <code>{preproc_cmd, "erl"})</code>.</p> <p>Example option: <code>{preproc_cmd, "gcc -x c++ -E"}</code>.</p> </item> <dt><em>preproc_flags</em></dt> <item> <p>Flags given to the preprocessor.</p> <p>Example option: <code>{preproc_flags, "-I../include"}</code>.</p> </item> </taglist> </div> <div class="section"> <h4>Code options</h4> <taglist> <dt><em>light_ifr</em></dt> <item> <p>Currently, the default setting is <code>false</code>. To be able to use this option Orber must be configured to use Light IFR (see Orber's User's Guide). When this options is used, the size of the generated files used to register the API in the IFR DB are minimized.</p> <p>Example option: <code>{light_ifr, true}</code>.</p> </item> <dt><em>gen_hrl</em></dt> <item> <p>Generate header files. Default is true.</p> </item> <dt><em>serv_last_call</em></dt> <item> <p>Makes the last <code>gen_server handle_call</code> either raise a CORBA exception or just exit plainly. Default is the exception. </p> </item> <dt><em>{{impl, IntfName}, ModName}</em></dt> <item> <p>Assumes that the interface with name <em>IntfName</em> is implemented by the module with name <em>ModName</em> and will generate calls to the <em>ModName</em> module in the server behavior. Note that the <em>IntfName</em> must be a fully scoped name as in <code>"M1::I1"</code>.</p> <p /> </item> <dt><em>this</em></dt> <item> <p>Adds the object reference as the first parameter to the object implementation functions. This makes the implementation aware of its own object reference. <br /> The option comes in three varieties: <code>this</code> which activates the parameter for all interfaces in the source file, <code>{this, IntfName}</code> which activates the parameter for a specified interface and <code>{{this, IntfName}, false}</code> which deactivates the parameter for a specified interface.</p> <p>Example option: <code>this)</code> activates the parameter for all interfaces.</p> <p>Example option: <code>{this, "M1::I1"}</code> activates the parameter for all functions of <code>M1::I1</code>.</p> <p>Example options: <code>[this, {{this, "M1::I2"}, false}]</code> activates the parameter for all interfaces except <code>M1::I2</code>.</p> </item> <dt><em>from</em></dt> <item> <p>Adds the invokers reference as the first parameter to the object implementation two-way functions. If both <code>from</code> and <code>this</code> options are used the invokers reference parameter will be passed as the second parameter. This makes it possible for the implementation to respond to a request and continue executing afterwards. Consult the <code>gen_server</code> and <code>Orber</code> documentation how this option may be used. <br /> The option comes in three varieties: <code>from</code> which activates the parameter for all interfaces in the source file, <code>{from, IntfName}</code> which activates the parameter for a specified interface and <code>{{from, IntfName}, false}</code> which deactivates the parameter for a specified interface.</p> <p>Example option: <code>from)</code> activates the parameter for all interfaces.</p> <p>Example options: <code>[{from, "M1::I1"}]</code> activates the parameter for all functions of <code>M1::I1</code>.</p> <p>Example options: <code>[from, {{from, "M1::I2"}, false}]</code> activates the parameter for all interfaces except <code>M1::I2</code>.</p> </item> <dt><em>handle_info</em></dt> <item> <p>Makes the object server call a function <code>handle_info</code> in the object implementation module on all unexpected messages. Useful if the object implementation need to trap exits.</p> <p>Example option: <code>handle_info</code> will activates module implementation <code>handle_info</code> for all interfaces in the source file.</p> <p>Example option: <code>{{handle_info, "M1::I1"}, true}</code> will activates module implementation <code>handle_info</code> for the specified interface.</p> <p>Example options: <code>[handle_info, {{handle_info, "M1::I1"}, false}]</code> will generate the <code>handle_info</code> call for all interfaces except <code>M1::I1</code>.</p> </item> <dt><em>timeout</em></dt> <item> <p>Used to allow a server response time limit to be set by the user. This should be a string that represents the scope for the interface which should have an extra variable for wait time initialization.</p> <p>Example option: <code>{timeout,"M::I"})</code> produces server stub which will has an extra timeout parameter in the initialization function for that interface.</p> <p>Example option: <code>timeout</code> produces server stub which will has an extra timeout parameter in the initialization function for all interfaces in the source file.</p> <p>Example options: <code>[timeout, {{timeout,"M::I"}, false}]</code> produces server stub which will has an extra timeout parameter in the initialization function for all interfaces except <code>M1::I1</code>.</p> </item> <dt><em>scoped_op_calls</em></dt> <item> <p>Used to produce more refined request calls to server. When this option is set to true, the operation name which was mentioned in the call is scoped. This is essential to avoid name clashes when communicating with c-servers. This option is available for the c-client, c-server and the Erlang gen_server back ends. <code>All</code> of the parts generated by ic have to agree in the use of this option. Default is <code>false</code>. </p> <p>Example options: <code>[{be,c_genserv},{scoped_op_calls,true}])</code> produces client stubs which sends "scoped" requests to a gen_server or a c-server.</p> </item> <dt><em>user_protocol</em></dt> <item> <p>Used to define a own protocol different from the default Erlang distribution + gen_server protocol. Currently only valid for C back-ends. For further details see <a href="ic_c_protocol.html" class="seealso">IC C protocol</a>.</p> <p>Example options: <code>[{be,c_client},{user_protocol, "my_special"}])</code> produces client stubs which use C protocol functions with the prefix "my_special".</p> </item> <dt><em>c_timeout</em></dt> <item> <p>Makes sends and receives to have timeouts (C back-ends only). These timeouts are specified in milliseconds. </p> <p>Example options: <code>[{be,c_client},{c_timeout, {10000, 20000}}])</code> produces client stubs which use a 10 seconds send timeout, and a 20 seconds receive timeout.</p> </item> <dt><em>c_report</em></dt> <item> <p>Generates code for writing encode/decode errors to <code>stderr</code> (C back-ends only). timeouts are specified in milliseconds. </p> <p>Example options: <code>[{be,c_client}, c_report])</code>.</p> </item> <dt><em>scl</em></dt> <item> <p>Used for compatibility with previous compiler versions up to <code>3.3</code>. Due to better semantic checks on enumerants, the compiler discovers name clashes between user defined types and enumerant values in the same name space. By enabling this option the compiler turns off the extended semantic check on enumerant values. Default is <code>false</code>. </p> <p>Example option: <code>{scl,true}</code></p> </item> <dt><em>precond</em></dt> <item> <p>Adds a precondition call before the call to the operation implementation on the server side.</p> <p>The option comes in three varieties: <code>{precond, {M, F}}</code> which activates the call for operations in all interfaces in the source file, <code>{{precond, IntfName}, {M, F}}</code> which activates the call for all operations in a specific interface and <code>{{precond, OpName}, {M, F}}</code> which activates the call for a specific operation.</p> <p>The precondition function has the following signature <code>m:f(Module, Function, Args)</code>.</p> <p>Example option: <code>{precond, {mod, fun}}</code> adds the call of m:f for all operations in the idl file.</p> <p>Example options: <code>[{{precond, "M1::I"}, {mod, fun}}]</code> adds the call of <code>m:f</code> for all operations in the interface <code>M1::I1</code>.</p> <p>Example options: <code>[{{precond, "M1::I::Op"}, {mod, fun}}]</code> adds the call of <code>m:f</code> for the operation <code>M1::I::Op</code>.</p> </item> <dt><em>postcond</em></dt> <item> <p>Adds a postcondition call after the call to the operation implementation on the server side. </p> <p>The option comes in three varieties: <code>{postcond, {M, F}}</code> which activates the call for operations in all interfaces in the source file, <code>{{postcond, IntfName}, {M, F}}</code> which activates the call for all operations in a specific interface and <code>{{postcond, OpName}, {M, F}}</code> which activates the call for a specific operation.</p> <p>The postcondition function has the following signature <code>m:f(Module, Function, Args, Result)</code>.</p> <p>Example option: <code>{postcond, {mod, fun}}</code> adds the call of m:f for all operations in the idl file.</p> <p>Example options: <code>[{{postcond, "M1::I"}, {mod, fun}}]</code> adds the call of <code>m:f</code> for all operations in the interface <code>M1::I1</code>.</p> <p>Example options: <code>[{{postcond, "M1::I::Op"}, {mod, fun}}]</code> adds the call of <code>m:f</code> for the operation <code>M1::I::Op</code>.</p> </item> </taglist> </div> <div class="section"> <h4>Warning options</h4> <taglist> <dt><em>'Wall'</em></dt> <item> <p>The option activates all reasonable warning messages in analogy with the gcc -Wall option. Default value is true.</p> </item> <dt><em>maxerrs</em></dt> <item> <p>The maximum numbers of errors that can be detected before the compiler gives up. The option can either have an integer value or the atom <code>infinity</code>. Default number is 10.</p> </item> <dt><em>maxwarns</em></dt> <item> <p>The maximum numbers of warnings that can be detected before the compiler gives up. The option can either have an integer value or the atom <code>infinity</code>. Default value is infinity.</p> </item> <dt><em>nowarn</em></dt> <item> <p>Suppresses all warnings. Default value is false.</p> </item> <dt><em>warn_name_shadow</em></dt> <item> <p>Warning appears whenever names are shadowed due to inheritance; for example, if a type name is redefined from a base interface. Note that it is illegal to overload operation and attribute names as this causes an error to be produced. Default value is true. </p> </item> <dt><em>pedantic</em></dt> <item> <p>Activates all warning options. Default value is false.</p> </item> <dt><em>silent</em></dt> <item> <p>Suppresses compiler printed output. Default value is false.</p> </item> </taglist> </div> <div class="section"> <h4>Back-End options</h4> <p>Which back-end IC will generate code for is determined by the supplied <code>{be,atom()}</code> option. If left out, <code>erl_corba</code> is used. Currently, IC support the following back-ends:</p> <taglist> <dt><em>erl_corba</em></dt> <item> <p>This option switches to the IDL generation for CORBA.</p> </item> <dt><em>erl_template</em></dt> <item> <p>Generate CORBA call-back module templates for each interface in the target IDL file. Note, will overwrite existing files.</p> </item> <dt><em>erl_plain</em></dt> <item> <p>Will produce plain Erlang modules which contain functions that map to the corresponding interface functions on the input file.</p> </item> <dt><em>erl_genserv</em></dt> <item> <p>This is an IDL to Erlang generic server generation option.</p> </item> <dt><em>c_client</em></dt> <item> <p>Will produce a C client to the generic Erlang server.</p> </item> <dt><em>c_server</em></dt> <item> <p>Will produce a C server switch with functionality of a generic Erlang server.</p> </item> <dt><em>java</em></dt> <item> <p>Will produce Java client stubs and server skeletons with functionality of a generic Erlang server.</p> </item> <dt><em>c_genserv</em></dt> <item> <p>Deprecated. Use <code>c_client</code> instead.</p> </item> </taglist> </div> <div class="section"> <h4>Preprocessor</h4> <p>The IDL compiler allows several preprocessors to be used, the <code>Erlang IDL preprocessor</code> or other standard <code>C</code> preprocessors. Options can be used to provide extra flags such as include directories to the preprocessor. The build in the Erlang IDL preprocessor is used by default, but any standard C preprocessor such as <code>gcc</code> is adequate.</p> <p>The preprocessor command is formed by appending the prepoc_cmd to the preproc_flags option and then appending the input IDL file name.</p> </div> <div class="section"> <h4>Configuration</h4> <p>The compiler can be configured in two ways:</p> <list type="ordered"> <item> <p>Configuration file</p> </item> <item> <p>Command line options</p> </item> </list> <p>The configuration file is optional and overrides the compiler defaults and is in turn overridden by the command line options. The configuration file shall contain options in the form of Erlang terms. The configuration file is read using <code>file:consult</code>.</p> <p>An example of a configuration file, note the "." after each line.</p> <pre class="sh_erlang"> {outdir, gen_dir}. {{impl, "M1::M2::object"}, "obj"}. </pre> </div> <div class="section"> <h4>Output files</h4> <p>The compiler will produce output in several files depending on scope declarations found in the IDL file. At most three file types will be generated for each scope (including the top scope), depending on the compiler back-end and the compiled interface. Generally, the output per interface will be a header file (<code>.hrl</code>/ <code>.h</code>) and one or more Erlang/C files (<code>.erl</code>/<code>.c</code>). Please look at the language mapping for each back-end for details.</p> <p>There will be at least one set of files for an IDL file, for the file level scope. Modules and interfaces also have their own set of generated files.</p> </div>
